A Pod is a shared apartment. The containers inside are roommates sharing the kitchen and bathroom (network + IPC), with handymen who finished setup before move-in (init) and helpful housemates running alongside (sidecars).

Section 1 · Concept

The Pod is the atom of scheduling

A Pod is the atomic unit of scheduling in Kubernetes. Not a container — a Pod. The container is the unit of packaging; the Pod is what K8s actually places on a node and reasons about. 90% of pods have exactly one container, but the rule that says "one or more containers, scheduled together, sharing some namespaces" enables the rest of K8s.

What's shared inside a Pod. Network namespace (same Pod IP, same port space — containers talk via localhost:port), IPC namespace (POSIX shared memory, semaphores), optionally PID namespace (shareProcessNamespace: true), and any pod-level Volumes.

What's NOT shared by default. Each container has its own filesystem (mount) namespace and its own PID namespace. Each container is independently restartable.

The pause container. Under the hood, every pod has an invisible pause container that's PID 1 in the pod's network/IPC namespaces. Its only job: hold those namespaces open so individual app containers can come and go without losing the pod's IP. You'll see it on a node with crictl ps; you don't write YAML for it.

Init containers run sequentially to completion BEFORE any main containers start. Sidecar containers run alongside the main one (log shipping, mesh proxy). Native sidecars (K8s 1.28+): declare in initContainers with restartPolicy: Always — starts before main, stays running, properly handled at shutdown. Ephemeral containers are added at runtime via kubectl debug, critical for debugging distroless containers.

Resources. requests = scheduler-reserved floor; limits = hard ceiling (CPU throttled, memory OOM-killed). The combination determines the pod's QoS class (Guaranteed / Burstable / BestEffort), which controls eviction priority when the node is under pressure.

Section 1.7 · Multi-container patterns

Three flavors of co-tenant container

Brendan Burns's classic taxonomy. Most production multi-container pods are sidecars; ambassador and adapter show up less often but explain a lot of common architecture.

Sidecar

Augments the main container — log shipper, mesh proxy, metrics scraper. Most common pattern.

Ambassador

Proxies an external service for the main container — DB connection pool, OAuth proxy, retry logic.

Adapter

Transforms the main's output — translates app metrics into Prometheus format, normalizes log shape.

Section 1.8 · requests vs limits

The floor and the ceiling

request is what the scheduler reserves for you — the floor. limit is the hard ceiling — CPU above limit is throttled, memory above limit is OOM-killed.

CPU above limit is throttled (slow); memory above limit is killed (OOM). Set requests close to your steady-state usage so the scheduler packs efficiently; set limits with enough headroom for legitimate spikes.

Section 1.9 · QoS classes

Three eviction-priority tiers

QoS class is computed automatically from your requests/limits and decides who gets evicted first when the node runs short on memory.

Last evicted

Guaranteed

Every container has CPU and memory request == limit. Highest priority during node pressure. Use for: latency-critical, predictable workloads.

Mid eviction

Burstable

At least one container has a request, and request < limit (or limit unset). Most production pods. Can spike above request when capacity is free.

First evicted

BestEffort

No requests or limits anywhere. Scheduler can't reserve capacity. First to be evicted under node pressure. Don't use for production workloads.

Important: QoS doesn't grant extra memory. Even Guaranteed pods get OOM-killed when they exceed their own limit. Class only affects eviction order when the node runs short.

Section 1.10 · restartPolicy

Three policies, three workload types

Always

Default. Restart any exited container regardless of exit code. Used by Deployments, StatefulSets, DaemonSets — long-running services.

OnFailure

Restart only on non-zero exit. Used by Jobs that should retry on error but stop on success.

Never

Don't restart, ever. Used by Jobs you want to fail loudly so a human investigates. Pod stays in Failed phase.

restartPolicy is pod-level — applies to all containers. Note: it controls kubelet's local restarts. Pod-level controllers (Deployments) have their own logic on top — they create new pods if a pod dies entirely.

Section 2 · Before & After

Multi-process container vs Pod with sidecar

The classic mistake first-time K8s users make: cramming app + log shipper into one Docker image with a shell wrapper.

Same workload (app + log shipper) — left bundles them in one image with a shell, right separates them as containers in one Pod. Independent images, independent restarts, proper PID 1 in each.
Before · one image, shell wrapper

Single Docker image with ./app & ./fluent-bit in a shell entrypoint. bash is PID 1, swallows SIGTERM, neither process is independently restartable, can't version them separately, scaling means scaling both.

After · Pod, two containers

Two separate images in one Pod. Main is just the app. Sidecar is just the log shipper. Each has its own image, version, lifecycle, resource requests, and restart count. Pod's pause container holds the IP open.

What didn't change: the app and the log shipper still cooperate. What changed: they're cleanly separated, independently versioned, and properly lifecycle-managed.

Section 3 · Analogy

The shared studio apartment

A Pod is a shared studio apartment. The containers are roommates. They share the kitchen (network namespace — they can chat from across the room via localhost) and the bathroom (IPC), but each has their own bedroom (separate filesystem, often separate PID namespace). The apartment has one doorbell (the Pod IP) that any roommate can answer.

Before anyone moves in, the handyman (init container) shows up, finishes the wifi setup, hangs the curtains, and leaves. Only then do the roommates move in.

The main roommate does the actual work. The sidecar housemate does laundry quietly in the background, or answers the door (ambassador), or translates when guests speak a different language (adapter).

When something breaks and the main roommate doesn't have a screwdriver (distroless image, no shell), a plumber can drop in temporarily (ephemeral debug container). They don't move in — they visit, fix, and leave. The apartment doesn't get evicted to bring the plumber in.

The building's superintendent (the kubelet) decides whether to give your apartment guaranteed rent control (QoS Guaranteed, last to evict), flexible rent (Burstable), or first to be asked to leave (BestEffort) based on how much electricity and water you reserved.

The mapping:

  • Apartment = Pod (atomic unit, shares an address)
  • Roommates = containers in the pod
  • Shared kitchen = network namespace (talk via localhost)
  • Shared bathroom = IPC namespace
  • Separate bedrooms = filesystem & PID namespaces (per container)
  • Doorbell at the front = Pod IP
  • Handyman who came first = init container
  • Helpful housemate = sidecar
  • Doorman = ambassador container
  • Translator = adapter container
  • Visiting plumber = ephemeral debug container
  • Building superintendent = kubelet
  • Reserved utilities = resource requests
  • Maximum utilities allowed = resource limits
  • Rent-control class = QoS class

Section 5 · Real-world scenarios

Where pod patterns pay off

Init container blocked startup until the database was reachable

Web app needed PostgreSQL ready before starting. Engineers had sleeps in the entrypoint, sometimes hit races. Fix: initContainers: [{name: wait-db, image: busybox, command: ["sh","-c","until nc -z db 5432; do sleep 1; done"]}]. Pod stays Pending until the init exits 0 — guaranteed db-ready. No more flaky cold starts.

Log-shipping sidecar replaced 12 different agents on each VM

Pre-K8s, every team installed their own log agent on each VM (different versions, different configs). Audit nightmare. After K8s: one standardized fluent-bit sidecar in every pod template (or DaemonSet — Lesson 16). Logs flow to one collector. Independent images, independent versioning, one mechanism.

kubectl debug saved an outage when the main container had no shell

Production app started returning 500s. Image was gcr.io/distroless/python3 — no /bin/sh to exec into. kubectl exec ... bash failed. Engineer ran kubectl debug -it broken-pod --image=busybox --target=app -- sh. Got a shell in a sibling container sharing the broken container's PID namespace. Found the bad file descriptor in seconds. Distroless safety + ephemeral debug = best of both.

QoS confusion: Guaranteed pods kept getting OOM-killed

Team set requests.memory: 1Gi, limits.memory: 1Gi (Guaranteed). App spiked to 1.2Gi during bursts. OOM-kills every burst. They thought Guaranteed meant "always gets memory." It doesn't — exceeding limit = kill, regardless of class. Fix: raise limit (becomes Burstable) or right-size the app so peak fits under limit. QoS only affects eviction priority when the node is under pressure.

Your web app needs PostgreSQL ready before it starts, otherwise it crashes on its first query. Current setup: single container with sleep loops in the entrypoint, sometimes races and crashes anyway. K8s-native fix?

answerUse an init container. Add to the pod spec: initContainers: [{name: wait-db, image: busybox, command: ["sh","-c","until nc -z db.svc 5432; do sleep 1; done"]}]. The kubelet runs init containers sequentially before any main containers start. The pod stays in Pending phase until every init exits 0. This guarantees the DB is reachable before your app's first byte runs — no sleep races, no entrypoint hacks. If you have multiple ordered prerequisites (wait-db, run-migration, seed-data), each is its own init container in order. Bonus: the app container's image stays clean — no netcat, no init logic.

Your production app uses a distroless base (gcr.io/distroless/python3) for security. It just started returning 500s. kubectl exec -it ... bash fails with "executable file not found in $PATH". What do you do?

answerkubectl debug. Run: kubectl debug -it broken-pod --image=busybox --target=app -- sh. This adds an ephemeral container to the running pod. The ephemeral container is a sibling to your main "app" container, sharing the pod's network namespace AND (because of --target) the main container's PID namespace. You get a busybox shell that can ps, ls /proc/<main-pid>/, and tcpdump as if you were inside the broken container. The pod doesn't restart. Once you exit, the ephemeral container is gone but the rest of the pod is unchanged. This is the right answer for distroless: keep the production attack surface tiny and use ephemeral debug containers when you need to investigate.

Team set requests.memory: 1Gi and limits.memory: 1Gi (QoS class: Guaranteed). App spikes to 1.2Gi during bursts and gets OOM-killed every time. They thought Guaranteed meant "always gets memory." Actual behavior?

answerGuaranteed QoS does NOT mean "this pod always gets the memory it asks for." It means "request == limit, AND this pod is the LAST to be evicted when the NODE is under memory pressure." Within the pod, the limit is still a hard cap — exceeding it = OOM kill, no exceptions. Class is irrelevant to your own limit. Two fixes: (1) Raise the limit. e.g., request: 1Gi, limit: 2Gi. Now the pod is Burstable but it can use up to 2Gi when needed. Cost: it can be evicted before Guaranteed pods if the node is under pressure (rare for healthy clusters). (2) Right-size the app. Profile peak memory and set both request and limit above the observed P99. Stay Guaranteed, but make the limit reflect reality. Lesson: QoS is about NODE-level eviction priority, not POD-level memory grants.
