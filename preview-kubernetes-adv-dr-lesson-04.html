🚨 The 3 AM Nightmare

It's 3 AM. Velero restored apps; PVCs back; but apps fail to start — secrets unavailable. Vault was in same region; it's also down. Today's lesson: each subsystem has its own DR pattern; coordinate the timeline.

🎯 If you remember nothing else: Each subsystem has DR: secrets via Vault + KMS; DNS via Route 53; stateful via DB replication; managed services have built-in DR with limits. Understand each; coordinate timeline.

Section 1.1 · secret recovery

Vault + KMS + ESO chain

External Vault: HA across multiple regions; raft replication for snapshot; quarterly snapshot to immutable storage. KMS keys: cross-region replicas (AWS / Azure / GCP all support); root-of-trust survives single-region loss.

Recovery: rebuild Vault in DR region from snapshot; rotate seal keys; KMS keys auto-replicated; External Secrets Operator (ESO) in new cluster re-syncs from Vault; K8s Secrets re-materialise.

Without external Vault: K8s Secrets can be Velero-backed but tightly coupled to cluster; less flexible.

⏸ Pause and check

Why external Vault for secret recovery?

External Vault decouples secrets from cluster lifecycle. K8s Secrets in cluster die with cluster; external Vault lives on.

Section 1.2 · DNS failover

Route 53 / Azure DNS + TTL + health checks

DNS-based failover: Route 53 health-check on primary endpoint; on failure, swap A record to DR LB. Azure DNS + Traffic Manager equivalent. GCP Cloud DNS + Global LB.

TTL tuning: lower TTL (30-60s) on critical records for faster failover; trade DNS query load for fast failover. Pre-stage low TTL before known maintenance windows.

External-DNS K8s controller automates DNS records from Service / Ingress; per-cluster setup syncs to managed DNS.

Section 1.3 · stateful workload DR

DB replication + restore + consistency

Self-managed DBs (PostgreSQL / MySQL operators):

  • Multi-AZ replication (primary + 2 standbys via streaming).
  • Cross-region streaming replica.
  • Backup to S3 / Blob via cron-job (pg_basebackup / wal-g / pgbackrest / Percona XtraBackup).
  • Restore-tested quarterly; consistency model (eventual / strong) understood.

K8s operators (CloudNativePG / Zalando postgres-operator / Strimzi for Kafka): handle replication lifecycle. PVC + CSI snapshot for periodic capture.

Section 1.4 · managed-service DR limits

RDS / Cosmos / Spanner — built-in but limited

AWS RDS / Aurora: multi-AZ standby (synchronous; in-region); cross-region read replica (async; promote on disaster); Aurora Global Database (async cross-region; minute RPO; global LB). RTO: minutes for promote.

Azure Cosmos DB: multi-region built-in; configurable consistency (Strong / Bounded Staleness / Session / Eventual / Consistent Prefix); auto-failover.

GCP Spanner: regional (1 region) or multi-region (5+ regions strong consistency); built-in; expensive.

Limits: cross-region promotion has data-loss window (RPO > 0); restore-from-snapshot can be slow (TB-scale = hours); test for your data size.

⏸ Pause and check

Aurora cross-region read replica RPO?

Async = data not in DR until next replication cycle. Aurora Global Database = single-second RPO; standard cross-region read replica = minutes.

Section 5 · Real-world scenarios

How DR architects do this

External Vault saved DR

Cluster lost in us-east-1; Vault HA in us-east-1 + us-west-2; rebuild cluster in us-west-2; ESO re-syncs from Vault us-west-2; secrets immediately available.

Route 53 failover automated DNS

Primary cluster ALB unhealthy; Route 53 health check fires; A record swaps to DR ALB within 60s. Tenants notice brief blip.

Aurora Global Database for tier-1

Fintech runs Aurora Global across us-east-1 + us-west-2; RPO < 1s; on disaster, promote DR to primary; RTO < 1 min. Cost premium accepted for tier-1.

Outage — assumed RDS DR worked

Pre-test: assumed RDS multi-AZ = enough. Real disaster: cross-region promotion took 2h. Postmortem: Aurora Global Database; cross-region promotion drilled.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

A few common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: "K8s Secrets are sufficient."
Truth: Cluster loss = K8s Secrets gone. External Vault is the foundation; K8s Secrets are derived (via ESO) + ephemeral.
Myth: "DNS TTL of 24h is OK."
Truth: High TTL = slow DNS failover. Critical records: TTL 30-60s + accept higher DNS query cost.
Myth: "Managed-service DR is automatic."
Truth: Managed services offer DR features but require deliberate config + tested promotion. Aurora multi-AZ ≠ Aurora Global Database; understand the difference.
External Vault — why for DR?
tap to flip
Survives cluster loss. Cross-region HA + raft replication. KMS for seal keys cross-region replicated. ESO re-syncs on new cluster.
DNS failover tools per cloud?
tap to flip
AWS Route 53 + health check. Azure DNS + Traffic Manager. GCP Cloud DNS + Global LB. Lower TTL (30-60s) for fast failover.
DB DR options self-managed?
tap to flip
Multi-AZ streaming replication; cross-region read replica; backups to S3 (pg_basebackup / wal-g / pgbackrest); CSI snapshots.
Aurora Global Database vs cross-region read replica?
tap to flip
Aurora Global: dedicated cross-region replication; RPO < 1s; faster promote (~1 min). Cross-region read replica: standard async; RPO seconds-minutes; promote takes longer.
Cosmos DB consistency levels?
tap to flip
Strong (linearizable; highest cost), Bounded Staleness, Session, Consistent Prefix, Eventual (lowest cost).
External-DNS K8s controller?
tap to flip
Watches Service / Ingress; syncs records to managed DNS (Route 53 / Cloud DNS / etc.). Automates DNS lifecycle.
Spanner DR shapes?
tap to flip
Regional (single region); multi-region (5+ regions; strong consistency; expensive).
K8s DB operators?
tap to flip
CloudNativePG (CNCF), Zalando postgres-operator, Strimzi (Kafka). Handle replication + backup lifecycle.

Design DR for an app with Postgres + Redis + Vault secrets + Route 53 DNS.

answer(1) Postgres: CloudNativePG operator + cross-region streaming replica + pgbackrest to S3 + restore-tested quarterly. (2) Redis: Redis cluster mode + cross-region replication; ephemeral data accepts loss; alternative: Elasticache cross-region. (3) Vault: HA in 3 regions; auto-unseal via KMS (cross-region replicated). (4) DNS: Route 53 health check on app endpoint; auto-failover to DR LB; TTL 60s. (5) Coordination: runbook orders restore: Vault first → DB replicas promote → app + ESO syncs secrets → DNS swap. (6) Drill quarterly: measure end-to-end RTO.

A team uses RDS multi-AZ. Region outage. Walk failover.

answer(1) RDS multi-AZ alone: synchronous standby in same region; doesn't help cross-region. (2) If cross-region read replica configured: aws rds promote-read-replica in DR region; takes 5-15 min. (3) App reconfigure: connection strings point at promoted replica's endpoint. (4) Aurora Global Database would have been faster (~1 min promotion). (5) Postmortem: upgrade to Aurora Global for tier-1; document promote runbook + drill.
🎬 Choose Your Own Adventure

The CFO sees Aurora Global cost premium (~30% over standard). Defend.

how the DR architect framed Aurora Global cost"Aurora Global Database is the difference between 1-min RTO + 2h RTO during region failure. Three considerations: (1) Tier-1 workloads: 30% premium = small vs cost of 2h customer-facing outage. (2) Synchronous replication: RPO < 1s vs minutes for standard cross-region. (3) Auto-promote: graceful failover without scripts. For tier-2 / tier-3: standard cross-region read replica is fine. Cost-by-tier: tier-1 gold = Aurora Global; tier-2 silver = standard async; tier-3 bronze = backup-only."
