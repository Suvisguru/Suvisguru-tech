Storage Part 1 · Persistent Volumes, Persistent Volume Claims, StorageClass

A Pod's filesystem disappears when the Pod restarts. Persistent storage survives. Three roles — the PV is the disk, the PVC is the request, the StorageClass is the recipe.

🚨 The 3 AM Nightmare

Your team's PostgreSQL Pod stores all data on the Pod's local filesystem. The Pod restarts at 3 AM (a routine event — a node taint, an image update, anything). It comes back up clean — fresh filesystem, no data. Years of customer records, gone in fifteen seconds. The recovery story is ugly. This lesson is about the abstraction that prevents this — and the three roles you need to understand to use it.

🎯 If you remember nothing else: Pod storage is ephemeral. To survive a Pod restart, declare a PVC (the request). A StorageClass (the recipe) tells K8s how to provision a PV (the actual disk). The PV binds to the PVC and the Pod mounts it.

Section 1 · Concept

Ephemeral by default; persistent on request

A Pod's container has a writable layer on top of the image's read-only layers. Anything the app writes there — log files, uploaded user data, cached computation — lives in that writable layer. That writable layer is destroyed when the container or Pod is removed. The next Pod gets a fresh writable layer; nothing carries over. Even emptyDir, the default scratch volume, is destroyed when the Pod itself is removed.

For data that must survive Pod churn (databases, file uploads, queue state), Kubernetes provides a layer of indirection. The Pod doesn't bind to a specific disk; it declares what it needs. A controller binds the declaration to actual storage. That's the PV / PVC / StorageClass trio. Each plays one role in the negotiation between "what this Pod wants" and "what disk the cluster can give it."

Section 1.5 · Three roles, three K8s objects

PV, PVC, and StorageClass — each plays one part

Understanding storage in Kubernetes is mostly understanding which object plays which role. Memorise these three and you've got the spine of the whole module.

📋

PersistentVolumeClaim (PVC)

the request

App-level: written by whoever owns the workload. Says "I need 10Gi of ReadWriteOnce storage from the fast-ssd StorageClass." Lives in a namespace. Mentioned by the Pod under volumes:.

Written by: app teams

💾

PersistentVolume (PV)

the actual disk

Cluster-level: a real piece of storage (an EBS volume, a Ceph RBD, an NFS export). Has capacity, access modes, a backing CSI driver, a reclaim policy. PVCs and PVs bind to each other, 1:1.

Created by: dynamic provisioning (StorageClass) or hand-rolled by the cluster admin

📜

StorageClass

the recipe

Cluster-level: tells K8s how to provision a PV when a PVC arrives that asks for this class. Names the CSI provisioner, sets parameters (disk type, IOPS, encryption), the reclaim policy, and the binding mode.

Written by: cluster operators

The flow: a PVC arrives → its storageClassName picks a StorageClass → the StorageClass's provisioner creates a PV with the right capacity and parameters → K8s binds the PV to the PVC → the Pod mounts the PVC and gets the volume at the path it asked for.

Section 1.7 · Access modes

Who can mount the volume — and how many at once

Access modes describe how many Pods (and on how many nodes) can mount the same volume simultaneously, and whether they get read-write or read-only access. The PVC asks for a mode; the PV must support it; the underlying storage must actually allow it.

ModeShortMeansTypical use
ReadWriteOnceRWOOne node mounts it for read-write. Only Pods on that node access it. The default.Single-replica databases, workloads that own their disk.
ReadOnlyManyROXMany nodes mount the same volume read-only.Shared static assets, ML model files.
ReadWriteManyRWXMany nodes mount it for read-write. Requires a network filesystem (NFS, CephFS, Azure Files).Shared content stores, log aggregators that multiple Pods write into.
ReadWriteOncePodRWOPStricter than RWO — only one Pod (not just one node) gets read-write access. K8s 1.27+.Single-writer guarantees, leader-election storage.

Most cloud block storage (EBS, GCE PD, Azure Managed Disk) supports only RWO. For RWX you need a network filesystem (EFS, Filestore, Azure Files) or an in-cluster solution (Rook/Ceph, Longhorn, OpenEBS).

Section 1.9 · Reclaim policies & binding modes

What happens when the PVC is deleted, and when the PV is provisioned

Reclaim policy on the PV decides what happens to the underlying disk when the PVC is deleted:

  • Delete (default for dynamic) — the PV and the underlying cloud disk are deleted with the PVC. Tidy. Risk: an accidental PVC delete loses your data.
  • Retain — the PV stays around in Released state with the data intact. A human reclaims or deletes it later. Safer for production data.
  • Recycle — deprecated, removed. Don't use.

Volume binding mode on the StorageClass decides when the PV is provisioned:

  • Immediate (the historical default) — the PV is provisioned the moment the PVC is created, before any Pod has been scheduled. In a multi-zone cluster this can put the disk in a zone the Pod can never reach.
  • WaitForFirstConsumer — the PV is only provisioned once a Pod that uses the PVC has been scheduled. The scheduler picks the node first; storage gets provisioned in the right zone. This is the right default for any multi-zone cluster.

[ deep dive — skip if new ]Static provisioning (the cluster admin creates PVs in advance, manually) is rare in modern clusters but still used for hand-managed network storage, NFS exports, or pre-existing SAN volumes. Dynamic provisioning via StorageClass is the default. When you read about an "in-tree" provisioner versus a "CSI" provisioner, that's a generational distinction — older Kubernetes shipped storage drivers inside the kube-controller-manager (in-tree); modern Kubernetes loads them as separate CSI drivers (out-of-tree). All in-tree drivers are being migrated to CSI. Lesson 19 covers CSI in detail.

⏸ Pause and check

A Pod has been writing user uploads to /data in its container's filesystem for an hour. The Pod restarts. What survives?

Answer: c. Container filesystem is wiped on container restart. emptyDir is wiped on Pod removal. Only PVC-backed volumes persist across the Pod's full lifecycle and across rescheduling.

Section 2 · Before & After

Manual disk wrangling vs declarative storage

Before · hand-managed

You SSH into a cloud VM and attach an EBS volume by hand. You mkfs.ext4, mount it at /mnt/data, edit /etc/fstab. You document the steps in a runbook nobody reads. When the VM moves, someone has to re-attach the volume. Multiply by 50 services and 3 environments and you have a permanent backlog.

After · PVC + StorageClass

You write a PVC manifest declaring 10Gi of fast-ssd class. kubectl apply. K8s provisions an EBS volume, formats it, attaches it to the right node, mounts it into the Pod at /data. The Pod restarts → the same disk is re-attached. The Pod moves to another node → the disk is detached and re-attached there. You wrote no scripts. The cluster handled it.

What didn't change: the disk itself. EBS, GCE PD, Azure Disk — all still real cloud volumes. What changed is the layer between the Pod's wish and the cloud's reality. Declarative storage is just declarative scheduling, applied to disks.

Section 3 · Analogy

The Customs Warehouse

You arrive at K-Town's Customs Warehouse with goods you need to store. You don't pick a locker yourself — you fill out a rental form describing what you need: how much space (10Gi), whether you alone need access or several teammates, and how long you'll need it. The clerk reads your form, checks the locker manifest, and either assigns you an unused locker that fits or orders a new one built to spec. The clerk hands you a key. You use the locker for as long as you need; when you check out, the warehouse policy decides whether to clear the locker or keep it pending pickup.

The warehouse runs by three roles. The rental form (PVC) is yours — every team that wants storage submits one. The locker (PV) is the warehouse's — physical, with capacity and locks. The warehouse policy (StorageClass) is set by the city — what kinds of lockers exist, how to build new ones, what happens when a renter leaves. Renters never deal with policy; clerks never deal with apps; the city writes policy once.

In the story……in Kubernetes
The rental form you fill outA PersistentVolumeClaim (PVC)
The locker assigned to youA PersistentVolume (PV)
The warehouse policy bookA StorageClass
The clerk reading your formThe provisioner (CSI driver)
"Just me / many of us / read-only"Access modes (RWO / RWX / ROX / RWOP)
Whether the locker is wiped or saved at checkoutReclaim policy (Delete vs Retain)
"Assign now" vs "wait until you arrive at your destination"Volume binding mode (Immediate vs WaitForFirstConsumer)

⚠️ The analogy stops here: real lockers don't move. K8s persistent volumes can be detached from one node and re-attached to another (within the same zone for cloud block storage), so a Pod can reschedule and keep its disk.

Section 4 · Two-level explanation

Two ways to hear it

ELI5 · explain like I'm 5

Your Pod is like a kid at school. The desk they sit at gets cleared at the end of every day — anything they leave on it is gone tomorrow. If they want to keep their drawings, they need a cubby. They ask the teacher for a cubby ("I want a cubby!"). The teacher gives them one. The drawings live in the cubby. The kid can move desks every day, but the cubby stays theirs.

ELI10 · explain like I'm 10

A Pod's container filesystem is wiped on restart. To keep data, you declare a PVC (a request for storage) referencing a StorageClass (the recipe). The cluster's provisioner creates a PV (a real disk) matching the PVC, binds them, and mounts the volume into the Pod at the path you specified. Pod restarts re-attach the same PV. Choose access mode based on how many Pods need it (RWO for single-writer, RWX for shared); choose reclaim policy based on whether deleting the PVC should delete the data (Delete) or keep it (Retain); choose WaitForFirstConsumer binding mode in any multi-zone cluster.

⏸ Pause and check

A team has 5 stateless web Pods that need to share a directory of static assets — read-only. What access mode fits?

Answer: b. ReadOnlyMany fits the pattern: many readers, no writers. Most cloud block storage supports it. ReadWriteMany would work too but requires a network filesystem and adds cost without need.

Section 5 · Real-world scenarios

How real teams use these three objects

A SaaS startup running PostgreSQL

Single primary, two read replicas, all on EKS. Storage class: gp3 with 4000 IOPS, WaitForFirstConsumer, Retain reclaim. Each Pod gets a 200 GiB PVC via StatefulSet's volumeClaimTemplates. Pod restarts re-attach the same disk. When they decommission a replica, the PV stays in Released state — a human verifies and deletes manually. Zero data loss in 18 months.

A bank running shared log aggregation

Five Fluentd Pods need to write into the same staging directory before forwarding to a SIEM. ReadWriteMany on Azure Files (SMB-based). Slower than local SSD but everyone can write concurrently. The team accepted the latency hit for the pipeline simplicity.

A media company training ML models

Training Pods need read access to a 4 TB dataset of pre-processed images. Their CSI driver supports ReadOnlyMany from object storage (S3-via-CSI). Each new training run mounts the same PVC; no copying, no duplication. Cost: pennies. Updates: re-mount when the dataset version changes.

A retail platform migrating from local disk

Legacy app wrote to /var/lib/myapp/sessions on the host. New PVC mounted at the same path via subPath. Code unchanged. The PVC handles the persistence; the app code stayed identical. Migration was a YAML change, not a refactor.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: A PVC is the disk.
Truth: The PVC is the request. The PV is the disk. They're separate objects that bind 1:1. You can delete a PVC without deleting the PV (with Retain reclaim policy).
Myth: WaitForFirstConsumer is just an optimisation.
Truth: It's a correctness fix in any multi-zone cluster. With Immediate binding the disk gets provisioned in some random zone before the Pod is scheduled — and cloud block storage can't cross zones. The Pod ends up Pending forever.
Myth: Deleting a PVC deletes the data instantly.
Truth: Depends on the PV's reclaim policy. Delete (default for dynamic) → yes, the disk is gone. Retain → the PV moves to Released; data still there until a human decides. For production data, prefer Retain.
PV vs PVC?
tap to flip
PVC = Persistent Volume Claim, the request (namespace-scoped, written by app teams). PV = Persistent Volume, the actual disk (cluster-scoped, created by provisioner or admin). They bind 1:1.
What is StorageClass for?
tap to flip
A "recipe" telling K8s how to provision PVs on demand. Names the CSI provisioner, sets parameters (disk type, IOPS, encryption), reclaim policy, volume binding mode.
Four access modes?
tap to flip
RWO ReadWriteOnce — one node, read-write. ROX ReadOnlyMany — many nodes, read-only. RWX ReadWriteMany — many nodes, read-write (needs network FS). RWOP ReadWriteOncePod — one Pod, read-write (1.27+).
Reclaim policies?
tap to flip
Delete (default for dynamic) — PV and underlying disk deleted with the PVC. Retain — PV stays in Released state; human reclaims. Recycle — deprecated; don't use.
Volume binding modes?
tap to flip
Immediate — provision PV when PVC created (problematic in multi-zone). WaitForFirstConsumer — provision PV when first Pod scheduled (correct for multi-zone — disk lands in the right zone).
Static vs dynamic provisioning?
tap to flip
Static — admin pre-creates PVs by hand; PVCs match against existing PVs. Dynamic — StorageClass + provisioner creates PVs on demand. Dynamic is the modern default.
When does data actually get cleaned?
tap to flip
Container filesystem: at every container restart. emptyDir: at Pod removal. PVC-backed volumes: per the PV's reclaim policy when the PVC is deleted (Delete = now; Retain = manual).
Why doesn't RWX work on EBS?
tap to flip
EBS (and most cloud block storage) attaches to one node at a time. RWX requires multiple nodes mounting concurrently — only network filesystems (EFS, Filestore, Azure Files, Ceph, Longhorn) support it.

A team writes storageClassName: "" (empty string) on their PVC. What does this mean?

answerIt explicitly opts out of dynamic provisioning. The PVC will only bind to a pre-existing PV that also has empty storageClassName. Different from omitting the field, which uses the cluster's default StorageClass. Used when you've manually created a PV and want this PVC to claim it specifically.

You delete a PVC with kubectl delete pvc my-data. The underlying disk should be preserved. What's the right setup?

answerUse a StorageClass with reclaimPolicy: Retain (or set the PV's reclaim policy to Retain after the fact with kubectl patch pv ... --type=json -p='[{"op":"replace","path":"/spec/persistentVolumeReclaimPolicy","value":"Retain"}]'). On PVC deletion the PV moves to Released state with the data intact. A human verifies before deleting it manually.

Words from this lesson · open if you want a quick reference
PersistentVolume (PV)
A piece of storage in the cluster — an actual disk. Cluster-scoped object. Has capacity, access modes, reclaim policy, and a backing CSI driver.
PersistentVolumeClaim (PVC)
A request for storage by a workload. Namespace-scoped. Says "I need X Gi of Y StorageClass with Z access mode."
StorageClass
Cluster-scoped recipe that tells K8s how to provision a PV when a PVC arrives. Names the provisioner, parameters, reclaim policy, binding mode.
Access mode
RWO / ROX / RWX / RWOP — describes how many concurrent mounters and what permissions they get.
Reclaim policy
What happens to the underlying disk when its PVC is deleted. Delete = removed; Retain = preserved (manual cleanup).
Volume binding mode
When the PV is provisioned. Immediate = on PVC creation. WaitForFirstConsumer = when the first consuming Pod is scheduled.
Static provisioning
Admin manually creates PVs ahead of time; PVCs claim from this pool. Used for hand-managed network storage or pre-existing volumes.
Dynamic provisioning
PVs created on-demand by a StorageClass's provisioner when a matching PVC arrives. The modern default.
In-tree provisioner
Storage driver historically built into Kubernetes itself (kube-controller-manager). Being phased out in favour of CSI.
CSI driver
Container Storage Interface — the modern standard for plugin-based storage drivers. Covered in detail in Lesson 19.
