Module D2 · Ship Rebuild Yard — rebuild from Git + backups in < 30 min

D2 · GitOps-Driven Recovery + Cluster Rebuild + Application Restore

Total cluster loss → rebuild path. Cluster IaC: Terraform / Crossplane / Pulumi rebuilds the cluster (control plane + node pools + IAM + networking). GitOps (Argo CD / Flux): syncs all apps from Git as soon as cluster is up. Velero restore: PVC + Secret + ConfigMap data restored. Order matters: cluster → CRDs → operators → apps → restore data. End-to-end RTO < 30 min for well-built systems.

🚨 The 3 AM Nightmare

It's 3 AM. Cluster is gone. Team has Terraform for the cluster + Velero backups for data + Argo CD for apps. But: the order matters; without a tested runbook, the rebuild is chaos. Today's lesson: the order + the runbook + the drill.

🎯 If you remember nothing else: Recovery order: cluster IaC → CRDs → operators → apps (GitOps) → Velero restore. Tested via quarterly drill. RTO < 30 min for well-built systems.

Section 1.1 · cluster IaC rebuild

Terraform / Crossplane / Pulumi

Cluster itself is rebuildable from IaC. Terraform / Crossplane / Pulumi declares: control plane (managed K8s service) + node pools + IAM + networking + DNS.

Rebuild time: 5-15 min on managed K8s; 30-60 min on self-managed (etcd restore + control-plane bootstrap). Cluster name + cloud account + region preserved (or new region for disaster).

Pre-prep: IaC tested + drill-rebuilt quarterly to non-prod; runbook updated.

⏸ Pause and check

Why IaC for cluster rebuild?

IaC encodes cluster spec; new cluster matches old cluster's shape automatically. Manual rebuild = hours + drift.

Section 1.2 · GitOps sync

Argo CD / Flux re-creates everything

Once cluster is up: install Argo CD / Flux (one Helm chart). Argo CD reads Git; reconciles all Apps. Within 5-10 min, all workloads spawn.

Sync waves matter: CRDs first (sync wave -2), operators (sync wave -1), apps (sync wave 0+). Without ordering, apps fail because their CRDs don't exist yet.

ApplicationSet generators replicate fleet-wide; new cluster picks up automatically.

Section 1.3 · Velero restore

PVCs + Secrets + ConfigMaps

Velero restore per backup. Restore order: per-namespace; high-priority first (databases / auth services); low-priority later. velero restore create --from-backup <name>.

Mapping: --namespace-mappings if restoring to a different namespace; --include-resources to limit scope.

External services (RDS / Vault) restore separately via their own DR procedures. Coordinate the timeline.

Section 1.4 · runbook + drill

Tested quarterly; RTO measured

Runbook: numbered steps; per cluster shape; tested. Common skeleton:

  1. Provision cluster via Terraform / Crossplane (~5-15 min).
  2. Install Argo CD / Flux (~2 min).
  3. Argo CD syncs CRDs + operators (sync wave) (~5 min).
  4. Argo CD syncs apps (~5 min).
  5. Velero restore PVCs + Secrets + ConfigMaps (~5 min).
  6. Validate: SLO metrics return.
  7. DNS swap to new cluster (Route 53 / Azure DNS).
  8. Tenant communications.

Quarterly drill: full rebuild to non-prod; measure RTO; update runbook. Find issues before real loss.

⏸ Pause and check

Why sync waves matter in restore?

CRDs (sync wave -2) → operators (sync wave -1) → apps (sync wave 0+). Without order, apps fail because CRDs don't exist yet.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

A few common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: "Velero alone is enough."
Truth: Velero handles K8s objects + PVCs; cluster itself needs IaC; secrets often need separate ESO + Vault recovery; DNS + external services need own DR. Coordinate.
Myth: "Argo CD will figure out the order."
Truth: Without sync waves, Argo CD applies in graph order — apps before CRDs may fail. Annotate sync-wave for deterministic ordering.
Myth: "GitOps recovery is too complex; just snapshot the cluster."
Truth: Snapshot-based recovery has limits (state drift, can't cross-region, large size). GitOps + IaC + Velero is more flexible + recoverable to any cluster shape.

Walk total-cluster-loss recovery for an EKS cluster.

answer(1) Trigger runbook: cluster destroyed (real or drill). (2) Provision new EKS cluster via Terraform (~10 min). (3) Install Argo CD via Helm (~2 min). (4) Argo CD ApplicationSet auto-discovers new cluster + syncs Apps. (5) Sync wave order: CRDs (-2) → operators like cert-manager / ESO (-1) → apps (0+) → ~10 min. (6) Velero restore: per-namespace; high-priority first. (7) External Secrets: ESO re-syncs from Vault (kept external). (8) DNS swap: Route 53 record points at new ALB. (9) Validate: SLO metrics return; tenant tests pass. (10) Postmortem: timeline + RTO measured + runbook updates.

A team's drill reveals 90-min rebuild (target 30 min). Walk improvement.

answer(1) Profile each phase: cluster IaC (15 min), CRDs (5 min), operators (15 min), apps (10 min), Velero (40 min — bottleneck), DNS swap (5 min). (2) Optimise Velero: parallel restore; CSI snapshots faster than Restic / Kopia for large volumes; smaller backup units (per-namespace). (3) Pre-warm cluster: keep DR cluster warm with same cluster shape; on disaster, just sync apps + restore data — skip cluster IaC. (4) Iterate: drill again next quarter; measure improvement.

The CFO asks: "why pay for DR engineering + drills?" Defend.

how the DR architect defended drills"Without tested DR, an outage is open-ended; with tested DR, RTO is bounded. Three reasons: (1) Customer expectations: enterprise customers want 99.9-99.99% SLA; without DR, single failure breaks SLA. (2) Compliance: SOC2 / HIPAA / PCI all require tested DR. (3) Cost of bad outage: customer churn + lawsuits + reputation. DR engineering cost: ~1-2 engineers + Velero / K10 + drill time. Cost of one severe outage: orders of magnitude more."
