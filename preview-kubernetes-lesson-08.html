Module 2 · Lesson 08 · how containers actually work

Linux Namespaces · cgroups · Capabilities

A container is just a process with walls and meters — Linux features that make it think it's alone on the machine, with limits on how much it can use. No magic. Just well-named kernel features.

🚨 The 3 AM Nightmare

Your container keeps getting killed. The logs say OOMKilled. You set the memory limit to 256Mi. The app reports it only uses 200MB. Why is it dying? Because the kernel — the real boss of the whole machine — is enforcing limits you didn't fully understand. This lesson is about the five Linux features that decide what your container can and can't do. No magic. Just plumbing.

🎯 If you remember nothing else: A container is a Linux process with walls (namespaces), meters (cgroups), and rules about what it can do (capabilities, seccomp, LSM). Five kernel features, no magic.

Section 1 · Concept

What's actually inside a container?

A container isn't a magical sandbox. It's a regular Linux process that the kernel tricks into thinking it's alone on the machine. Five Linux features make this work, and they all came from the kernel — not from Docker, not from Kubernetes. Docker just made them easy to use.

Namespaces give each container its own private view of system resources. Process IDs, network interfaces, mount points, hostnames, user IDs — they all look private to the container, but they're really shared kernel resources sliced into isolated views. There are 8 namespace types in modern Linux: PID, NET, MNT, UTS, IPC, USER, cgroup, and time.

cgroups (control groups) are the meters. Each container gets a budget for CPU, memory, disk I/O, and network bandwidth. Try to use more than your budget, and the kernel throttles you (or kills the process for memory).

[ deep dive — skip if new ]Two versions of cgroups exist — cgroup v1 (legacy, hierarchies per resource) and cgroup v2 (unified hierarchy, the modern default since ~2022). Kubernetes 1.25+ requires v2 for certain features (PSI-based memory pressure, the memory.high soft cap).

Capabilities chop the old "root vs not-root" model into ~40 fine-grained permissions. Instead of giving a container full root, you give it just NET_BIND_SERVICE (bind to port < 1024) or just SYS_TIME (change the clock). Containers should drop everything they don't need.

seccomp filters which Linux system calls a container is allowed to make. Modern container runtimes apply a default profile blocking roughly 40–65 dangerous syscalls (varies by runtime version) out of ~300 total. You can write custom profiles for tighter restriction.

AppArmor / SELinux are mandatory access control (MAC) systems. They define explicit policies — "this container can read these files but not those" — at a layer below normal Linux permissions. Different distros pick different ones (Ubuntu/Debian default to AppArmor; RHEL/Fedora default to SELinux).

That's it. Five Linux features. Stack them together and you have a container. Take them away and a container is just a normal process.

Section 1.7 · The 8 namespaces, in detail

One row per namespace type

These are the 8 namespace types Linux ships today. Each gives a container a private view of one specific kind of resource.

PID
Own process ID tree. Container sees its own PID 1, 2, 3 — even though those are different real PIDs on the host.
NET
Own network interfaces, IP addresses, routing table, firewall rules. Each container has its own private network stack.
MNT
Own filesystem mount table. Container sees its own root filesystem (built from image layers).
UTS
Own hostname and domain name. Each container can have its own hostname like a real machine.
IPC
Own inter-process communication (System V IPC, POSIX message queues). Containers can't talk to each other through shared memory by default.
USER
Own user/group ID mapping. Container's UID 0 (root) can map to host UID 100000 (unprivileged). Powerful for security.
cgroup
Own view of the cgroup hierarchy. Container sees its own cgroup root, not the host's.
time
Newest. Own monotonic and boot clocks (not wall clock). Useful for testing time-sensitive code in containers.

When you create a container, the runtime calls clone() with flags like CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS to put the new process in fresh namespaces. That single syscall is the moment a "container" comes into existence.

⏸ Pause and check

What stops one container from seeing another container's processes?

Answer: b. Each container gets a private view of process IDs. Without that namespace, ps would show every process on the host.

Section 2 · Before & After

Without these primitives vs with them

Linux had multitasking forever — multiple processes could run on the same machine. But until namespaces and cgroups, "isolation" meant Unix permissions and prayer. Containers required the kernel to grow new tools.

Same machine, same processes. Namespaces give them privacy. cgroups give them budgets.
Before · pre-2008

Multiple processes on one Linux machine shared everything: same PID space, same network, same filesystem, same hostname. Process A could see B's PIDs, ports, files. If A started a runaway loop, it stole all the CPU. Isolation was Unix permissions and prayer.

Companies that wanted real isolation used VMs — heavy, slow to start, but each VM had its own OS so isolation was guaranteed.

After · namespaces + cgroups

Linux added namespaces (private views) and cgroups (resource budgets) over the late 2000s. By 2013, Docker showed the world how to combine them into something usable. Now multiple processes can run on the same kernel but think they're alone, with hard limits on CPU and memory.

That's a container. Process-level isolation, milliseconds to start, hundreds per host. Cheaper than VMs. Better isolation than mere Unix permissions.

What didn't change: containers still share the kernel. A kernel exploit affects everything. For untrusted multi-tenancy, you still want VMs or sandboxed runtimes (gVisor, Kata).

Section 3 · Analogy

An office building, with utility meters per office

Think of a 4-story office building. Many tenants share the same building — same foundation, same plumbing, same electricity. But each tenant rents a private office: their own door, their own desk, their own walls, their own filing cabinet. From inside, it feels like their own space. They don't trip over the next tenant's stuff.

The building has a utility meter for each office. Office A pays for the electricity it uses. Office B has its own meter. The landlord caps how much power each office can pull, so one office running a giant printer doesn't blow the whole building's circuit. There's a security guard at each door checking IDs to make sure only the right people get in.

Linux containers work exactly the same way. Namespaces are the office walls (private views of resources). cgroups are the utility meters (resource budgets). Capabilities and seccomp and AppArmor/SELinux are the security guards (which actions are allowed). The building (the host kernel) is shared by everyone, but each container thinks it's alone.

⚠️ The analogy stops here: an office building has one electrical panel that one tenant can overload. The kernel can be DoS'd by a noisy container too — but cgroups make this much harder than the real-world equivalent.

Section 4 · Two-level explanation

Two ways to hear it

Same machinery, two depths.

ELI5 · explain like I'm 5

Imagine a big shared playroom. Linux puts up little curtains so each kid has their own pretend room. Each kid has a snack bowl with rules on how many cookies they can eat. And there's a teacher checking what they're allowed to do. The kids share the same playroom floor (the kernel) but they each think they have their own little space.

ELI10 · explain like I'm 10

A container is just a regular Linux process with three things added by the kernel: namespaces (it sees its own private slice of process IDs, network, files, hostname, etc.), cgroups (it has hard limits on CPU, memory, disk I/O), and security policies (capabilities, seccomp, AppArmor/SELinux — fine-grained control over what kernel actions it can perform). When you "start a container" with Docker or containerd, the runtime calls clone() with namespace flags, configures cgroups, drops most capabilities, and applies a seccomp profile. The result is a process that thinks it's alone on the machine. Hundreds of these can run on one host because they all share the host's kernel — way more efficient than VMs.

A container hits 257MB of memory; its limits.memory is 256Mi. What happens?

Answer: b. cgroup memory limits aren't suggestions. The kernel enforces them with OOMKilled.

Section 5 · Real-world scenarios

Where these primitives matter in practice

Four real situations where understanding namespaces, cgroups, or capabilities saves the day.

Container OOM-killed at 256 MiB

An engineer notices their Pod keeps restarting. kubectl describe Pod shows reason: OOMKilled. They had set memory limit to 256Mi in the Pod spec. The app legitimately needed 350Mi at peak. The kernel's cgroup-v2 memory controller saw memory usage exceed the cgroup's limit and killed PID 1 in the container — exactly as designed. Fix: bump the limit. The lesson: cgroup limits aren't suggestions. The kernel will kill you.

Container can't bind to port 80

A developer's containerized web app fails: permission denied binding to port 80. The container is running as a non-root user (good practice). Ports below 1024 require CAP_NET_BIND_SERVICE on Linux. By default, modern container runtimes drop this capability. Fix: either run on port 8080 and let the load balancer rewrite, or explicitly add NET_BIND_SERVICE to the container's capability set. The lesson: container runtimes drop capabilities aggressively for safety; you have to explicitly opt back in.

Container A can see container B's processes

A security audit finds container A can see container B's process list via ps. Investigation: someone set hostPID: true in container A's Pod spec, sharing the host's PID namespace. This was a debugging convenience that got merged. Fix: remove hostPID: true. The lesson: every "host*" Pod field (hostNetwork, hostPID, hostIPC) drops a namespace and gives the container access to host resources. Use sparingly, audit regularly.

Custom seccomp profile for a sandboxed workload

A team runs untrusted user code (a code-execution playground). The default seccomp profile blocks roughly 40–65 dangerous syscalls (varies by runtime version) but still allows things like ptrace. They write a custom seccomp profile that blocks ptrace, ioctl, and several others. The user code can still run normal Python; it can't escape the sandbox. The lesson: seccomp default profiles are good baseline; for hostile workloads, write a tighter custom profile.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: "Container" is a Linux feature.
Truth: Containers are made by combining five kernel features (namespaces + cgroups + capabilities + seccomp + LSM). The runtime (Docker, containerd) bundles them. The kernel does the work.
Myth: Each container has its own kernel.
Truth: All containers on a host share the host's kernel. That's the whole point — and the whole risk. A kernel exploit can break out of any container.
Myth: limits and requests are the same thing.
Truth: requests is what the scheduler reserves (the floor). limits is the hard ceiling (CPU throttled, memory OOM-killed above).
What is a container, technically?
tap to flip
A regular Linux process placed in fresh namespaces (private views), constrained by cgroups (resource budgets), and restricted by capabilities + seccomp + LSM (security policies). All kernel features.
Name the 8 namespace types
tap to flip
PID, NET, MNT, UTS, IPC, USER, cgroup, time. Each gives a private view of one specific kind of resource.
cgroups v1 vs v2?
tap to flip
v1: separate hierarchies per resource (CPU hierarchy, memory hierarchy, etc.). v2: unified single hierarchy. v2 is the modern default since ~2022 (Kubernetes 1.25+ requires it on certain features).
What are Linux capabilities?
tap to flip
~40 fine-grained permissions chopping the old "root vs not-root" model. Examples: NET_BIND_SERVICE (bind to port < 1024), SYS_TIME (change clock), SYS_ADMIN (most-dangerous, do not grant). Drop everything you don't need.
What is seccomp?
tap to flip
Linux feature filtering which system calls a process can make. Modern container runtimes apply a default profile blocking roughly 40–65 dangerous syscalls (varies by runtime version). Custom profiles allow tighter restriction for sensitive workloads.
AppArmor vs SELinux?
tap to flip
Both are Linux Security Modules (LSM) providing mandatory access control. AppArmor is path-based, simpler. SELinux is label-based, more powerful but harder. Ubuntu/Debian default to AppArmor; RHEL/Fedora default to SELinux.
What does hostPID: true do?
tap to flip
Disables the PID namespace for the container, sharing the host's PID namespace instead. Container can see all host processes via ps. Useful for debugging, dangerous for security. Same pattern: hostNetwork, hostIPC.
Why is the kernel shared?
tap to flip
Containers are processes on the host kernel — that's the whole point of containers vs VMs. It makes them lightweight (no boot time, no OS overhead per container). The trade-off: a kernel exploit can break out of any container. For untrusted multi-tenancy, use VMs or sandboxed runtimes (gVisor, Kata).

A container running as non-root user keeps failing with "permission denied" when binding to port 80. What's happening, and what are two fixes?

answerPorts below 1024 require CAP_NET_BIND_SERVICE on Linux. Modern container runtimes drop this capability by default for security. Two fixes: (1) bind to port 8080+ and let your load balancer/Service map external port 80 to internal 8080 (preferred — keeps the container low-privilege). (2) Explicitly add NET_BIND_SERVICE to the container's capability set in the Pod spec under securityContext.capabilities.add. The first is the cleaner pattern.

A Pod has memory limit 256Mi and the container keeps getting OOMKilled. The app reports it only uses ~200MB. Why might it still be OOMKilled?

answerThe cgroup memory accounting includes more than the app's heap: page cache, anonymous mappings, kernel memory, etc. The 256Mi limit is enforced on the cgroup's total memory.usage_in_bytes, which includes everything the container has caused the kernel to allocate. Common causes of "I only use 200MB but it's OOMKilled at 256MB": JVM/Node off-heap, large file caches, memory-mapped files. Fix: bump the limit or set memory.high (cgroup v2) for soft pressure before the hard kill.
