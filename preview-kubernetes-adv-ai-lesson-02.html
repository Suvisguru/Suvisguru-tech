I2 · Kueue + MultiKueue + Volcano Gang Scheduling

🚨 The 3 AM Nightmare

It's 3 AM. Distributed training of 16 Pods stalled with 14 running + 2 stuck Pending. K8s schedules Pods independently; multi-Pod jobs need all-or-nothing. Today's lesson: Kueue + Volcano give batch admission + gang scheduling so multi-Pod jobs work.

Section 1.1 · Kueue model

ResourceFlavor + ClusterQueue + LocalQueue + Workload

ResourceFlavor: cluster's capacity slice (e.g., "H100-flavor"). ClusterQueue: cluster-scoped queue declaring quota per ResourceFlavor + cohort + preemption rules. LocalQueue: namespace-scoped pointer to ClusterQueue. Workload: Kueue-tracked job (Job / JobSet / RayJob / MPIJob).

Tenant submits Job to LocalQueue; Kueue admits if quota available; Pods then schedule normally.

⏸ Pause and check

Why use Kueue ClusterQueue?

Kueue ClusterQueue is admission with cohort + preemption + waiting-queue semantics; ResourceQuota is per-namespace static. Different concerns; both useful.

Section 1.2 · Volcano gang scheduling

all-or-nothing + min-member + queue priorities

Volcano scheduler runs alongside or replaces kube-scheduler. Schedules PodGroup (gang) atomically: all min-member Pods schedule together or none do.

Avoids the half-running half-pending failure mode for distributed training. PodGroup priorities + queue admission supplement default scheduling.

Section 1.3 · MultiKueue federation

Submit once; runs in any cluster with capacity

MultiKueue: control-plane Kueue cluster federates worker Kueue clusters. Tenant submits Workload to control plane; MultiKueue routes to worker cluster with capacity.

Wins: burst across clouds, multi-region GPU pools, spot-fallback. ML platform spans many clusters; tenants see one queue.

Section 1.4 · workload kinds + integrations

Job / JobSet / RayJob / MPIJob + Kubeflow integration

Kueue admits standard K8s Job + JobSet (multi-pod) + RayJob (KubeRay) + MPIJob (Kubeflow MPI Operator) + PyTorchJob / TFJob (Kubeflow Training Operator).

Pattern: Kubeflow Training Operator submits PyTorchJob → Kueue admits → Volcano gang-schedules → distributed training runs. End-to-end batch ML pipeline.

⏸ Pause and check

Why gang scheduling for distributed training?

Distributed training needs all participants up to start; without gang, partial start hangs. Volcano gang-schedules atomically.

Section 5 · Real-world scenarios

How AI / ML teams do this

Distributed training — Volcano gang

ML team's 16-Pod PyTorchJob hung pending half-up. Migrated to Volcano: PodGroup admits 16 Pods together; training starts cleanly. Throughput up; partial-start incidents zero.

Kueue cohort lending

Two teams share a GPU pool. Each has 8-GPU quota; cohort lends idle quota across teams. Team A bursts to 16 GPU when Team B idle; reverts when B requests. Utilization rises without contention.

MultiKueue across clouds

ML platform fleets across AWS + GCP + on-prem. Tenants submit RayJobs to control-plane Kueue; MultiKueue routes per available capacity. Spot-fallback across clouds; tenant doesn't care which cloud.

Outage — non-Kueue partial-fill

Pre-Kueue, Job's Pods scheduled greedily; partial-fill during high-demand; jobs deadlocked. Postmortem: adopt Kueue + Volcano; gang admit prevents.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

A few common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: "Kueue replaces ResourceQuota."
Truth: They're complementary. ResourceQuota = per-namespace static cap. Kueue = batch admission + cohort + preemption. Both useful for different patterns.
Myth: "Volcano replaces kube-scheduler."
Truth: Volcano can either replace or run alongside kube-scheduler. Modern pattern: Volcano for gang-scheduled batch; kube-scheduler for everything else. Coexist.
Myth: "MultiKueue is required for multi-cluster ML."
Truth: Optional. MultiKueue is the K8s-native answer. Some platforms use Argo Workflows + custom routing; both work; MultiKueue is the standard for new builds.
Kueue four concepts?
tap to flip
ResourceFlavor (capacity slice), ClusterQueue (cluster quota + cohort + preemption), LocalQueue (namespace-scoped), Workload (Kueue-tracked job).
What is gang scheduling?
tap to flip
All-or-nothing scheduling for multi-Pod jobs. N Pods (PodGroup min-member) start together or none start. Volcano implements.
MultiKueue what does it federate?
tap to flip
Workloads across worker Kueue clusters; control-plane Kueue routes per available capacity. Tenants see one queue; platform spans many clusters.
Cohort lending in Kueue?
tap to flip
Multiple ClusterQueues in same cohort can borrow idle quota from each other. Per-team quota stays the contract; bursts use cohort spare.
Workload kinds Kueue admits?
tap to flip
Job, JobSet, RayJob (KubeRay), MPIJob (MPI Operator), PyTorchJob / TFJob (Training Operator), plus custom CRDs via Kueue integration.
When use Volcano?
tap to flip
Multi-Pod jobs needing all-or-nothing (distributed training, MPI). Single-Pod jobs don't need it. Often run alongside kube-scheduler.
Kueue preemption?
tap to flip
Higher-priority Workload can preempt running lower-priority Workloads in the same cohort. Configurable per ClusterQueue.
Pattern for distributed training?
tap to flip
Kubeflow PyTorchJob / TFJob → Kueue admits → Volcano gang-schedules → training runs. End-to-end batch ML.

Walk Kueue + Volcano setup for a multi-tenant ML cluster.

answer(1) Install Kueue Helm + Volcano Helm. (2) ResourceFlavor per accelerator type (H100, A100). (3) ClusterQueue per team with quota + cohort "ml-cohort" for cohort lending. (4) LocalQueue in each tenant namespace. (5) PyTorchJob / RayJob annotation: kueue.x-k8s.io/queue-name: .... (6) Volcano: PodGroup auto-created from Job; gang-schedules. (7) Validate: Kueue dashboards show admission queue depth + cohort utilization.

MultiKueue across 3 clusters — design federation.

answer(1) Pick control-plane cluster: small dedicated cluster running MultiKueue Admission webhook. (2) Worker clusters: each runs Kueue + ClusterQueue + ResourceFlavor matching the federation's capacity. (3) MultiKueue config: control plane registers worker cluster connections (kubeconfig + auth). (4) Tenant submits Workload to control plane; MultiKueue routes to worker. (5) Spot fallback: configure cohort priorities so on-demand cluster gets first dibs; spot cluster receives overflow.
