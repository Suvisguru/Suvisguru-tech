Module 2 · Lesson 09 · the standard format + the chefs that read it

Container Runtimes & OCI Standard

An OCI image is a standardized box. A registry is a warehouse. A runtime is the forklift that pulls a box and turns it into a running container. Many forklifts. One standard.

🚨 The 3 AM Nightmare

Your team built an app on an ARM Mac (M-series). You deploy to AMD64 nodes in EKS. The Pod crashes immediately: exec format error. You can't tell from the message that the binary is the wrong architecture. You stare at logs for an hour. This lesson is about the format that makes containers portable — and the standard that lets one image work on every CPU and every cloud.

🎯 If you remember nothing else: OCI is the standard. Runtimes are forklifts. Layers stack. Tags move; digests don't. Pin to digests in production.

Section 1 · Concept

The standard, the runtimes, and the storage

In 2015, the container ecosystem was at risk of fragmenting — Docker had its own format, CoreOS had rkt with a different format, and every cloud was building incompatible variations. The industry agreed to a common standard hosted by the Linux Foundation: OCI (Open Container Initiative). Today, every serious container tool implements OCI specs.

OCI defines three specifications: the image spec (what's in a container image — layers, manifest, config), the runtime spec (how to start a container from an unpacked filesystem), and the distribution spec (how registries serve images via HTTP). Together they form a complete contract that lets you build with one tool, store in any registry, and run with any runtime.

[ deep dive — skip if new ]Container runtimes are the programs that actually start containers. Two layers exist: a high-level runtime like containerd or CRI-O handles image pulling, lifecycle, and the Kubernetes CRI (Container Runtime Interface). A low-level runtime like runc, crun, or youki does the actual clone() + cgroups + capabilities setup we saw in Lesson 08. runc is the reference implementation (Go); crun is a faster C rewrite; youki is a Rust rewrite.

An image is a stack of layers (read-only filesystem snapshots), a manifest (JSON listing the layers + their digests), and a config (how to run it: command, env, ports). When the runtime "runs an image," it stacks those layers into a unified filesystem, adds a writable top layer, and starts the container.

Tags are mutable friendly names like nginx:1.27 or my-app:latest. They point at images. A tag can be moved tomorrow. A digest is a cryptographic hash like sha256:abc123… that identifies a specific image content. Same content = same digest, always. For reproducibility and supply-chain safety, prefer digests in production.

Registries are HTTP services that store and serve images. Big public ones: Docker Hub, GitHub Container Registry (GHCR), AWS ECR, Google Artifact Registry, Azure ACR, Red Hat Quay, JFrog Container Registry, Harbor (open-source self-hosted). All speak the OCI distribution spec, so the same image works across them.

Section 1.7 · The runtime landscape (2026)

Who's who in the runtime world

Five tools you'll hear about. They split into two layers — the high-level runtime that talks to Kubernetes, and the low-level runtime that talks to the kernel.

containerd
High-level · default in K8s
CNCF graduated. Default container runtime in most Kubernetes distributions. Handles image pulling, container lifecycle, CRI for kubelet.
CRI-O
High-level · K8s-only
Built specifically for Kubernetes (no general-purpose API). Smaller than containerd. Common on OpenShift and bare-metal K8s.
runc
Low-level · reference
The original OCI runtime. Written in Go. Default low-level runtime under containerd and CRI-O. Reference implementation everyone trusts.
crun
Low-level · faster
C rewrite of runc by Red Hat. Smaller binary, lower memory, faster startup. Drop-in replacement for runc on most distros.
youki
Low-level · Rust
Rust rewrite of runc. Memory-safe. Newer (gaining maturity). OCI-compatible drop-in. Picked up where speed + safety matter.
gVisor / Kata
Sandboxed runtimes
Heavier-isolation alternatives. gVisor intercepts syscalls in user space; Kata runs each container in a tiny VM. Used for hostile multi-tenancy.

Most Kubernetes clusters run containerd + runc by default. You generally don't need to think about runtime choice unless you have specific isolation or performance requirements.

Section 1.8 · Tag vs digest · the most important distinction

Why production should pin to digests

This one trips up almost everyone. Read both cards.

TAG
nginx:1.27

Mutable friendly name. The tag is a label that points at an image. The owner of the image can move the tag tomorrow.

Today nginx:1.27 might point at one image. Next week, after a CVE fix, it points at a different image (with the same "1.27" version number). That's a feature for ops (auto-updating patches) but a footgun for reproducibility.

⚠ :latest is the worst offender. Avoid in production.

DIGEST
nginx@sha256:abc123def456…

Immutable cryptographic fingerprint. The SHA256 hash of the image manifest. Same content = same digest, forever. Different content = different digest. Cannot be moved.

In production, pinning to a digest guarantees you'll get exactly the bits you tested. CI builds an image, computes its digest, deploys that digest. The deployment is now reproducible — six months later, the same digest still resolves to the same bits.

✓ Use in production. Get bytes, not vibes.

In Kubernetes manifests, write image: nginx@sha256:abc... instead of image: nginx:1.27. Tools like Renovate and Dependabot can auto-update digests via PR — keeping reproducibility AND staying current.

Answer: b. Three specs (image, runtime, distribution). The contract that lets any runtime work with any registry.

Section 2 · Before & After

Before OCI vs after

OCI didn't make containers possible — it made them portable. Before the standard, every tool had its own format and no two tools could read each other's images.

Same idea (containers), different freedom levels. OCI standardization unlocked the ecosystem.
Before · pre-OCI (2015)

Docker images only worked with Docker. CoreOS rkt had its own format. Cloud vendors built proprietary variants. Switching tools meant rebuilding everything — different image format, different registry protocol, different runtime API.

After · OCI standard

One image format works with any runtime (containerd, CRI-O, Podman). One registry protocol means any image hosts on any registry (Docker Hub, GHCR, ECR, GCR, Harbor). The image you build today still works in 5 years on a runtime that doesn't exist yet — as long as it's OCI-compliant.

What didn't change: there's still operational complexity around image signing, supply-chain security (SBOMs, provenance), and registry credential management. OCI standardized the format, not the policy.

Section 3 · Analogy

Standardized warehouses and forklifts

Imagine the global shipping industry. There's one standard for shipping containers (ISO sizes, lifting points, manifest format). Any container fits on any truck, ship, or train. Any warehouse can store any container. Any forklift can pick one up. The standard is what makes the network work.

OCI is that standard for software containers. The image format is the standard "container size." The distribution spec is how warehouses communicate. The runtime spec is what the forklift does to pick a box up and get it moving. Different forklifts (runc, crun, youki) all work with the same boxes. Different warehouses (Docker Hub, GHCR, ECR) all store the same boxes.

Inside each box, there are smaller boxes stacked together — the layers. The manifest is the packing list nailed to the side. The tag is the friendly label ("v1.27") and the digest is the unique tracking number. You can move the friendly label, but the tracking number stays with the box forever.

In the story……in Kubernetes
A standard ISO shipping containerAn OCI image (the format)
Layers stacked inside the containerImage layers (each a tarball with its own SHA-256)
The packing list on the doorThe image manifest (JSON describing the layers)
A friendly label on the outsideA tag (mutable — can be moved by the publisher)
The unique tracking numberThe digest (immutable SHA-256 hash of the manifest)
The warehouseA container registry (Docker Hub, GHCR, ECR, GCR, ACR, Harbor, Quay)
The forklifts that move boxesContainer runtimes (containerd / CRI-O at high level; runc / crun / youki at low level)

⚠️ The analogy stops here: a real shipping container has fixed dimensions. A software image is whatever size the layers add up to. The "standard" is the format, not the size.

Section 4 · Two-level explanation

Two ways to hear it

ELI5 · explain like I'm 5

There are giant warehouses full of boxes. Every box is the same shape, with a label on the outside and a packing list inside. Forklifts can pick up any box from any warehouse, because all the boxes are the same shape. Some boxes have stickers like "v1.27" — those stickers can be moved. Some boxes have a special bar code that nobody can change. That's how containers work too.

ELI10 · explain like I'm 10

OCI (Open Container Initiative) is a Linux Foundation standard with three specs: image format (what's in a container image), runtime spec (how to start a container from an unpacked filesystem), and distribution spec (how registries serve images over HTTP). Container runtimes split into high-level (containerd, CRI-O — pull images, manage lifecycle, talk to Kubernetes via CRI) and low-level (runc, crun, youki — make the kernel clone() + cgroups + capabilities calls from Lesson 08). Images are stacks of read-only layers + a manifest. Tags like nginx:1.27 are mutable labels; digests like sha256:abc... are immutable hashes. For production reproducibility, pin to digests.

Section 5 · Real-world scenarios

Where OCI, runtimes, and registries trip people up

"latest" tag broke production

A team ran image: my-app:latest in their Kubernetes Deployments. Friday at 6 PM, someone published a new "latest" with a bad migration. Pods restarting through the night picked up the new "latest" tag. By Saturday morning, half the Pods had the bad image and half had the old one. The fix: pin to digests, and never use :latest in production manifests. Renovate auto-updates digests via PR (with tests + review).

Multi-arch image build (one tag, two architectures)

A team builds an app that needs to run on both ARM (Graviton EC2) and AMD64 (Intel laptops). They use docker buildx with --platform linux/amd64,linux/arm64. The result is a single tag my-app:v2 that contains two manifests — one per arch. When a node pulls the image, the runtime picks the right manifest. Same tag, right binary on every machine.

Switched runtime from runc to crun for performance

A high-density container platform (200+ containers per host) switched from runc to crun. Crun's smaller memory footprint and faster startup added up across thousands of containers — Pod startup latency dropped by ~30%, host memory savings of ~5%. The change: edit containerd config to set the runtime path. Application teams noticed nothing — same OCI image, just a different forklift underneath.

Image pull failed: ImagePullBackOff (private registry auth)

A team pushed their image to a private GHCR repo. Deploying it failed with ErrImagePull → ImagePullBackOff. Investigation: the kubelet didn't have credentials for GHCR. Fix: create a Kubernetes Secret of type kubernetes.io/dockerconfigjson with the GHCR token, and reference it in the Pod spec via imagePullSecrets. Or use a node-level credential provider for cleaner ops at scale.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: Docker is the only container runtime.
Truth: Modern Kubernetes runs containerd or CRI-O by default. Docker is a higher-level platform that uses containerd underneath.
Myth: :latest is fine for production.
Truth: :latest is the worst tag offender. It's mutable. Two Pods with the same :latest tag can run different code on different nodes after a re-push. Pin to a digest.
Myth: An image works on any CPU.
Truth: Only if it was built multi-arch (docker buildx --platform linux/amd64,linux/arm64). Otherwise an ARM build will not run on AMD64 nodes.

What is ImagePullBackOff?

Kubernetes Pod state when the kubelet repeatedly fails to pull an image. Common causes: typo in image name, wrong tag, missing registry credentials (need imagePullSecrets), or registry rate-limiting (Docker Hub anonymous limits).

A teammate says "let's just use :latest for our production deployments — it'll auto-update us to the newest version." What do you say?

answerIt's a footgun. :latest is a mutable tag — the image owner can move it at any time. When Pods restart in production (which happens constantly), they'll pull whatever :latest happens to point at then, not what you tested with. Half your fleet may end up on a different image than the other half. Pin to digests (nginx@sha256:abc...) in production manifests, and use Renovate or Dependabot to bump digests via PR (with tests). You get reproducibility AND stay current.

A Pod stays in ImagePullBackOff. kubectl describe Pod shows the kubelet is trying to pull from ghcr.io/myorg/my-app:v2 but failing with "denied". What's the likely fix?

answerMissing or wrong registry credentials. GHCR requires authentication for private images. Fix: create a Kubernetes Secret of type kubernetes.io/dockerconfigjson containing GHCR auth (a personal access token with read:packages, or a GitHub App token), then reference it in the Pod spec via imagePullSecrets: [{name: ghcr-creds}]. For larger fleets, use a node-level credential provider (cleaner ops than per-namespace secrets).
