Module 13 · Lesson 30 · what an image actually is, and how to trust it

Supply Chain Security · Cosign, Sigstore, SLSA, SBOM

"Container image" feels like a single thing. It's actually a chain of artefacts — source code, build provenance, the image itself, an SBOM, signatures. Each link can be tampered with. Modern supply-chain security signs each link and verifies the whole chain at admission. The toolkit: Sigstore + Cosign (signing), SLSA (build provenance), SBOMs (component inventory).

🚨 The 3 AM Nightmare

2024 — XZ Utils backdoor. A maintainer added obfuscated malicious code to a fundamental Linux library, hiding it through years of trusted contributions. Caught by accident days before stable Linux distros would have shipped it. The lesson the industry took: code provenance is necessary but not sufficient — every artefact in the chain needs to be signed, scanned, and verified at admission. K8s clusters that did supply-chain right caught the issue before it ran. Clusters that did "pin tags + trust upstream" were vulnerable. This lesson is the toolkit that gets you to the first column.

🎯 If you remember nothing else: Three artefacts plus signing: SBOM (Software Bill of Materials — what's in the image), SLSA provenance (how the image was built, attested), Cosign signature (the artefact is authentic). All stored alongside the image in the OCI registry. Admission verifies the chain via cosign verify or Kyverno verifyImages.

Section 1 · Concept

What's in an image, and what can go wrong

A container image (Lesson 9-11) is a stack of layers + manifest. The chain that produced it has many places to inject something malicious:

  • Source — git commits. A compromised maintainer commits malicious code.
  • Build environment — the runner. Someone with access to the CI runner can swap in a different binary.
  • Dependencies — npm/pip/Maven packages pulled at build time. A typosquatted package replaces a legitimate one.
  • Image registry — push/pull. Tag mutability means "nginx:1.21" today can mean a different image tomorrow.
  • Pull at runtime — registry → kubelet. MITM, registry compromise.

Each link in this chain needs a verifiable receipt. The modern toolkit signs each artefact (image, SBOM, attestations) with a key the cluster trusts, and admission policies verify before running.

Section 1.5 · Sigstore + Cosign

Signing made boring

Sigstore is a CNCF project that made signing artefacts simple. Three pieces:

  • Cosign — CLI for signing/verifying container images, blobs, attestations.
  • Fulcio — short-lived certificate authority that issues x.509 certs tied to OIDC identities (your GitHub Actions OIDC token, corp SSO token, etc.). "Keyless signing" — no key file to manage.
  • Rekor — public tamper-evident log of every signing event. Auditable. Append-only.

The keyless flow:

  1. CI obtains an OIDC token (e.g., from GitHub Actions).
  2. Fulcio exchanges it for a short-lived cert tied to the workflow identity.
  3. Cosign signs the image; signature + cert + transparency log entry stored in the OCI registry alongside the image.
  4. K8s admission verifies the cert chain + the workflow identity matches expectations.

No long-lived signing keys to rotate. No HSM to manage. The OIDC identity ("this image was signed by GitHub Actions on behalf of org/repo, branch main, workflow X") is the trust root.

Section 1.7 · SLSA — provenance levels

A grading system for build supply chains

SLSA (Supply-chain Levels for Software Artifacts; pronounced "salsa") is a graded standard from OpenSSF. It defines build-process maturity from level 0 ("we don't know how this was built") to level 4 ("hermetic, reproducible, two-person reviewed"). The output of a SLSA-compliant build is an attestation describing:

  • The exact source repo + commit.
  • The exact build steps + environment.
  • The exact dependencies pulled.
  • Who triggered the build (identity).

The attestation is itself a signed artefact in the OCI registry. SLSA Level 3 (the realistic target for most orgs) requires:

  • Build runs on an isolated, ephemeral environment (typical CI runner).
  • Provenance is generated by the build platform (not user code).
  • Provenance is signed.
  • Source platform is trusted (GitHub, GitLab, etc.).

GitHub Actions gained native SLSA L3 provenance generation in 2023. Buildah and BuildKit support it. The major cloud build services (Cloud Build, AWS CodeBuild) support attestation export.

⏸ Pause and check

A team uses Cosign keyless signing in GitHub Actions. The signature in the registry has a Fulcio-issued cert. What's the trust root?

Answer: b. Keyless signing's trust root is the OIDC identity. The Fulcio cert binds the signature to the identity ("this signature was created during the GitHub Actions run for repo X, branch Y"). Verification at admission checks that identity matches expectations. Rekor adds tamper-evidence on top.

Section 1.9 · SBOMs and admission verification

What's actually in the image

An SBOM (Software Bill of Materials) lists every package + version inside an image. Two main formats: SPDX (Linux Foundation) and CycloneDX (OWASP). Both are JSON; both are widely tooled.

Generated at build time by tools like Syft (Anchore) or Trivy. Stored as an attestation in the registry. Used downstream by:

  • Vulnerability scanners — Trivy, Grype consume SBOMs to flag known CVEs without re-scanning the image.
  • License compliance — "does any package have a license we can't use?"
  • Forensics — "after the log4j CVE, which images contain a vulnerable version?" SBOM lookup, not code reading.

The complete admission policy looks like:

# Kyverno verifyImages (simplified)
- imageReferences: ["prod-registry.corp/*"]
  attestations:
  - type: https://slsa.dev/provenance/v1
    attestors: [{keys: {publicKeys: "..."}}]
    conditions: [{key: "buildType", operator: Equals, value: "github-actions-slsa3"}]
  - type: https://cyclonedx.org/bom
    attestors: [{keys: {publicKeys: "..."}}]
  attestors:
  - entries: [{keyless: {issuer: "https://token.actions.githubusercontent.com", subject: "https://github.com/myorg/.*"}}]

This says: image must be from prod-registry, must have a SLSA L3 attestation from our CI, must have a CycloneDX SBOM, and the image must be signed by a Fulcio cert from our GitHub org. Every link in the chain verified at admission.

Section 2 · Before & After

What changes when you adopt this

Before

Old practice: pin nginx:1.21, hope upstream is honest, scan with Trivy nightly, react when CVE drops. "Trust" was a mix of a registry password + a tag string. SBOMs were a wishful spreadsheet. Build provenance was the assumption that GitHub's CI ran what was committed.

After

Modern: pin by digest (sha256:abc...), verify a Fulcio-signed cert tying the image to a specific OIDC identity, verify a SLSA L3 provenance attestation describing the build, verify an SBOM attestation, all at admission. Tag mutability is irrelevant. Compromise of a registry doesn't bypass admission. Compromise of a maintainer triggers a verification mismatch.

XZ Utils 2024 was the wake-up call. The industry is moving from "trust the registry" to "verify each artefact in the chain." Sigstore + SLSA is the toolkit; admission policies are the enforcement.

Section 3 · Analogy

The story for this district

Bank Vault Quarter holds K-Town's ledgers. Every package coming into the city has a paper trail. A signed declaration of origin (Cosign signature) — "this came from this CI workflow on this date." A build manifest (SLSA provenance attestation) — "these are the steps used to produce it; here's the runner identity that signed off." A contents inventory (SBOM) — "the package contains these specific component versions." All stamped by a notary (Sigstore / Fulcio) whose stamps are themselves logged in a public ledger (Rekor) that nobody can edit retroactively. At the city gate (K8s admission), an inspector checks every package's paper trail against the city's allow-list. No paper trail = denied entry.

⚠️ The analogy stops here: Sigstore notarisation is cryptographic, not paper. The trust root is OIDC + Rekor's tamper-evidence; if either is compromised, signatures can't be trusted. Pin to a known Rekor log entry for full assurance.

Section 5 · Real-world scenarios

How real teams use this

A SaaS shipping production with full chain

GitHub Actions builds + signs images via Cosign keyless. SLSA L3 provenance generated by the GitHub-provided action. Syft generates CycloneDX SBOM and uploads as OCI attestation. Kyverno verifyImages policy checks signature + provenance + SBOM presence. Attempt to deploy a hand-built image fails admission. Verifies the workflow that signed it.

A bank doing forensic SBOM lookups

When CVE drops, query cosign download attestation --predicate-type cyclonedx ... for every running image. JSON parse against the SBOM, find affected ones in seconds. Pre-SBOM equivalent: re-scan every image with Trivy, hours of work. Reduces incident MTTR for known CVEs from hours to minutes.

A startup using Chainguard Images

Migrated to cgr.dev/chainguard/... images for distroless minimal-CVE base layers. All Chainguard images come pre-signed + with SLSA L3 provenance + SBOMs. Their Kyverno policy reuses Chainguard's public key. Switching from a custom base image saved months of supply-chain plumbing.

A team that caught XZ-style supply chain

Routine SBOM diff on a base-image update flagged a new package not in the previous SBOM. Investigation revealed the package was an obfuscated payload from a transitively-included Go module. Caught at PR review, never deployed. The SBOM made the diff visible; without it, the new file would have been invisible.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

A few common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: Pinning images by tag is good enough.
Truth: Tags are mutable. nginx:1.21 can be republished with different content tomorrow. Pin by digest (sha256:abc...) for immutability. Or pin by tag + verify signature at admission so a stealth republish fails verification.
Myth: Sigstore keyless signing is less secure than key-based.
Truth: It's often more secure: short-lived certs (10 min TTL) tied to OIDC identities can't be exfiltrated and reused; the trust root is a federated identity (GitHub, corp SSO) that's typically better managed than a key file in CI. Long-lived signing keys are notoriously leaky.
Myth: An SBOM proves the image is safe.
Truth: An SBOM is an inventory, not a verdict. It tells you what's in the image. Combine with a vulnerability database to know which CVEs apply. SBOMs are evidence, not protection.

A vendor publishes a new image: vendor/app:2.5. Your cluster pulls it via Kyverno verifyImages. The vendor signed it with their key, which you've added to your policy. Should you trust the image now?

answerThe signature proves the image came from the vendor — but it doesn't tell you if the vendor is honest, or compromised, or shipping malicious code. Cosign verification = authenticity, not safety. Defense in depth: (1) verify the SBOM has expected components (no surprise binaries). (2) Verify SLSA provenance (was built by the vendor's standard CI, not a one-off run). (3) Run vulnerability scans on the SBOM. (4) Track Rekor entries for unusual patterns (signing surge after a quiet period?). Signing solves "is this from who they say it's from"; supply-chain assurance needs all four signals.

Your CI signs images with Cosign keyless. The Kyverno policy verifies issuer: token.actions.githubusercontent.com + subject: github.com/myorg/.*. An attacker steals a developer's GitHub credentials and pushes a malicious change. What still saves you?

answerThe malicious image gets built + signed by a legitimate workflow on the legitimate org. Verification passes. Cosign keyless doesn't protect against authenticated insider threats — it protects against unauthenticated attackers and registry compromise. What saves you: (1) PR review on the source code change. (2) Branch protection on main (no direct push). (3) Required code-owner approval. (4) PR-time SLSA provenance check (was the signing workflow run on main, or a fork?). (5) SBOM diff review — new dependencies flagged at PR. (6) Runtime detection (Falco rules, EDR). Supply-chain hardening is necessary; it's not sufficient against compromise of the source identity itself.
🎬 Choose Your Own Adventure

The CISO mandates: "every running image in production must have a signed SBOM available." Build the playbook. Click for the rollout. ▼

the rollout playbookPhase 1 — produce SBOMs in CI. Add Syft (or Trivy) to every build pipeline. Output CycloneDX. Push as OCI attestation alongside the image: cosign attest --predicate sbom.json --type cyclonedx .... Phase 2 — Kyverno policy in audit mode. Require SBOM attestation on all images in prod namespaces. Audit-mode = log violations, not block. Run for 2-4 weeks. Phase 3 — clean up. Audit reports list every image without an SBOM. Most are vendor / legacy images. For each: rebuild with SBOM, switch to a Chainguard-equivalent that ships SBOMs, or accept as a documented exception via PolicyException. Phase 4 — switch to Enforce mode. No more violations possible. Phase 5 — wire SBOMs into vulnerability response. Daily job pulls SBOMs from registry, runs against vulnerability DB, alerts on new CVEs affecting deployed images. Total time: ~6 weeks for a mature org with mature CI; longer if CI needs uplift first. Payoff: when the next log4j drops, you know which Pods are affected in minutes, not days.
