Module 4 · Lesson 16 · how Kubernetes runs your apps

Workload Controllers · Deployment, StatefulSet, DaemonSet, Job, CronJob

Pods come and go; workload controllers keep your apps running at the right shape. Five flavours, one pattern: declare what you want, the controller reconciles. Forever.

🚨 The 3 AM Nightmare

You manually create five Pods for your web app and walk away. By 3 AM, traffic doubles. The Pods can't keep up. One crashes from memory pressure. Now you have four. Nobody's awake to notice. By 8 AM, two more have crashed. You're at two Pods serving rush-hour traffic, customers are seeing timeouts, and the on-call engineer's first task of the day is figuring out what just happened. This lesson is about the abstraction that makes that whole class of pain go away — and the five flavours of it for the five shapes of work.

🎯 If you remember nothing else: Pods come and go. Workload controllers keep your apps running at the right shape — five flavours: Deployment for stateless apps, StatefulSet for apps with identity, DaemonSet for one-per-node, Job for run-once, CronJob for run-on-schedule.

Section 1 · Concept

What a workload controller is

A Pod is the unit of work — one or more containers scheduled together. But a Pod by itself is fragile: if it crashes, nothing brings it back. If you want three copies, you have to create three Pods. If demand doubles, you create more by hand. If a node dies, every Pod on that node disappears.

A workload controller is the loop that solves this. You declare what you want — "five copies of my-app:v2, healthy" — and the controller compares that against what's running, and creates or deletes Pods to close the gap. Forever. If a Pod crashes, the controller notices and creates a replacement. If a node dies, the controller reschedules the missing Pods elsewhere. If you change the desired count, the controller adjusts.

Kubernetes ships with five built-in workload controllers, one for each common shape of work:

Section 1.5 · The five flavours

One controller per shape of work

Each controller has the same loop pattern (declare, observe, reconcile) but a different idea of what the right Pod shape looks like. Pick the one that matches the work.

🔁

Deployment

rotating shifts

Manages a set of identical, interchangeable Pods. Rolling updates create new Pods one batch at a time and retire old ones once the new are ready. The default for stateless web apps, APIs, workers.

When: any app where Pods don't care about their own identity.

🪑

StatefulSet

assigned seats

Each Pod has a stable name (db-0, db-1, db-2) that survives restarts, plus a stable persistent volume per replica. Startup and shutdown are ordered. The right choice for databases, queues, anything that needs identity.

When: Postgres, Cassandra, MongoDB, Kafka, Elasticsearch, ZooKeeper.

👀

DaemonSet

one-per-node watchman

Runs exactly one Pod on every node (or every node matching a selector). Add a node, the Pod auto-deploys; remove a node, the Pod disappears. The pattern for log collectors, monitoring agents, network plugins.

When: Fluent Bit, Prometheus node-exporter, Calico CNI, kube-proxy.

📋

Job

one-time work order

Runs one or more Pods to completion. Once a Pod exits cleanly, the Job is done. If a Pod fails, the Job restarts it (up to a backoff limit). For batch processing, migrations, image renders, reports.

When: database migrations, video transcodes, ML training runs.

⏰

CronJob

scheduled round

A Job on a cron schedule. 0 2 * * * runs the Job at 2 AM every day. The CronJob controller creates a fresh Job at each tick; the Job creates the Pod; the Pod runs to completion.

When: nightly backups, hourly cache warms, monthly reports.

Section 1.7 · Deployment in detail

The most common controller — Deployment + ReplicaSet

A Deployment is a layer above the simpler ReplicaSet. A ReplicaSet keeps N identical Pods alive — that's the simplest reconciliation loop. A Deployment manages ReplicaSets so that you can update the Pod template (e.g. change the image tag) and the cluster transitions smoothly from the old version to the new one.

The rolling update is the key trick. When you update a Deployment's image:

  1. The Deployment creates a new ReplicaSet (with the new image) at zero replicas.
  2. It scales the new ReplicaSet up by maxSurge (default 25%) and waits for the new Pods to report ready.
  3. Once the new Pods are ready, it scales the old ReplicaSet down by maxUnavailable (default 25%).
  4. Repeat until the new ReplicaSet has all replicas and the old one has zero.

If a new Pod fails its readiness probe, the rollout pauses — old Pods stay up. kubectl rollout undo deployment/my-app reverts to the previous ReplicaSet (which still exists, just at zero replicas). Rollback is one command.

[ deep dive — skip if new ]You'll rarely create a ReplicaSet directly — Deployments hide them. The ReplicaSet hash appears in Pod names (my-app-7d4b9c-xkz2p) so you can identify which version a Pod belongs to. kubectl get rs shows the active and historical ReplicaSets the Deployment has produced.

Section 1.9 · StatefulSet in detail

When Pods need a name

A StatefulSet is a workload controller that gives each Pod a stable identity that survives restarts. It's the right choice for any workload where Pods know about each other by name — replicated databases, message queues, distributed file systems.

Three guarantees a Deployment doesn't make:

Deployment (stateless)

  • Pods get random names (app-7d4b9c-xkz2p)
  • Restart → fresh name, fresh IP, fresh PVC
  • Startup order: any order, in parallel
  • Rolling update: replace any Pod at any time
  • Storage: ephemeral or shared

StatefulSet (stateful)

  • Pods get ordinal names (db-0, db-1, db-2)
  • Restart → same name, same PVC re-attached
  • Startup: ordered (db-0 before db-1 before db-2)
  • Shutdown: reverse-ordered
  • Storage: volumeClaimTemplates creates a PVC per Pod

The trade-off is operational rigidity. StatefulSets are slower to scale (one Pod at a time, sequentially), slower to roll out updates, and more complex to debug. Use them when identity matters — and stick with Deployments otherwise.

⏸ Pause and check

What's the difference between a Pod and a Deployment?

Answer: b. The Pod is what runs your container. The Deployment is the loop that creates Pods, watches them, and replaces failed ones — automatically and forever.

Section 2 · Before & After

Manual Pod management vs declared workloads

Before · raw Pods

You write five YAML files for five Pods. You apply them. One crashes — you re-apply. Demand doubles — you write five more YAML files. A node fails — you SSH in, find which Pods were on that node, recreate them on healthy nodes by hand.

Updates are surgery. To roll out a new version: stop one Pod, apply the new spec, wait, stop the next one, apply, wait. If something breaks halfway, you have a mix of old and new Pods and no clean rollback.

After · workload controllers

You write one Deployment YAML with replicas: 5. The controller creates the five Pods. A Pod crashes — replaced in seconds. A node fails — the missing Pods are rescheduled. Demand doubles — change one number, the controller scales.

Updates are kubectl set image deployment/my-app web=my-app:v2. Rolling update happens automatically — new Pods come up healthy before old ones go down. Rollback is kubectl rollout undo. The cluster never has more than 25% unavailable during the transition.

What didn't change: the underlying desire. You still want N copies of your app running. The controller turns that desire into a never-ending loop instead of a one-time imperative command.

Section 3 · Analogy

The K-Town Dispatch Office

Imagine the city's Dispatch Office — a corner building near city hall, full of clipboards, ticket-takers, and a wall of work orders. Businesses send standing orders to the dispatcher: "I need five baristas at all times in the morning shift," "I need a permanent night-watchman at the Bank Vault Quarter," "We need a one-time crew to repaint the harbour fence next Tuesday."

The dispatcher reads each order, sends workers to fill the slots, and watches. If a barista calls in sick, the dispatcher sends a replacement — same role, different person. If a watchman quits, the dispatcher posts the position again. If a one-time crew finishes the harbour fence, that work order closes. New shift requirements arrive, the dispatcher adjusts headcounts.

Different businesses have different needs, so the Dispatch Office issues five kinds of standing work order. Some bring rotating shift workers (any qualified person fills the slot — like baristas); others bring assigned-seat employees (Anna at desk 0, Brian at desk 1 — like the bank's tellers); others bring the one-per-building watchman; others bring one-time crews; others bring scheduled maintenance teams. Same dispatcher, same loop, five shapes of work.

⚠️ The analogy stops here: real workforce dispatch keeps the same people across shifts. Kubernetes Deployments routinely terminate and replace Pods — same role, different instances. The "shift" is permanent; the workers are not.

Section 4 · Two-level explanation

Two ways to hear it

ELI5 · explain like I'm 5

Imagine you have toy soldiers and a friend who watches them for you. You tell your friend, "I want five soldiers standing on the rug at all times." If a soldier falls over, your friend stands a new one up. If you say, "I want eight now," your friend gets three more. If you say, "Switch them all to blue uniforms," your friend swaps them in one at a time. Your friend never stops watching. That's a workload controller.

ELI10 · explain like I'm 10

A Pod runs your app, but a single Pod is fragile — if it crashes or its node dies, nothing brings it back. A workload controller is a small program (one of five flavours) that keeps a declared number of Pods running, reconciling reality against your spec forever. Most apps use a Deployment: declare replicas: 5 and the controller maintains exactly five healthy copies. Apps that need stable identity (databases, queues) use a StatefulSet. Workloads that should run on every node (logging, monitoring) use a DaemonSet. One-time tasks use Jobs; scheduled tasks use CronJobs. Same loop, five shapes.

⏸ Pause and check

Your team is running PostgreSQL with three replicas. The replicas need to find each other by stable hostname for replication. Should they use a Deployment or a StatefulSet?

Answer: b. StatefulSets give each Pod a stable name and a per-Pod persistent volume that survives restarts. Deployments treat Pods as interchangeable, which breaks replication.

Section 5 · Real-world scenarios

Where each controller shows up

A SaaS startup running 30 stateless services

Three engineers, weekly releases. Every service is a Deployment with 3-10 replicas. Releases are kubectl set image; the cluster rolls 25% at a time, never serving more than 25% old. Rollbacks happen via kubectl rollout undo; they happen monthly and are non-events. "We never write kubectl scale by hand — we change the YAML and let it reconcile."

A regional bank running Cassandra across three AZs

Cassandra needs each node to know its peers by stable hostname. The team uses a StatefulSet with 6 replicas spread across three availability zones (cassandra-0 through cassandra-5). Each Pod's PVC is recreated on the same persistent volume during restarts, so the Cassandra cluster's state survives Pod churn. Bootstrap took planning; daily ops are quiet.

A telecom running a metrics agent on every node

A Prometheus node-exporter scrapes per-node metrics. The team uses a DaemonSet — one agent per node, scheduled automatically when new nodes come up. Decommissioning a node automatically removes the agent. Adding a 50th node took zero ops effort; the DaemonSet noticed and deployed.

A media company encoding videos overnight

Every night at 1 AM, a CronJob fires that creates a Job for each video uploaded the previous day. Each Job runs a Pod with FFmpeg, transcodes the video into multiple bitrates, uploads to S3, and exits. Failed transcodes retry up to three times (the Job's backoffLimit). The team wakes up to a queue of completed transcodes, with failures already escalated.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: Pods and Deployments are the same thing.
Truth: A Pod is a single running instance. A Deployment is a controller that maintains a declared number of identical Pods, replacing failed ones automatically and rolling out updates safely.
Myth: I can use a Deployment for my database.
Truth: Use a StatefulSet for stateful workloads (databases, queues, file servers). Deployments treat Pods as interchangeable; databases need stable identity, predictable startup order, and per-Pod persistent storage.
Myth: Rolling updates restart all Pods at once.
Truth: Rolling updates create new Pods one batch at a time and only retire old Pods after the new ones report ready. The cluster stays available throughout, with at most maxUnavailable (default 25%) gone at any moment.

Deployment vs ReplicaSet?
tap to flip

When use StatefulSet?
tap to flip

What is a DaemonSet?
tap to flip

Job vs CronJob?
tap to flip
Job = run a Pod to completion once (with retries on failure). CronJob = create a fresh Job on a cron schedule (0 2 * * * = daily at 2 AM).
Rolling-update knobs?
tap to flip

How do I roll back?
tap to flip

A team wants to roll out a new image version with zero downtime. They edit the Deployment's image tag and apply. What does Kubernetes do?

answerThe Deployment creates a new ReplicaSet at zero replicas, scales it up by maxSurge (default 25%) — those new Pods pull v2 and report ready. Once ready, the Deployment scales the old ReplicaSet down by maxUnavailable (default 25%). It alternates until the new ReplicaSet has all replicas and the old has zero. If a new Pod fails its readiness probe, the rollout pauses; the old Pods keep serving. kubectl rollout status shows progress; kubectl rollout undo reverts.

You need to run a Postgres backup every night at 1 AM. Which workload controller, and what does the spec look like at a high level?

answerA CronJob. Spec carries the cron schedule (0 1 * * *) and a Pod template that runs pg_dump, uploads to S3, and exits. The CronJob controller fires at each scheduled tick by creating a Job, which creates a Pod, which runs the backup script and terminates. Set backoffLimit for retries on failure, concurrencyPolicy: Forbid if a previous run is still going, and successfulJobsHistoryLimit to keep recent runs visible.

Your team uses a Deployment for their three Postgres replicas. During a rolling update at 2 PM, all three Pods get terminated and replaced with fresh ones — in random order, with fresh IPs. Click to see what happens. ▼

2 PM and 30 seconds laterReplication breaks. The "new primary" doesn't know who its secondaries are; the "new secondaries" can't find the old primary's IP. Reads and writes are inconsistent. Half the application dashboard shows yesterday's data. Fix: switch to a StatefulSet. Words from this lesson · open if you want a quick reference
Workload controller
A controller that maintains a declared number of Pods of a particular shape. Five built-in flavours: Deployment, StatefulSet, DaemonSet, Job, CronJob.
Deployment
Manages a set of identical, interchangeable Pods. Default for stateless apps. Supports rolling updates and rollback.
ReplicaSet
Lower-level controller that keeps N Pods of a single template alive. Deployments manage ReplicaSets; you rarely create one directly.
StatefulSet
Workload controller giving each Pod a stable identity (name-N), per-Pod PVC, and ordered startup / shutdown.
DaemonSet
Workload controller that runs one Pod on every node (or every matching node). Scales automatically as nodes come and go.
Job
Workload controller that runs Pods to completion. Used for batch tasks. Retries failed Pods up to backoffLimit.
CronJob
Creates a fresh Job on a cron schedule. Used for scheduled batch tasks (backups, reports, cache warms).
Rolling update
Deployment's default update strategy: bring up new Pods in batches, retire old Pods only after new ones are ready.
maxSurge / maxUnavailable
Knobs on the rolling-update strategy. Surge = max extra Pods above desired during the update. Unavailable = max Pods that can be down. Both default to 25%.
HorizontalPodAutoscaler
A separate controller that auto-scales a Deployment's replicas based on CPU, memory, or custom metrics. Scales up under load, down at idle.
