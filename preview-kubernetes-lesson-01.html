Cloud-Native and Kubernetes Foundations

Kubernetes' single biggest idea is the reconciliation loop: you describe what you want, and a controller continuously moves the world toward that description. Once that lands, every other Kubernetes concept — pods, deployments, services, GitOps, operators — is just a special case of the same loop. This lesson is the spine of the K-COM track. By the end you'll know what cloud-native really means, when Kubernetes fits and when it doesn't, how mature teams operate it, and where the platform came from.

Subtopic 1 of 5

Cloud-native principles — reconciliation, desired state, pets vs cattle

Five ideas that everything else in the lesson stacks on. Get this and the rest is layering vocabulary on the loop.

📖

Words you'll meet in this section

First-time learner? Read these once. They'll be on every Kubernetes page you ever look at.

Cloud-native concept
What it is: A way of building software that assumes the infrastructure is dynamic (machines come and go), failure is routine, and operators describe outcomes instead of steps.
Cloud-native ≠ "running in the cloud." Cloud-native is a design style. A monolith on a hand-managed VM in AWS is cloud-hosted, not cloud-native.
Declarative concept
What it is: You describe what you want — "3 replicas of nginx on port 80" — and let the system figure out how. The opposite is imperative: "install nginx, then start it, then do …"
Kubernetes is declarative end-to-end. You write YAML; controllers turn it into reality. The YAML is the plan and the audit trail at the same time.
Reconciliation loop concept
What it is: A small program — a controller — that runs forever: read what you want, look at what exists, take the smallest action to close the gap, repeat.
This is the heart of Kubernetes. Every other concept (Deployments, ReplicaSets, Jobs, custom operators) is a specialised reconciliation loop.
Desired state concept
What it is: The picture of the world the operator declared. Stored in the Kubernetes API (etcd, behind kube-apiserver).
Whatever you can write down in a manifest is the truth, as far as the controllers are concerned. Drift gets corrected, not promoted.
Actual state concept
What it is: What's actually running, right now, on the machines. Reported back to the API by node-side agents (the kubelet).
Desired and actual are constantly compared. Their difference is what triggers controller action.
Controller tool
What it is: A program that watches a kind of object (Deployments, Nodes, Jobs, …) and runs the loop for it. Lives in kube-controller-manager for built-ins, or as a separate operator for custom kinds.
A typical cluster runs dozens of controllers in parallel. Each one cares about its own resource type and ignores the rest.
Pets vs cattle concept
What it is: A pet is a hand-tuned named server you'd grieve. Cattle are identical, replaceable, numbered. Cloud-native treats servers and pods as cattle.
Reliability comes from "any machine can do this job and the platform replaces any that dies," not from "this specific machine never fails."
Horizontal scaling concept
What it is: Adding more identical copies (scale out) instead of making one machine bigger (scale up). The replica count is the dial.
Bigger machines have ceilings; replica counts don't. And horizontal scale survives the loss of any one replica gracefully.

The reconciliation loop, drawn

Controllers compare desired and actual on every tick. The minimum-diff principle is what makes the loop converge cleanly without thrashing.

Pets vs cattle

A pet has unique state and unique knowledge that lives only on that machine. Cattle are described declaratively — losing one is a routine event the platform handles.
★
The single most important sentence in this lessonOnce you internalise that everything in Kubernetes is a reconciliation loop watching some kind of resource, the rest of the curriculum stops feeling like memorisation and starts feeling like patterns. Pods are loops on Pod objects. Deployments are loops on Deployment objects (which create ReplicaSet loops, which create Pod loops). Operators are loops on your own custom kinds.

Concept

"Cloud-native" is a way of building software that assumes the underlying infrastructure is dynamic, that machines fail routinely, and that operators describe outcomes rather than steps. It rests on a small, repeating set of ideas: immutable infrastructure (rebuild, don't patch in place), declarative configuration, reconciliation loops, the constant comparison of desired vs actual state, treating servers as cattle, not pets, scaling horizontally rather than making single machines bigger, and treating failure as normal.

The reconciliation loop is the spine of all of it. A controller reads what you said you wanted, looks at what actually exists, takes the smallest action to close the gap, and then does it again. The Deployment controller does this for "how many pod replicas should exist." The Node controller does this for "is this machine still healthy." The Job controller does this for "did this batch task finish." Each runs its own loop, on its own resource, forever.

Before / After

Before · the imperative world

Ops engineer SSHes into web-01, runs apt install, edits a config file, restarts the service. Next quarter they SSH back in, can't remember what they changed, and production has slowly drifted from documentation into a place no one can reproduce.

→
After · the declarative world

Engineer commits a YAML manifest. A controller reads it, sees zero replicas exist, creates three. A node fails — actual drops to two; the controller creates a third on a healthy node. Nobody SSHes anywhere. The YAML is the source of truth.

What did not get easier: the application itself. The reconciliation loop won't save you from a buggy image, an undersized resource request, or a missing health check. The platform automates operations around the application, not the application's own logic.

AnalogyThe thermostat

A reconciliation loop is a thermostat. You set the dial to 21°C — that's the desired state. The thermostat reads the room (say it's 18°C) — that's the actual state. It computes the difference (a 3°C gap) and runs the heater (the controller action). It keeps reading the room. When the room hits 21°C, the heater stops. When the room cools again, the heater fires again. The thermostat never declares victory and walks away.

Kubernetes is a building full of thermostats, one for every kind of resource. The Deployment thermostat watches replica count. The Node thermostat watches machine health. The Job thermostat watches task completion. Each runs its own loop on its own resource, forever.

Two ways to think about it

ELI5

A thermostat watches the room and turns the heater on when it's too cold and off when it's warm enough. It never stops watching. Kubernetes is a building full of tiny thermostats — one watches "how many copies of this app are running," another watches "is this computer still alive." When something doesn't match what you asked for, the thermostat fixes it.

ELI10

A reconciliation loop is a small program that does one thing forever: it reads what you said you wanted, looks at what actually exists, and makes a small change to close the gap. The Deployment controller does this for replica count: if you said three replicas and one died, it creates a fourth pod to bring the count back up. The Node controller does this for node health: if a node hasn't checked in for a while, it marks it unschedulable. This is why Kubernetes survives node failures, network blips, and pods crashing — nothing is "set up once and forgotten." Treating servers as cattle (identical, replaceable) instead of pets (named, hand-tuned) is what makes it survivable.

A SaaS startup running on three hand-built EC2 instances kept losing a weekend every quarter to "the database VM rebooted and the app didn't come back." Migrating to Kubernetes meant they wrote a Deployment for the app and a StatefulSet for the database. After the move, when an EC2 host was rotated by AWS, the controllers rescheduled the pods onto a healthy host and the on-call engineer woke up to a Slack message saying "self-healed at 3:14am — no action needed." The reconciliation loop ate the page.

Flashcards · cloud-native principles (3)

What is a reconciliation loop?
tap to flip
A controller program that reads desired state, observes actual state, and takes the minimum action to close the gap — repeatedly, forever. The spine of Kubernetes operation.
Declarative vs imperative configuration?
tap to flip
Declarative says what the world should look like (3 replicas of nginx). Imperative says how to get there (install nginx, then start 3 copies). Kubernetes is declarative.
What does pets vs cattle mean?
tap to flip
A pet is a hand-tuned named server you'd mourn. Cattle are identical replaceable units. Cloud-native systems treat servers and pods as cattle so failure is routine and recoverable.

Quiz · cloud-native principles (2)

⏸ pause animation in 'scale-up' mode
The animation shows desired = 3 replicas. Actual = 1. The controller is about to fire. What action will it take, and why?
Answer

The controller will create two more pods of the same template, bringing actual to 3. The action is the minimum needed to close the gap — it doesn't recreate the existing pod, doesn't delete anything, doesn't restart. It just creates what's missing. That minimum-diff principle is what makes the loop converge cleanly without thrashing.

Why does treating servers as cattle make a system more reliable than treating them as pets?
Answer

Pets have unique state and unique knowledge that lives only on that machine. When a pet dies, you have to recreate it from incomplete documentation. Cattle are identical and described declaratively — when one dies, the platform replaces it from the same description. Reliability comes from "any machine can do this job and the platform will replace any that dies," not from "this specific machine never fails."

Subtopic 2 of 5

Twelve-factor apps; microservices vs modular monoliths

The shape of an app that drops cleanly into a pod, and the question of how big each app should be.

📖

Words you'll meet in this section

Two ideas: a checklist for app shape, and a debate about app size.

Twelve-factor app concept
What it is: A 12-rule checklist (Heroku co-founder Adam Wiggins, 2011, 12factor.net) for writing apps that move cleanly between environments.
An app that follows it drops into a pod with almost no work. An app that violates it fights the platform forever.
Stateless process concept
What it is: An app that holds no important state in memory or on local disk. State lives in attached backing services (DB, cache, object store).
The platform can stop, move, restart, or replace a stateless process at any time. Anything that lives only inside the process disappears with it.
Backing service concept
What it is: Any external resource the app uses over a network — DB, cache, queue, search index, object store, third-party API.
Treated as a swappable URL + credentials. You can change "Postgres on RDS" to "Postgres on a self-run cluster" by changing one env var.
Disposability concept
What it is: Fast startup (seconds, not minutes) and graceful shutdown (catch SIGTERM, finish in-flight work, exit).
Pods get restarted constantly — node drains, rolling updates, autoscaling. Slow-starting apps create rolling outages.
Microservice concept
What it is: A small service responsible for one bounded capability, deployed and scaled independently. Communicates over the network.
Independence has a cost — each call is now a network call with its own retries, timeouts, and failure modes.
Monolith concept
What it is: One deployable unit that contains all (or most) of an app's logic. Everything lives in one process.
Operationally simple — one image, one deploy, one set of dashboards. Can grow tangled if module boundaries aren't enforced.
Modular monolith concept
What it is: One deployable, but the codebase is split into modules with strict, code-enforced boundaries. Modules call each other in-process, not over the network.
Often the best starting architecture: monolith's operational simplicity + microservices' boundary discipline.
Sidecar pattern concept
What it is: A second container in the same pod that augments the main app — for logging, proxying, secrets injection, etc.
Lets you bolt platform features onto an app without modifying the app's code. Common with service meshes.

The 12 factors, at a glance

Source: 12factor.net by Adam Wiggins (2011). Most violations of these rules turn into Kubernetes pain in predictable ways.

Monolith · microservices · modular monolith

Most teams should reach for "modular monolith first, microservices when org or scale demands it." Kubernetes runs all three; the choice is about your org, not the platform.

Concept

The twelve-factor methodology is twelve rules for writing applications that run well on dynamic platforms — Heroku originally, but the rules generalise. The factors cover: codebase, dependencies, config, backing services, build/release/run, processes, port binding, concurrency, disposability, dev/prod parity, logs, and admin processes. A twelve-factor app drops into a Kubernetes pod almost without modification. A non-twelve-factor app — one that writes config to disk, stores session in local memory, expects a long restart, or reads logs from a fixed file — fights the platform.

Microservices split an application into many small services that communicate over the network. Modular monoliths keep one deployable but enforce strict module boundaries internally. Microservices pay a coordination cost (network calls, deployment fan-out, distributed tracing); monoliths pay a discipline cost (boundaries that must be preserved by code review, not by network). For most teams, a well-structured modular monolith is the right starting point; microservices are an answer to organisational scale, not a starting architecture.

Before / After

Before · the bespoke crate

App reads its DB password from /etc/myapp/db.conf, stores user sessions in session/, writes logs to /var/log/myapp.log rotated by cron. Moving servers means recreating that exact filesystem layout. Running on a platform that gives a fresh container every deploy means losing sessions every deploy.

→
After · the standard container

App reads DB password from $DB_PASSWORD, stores sessions in Redis, writes logs to stdout. Moving to Kubernetes is a Dockerfile, a Deployment, an injected Secret. Sessions survive deploys because they're not on the pod's filesystem. Logs are a stream the platform collects.

What did not get easier: twelve-factor doesn't make the app correct or fast. A buggy stateless app is still a buggy app. The methodology removes friction with the platform; the application logic is still your problem.

AnalogyShipping containers

A twelve-factor app is a standard shipping container. Fixed dimensions, standard lift points, manifest on the side, plug points for refrigeration. It rides on a truck, then a ship, then a train, then another truck — and at each transfer the equipment knows what to do because the container is standard.

A non-twelve-factor app is a one-off bespoke crate. Custom rigging, a specific kind of forklift, careful loading instructions written by the engineer who built it. Every transfer is a project. The platform can't move it without help.

Microservices are many small containers. Modular monoliths are one big container with internal compartments — the compartments are still real, but the container moves as one piece.

Two ways to think about it

ELI5

Imagine all the boxes that ship around the world look exactly the same — same size, same lift points, same labels. Now any truck or ship or warehouse can handle any box without instructions. A twelve-factor app is one of those boxes. A normal app is a one-off package the post office has to figure out every time.

ELI10

Twelve-factor is a checklist for writing apps that move easily between environments. The rules — read config from environment variables, write logs to stdout, don't store state on local disk, start fast, shut down gracefully — together remove almost every reason an app would refuse to run on a dynamic platform. Microservices and modular monoliths are two answers to "how do I split a big app." Microservices split across the network — independent deploy, independent scale, independent failure, with the cost of coordination. A modular monolith splits by code module inside one deployable — easier to deploy and reason about, with the cost of needing real discipline. Most teams should reach for "modular monolith first, microservices when org or scale demands it." Kubernetes runs either, and isn't a reason to choose microservices.

A 30-engineer SaaS team kept hitting deployment slowdowns on a Rails monolith — a single bundle install + rails s took 4 minutes, and cross-team merges blocked each other. They considered breaking into microservices but instead split the codebase into well-defined modules with explicit interfaces, kept the monolith, and added a CI gate that prevented module-boundary violations. Deploys stayed fast (one image), team independence improved (modules owned by teams), and they avoided the operational tax of running 12 services with 12 deployments and 12 dashboards. They eventually extracted one high-traffic module into its own service when traffic justified it.

Flashcards · 12-factor & services (3)

What is a twelve-factor app?
tap to flip
An app written to twelve specific guidelines (codebase, dependencies, config, backing services, build/release/run, processes, port binding, concurrency, disposability, dev/prod parity, logs, admin processes) that make it run cleanly on dynamic platforms.
Microservices vs modular monolith — what's the trade?
tap to flip
Microservices split across the network for independent deploy/scale/fail at the cost of coordination overhead. A modular monolith splits inside one deployable for simpler operations at the cost of needing strict discipline to keep boundaries clean.
Why are stateless processes a twelve-factor rule?
tap to flip
Because the platform can stop, move, restart, or replace any process at any time. Anything that lives only inside the process disappears with it. State must live in attached backing services — DB, cache, object store.

Quiz · 12-factor & services (2)

A team is moving a Python app to Kubernetes. The app reads database credentials from /etc/app/db.conf, written by a Chef recipe at provisioning time. Which twelve-factor rule does this violate, and what's the cleanest fix?
Answer

Violates "Config — stored in the environment, never in code or in baked-in files." Cleanest fix: store credentials as a Kubernetes Secret, mount or inject as environment variables. The app reads os.environ['DB_PASSWORD']. The Chef recipe goes away. The platform supplies config; the image stays generic.

Why is a modular monolith often a better starting architecture than microservices for a young team?
Answer

Microservices push complexity from inside the codebase to the network — distributed tracing, retries, idempotency, service discovery, separate dashboards, separate on-call. A young team rarely has the operational maturity to absorb that. A modular monolith keeps clear internal boundaries with a fraction of the operational tax. You can split out a service later when one module's scale or team boundary actually justifies it.

Subtopic 3 of 5

When Kubernetes is useful — and when it's overkill

A power tool answers some questions and creates others. The honest answer to "should we use Kubernetes?" is "depends on your workload's shape and your team's bandwidth."

📖

Words you'll meet in this section

Sizing terms first.

Operational scale concept
What it is: The count of services to run, teams to support, and environments to maintain. Distinct from traffic scale.
Kubernetes pays back in proportion to operational scale. Lots of services + lots of teams = high payback. Two services + one team = low payback.
Platform tax concept
What it is: The ongoing cost of running the platform itself — cluster upgrades, observability stack, security patches, on-call for the platform.
A real number. For tiny workloads it can exceed the value the platform delivers. For big ones it's amortised across many services.
Paved road concept
What it is: The standard, supported way the platform team has decided things should be done. Easier than the alternative on purpose.
A small team can't afford 10 ways to do everything. The paved road exists so most teams pick it without thinking.
Latency-sensitive workload concept
What it is: A workload where sub-millisecond per-hop latency matters — high-frequency trading, real-time bidding, certain telecom paths.
Kubernetes' network and scheduling layers add microseconds. Negligible at 50ms, expensive at 50µs.
Managed service concept
What it is: Heroku, Render, Fly, Railway — platforms where you push code and get an HTTPS URL. The platform owns the cluster.
Often the right choice for small teams. You give up some control; you save weeks of operational work.
Serverless / FaaS concept
What it is: Functions that run only when invoked. AWS Lambda, Cloud Functions, Azure Functions. No servers to manage; pay per invocation.
Wins for spiky / event-driven workloads. Loses for steady-state long-running services where idle time is rare.

The fit decision · a flowchart you can run yourself

Three "yes" answers and Kubernetes is almost certainly the right call. Three "no" answers and you should look hard at simpler options first.
!
"Modern" is not a reason"We should use Kubernetes because it's the modern way" is the most common bad reason teams adopt it. Modern doesn't mean appropriate. Adopt because the workload demands it, not because the technology is in vogue.

Concept

Kubernetes pays back when you run many services that each need lifecycle automation; when workloads are dynamic (scale up and down, get rescheduled, get replaced); when multiple teams share infrastructure and want a common platform with isolation; when you need portable workloads (same manifests on local kind, EKS, AKS, GKE, OpenShift); and when you've already accepted the cost of distributed systems and want a standard control plane.

Kubernetes is overkill when a single application runs fine on a single VM and rarely changes; when the team has no platform engineering capacity; when workloads are batch only; when latency is microsecond-sensitive; or when the org's pain is application architecture, not orchestration. Moving a bad app to Kubernetes makes it a bad app on Kubernetes.

The honest summary: Kubernetes solves operational scale and heterogeneity. If you don't have operational scale or heterogeneity, you don't need it.

Before / After

Before · small startup

Two services on two VMs, one Postgres on RDS, deployments via a shell script. Total operational load: one engineer, two hours per week. Adding Kubernetes here adds ten hours per week of cluster maintenance, training, debugging, on-call. The startup loses time without gaining anything.

→
After · series-B startup

Eight services, four teams, each wanting to deploy independently, share secrets, run staging environments, add new services without filing a ticket. Now the platform is the bottleneck. Kubernetes (with a small platform team) replaces 8 deploy scripts and 4 staging tents with one common system. The ten hours of cluster work pays for itself.

What did not change: Kubernetes never makes a single service simpler. It pays off only when the count of things to operate is the bottleneck.

AnalogyIndustrial kitchen vs home toaster

Kubernetes is an industrial kitchen. Many cooks, multiple ovens, walk-in fridges, line orders, prep stations, plating stations, expediter, head chef coordinating, cleaning crew, written service standards. When you're feeding 800 people for a wedding, you need this.

When you're making yourself toast in the morning, you need a toaster and a counter. If someone built you an industrial kitchen for your toast, the kitchen would consume more attention than the toast. The kitchen isn't bad — it's wrong for the use case.

The question isn't "is Kubernetes good?" It's "are you in the wedding-catering situation, or the morning-toast situation?"

Quick check

A two-person startup ships a Rails app, one PostgreSQL database, and a simple background worker. They get a few hundred users a day. Should they adopt Kubernetes?

Two ways to think about it

ELI5

A big restaurant kitchen with many ovens and many cooks is amazing if you're feeding hundreds of people. It's silly if you're making one piece of toast. Kubernetes is the big kitchen.

ELI10

Kubernetes is a platform — it solves the problem of running lots of services across many machines, with deployments, scaling, restarts, secrets, networking, and observability all standardised. When you have lots of services and multiple teams, that standardisation is gold; you don't want every team inventing their own deploy script. When you have one service, the platform itself becomes the work — you spend time keeping the cluster healthy instead of building product. A useful test: count the services you'll deploy and the teams that own them. Below ~5 services and ~2 teams, simpler tools (managed platforms, single-VM systemd, serverless) usually win. Above that, the per-service cost of Kubernetes drops sharply. Latency-sensitive workloads (sub-millisecond) are a separate category — Kubernetes' network and scheduling layers add per-hop cost that's irrelevant for HTTP APIs but expensive for sub-ms paths.

A media company with 40 microservices, 6 product teams, and a custom Capistrano deploy from 2014 was spending most of its platform engineering time on "make a deploy work." Migrating to Kubernetes (managed EKS) over six months collapsed 40 deploy scripts into one Helm pattern, replaced six different staging setups with namespaced previews, and freed two engineers to work on developer experience instead of plumbing. Conversely: a four-engineer fintech tried Kubernetes for a single core service plus a job worker, spent a year on cluster operations, and later moved back to ECS Fargate — the platform tax wasn't earning anything for one service.

Flashcards · when K8s fits (3)

Three signals that Kubernetes is the right fit?
tap to flip
Many services to operate; multiple teams sharing infrastructure; dynamic workloads that scale and reschedule; need for workload portability across environments; existing operational maturity to run a cluster.
Three signals that Kubernetes is overkill?
tap to flip
Single application that rarely changes; no platform engineering capacity; latency-sensitive sub-millisecond workloads; the team's current pain is application architecture rather than orchestration.
Why doesn't moving a small app to K8s make it easier to operate?
tap to flip
Kubernetes solves the count problem — running many things — not the per-app complexity problem. A single app on Kubernetes pays the cluster operational cost without the benefit; on a single VM, you'd skip the cost entirely.

Quiz · when K8s fits (2)

A 6-engineer team has one Django app, one Postgres, two background workers, and a Redis cache. They're considering Kubernetes. What questions should drive the decision?
Answer

How many services do they expect to add in 12 months? How many independent teams will own deploys? Do they need workload portability (cloud A → cloud B → on-prem)? Do they have time for cluster operations (upgrades, observability, cost)? If the answers are mostly "stays small," "one team," "no portability need," "no platform time" — they should not adopt yet. If at least three answers point toward growth, adopt.

Why is "we want to use Kubernetes because it's the modern way" a poor reason to adopt it?
Answer

Because adoption isn't free — the cluster itself becomes work. Kubernetes pays back in proportion to operational scale (many services, many teams). For workloads that don't have that scale, the operational cost of the platform exceeds the value it delivers. Adopt because the workload shape demands it, not because the technology is in vogue.

Subtopic 4 of 5

GitOps · platform engineering · SRE · service ownership · multi-tenancy

Five ideas mature Kubernetes orgs converge on. None is a Kubernetes feature; all are practices Kubernetes enables.

📖

Words you'll meet in this section

These five terms describe how mature teams actually operate Kubernetes once they have it.

GitOps tool
What it is: An operating model where the desired state of the system lives in a Git repository, and an automated agent in the cluster continuously pulls from Git and reconciles. Coined by Weaveworks (Alexis Richardson, 2017).
Tools: Argo CD, Flux CD. Formal four principles at opengitops.dev.
Platform engineering concept
What it is: The discipline of building an internal platform — a "paved road" — that product teams consume. The platform team owns Kubernetes, observability, secrets, CI/CD, ingress.
Codified in Team Topologies and the platform-engineering community of the late 2020s. Replaces "ops as a ticket queue."
SRE · Site Reliability Engineering concept
What it is: Google's operating model from the 2016 SRE Book. Defines reliability with SLIs (indicators), SLOs (objectives), SLAs (agreements). Spends an error budget on velocity until reliability drops.
Treats infrastructure as a software problem. Reliability becomes a feature you trade against velocity.
Service ownership concept
What it is: "You build it, you run it." The team that wrote the service is on its on-call rotation. They feel both the deploy pleasure and the 3am page.
Aligns incentives toward reliability without bureaucratic enforcement. Teams that get paged invest in better tests, observability, and deploy patterns.
Multi-tenancy concept
What it is: Sharing one cluster (or one platform) across multiple teams or customers. Soft = trust between tenants, namespace isolation. Hard = no trust, separate clusters or vClusters.
The right point on the spectrum depends on your threat model. "Just use namespaces" is fine for trusted teams; insufficient for hostile tenants.
SLI / SLO / error budget concept
What it is: SLI = a measured indicator (request success rate). SLO = the target (99.9%). Error budget = how much unreliability you can spend (0.1% = 43 min/month).
Stops the eternal "more reliability is always better" debate. If you're hitting your SLO, ship faster; if you're missing it, slow down and fix.
Drift concept
What it is: When the live cluster state diverges from the source of truth (Git). Caused by hand edits, expired certs, manual hot-fixes.
GitOps controllers detect drift and either alert or auto-correct. The argument for GitOps is that drift is visible, not silent.

The GitOps loop

Same loop as before — desired vs actual, controller in between — but the desired state is in Git, not in etcd. The platform team's job becomes "make the agent's job easy."

Multi-tenancy spectrum

Most teams sit in "namespaces" or "namespaces + vCluster for risky tenants." Dedicated clusters are reserved for genuinely untrusted tenants (compliance, regulated industries, public PaaS).

Concept

These five terms describe how mature organisations operate Kubernetes once they have it.

  • GitOps moves the source of truth from the cluster to a Git repository. An automated agent inside the cluster watches Git and reconciles — same loop pattern as a Kubernetes controller, just with Git as the desired state. Four formal principles: declarative; versioned and immutable in Git; pulled automatically by an agent; continuously reconciled.
  • Platform engineering is the discipline of building an internal platform — a "paved road" — that product teams consume. The platform team owns Kubernetes, observability, secrets, CI/CD, ingress; product teams ship apps onto it via a thin interface (manifests, a CLI, a portal).
  • SRE popularised by the 2016 SRE Book. Defines reliability with SLIs, SLOs, SLAs; spends an error budget on velocity until reliability drops; automates operations as code; treats infrastructure as a software problem.
  • Service ownership puts the team that wrote the service on its on-call rotation. The team feels both the deploy pleasure and the 3am page.
  • Multi-tenancy is sharing one cluster (or platform) across multiple teams. Soft tenancy = trusted teams, isolation by namespace + RBAC + quotas. Hard tenancy = untrusted, isolation by separate clusters or virtual clusters + dedicated nodes + strict admission policy.

These five compound. GitOps without service ownership becomes a bottleneck on the platform team. Service ownership without SRE becomes burnout. SRE without a platform turns every team into a Kubernetes operator. Multi-tenancy without RBAC and quotas is a noisy-neighbour disaster. Together they define how a healthy Kubernetes-using org runs.

Before / After

Before · ops as a ticket queue

To deploy, a developer files a ticket. The ops team patches a server, runs a script, closes the ticket. When the system breaks at 3am, ops gets paged — the developer doesn't. Knowledge of how things actually run lives only in ops; developers ship code blind.

→
After · platform + ownership

The platform team builds and runs the cluster. Product teams own their service end-to-end: write the YAML, commit to Git, an agent reconciles, they're on the rotation when their service pages. The platform team monitors error budgets and provides paved roads. Ops as a separate function disappears.

What did not change: people still need to be on call. Software still breaks. The 3am page still happens. What changes is who answers it (the team that owns the code) and how it gets fixed (with the same observability and tools the team uses every day).

AnalogyThe library

GitOps is a library where every change to the shelves starts with a card written in the catalog (a Git commit). A librarian (the controller — Argo CD, Flux) reads new cards, walks to the shelf, and makes the change. Nobody walks past the librarian and rearranges books — and if they do, the librarian notices the drift and puts the books back the way the catalog says.

Platform engineering is the central catalog desk and reading rooms — built once, used by everyone. Service ownership is each section ("Fiction," "Reference," "Periodicals") having its own librarian who knows that section deeply. SRE is the head librarian who watches how often books get misshelved and decides how much risk the library can absorb this quarter. Multi-tenancy is the fact that schools, scholars, and the public all share the same building — separated by hours, by reading rooms, by privileges.

Two ways to think about it

ELI5

A library with rules: every book change starts with a written card in the catalog. A librarian reads new cards and moves the books. Nobody is supposed to rearrange books by walking up to the shelf — and if they do, the librarian puts the books back the way the catalog says.

ELI10

GitOps moves the source of truth from the cluster to a Git repository. An agent inside the cluster watches Git and reconciles the cluster to match — same loop pattern as a Kubernetes controller, just with Git as the desired state. Platform engineering is the discipline of building the paved road that product teams ride on, so each team isn't reinventing CI, secrets, ingress, observability. SRE adds the language of measurable reliability — SLIs, SLOs, error budgets — and the principle that reliability is a feature you trade off against velocity. Service ownership puts the team that built a service on call for it, so the people who can fix a problem are also the people who feel it. Multi-tenancy lets multiple teams share one platform with isolation tuned to the level of trust between them. None of these is a Kubernetes feature — they are operating practices that Kubernetes enables.

A retail company with 20 product teams adopted GitOps via Argo CD over a year. Pre-GitOps, their average production change took 2 days from commit to live and required three approvals; post-GitOps, the same change took 14 minutes via PR merge to the env repo. The platform team published a paved-road template — Helm chart skeleton + Argo Application + ServiceMonitor + AlertmanagerConfig — and any new service got all of those for free. Two years later they introduced soft multi-tenancy: each team got its own namespace, ResourceQuotas, NetworkPolicies, and an Argo Project scoped to its repos. They still ran one shared cluster per environment.

What are the four GitOps principles?

Declarative; versioned and immutable in Git; pulled automatically by an agent; continuously reconciled. Per opengitops.dev (the formal definition from the CNCF GitOps Working Group).
Platform engineering vs SRE — what's the difference?
tap to flip
Platform engineering builds the platform product (the paved road that product teams ride on). SRE is a reliability practice (SLIs / SLOs / error budgets, automate ops as code). They overlap heavily but answer different questions.
Soft vs hard multi-tenancy?
tap to flip
Soft: trusted tenants share one cluster, isolation via namespaces + RBAC + ResourceQuotas + NetworkPolicy. Hard: untrusted tenants need separate clusters or virtual clusters (vCluster), strict admission policy, dedicated nodes.

Quiz · operating model (2)

⏸ pause animation in 'drift-correction' mode
A developer SSHes into a production node and changes a ConfigMap by hand. The cluster is GitOps-managed by Argo CD. Within minutes, the change disappears. Why?
Answer

Argo CD continuously reconciles the live cluster state to the Git source of truth. The hand-edited ConfigMap is drift; Argo CD detects it and reverts the resource to whatever Git says. The right path is to commit the change to Git (and let Argo CD apply it), or to mark the resource as ignored by Argo CD if it's intentionally externally managed.

What's the practical reason that "you build it, you run it" makes services more reliable over time?
Answer

Incentive alignment. The team that wrote the code feels the on-call pages. They cannot externalise the cost of bugs onto a separate ops team. Over weeks, they invest in better tests, better observability, better deploy patterns — because they will be paged otherwise. Reliability emerges from incentive, not from process.

Subtopic 5 of 5

Lineage, CNCF, release cadence, KEPs, feature gates

Where Kubernetes came from, who governs it, how often it ships, and why "alpha → beta → GA" matters more than it sounds.

📖

Words you'll meet in this section

Governance and release vocabulary. Some of these (KEP, feature gate) you'll see in every release notes page.

Borg concept
What it is: Google's internal cluster manager (in production since the early 2000s, paper published 2015). Schedules containers across hundreds of thousands of machines.
Kubernetes was built by Borg veterans. Its core ideas — declarative config, controllers, labels — come straight from Borg.
Omega concept
What it is: A Google research successor to Borg (paper 2013) that explored shared-state scheduling.
Many of Omega's ideas (optimistic concurrency on a shared store, multiple specialised schedulers) shaped Kubernetes' architecture.
Kubernetes / K8s tool
What it is: Open-source container orchestration system. Greek for "helmsman." "K8s" = K + 8 letters + s. Open-sourced June 2014 · 1.0 July 2015.
The de facto standard. Every major cloud has a managed Kubernetes service.
CNCF · Cloud Native Computing Foundation tool
What it is: A Linux Foundation project (formed 2015) that hosts vendor-neutral cloud-native open-source projects. Kubernetes was its first graduated project (March 2018).
Owns the Kubernetes trademark, runs the conformance program, hosts hundreds of related projects (Prometheus, Envoy, Helm, Argo, …). See landscape.cncf.io.
Release cadence config
What it is: How often new minor versions ship. Currently ~3 minor releases per year (since 2023; was 4/year before).
Each minor release is supported for ~14 months (12 months active + ~2 months EOL transition). The most recent 3 minors are patched at any time.
KEP · Kubernetes Enhancement Proposal protocol
What it is: The formal change-management process. A markdown document in kubernetes/enhancements, reviewed by the relevant SIG.
Any non-trivial change goes through a KEP. Reading the KEP for a feature you care about tells you the design, the alternatives, and the rollout plan.
Feature gate config
What it is: A runtime flag that turns an individual feature on or off. Default depends on the feature's stage (alpha / beta / GA).
Per-component (kube-apiserver, kubelet, kube-scheduler, …) usually with the --feature-gates flag. Cluster-wide on managed services.
Alpha · Beta · GA config
What it is: Three feature lifecycle stages. Alpha = off by default, may break or be removed. Beta = since 1.24+, new beta features off by default. GA = on by default, backward-compatibility guaranteed.
The "production rule" is "GA features only, plus explicitly vetted beta." Reserve alpha for dev/test clusters.
Conformance tool
What it is: The CNCF Certified Kubernetes program. Distributions that pass the test suite (run via hydrophone, historically sonobuoy) can call themselves "Certified Kubernetes."
Conformance covers the API. Distributions add their own features on top — those are not conformance, they're product additions.

Why this matters operationallyThe "GA only in prod" rule isn't conservatism — it's amortising the cost of upgrades. A cluster that runs only GA features can be upgraded between minor versions almost mechanically. A cluster that runs alpha features may need refactoring on every upgrade. Pick which lane you want to sit in.

Concept

Kubernetes was created at Google in 2014 as an open-source descendant of two internal cluster-management systems: Borg (in production since the early 2000s; paper published 2015) and Omega (paper 2013). Borg taught Google how to schedule containers at massive scale; Omega was a research successor that explored shared-state scheduling. Kubernetes was open-sourced in June 2014 and reached 1.0 in July 2015. That same month it was donated to the Cloud Native Computing Foundation (CNCF); in March 2018 it became the CNCF's first graduated project.

Release cadence. Kubernetes ships ~3 minor releases per year (it was 4/year through 2022). Each minor release is supported with patches for ~14 months (12 months of active support + ~2 months of EOL transition), so the last 3 minor versions are actively patched at any time.

KEPs and feature gates. Any non-trivial change to Kubernetes goes through a KEP (Kubernetes Enhancement Proposal). Feature gates are the runtime flags that turn individual features on and off. A feature progresses through three stages: alpha (off by default, may break or be removed without deprecation), beta (since K8s 1.24 / KEP-3136, new beta features are also off by default; API may still change), and GA / stable (on by default, backward-compatibility guaranteed, removal requires multi-release deprecation).

Conformance. The CNCF Certified Kubernetes program defines what "Certified Kubernetes" means via a test suite (hydrophone, historically sonobuoy). Vendor-specific features sit outside conformance — they're product additions, not the standard.

Before / After

Before · home-grown schedulers

Running containers at scale meant either home-grown schedulers (most companies stopped at "a script that SSHes to N machines") or one of a few competing systems — Mesos + Marathon, Docker Swarm, Nomad. Each company recreated the same primitives differently. Portability across clouds and on-prem was a custom integration job.

→
After · the de facto standard

A single API for "schedule a container, give it an IP, expose it as a service, scale it, restart it on failure" became standard across clouds, on-prem, and edge. Every major cloud provides a managed Kubernetes service (EKS, AKS, GKE) with the same upstream API. Workload portability is roughly real for the parts that stick to upstream APIs.

What did not get easier: Kubernetes is not simple. Its API surface is enormous, conformance only covers the API not the operational reality, and the variation between distributions in defaults, networking, storage, and security is still a real cost when moving between them.

AnalogyTrain-line testing

New Kubernetes features go through alpha → beta → GA the way new train signals get tested on a quiet branch line first, then on a moderately busy line, then on the main intercity route. By the time you ride the main line, the signals are boring on purpose — they've been observed long enough to be predictable. Alpha is the branch line: things break, that's the point. Beta is the moderately busy line: most things work, but the railroad tells you "we may still tweak the signal pattern." GA is the main line: stable, predictable, and removing it requires its own multi-year process.

Two ways to think about it

ELI5

Kubernetes is built by lots of people working together. Whenever they want to add something new, they write down what they want to add, they try it carefully in a test version (alpha), then a slightly more careful version (beta), then a full version (GA). By the time it's "GA," lots of people have used it and it's safe.

ELI10

Kubernetes' lineage is Borg → Omega → Kubernetes (open-sourced 2014, 1.0 in 2015, CNCF first graduated 2018). It now ships about three minor releases per year, with each release supported for around 14 months — so at any moment, the most recent three minors are patched. New features are proposed as Kubernetes Enhancement Proposals (KEPs), reviewed by the SIG that owns the area, and shipped through the alpha → beta → GA pipeline. Feature gates are the per-feature on/off switches; their default depends on the stage. Alpha is opt-in and unsafe; beta is opt-in for new features (since K8s 1.24) or opt-out for older ones; GA is on and stable. Conformance is the CNCF program that defines what "Certified Kubernetes" means — the API contract that distributions agree to honor — but distributions add their own features on top, which is why one EKS cluster and one OpenShift cluster will look familiar at the API level and very different in operation.

A platform team standardised on "we ship workloads against GA APIs only; beta features are evaluated case by case; alpha is dev-only." This rule meant they were never burned by a beta change between Kubernetes 1.27 and 1.28 (when the InPlacePodVerticalScaling feature changed shape during beta), but they did wait an extra year to use Gateway API in production until the v1 GA. Their next-door neighbour team made the opposite call (run alpha aggressively for early features) and spent two on-call weeks rebuilding workloads when an alpha API was renamed in a minor release.

Quiz · history & releases (2)

Why is "we use a feature only after it goes GA" usually the right operational rule?
Answer

Because GA carries the strongest compatibility guarantee Kubernetes offers — backward-compatible behavior, multi-release deprecation if it's ever removed, on by default. Beta APIs can still change shape; alpha can vanish. Operating production on GA features means upgrades won't surprise you; operating on beta or alpha means you accept upgrade-time refactoring as a routine cost.

A vendor's distribution says it's "Kubernetes-compatible" but not "Certified Kubernetes." What does this difference signal?
Answer

Certified Kubernetes is a CNCF program with a specific test suite (run via hydrophone, historically sonobuoy) that the distribution has passed. "Kubernetes-compatible" is not a defined term — it might mean the distribution implements most of the Kubernetes API but has skipped or modified parts of it. The practical risk is portability: your manifests may work, or they may hit an unimplemented API and break in a way that wouldn't happen on a certified cluster. Prefer Certified Kubernetes when portability matters.
