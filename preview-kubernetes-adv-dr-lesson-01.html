🚨 The 3 AM Nightmare

It's 3 AM. A bad migration deleted half the production namespaces. The team has Velero installed but never tested restore. The first restore attempt fails because the StorageClass doesn't exist on the new cluster. Today's lesson: backup tools + tested restores; backup that's never restored is a hope, not a plan.

Section 1.1 · etcd backup

Foundation of cluster rebuild

etcd holds all K8s API state. etcdctl snapshot save creates a point-in-time snapshot. Per-cluster backup pattern: cron-job snapshot every 6h to S3 (encrypted) + retention policy.

Restore: provision fresh cluster; etcdctl snapshot restore the snapshot; control plane reconstructs from etcd. Use case: total cluster loss, control-plane corruption.

Managed K8s (EKS / GKE / AKS): cloud handles etcd backup; you can't directly snapshot. Backup workloads via Velero instead.

⏸ Pause and check

Why etcd snapshot necessary?

etcd snapshot = control-plane state. Velero = workloads. Both required for self-managed clusters.

Section 1.2 · Velero

K8s-native backup; CSI / Kopia / Restic

Velero: K8s-native backup framework. Backs up: K8s objects (Deployment / Service / etc.) + PVCs (via CSI snapshot or filesystem-backup with Restic / Kopia).

Per-cluster: Backup CR with namespace selector + schedule; Restore CR with mapping. BackupStorageLocation = S3 / Azure Blob / GCS. VolumeSnapshotLocation per CSI driver.

Patterns: full-cluster nightly + per-namespace hourly + per-tenant on-demand. Restore: namespace-by-namespace; selective resource restore.

Section 1.3 · Kasten K10

Commercial app-aware backup

Kasten K10 (Veeam): commercial K8s backup. Application-aware (knows database state needs quiescing); rich policies (RBAC + RPO + RTO + compliance reports).

Trade self-host for richer dashboards + commercial support + multi-cluster aggregation.

Section 1.4 · CloudCasa + selection

SaaS DR-as-a-service + how to choose

CloudCasa: SaaS DR — install agent in cluster; backups + restores managed by CloudCasa platform. Trade engineering time for hosted simplicity.

Selection: Velero for K8s-shop self-host; K10 for commercial-support + richer features; CloudCasa for managed-DR-as-a-service. etcd snapshot always required for self-managed clusters.

⏸ Pause and check

When pick CloudCasa?

CloudCasa is hosted DR-as-a-service; trade engineering time for hosted ease. Velero / K10 if you self-host.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

A few common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: "Velero backs up everything."
Truth: Velero backs up K8s objects + PVCs; doesn't back up etcd directly (managed K8s = cloud-managed; self-managed = use etcdctl). Doesn't backup external services (RDS, Vault, S3 outside cluster).
Myth: "Backup is the hard part."
Truth: Restore is the hard part. Quarterly restore drills find issues that backup verification doesn't — StorageClass mismatches, NetworkPolicy misalignment, dependency order.
Myth: "Cloud-managed K8s makes backup unnecessary."
Truth: Cloud manages etcd; doesn't back up your workloads + PVCs + secrets. Velero / K10 / CloudCasa still required.
etcd snapshot — when needed?
tap to flip
Self-managed clusters; foundation for total-cluster-loss rebuild. Managed K8s (EKS / GKE / AKS) = cloud handles.
Velero backup what?
tap to flip
K8s objects (Deployment / Service / Secret / etc.) + PVCs (via CSI snapshot or Restic / Kopia).
BackupStorageLocation — what is it?
tap to flip
Velero CR pointing at S3 / Azure Blob / GCS bucket for backup destination.
Restic vs Kopia in Velero?
tap to flip
Restic: longer history; widely deployed. Kopia: newer; faster; recommended default in modern Velero.
Kasten K10 differentiator?
tap to flip
App-aware (knows DB quiescing); commercial support; richer policies; multi-cluster aggregation.
CloudCasa — what does it offer?
tap to flip
SaaS DR; agent-based; managed by CloudCasa team; hosted simplicity vs self-host.
Restore-test cadence?
tap to flip
Quarterly minimum; semi-annually for critical clusters. Find issues before real loss.
What backup doesn't cover?
tap to flip
External managed services (RDS / Vault / S3 / SaaS) — those have their own DR. Backup your K8s state; align with managed-service DR for the full picture.

Design backup for a 50-namespace EKS cluster.

answer(1) Velero install via Helm. (2) BackupStorageLocation: S3 bucket per environment + KMS-encrypted. (3) VolumeSnapshotLocation: AWS EBS CSI driver. (4) Schedule: nightly full backup + hourly per-namespace + on-demand pre-deploy. (5) Retention: 7-day daily + 4-week weekly + 12-month monthly. (6) Restore-test quarterly: pick 1 namespace; restore to dev cluster; validate. (7) Documented RPO + RTO: per-tier (gold = 1h RPO / 4h RTO; bronze = 24h).

A team's prod cluster died at 2 AM. Walk recovery.

answer(1) Read runbook: total-cluster-loss procedure. (2) Provision new cluster via Terraform / Crossplane (5-15 min). (3) Restore etcd if self-managed (or skip on EKS — control plane is managed). (4) Velero restore: per-namespace; high-priority first. (5) Restore PVCs: CSI snapshots; mount; verify data. (6) Restore secrets: ESO re-syncs from external Vault. (7) DNS swap: Route 53 record updates point to new cluster Load Balancer. (8) Validate: SLO metrics return; tenants notified. (9) Postmortem: timeline + gaps + roadmap.
