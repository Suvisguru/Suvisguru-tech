Module 2 · Lesson 10 · how images are baked

Image Building · Multi-Stage · Distroless · SBOM & Signing

Building an image is like baking a cake. Recipe (Dockerfile) + oven (builder) + ingredients (base + layers) → a finished cake. The pros use less sugar, sign their work, and ship a list of every ingredient.

Section 1 · Concept

How a container image actually gets built

An image is not magic. You write a Dockerfile (or equivalent) describing the build steps, hand it to a builder, and out comes an OCI image. Each build instruction (FROM, COPY, RUN) becomes a layer. The builder hashes everything, deduplicates, and pushes to a registry.

Build tools beyond Docker. Docker's BuildKit is the dominant builder, but several alternatives exist. Buildah (Red Hat) builds images without a daemon. Kaniko (Google) builds images inside a container itself, useful for CI in Kubernetes. ko builds Go images without a Dockerfile (just point it at your Go module). Cloud Native Buildpacks (Heroku/Pivotal) detect your language and build automatically — no Dockerfile.

Multi-stage builds are the most important Dockerfile pattern. You have one stage that builds your app (compiler, dev dependencies, full toolchain) and another that copies only the artifacts into a tiny final image. The builder is huge; the runtime is small.

Base image strategies. Three trade-offs: full distros like ubuntu (large, full shell, every tool), slim variants like node:22-alpine (smaller, fewer tools), distroless images (just your app + minimal runtime, no shell, no package manager), and scratch (literally empty — just your statically-linked binary). Each strip-down reduces image size and attack surface.

Multi-arch with buildx. Build one image that works on both AMD64 and ARM64 with docker buildx build --platform linux/amd64,linux/arm64. The result is a single tag containing manifests for each architecture (we covered this in Lesson 09).

Supply chain. Three concerns. CVEs — known vulnerabilities in the image's components. Tools like Trivy, Grype, Snyk scan your image and flag them. SBOM (Software Bill of Materials) — a machine-readable list of everything inside your image, in formats like SPDX or CycloneDX. Image signing — cryptographically prove the image came from your build pipeline (cosign, sigstore, Notary v2). Together they form your image supply-chain story.

Section 1.7 · The build tool landscape

Pick the right builder for the job

Five tools you'll see in the wild. Most teams use BuildKit (the default Docker builder) and never look further. Specialized cases use the others.

BuildKit
Default · Docker
Modern Docker builder. Parallelism, layer caching, secret mounts. The default behind docker build. Use unless you have a reason not to.
Buildah
Red Hat · daemonless
Builds OCI images without a daemon. Doesn't need privileged container. Common on RHEL / OpenShift CI.
Kaniko
Google · in-cluster
Builds images inside an unprivileged container. Perfect for Kubernetes-based CI (Tekton, Argo Workflows). No Docker socket needed.
ko
Go · no Dockerfile
Builds images directly from Go modules — no Dockerfile, no Docker daemon. Tiny images. Used by major K8s projects.
Buildpacks
Heroku · auto-detect
Cloud Native Buildpacks detect your language (Python, Node, Java, Go, etc.) and build an image automatically. No Dockerfile needed. Used by Heroku, Cloud Foundry, Cloud Run.
Bazel · Nix
Reproducible
Bytewise-reproducible image builds. Heavy investment, big payoff for orgs that need every build to produce identical bits.

For 90% of teams: BuildKit. For Kubernetes-based CI: Kaniko. For Go: ko. For "just build my Python app please": Buildpacks.

Section 1.8 · Base image strategies

Pick the smallest base that works

Smaller base = smaller image = faster pull + smaller attack surface + fewer CVEs to track. The trade-off: less stuff to debug with.

Full distro

~300 MB

ubuntu, debian, fedora. Everything you'd expect on a real machine. Easy to debug; lots of CVEs to track.

Slim / Alpine

~5–80 MB

node:22-alpine, python:3.13-slim. Stripped-down distros. Most apps work; some musl/glibc gotchas with Alpine.

Distroless

~10–50 MB

gcr.io/distroless/nodejs22 etc. Just your app + runtime. No shell, no package manager, no debugger. Production-grade.

scratch

app size only

Literally empty. Only works for static binaries (Go, Rust). Image = just your binary. Tiny. Bulletproof.

Default to distroless for production runtime, slim/alpine for build stage, full distro only when you need debugging. Use scratch only for static binaries (Go, Rust, statically-linked C).

Section 2 · Before & After

Naive build vs multi-stage + distroless

The same Node.js app, two ways. The first is what you build the first time you write a Dockerfile. The second is what you build after one CVE scan.

Same Node app. Same functionality. 17× smaller. 24× fewer CVEs. No shell for attackers.
Before · 1.4 GB

One-stage Dockerfile. Started FROM ubuntu for "familiar" feel. Ran apt install for build tools. Copied source code. Ran npm install + build. Final image contains: full ubuntu, gcc, make, source code, npm cache, build artifacts, your app.

Slow to pull, slow to push, lots of CVEs to track. An attacker who breaks in has a full shell, package manager, and your source code.

After · 80 MB

Multi-stage Dockerfile. Stage 1 (builder): node:22, install deps, build app, produce /app/dist. Stage 2 (runtime): FROM gcr.io/distroless/nodejs22, copy only /app/dist from stage 1. Final image: distroless + your built artifact. No source code, no build tools, no shell.

17× smaller. Pulls in seconds. Far fewer CVEs. An attacker who breaks in has nothing — no shell to exec, no package manager to install tools.

What didn't change: the app's behaviour. Multi-stage + distroless is purely operational improvement — no code change. The cost: harder to debug a running container (no shell to exec into), so you build "debug variants" with shell when needed.

Section 3 · Analogy

The bakery

Building a container image is like running a bakery. You have a recipe (Dockerfile) pinned to the wall. You have an oven (BuildKit, Buildah, Kaniko — different brands of ovens, all bake the same kind of cake). You stack ingredients (base image + your code + dependencies) and out comes a layered cake (the OCI image).

The pros do three more things. They use a multi-stage process: a messy prep kitchen for chopping and mixing, a clean finishing kitchen that only handles the final cake. The customer never sees the messy prep kitchen. They use the smallest possible cake base — distroless or scratch — because customers don't want to eat the cardboard box. And they ship a complete ingredient list (SBOM) plus a baker's signature seal (image signature) so customers know what's in the cake and that it really came from the official bakery.

That's all "image building" is. Recipe → oven → cake. Add some discipline and your cake is small, audited, and signed.

The mapping:

  • Recipe = Dockerfile (or Buildpacks descriptor, ko config, etc.)
  • Oven = builder (BuildKit, Buildah, Kaniko, ko, Buildpacks)
  • Multi-stage process = builder stage + runtime stage
  • Cake base = base image (full distro vs alpine vs distroless vs scratch)
  • Ingredient list on the side = SBOM (SPDX, CycloneDX)
  • Baker's signature seal = image signing (cosign / sigstore)
  • Food safety inspection = vulnerability scanning (Trivy, Grype, Snyk)
Section 4 · Two-level explanation

Two ways to hear it

ELI5 · explain like I'm 5

A bakery. The recipe is a list of steps. The oven is the machine that bakes. You can use a small base (just a cookie) or a big base (a whole cake). Pros use the smallest base that works because nobody wants to eat the box. They also write down what's in the cake so you know what you're eating, and they sign their work so you know it came from them.

ELI10 · explain like I'm 10

An OCI image is built by feeding instructions (a Dockerfile) to a builder (BuildKit, Buildah, Kaniko, ko, or Buildpacks). Each instruction becomes a layer. The most important pattern is multi-stage: one stage with all the build tools and dev dependencies, a final stage that copies just the compiled artifact into a tiny base image. Common bases: full distro (300MB), alpine (~5MB), distroless (~10-50MB, no shell), scratch (literally empty, only works for static binaries). Then you scan for CVEs (Trivy/Grype), generate an SBOM (machine-readable ingredient list in SPDX or CycloneDX format), and sign the image (cosign). Together those three give you a defensible supply-chain story.

Section 5 · Real-world scenarios

Where image-building decisions show up

Multi-stage cut image from 1.2 GB to 80 MB

A team's Node.js app shipped as a 1.2 GB image. Pulls took 90 seconds, deploys felt slow. They refactored to multi-stage: builder uses node:22 with all dev dependencies, final stage uses gcr.io/distroless/nodejs22 with only the built dist/ directory. Result: 80 MB image. Pulls take 6 seconds. CVE count dropped from 124 to 8. Same app behavior, 15× faster deploys.

Distroless prevented an attacker from getting a shell

A web app got compromised through an unsanitized input vulnerability. The attacker's payload tried to exec("/bin/sh") to escalate. The image used gcr.io/distroless/python3 — no shell included. The exec failed; the attacker had nothing to pivot to. The team fixed the input bug calmly during business hours. Distroless turned a "wake up at 3am" incident into a Tuesday afternoon ticket.

SBOM sped up the Log4j response

When the Log4Shell CVE landed in December 2021, every team scrambled to find which of their hundreds of services included the vulnerable Log4j version. Teams that had been generating SBOMs (SPDX/CycloneDX) for every image had a one-line answer: grep their SBOM database for "log4j-core". Teams without SBOMs spent days inspecting Dockerfiles and rebuilding images to check. SBOMs paid for themselves in a single incident.

Image signing caught a supply-chain attack

An attacker compromised a developer's CI credentials and pushed a malicious image to the team's registry under a legitimate tag. Production clusters had a Kyverno admission policy requiring images to be signed by the team's Cosign key. The malicious image wasn't signed (the attacker had registry creds but not the Cosign key). Kubernetes refused to admit the pod. Incident contained at the admission boundary, not after the malicious code ran.

Section 7 · Flashcards & quiz

Lock it in

What is a Dockerfile?
tap to flip
A text file with build instructions (FROM, COPY, RUN, etc.) that a builder turns into an OCI image. Each instruction usually becomes a cached layer. Most popular image-build format; not the only one.
What is multi-stage build?
tap to flip
A Dockerfile pattern with multiple FROM stages. Build stage has compilers + dev deps (large). Runtime stage copies only the final artifact into a tiny base. Final image is 10-100× smaller than single-stage.
What is distroless?
tap to flip
Base images from gcr.io/distroless that contain only your app + minimal runtime — no shell, no package manager, no debugger. ~10-50 MB. Tiny attack surface; harder to debug a running container.
What is scratch?
tap to flip
An empty base image — literally nothing. Only works for static binaries (Go, Rust, statically-linked C). Image = just your binary. Bulletproof but no debugging.
Buildah vs Kaniko vs ko?
tap to flip
Buildah = daemonless OCI builder (Red Hat). Kaniko = builds inside an unprivileged container, perfect for K8s CI. ko = builds Go images without a Dockerfile, just point at the Go module.
What is multi-arch with buildx?
tap to flip
docker buildx --platform linux/amd64,linux/arm64 builds one image tag containing manifests for both architectures. Each node pulls the right one. Built using QEMU emulation under the hood.
What is an SBOM?
tap to flip
Software Bill of Materials. A machine-readable list of every component inside an image (libraries, OS packages, language deps, versions). Standard formats: SPDX, CycloneDX. Critical for CVE response (e.g., Log4Shell).
What is image signing?
tap to flip
Cryptographically prove the image was built by you and hasn't been tampered with. Tools: cosign (Sigstore project, easiest), Notary v2. Admission controllers (Kyverno, OPA) verify signatures before allowing the image to run.

Your Node.js app image is 1.2 GB and CVE scans flag 124 vulnerabilities (mostly in libraries you don't use). Production deploys feel slow. What's the highest-leverage change?

answerRefactor to multi-stage Dockerfile + distroless. Stage 1: FROM node:22 with all dev dependencies, runs npm ci + build. Stage 2: FROM gcr.io/distroless/nodejs22, copies /app/dist from stage 1. Likely result: ~80 MB image (15× smaller), ~5-10 CVEs (90% fewer because no extra OS packages). Deploys speed up dramatically. No app code change required.

A CVE drops in a popular library (like the next Log4Shell). Your team manages 80 services. How do you find which images include the vulnerable library?

answerQuery your SBOM database. If you generate SBOMs (SPDX or CycloneDX) for every image at build time and store them centrally (e.g., in Dependency-Track or your registry's SBOM endpoint), you grep for the library name in seconds. Teams without SBOMs spend days manually inspecting Dockerfiles and rebuilding images to check. Generate SBOMs in CI; you'll thank yourself the day a critical CVE lands.

An engineer wants to use FROM ubuntu:latest as the base for a Go app. Why is that wrong, and what should they use instead?

answerTwo issues. (1) :latest is a moving tag — pin to a specific tag (ubuntu:24.04) or digest. (2) ubuntu (300 MB) is overkill for a Go app. Go binaries are statically linked — they need nothing from the OS. Use FROM scratch or FROM gcr.io/distroless/static in the runtime stage. Final image = just the Go binary. ~5-15 MB instead of 300+. Faster, safer, fewer CVEs.
