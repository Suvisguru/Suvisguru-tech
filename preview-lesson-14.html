Deploying and Configuring vSphere Clusters

A vSphere cluster is more than a folder of hosts. Turn it on with two flags — DRS (load balancer) and HA (auto-restart on host failure) — and your environment goes from "I have hosts" to "I have a self-healing, self-balancing fabric." This lesson walks the cluster from creation to monitoring, then deep-dives into DRS and HA, ending with Fault Tolerance for the rare workload that needs zero downtime.

Subtopic 1 of 11

Create a vSphere cluster with DRS and HA

A cluster is the unit of intelligence. Turn it on with two checkboxes.

📖

Words you'll meet in this section

Cluster vocabulary — three terms to lock in before you click New Cluster.

vSphere cluster concept
What it is: A logical group of ESXi hosts that share resources and can run cluster services (DRS, HA, vSAN). Hosts are added to the cluster and managed as a unit.
Without a cluster, every host is an island. Inside a cluster, vCenter can place / move / restart VMs across the host pool.
DRS · Distributed Resource Scheduler tool
What it is: The cluster's load balancer. Watches CPU and memory pressure on every host and vMotions VMs to even things out.
No DRS = manual placement. With DRS = self-balancing fabric. Every production cluster runs DRS.
HA · High Availability tool
What it is: Auto-restart on host failure. If a host dies, HA powers on its VMs on surviving hosts within minutes.
A failed server stops being a 2 a.m. page; it becomes a Slack notification. Pair with DRS for the full cluster experience.
Quickstart config
What it is: The cluster wizard's guided flow — pick datacenter, name the cluster, tick DRS / HA / vSAN, add hosts.
Use it. The wizard prevents most "I forgot to enable X" mistakes that haunt manual-built clusters.
EVC at cluster creation config
What it is: The Enhanced vMotion Compatibility baseline (lesson 13 sub-3). Set at cluster creation; harder to change later.
Always set EVC at cluster creation. Even uniform clusters benefit when you add a newer host two years from now.
Single Image vs baselines concept
What it is: Two ways to manage host updates (lesson 15). At cluster creation you choose which model the cluster uses.
Modern default: Single Image (vLCM image). Legacy: baselines. New clusters should default to Image.
Datacenter object concept
What it is: The container above clusters in the vCenter inventory tree. A vCenter can have multiple datacenters; each datacenter holds clusters and standalone hosts.
Clusters live inside a datacenter. Pick the right one before creating a cluster — moving clusters between datacenters is non-trivial.
Empty-cluster pattern concept
What it is: Best practice — create the cluster empty with all features enabled, THEN add hosts. The hosts inherit the cluster's configuration.
Avoids the awkward dance of enabling cluster features on a cluster that already has running VMs.

The New Cluster wizard at a glance

Five wizard steps: name and place, DRS on, HA on, EVC + image, finish. Hosts join afterward and inherit cluster settings automatically.
★

i
vSAN as a third checkboxIf your cluster will run vSAN, enable it during the wizard too. The wizard will then validate disk groups and network configuration as you add hosts.

What it is

A cluster is a logical container with cluster-wide intelligence. You name it, place it in a datacenter, tick DRS, tick HA, set the EVC baseline and vLCM image — then click Finish. The result is an empty cluster waiting for hosts. As soon as you drag hosts in (or right-click → Add Host), each host inherits the cluster settings: gets enrolled with HA agents, becomes eligible for DRS placement and load balancing, validates against the EVC baseline.

Two ways to think about it

ELI5

A cluster is a team of computers that work together. Two of them have superpowers: one balances who's working hard (DRS), another revives the team if one falls down (HA). Turning on the cluster turns on the team.

ELI10

A cluster is a vCenter object that aggregates hosts and unlocks coordinated services. DRS, HA, vSAN, vLCM are all per-cluster features. Created via the New Cluster Quickstart wizard inside a datacenter; the wizard sets EVC baseline, vLCM image, and HA / DRS policy. Hosts joining the cluster inherit configuration and gain HA agent + DRS eligibility.

You inherit eight ESXi hosts running standalone. Workflow: New Cluster Quickstart → name "cluster-prod-east" → DRS on (Fully Automated), HA on (Restart Priority Medium, Admission Cluster %), EVC Cascade Lake, Single Image ESXi 8.0u2 → Finish. Then right-click cluster → Add Hosts → bulk-add the eight. Each host is reconfigured automatically; hosts come up with HA agents, DRS eligibility, and a uniform image. From "fleet" to "fabric" in 25 minutes.

Answer

Why does vSphere recommend Single Image instead of baselines for new clusters?
Answer

Single Image is the modern vLCM workflow — declarative ("the cluster is THIS image: ESXi 8.0u2 + drivers + firmware") rather than imperative ("apply patches A, B, C"). Easier to reason about, easier to enforce drift detection, lighter operational burden. Baselines remain for legacy clusters.

Subtopic 3 of 11

How DRS decides VM placement on hosts

Initial placement at power-on plus ongoing rebalance — both driven by per-VM "happiness."

📖

Words you'll meet in this section

DRS speaks in scores and recommendations.

Initial placement concept
What it is: When a VM powers on, DRS picks the best host across the cluster.
Smart placement at power-on is half of DRS's value — every VM starts on the right host.
Ongoing balancing concept
What it is: Every 5 minutes (default), DRS recomputes the cluster state and may vMotion VMs to a happier host.
Workload patterns shift through the day. DRS shifts placement to match.
VM DRS Score concept
What it is: Per-VM "happiness" score 0-100% based on how well its CPU and memory demand is being served on its current host.
DRS's modern engine optimizes per-VM happiness, not host-level imbalance. The cluster DRS Score is the average.
CPU happiness concept
What it is: Sub-component of VM score — measures CPU demand vs delivered.
High CPU ready or co-stop drives CPU happiness down. DRS may move the VM to a less crowded host.
Memory happiness concept
What it is: Sub-component — memory demand satisfied without ballooning / compression / swap.
If a VM is ballooning, DRS sees memory unhappiness and may relocate.
Cost-benefit analysis concept
What it is: DRS only proposes a vMotion if predicted score improvement exceeds the migration cost.
Stops DRS from chasing tiny imbalances. Movement has cost (vMotion bandwidth, brief impact); DRS accounts for it.
Affinity rules config
What it is: Constraints — keep VMs together (affinity) or apart (anti-affinity), pin VMs to host groups.
Use anti-affinity for clustered apps (don't run both DC controllers on the same host). Use VM-host pinning for licensing constraints.
Soft vs hard rules concept
What it is: Soft = "should" (DRS may violate to keep cluster running); hard = "must" (DRS will fail rather than violate).
Use soft for preferences, hard only when violating would cause real damage (compliance, license constraint).
DRS recommendation concept
What it is: A proposed vMotion. In Manual mode, the admin approves; in Fully Automated, DRS executes.
In production almost everyone runs Fully Automated. DRS recommendations show what it would do — useful in audit and Manual modes.

How DRS picks a host

DRS computes a predicted VM happiness score on every candidate host and picks the highest. Same model on every rebalance, plus a cost-benefit check.
i
Modern DRS is per-VM, not per-hostOld DRS balanced host-level metrics. Modern DRS optimizes per-VM happiness scores. The result: one VM's pain matters even if hosts look balanced on average.

What it is

DRS computes a happiness score for each VM (CPU + memory demand satisfied without queuing or reclamation). At power-on, it picks the candidate host that gives the highest predicted score. Every 5 minutes, it re-evaluates the cluster: any VM whose current score is materially below what it would be on another host becomes a vMotion candidate. The cost-benefit gate stops it from chasing tiny imbalances.

Before / After

Before

Admin manually picks a host at power-on. Some hosts get hot; some sit empty. Performance varies day-to-day for no clear reason.

→
After

DRS picks the best host every time and rebalances on schedule. Cluster behaves like a single pool of resources.

Analogy

DRS is a host at a busy restaurant who seats parties at the table where they'll be most comfortable. Bigger party? Quieter section? They optimize per-party happiness, not "are tables evenly full." Sometimes they re-seat a couple mid-meal if a quieter booth opens up — but only if the move is worth the disruption.

Two ways to think about it

ELI5

Imagine choosing where to sit on a school bus. DRS picks the seat where you'll be happiest — quietest, with friends, near the window. It checks every seat before deciding.

ELI10

DRS uses a per-VM happiness model: predicted CPU + memory satisfaction on each candidate host, factoring contention, NUMA fit, ready time, balloon / swap. Initial placement picks max-predicted-score host. Continuous balancing scans every 5 min; recommends vMotions when predicted improvement clears a cost-benefit threshold (which accounts for migration cost and brief impact).

9 a.m. — batch jobs start, three hosts spike. By 9:05 DRS has vMotioned 4 VMs to quieter hosts. Admin doesn't notice. By 5 p.m., batch jobs end, DRS has rebalanced again to the new state. The cluster's DRS Score never drops below 87%.

tap to flip
Initial placement — scores every candidate host for predicted VM happiness (CPU + memory demand satisfied) and picks the highest. Smart selection means the VM starts in the right place.
When does DRS rebalance and what does it consider?
tap to flip
Every 5 minutes (default). Computes per-VM happiness scores; recommends vMotion if predicted improvement clears the cost-benefit threshold (accounts for migration cost vs benefit). Doesn't move for tiny gains.

tap to flip

Quiz · DRS placement (2)

⏸ pause animation in 'DRS placement' mode
Cluster DRS Score is 70% — flagged as not great. What might DRS do?
Answer

Identify the lowest-scoring VMs and recommend / execute vMotions to better-fit hosts. After moves, recompute. Cluster DRS Score should rise. If it can't (e.g., cluster is genuinely overloaded), DRS reports a recommendation it can't fulfill — capacity issue, not DRS issue.

Two domain-controller VMs were placed on the same host. Why is that bad and what rule fixes it?
Answer

If that host fails, both DCs go down — defeats the redundancy purpose. Add a VM-VM Anti-Affinity rule: "DC1 and DC2 must run on separate hosts." DRS will keep them split. Use Hard for compliance, Soft for preference.

Subtopic 4 of 11

Use cases for vSphere DRS settings

Automation level · Migration threshold · Predictive · Power management. Pick once, change rarely.

📖

Words you'll meet in this section

Each setting has a clear default. Knowing when to deviate is the skill.

Automation level config
What it is: Manual / Partial / Fully Automated. Determines whether DRS just recommends or also executes vMotions.
Fully Automated is the production default. Manual is used during cluster bring-up to vet recommendations. Partial = auto initial placement, manual ongoing.
Migration threshold config
What it is: 1 (very conservative) to 5 (aggressive). Default 3. Higher numbers = lower benefit threshold = more migrations.
Default is right for most clusters. Bump to 4-5 only for heavily uneven workloads. Drop to 2 if vMotion bandwidth is constrained.
Predictive DRS config
What it is: Optional integration with vRealize Operations / Aria — uses workload forecasts to pre-emptively rebalance.
Adds value for cyclical workloads (batch windows, business hours). Most clusters can skip without loss.
DPM · Distributed Power Management config
What it is: Automatically powers off lightly-used hosts and powers them back on when load returns. Off by default.
Saves real power, but startup latency means a load spike may briefly hit busy hosts before DPM brings idle ones online. Use in dev/test.
Aggressive vs conservative concept
What it is: Aggressive = many small vMotions for marginal balance. Conservative = only big-impact vMotions.
vMotion bandwidth is finite. Aggressive on a slow vMotion network means you spend it all on rebalancing.
Per-VM override concept
What it is: A specific VM can have its own automation level different from the cluster.
Use sparingly. A latency-sensitive VM might be Manual to prevent unwanted moves; everything else stays Fully Automated.
Maintenance Mode integration config
What it is: When you put a host in Maintenance Mode, DRS auto-evacuates the VMs (in Fully Automated). In Manual, it shows recommendations.
The single biggest argument for Fully Automated: you click Maintenance, DRS does the rest.
DRS rule conflicts concept
What it is: When affinity / anti-affinity rules contradict (e.g., two hard rules conflict), DRS reports a fault.
Audit rules occasionally — pile of conflicting rules means DRS can't optimize. Soft rules degrade gracefully; hard rules fail.

The DRS settings that matter

Three meaningful settings: Automation level (Fully Automated), Migration threshold (default 3), and a few advanced knobs (Predictive, DPM, per-VM overrides) you usually leave alone.
★
The 95% setting: Fully Automated · Threshold 3This combination matches the vast majority of production clusters. Don't second-guess it without a specific reason. Re-evaluate only if you see migration storms (drop to 2) or persistent imbalance (raise to 4).

What it is

DRS settings shape how aggressively the cluster rebalances and whether the admin is in the loop. Three big ones — Automation level, Migration threshold, and a handful of advanced toggles. Production defaults are Fully Automated + Threshold 3. The other knobs (Predictive DRS, DPM, per-VM overrides) are situational; most clusters never touch them.

Before / After

Before

Manual mode. DRS pings you 12 times a day with recommendations; you click Approve every time. Eventually you get tired and ignore them.

→
After

Fully Automated. DRS does its job; you check the score weekly. Hours saved, fewer mistakes, better balance.

Analogy

DRS settings are how you delegate to a smart assistant. Manual = "ask me before every move." Partially Automated = "you choose who sits where on day one, but ask me to swap people around." Fully Automated = "use your judgment, I'll review the weekly summary." The aggressiveness slider is how many small adjustments they make.

Quick check

Cluster has Threshold 5 and a slow vMotion network. Admin sees lots of vMotions and complaints of brief stalls. Best change?

Two ways to think about it

ELI5

You can let the smart helper decide everything (Fully Automatic), let it ask you first (Manual), or pick something in between. The aggressive slider says how often it cares about small differences vs only big ones.

ELI10

Automation level controls whether DRS executes vMotions or just recommends. Migration threshold (1-5) maps to a benefit-cost gate — higher = lower required net benefit per move. Production default Fully Automated + 3. Predictive DRS adds workload forecasting; DPM adds power-aware host shutdown; per-VM overrides exist for outliers.

A bank's main cluster runs Fully Automated, Threshold 3. A separate dev/test cluster runs Fully Automated, Threshold 4 with DPM enabled — happily powers off three of its eight hosts overnight, saving real datacenter dollars. One latency-sensitive trading VM in the prod cluster carries a per-VM override of Manual to prevent surprise moves during market hours.

Three DRS automation levels?
tap to flip
Manual (recommend only), Partially Automated (auto place at power-on, manual rebalance), Fully Automated (auto place + auto rebalance — production default).
What does Migration threshold control?
tap to flip

Quiz · DRS settings (2)

⏸ pause animation in 'DRS settings' mode
A new cluster is being commissioned. The team isn't sure DRS will do the right thing. What's a safe path to Fully Automated?
Answer

Start in Manual. Watch DRS recommendations for a week. If they look reasonable, switch to Partially Automated for another week (auto place; manual rebalance). Then Fully Automated. Most teams skip straight to Fully Automated and are fine — but progressive bring-up is fair when stakes are high.

Why is per-VM Manual override useful for a latency-sensitive VM in an otherwise Fully-Automated cluster?
Answer

That VM won't be vMotioned automatically. The brief stun phase of a vMotion is invisible to most workloads but unacceptable to a latency-critical one (sub-ms trading, real-time control). The cluster keeps balancing everything else; this VM is left alone unless an admin moves it.

Subtopic 5 of 11

Monitor a vSphere DRS cluster

Three signals: DRS Score, recent migration history, and faults / unfulfilled recommendations.

📖

Words you'll meet in this section

DRS exposes its decisions transparently. These are the views to know.

Cluster DRS Score view tool
What it is: Monitor → vSphere DRS → Cluster Score. Shows the cluster average plus per-VM scores.
First place to look. Persistent low cluster score = DRS can't fix it (capacity issue, rule conflict).
VM DRS Score breakdown tool
What it is: Per-VM happiness with CPU vs memory components.
Identify the unhappy VM. Memory unhappy + balloon active = host memory tight. CPU unhappy + high ready = oversubscribed.
DRS recommendations / history tool
What it is: Logs of every DRS-driven vMotion plus reason and predicted improvement.
Audit trail. "Why did DRS move that VM at 3 a.m.?" — answer is here.
DRS Faults tool
What it is: A "couldn't satisfy" log — failed vMotions, conflicting rules, capacity insufficient.
Persistent faults are red flags. Investigate before they become outages.
Migration storm concept
What it is: A spike of many vMotions in a short window. Often after a capacity event (host added, host failed, threshold change).
A storm is normal post-event. Sustained activity at high rates is suspicious — investigate threshold or workload changes.
Predictive recommendation concept
What it is: When Predictive DRS is on, future-load-based recommendations show alongside reactive ones.
Useful to confirm forecasted patterns; ignored ones suggest the forecast is off and Predictive should be tuned or disabled.
Cluster utilization vs balance concept
What it is: Two distinct views: how much of the cluster is used (utilization) and how evenly across hosts (balance).
High utilization + low balance = some hosts hot, others cool — DRS opportunity. High utilization + good balance = capacity issue (add hosts).
DRS faults you'll actually see concept
What it is: "Insufficient resources to satisfy reservation," "Affinity rule violated," "VM uses raw mapping in physical mode" — the common faults.
Each has a known fix. Reading the fault text saves you ten minutes of guessing.

What healthy vs unhealthy DRS looks like

Healthy: high cluster score, no low-VM outliers, modest balanced vMotion cadence, no faults. Trouble: low score, unhappy VMs (often memory-bound), migration churn, faults that don't clear.
i
Persistent low score = capacity issueIf the score stays under 80% even after DRS settles, the cluster is genuinely overloaded — add a host, retire low-priority VMs, or right-size oversized ones. DRS can't conjure capacity it doesn't have.

What it is

How vSphere HA responds to various failures

Host failure · isolation · partition · VM crash · APD/PDL — five flavors, five responses.

📖

Words you'll meet in this section

HA's job is auto-restart. The art is detecting WHAT failed.

Host failure concept
What it is: The host is dead — power loss, kernel crash. No heartbeats, no datastore I/O.
HA's classic case. Surviving hosts detect the failure and restart the dead host's VMs.
Host isolation concept
What it is: The host can't reach the management network and can't ping the isolation address — but it might still be running VMs and have datastore I/O.
Trickier than failure. The host can't tell if it's the only one disconnected. Default response: leave VMs running.
Network partition concept
What it is: Two groups of hosts can each see each other but not the other group. Both think the other side is dead.
Risk of split-brain. HA elects a master per partition; datastore heartbeats break the tie.
VM crash · OS hang concept
What it is: The VM stops responding while the host is healthy. Detected by VM Monitoring (VMware Tools heartbeat).
HA can restart just the VM, not the host. Catches OS hangs and BSODs.
APD · All Paths Down concept
What it is: The host briefly loses access to a datastore but the storage hasn't reported a permanent failure. Could be a network blip.
VMs hang waiting for I/O. HA can restart them on a host that has access (with VM Component Protection enabled).
PDL · Permanent Device Loss concept
What it is: The storage explicitly tells the host "this LUN is gone." Definitive — not a transient.
Clear signal. HA can immediately restart affected VMs on a host that has the datastore.
VMCP · VM Component Protection tool
What it is: The HA feature that handles APD and PDL. Optional; enable in HA settings.
Without VMCP, APD/PDL leave VMs hung. With VMCP, HA migrates / restarts them automatically.
Master / slave hosts concept
What it is: One host in the cluster is HA master (orchestrates failure response); others are slaves. Election runs automatically.
During a partition, each side may elect its own master. Datastore heartbeats reveal the real situation and prevent split-brain.

Five failure modes, five HA responses

Five failure modes, each detected through different signals. Heartbeats over network and datastores triangulate which scenario applies; HA responds appropriately.
⚠
Always enable VM Component ProtectionWithout VMCP, APD/PDL events leave VMs hung indefinitely. With it, HA restarts the VMs on a host that has access. It's a free safety net — turn it on.

What it is

HA is conservative on purpose — it triangulates evidence before declaring a failure. Network heartbeats every second tell the master who's alive; datastore heartbeats (lock files plus a heartbeat file) confirm whether a silent host is still running; an isolation address ping helps a host decide if IT is the disconnected one. Five failure modes — host failure, isolation, partition, VM crash, APD/PDL — each have a tailored response.

Before

→
After

Host dies. HA detects in < 30 s, restarts every affected VM on surviving hosts within 1-2 minutes. Pager rings only after the work is done.

If just one room loses lights (isolation), they don't evacuate immediately — they check first. If two halves of the building can't reach each other (partition), the larger half takes charge. They never just guess.

Network redundancy in a vSphere HA cluster

Two physical NICs · two switches · two datastore heartbeats · isolation address. Belt and suspenders.

📖

Words you'll meet in this section

HA's accuracy depends on redundant signals. Eliminate single points of failure in the heartbeat path.

Management VMkernel config
What it is: The host network interface that vCenter and HA use for control. Heartbeats ride here.
Single Mgmt NIC = single point of failure for HA. Always two physical uplinks for the Mgmt vSwitch.
Redundant Mgmt NICs config
What it is: Two physical NICs uplinked to the Mgmt port group, ideally on different physical switches.
A NIC failure or a switch failure no longer isolates the host. Mandatory for production HA clusters.
Datastore heartbeating config
What it is: HA configures lock files on datastores; updates them periodically. A silent host that's still updating these is "alive but partitioned."
Disambiguates network partition from real host failure. HA picks at least 2 datastores per host.
Isolation address config
What it is: The IP a host pings to decide "am I isolated?" — defaults to the gateway, configurable.
If a host can't reach this address either, it concludes itself isolated and follows the configured response.
Multiple isolation addresses config
What it is: You can set up to 10 (das.isolationaddress[0..9]). Host considers itself isolated only if all fail.

Slot model vs Cluster % concept
What it is: Two admission-control models. Slot = reserve N "slots" for failover. Cluster % = reserve a percentage of total CPU and RAM.
Cluster % is the modern default — simpler, less surprise. Slot model still common in legacy clusters.
Heartbeat datastores per host protocol
What it is: HA selects 2-5 datastores accessible by every host as heartbeat datastores.
Picking VMFS / vSAN / NFS datastores intentionally tells HA which to use; otherwise vCenter auto-selects.
Master election concept
What it is: Hosts elect one master (typically the host with the most datastores accessible). On master failure, a new election runs.
During a partition, each side may elect its own master. Datastore heartbeats prevent split-brain restarts.

Belt-and-suspenders heartbeat paths

Two physical NICs across two switches, two datastore heartbeats, multiple isolation addresses. HA needs all paths to fail before declaring a host dead.

What it is

The host doesn't conclude failure until all three signals point that way.

vSphere HA design considerations

Capacity for failover · uniform hosts · CPU compatibility · cluster size · failure tolerance.

📖

Words you'll meet in this section

HA design rules of thumb. Get these right at cluster creation; expensive to retrofit later.

N+1 capacity concept
What it is: Cluster sized so it can lose one host and still run all VMs. "N working hosts plus 1 spare's worth of capacity."
Bare minimum design. Without N+1, a single host failure means some VMs can't restart anywhere.
N+2 / N+more concept
What it is: Sized to tolerate two or more concurrent host failures.
For larger clusters or critical workloads. A 16-host cluster designed N+2 is a common high-resilience pattern.
Admission Control config
What it is: The cluster setting that enforces failover capacity at VM power-on. "Block power-on if cluster doesn't have enough headroom."
Without admission control, a cluster might over-commit so much that a host failure leaves VMs unable to restart. Always on in production.
Uniform host hardware concept
What it is: Same CPU generation, same RAM size, same NICs across the cluster. Reduces edge cases.
Mixed clusters work (with EVC) but uniform clusters are easier to capacity-plan and have fewer surprises.
CPU compatibility concept
What it is: Hosts must support the same CPU instruction set (or use EVC) for HA to restart VMs across them.
EVC eliminates this gotcha — always set EVC at cluster creation.
Storage shared by all hosts concept
What it is: Every host sees every datastore (vSAN, FC SAN, NFS, etc.).
A host can only restart a VM if it can see the VM's datastore. Asymmetric storage = HA holes.
Cluster size concept
What it is: ESXi 8 supports up to 96 hosts per cluster, ~8000 VMs. Practical sweet spot: 8-32 hosts.
Smaller clusters fail-fast; larger reduce N+1 overhead but increase blast radius. 8-16 is a common compromise.
Resource buffer concept
What it is: Extra capacity beyond N+1 — for transient spikes, vMotion overhead, occasional human error.
Running a cluster at 95% utilization gives HA nothing to work with. Aim for ≤70% sustained CPU/RAM.

Sizing for HA — N, N+1, N+2

N: each host runs at full capacity, no spare. A failure = some VMs can't restart. N+1: each host runs at 75%, has spare for one failure. N+2: tolerates two simultaneous failures.
⚠
Don't run hotA cluster running steady at 90%+ has nothing left for failover. Aim for ≤70% sustained CPU/RAM. Admission control reserves the right buffer automatically — don't override it.

What it is

HA design lives at the intersection of capacity, host uniformity, and storage symmetry. The rules: size for at least N+1 (one host can fail and everything still runs); keep hosts uniform (same CPU class via EVC, similar RAM); ensure every host sees every datastore (no asymmetric storage); enable Admission Control with the right model. Cluster size 8-16 hosts is the sweet spot.

Before

→
After

Same workload across more hosts. Steady utilization 65%. Admission control reserves 25%. A host fails — every VM restarts in minutes.

Analogy

Designing HA is like designing redundant elevators. If a building has the absolute minimum elevators it can fit, breaking one cripples the building. Add one or two extras (N+1, N+2) and life continues during repairs. The extra elevators cost real money — but they're the difference between a building and a useful building.

tap to flip

tap to flip

Quiz · HA design (2)

⏸ pause animation in 'HA design' mode
A 16-host cluster averages 85% CPU. Should it be sized N+1, N+2, or N+3?
Answer

At 85% steady CPU, even N+1 is precarious — losing one host pushes survivors over 90%. Either: drop utilization (right-size VMs, add a host to enable real N+1), or commit to N+2 by adding 2 hosts so the admission control reserves are meaningful. 16-host clusters often pick N+2 for resilience.

Why is asymmetric storage access an HA anti-pattern?
Answer

If host A can see datastore X but host B cannot, VMs whose vmdks live on X cannot restart on B. After a host A failure, those VMs are stuck. HA's promise of any-VM-on-any-host requires every host to see every datastore — the simplest, safest design.

Confirm both new hosts show HA agent: connected. Confirm they were added to heartbeat datastore set if you'd pinned datastores explicitly. Re-check failover capacity (more hosts may shift admission %). Quick chaos test optional but encouraged.

When to use vSphere Fault Tolerance (FT)

A second VM running in lockstep with the primary. Zero downtime on host failure. Use sparingly.

📖

Words you'll meet in this section

FT is special — far stronger than HA, far more constrained. The vocabulary captures the trade-offs.

vSphere Fault Tolerance (FT) tool
What it is: A feature that maintains a duplicate "secondary" VM on a different host, kept in lockstep with the primary's CPU and memory. If the primary's host dies, the secondary takes over instantly with zero downtime.
For workloads that cannot tolerate even the 30-90 second HA restart. Real zero-downtime instead of "minimal downtime."
Primary VM concept
What it is: The VM you actually use. All I/O originates from it.
Logically the "real" VM. The secondary is hidden from the user.
Secondary VM concept
What it is: A shadow VM on a different host running the same instructions in lockstep with the primary.
Invisible until the primary host dies; then it takes over and a new secondary spawns elsewhere.
Fast Checkpointing concept
What it is: The mechanism modern FT uses to keep primary and secondary identical — frequent (sub-second) snapshots of CPU/memory state replicated.
Replaced the older "record/replay" approach. Lets FT scale to multi-vCPU VMs.
FT VMkernel network config
What it is: Dedicated 10 GbE+ VMkernel for FT logging traffic — checkpoint state from primary to secondary.
FT generates significant network traffic. Slow FT network = primary slowdown. 10 GbE minimum, 25 GbE recommended.
vCPU limit concept
What it is: FT supports up to 8 vCPUs per VM (vSphere 8). Older versions had lower limits.
FT can't protect arbitrarily large VMs. Big DBs and SAP HANA may exceed the limit and need other HA strategies.
Constraints & unsupported features concept
What it is: Snapshots, hot-add CPU/memory, RDM-physical, vSAN stretched clusters, and others are restricted on FT VMs.
FT trades flexibility for instant failover. Read the constraint list before turning it on.
FT vs HA concept
What it is: HA = restart-on-failure (30-90 s downtime). FT = zero-downtime live failover.
FT is a strict superset for the protected VM but expensive (2× compute, dedicated bandwidth). Use HA for the cluster, FT for the few critical VMs.

FT in action

Primary and secondary run on different hosts, kept in lockstep via the FT logging network. If the primary host fails, the secondary becomes the new primary in milliseconds — zero perceived downtime.

What it is

FT runs a hidden secondary copy of a VM on a different host, kept in lockstep with the primary. The Fast Checkpointing engine captures CPU/memory state at sub-second cadence and replicates over a dedicated FT logging network. If the primary's host dies, the secondary becomes the new primary with zero perceived downtime; a fresh secondary spawns on another host. The cost: 2× compute, dedicated bandwidth, and a long list of constraints (max 8 vCPU, no snapshots, no hot-add, etc.).

Analogy

FT is having an identical twin doing your job in real time on a different floor. If you collapse, the twin steps in mid-sentence. Twins are expensive (you pay both salaries) and limited (you can't have one for every employee). Worth it for the CEO; not worth it for everyone.

Quiz · Fault Tolerance (2)

⏸ pause animation in 'FT' mode
Why isn't FT used as the default for every VM in production?
Answer

Cost. Every FT VM consumes 2× compute (primary + secondary) and dedicated FT network bandwidth. FT also has constraints (max vCPU, no snapshots, etc.). For the vast majority of VMs, the 30-90 second HA restart is acceptable; FT's overhead is reserved for the rare workload that truly can't tolerate it.

A FT VM's primary host dies. What sequence of events happens?
Answer

(1) Secondary detects loss of primary checkpoint stream. (2) Secondary promotes itself to primary in < 200 ms — applications continue without dropping connections. (3) FT spawns a fresh secondary on another host and begins re-establishing the lockstep. (4) Cluster returns to fully-protected state in seconds.
