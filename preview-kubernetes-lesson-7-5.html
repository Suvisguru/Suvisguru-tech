Lesson 7.5 · Level-0 primer

How a Linux Computer Works in 5 Minutes

A quick primer before Module 2. Skip if you already know what a kernel is.

Module 2 (Lessons 08–12) explains how containers work at the Linux level. That conversation assumes you know what a process is, what an operating system is, what the kernel is, and what root means. If those words feel foggy, this 5-minute tour will make Lesson 08 land cleanly. If you already know them, skip ahead — go straight to Lesson 08.

🎯 If you remember nothing else: A container is a Linux process the kernel has been tricked into thinking it's alone on the machine. The next 5 minutes explain how the kernel does the trick.

Section 1 · the basics

What a computer actually does

A computer runs programs. A running program is called a process. Your laptop, right now, has hundreds of them — your browser, the clock in the menu bar, the wifi connection, the music app, a few dozen background helpers you've never named. Every one is a process.

A process is just instructions plus the memory it uses. Most of the time, processes are doing tiny amounts of work — checking for a notification, drawing a pixel, waiting for input — and the computer juggles them so quickly that they all feel simultaneous.

Section 2 · the traffic cop

What an operating system is

An operating system is software that decides which process gets the CPU, the memory, the disk, and when. It's the traffic cop of the machine. Linux is one. macOS is another. Windows is another. They all do the same job — share the hardware among many processes — using slightly different rules.

When you click an app icon, the operating system picks a CPU core for it, allocates some memory, gives it access to your files, and starts the process running. When the app needs to read a file or send a network packet, it asks the operating system to do it. Apps don't talk to the hardware directly; the operating system stands between them.

Section 3 · the heart

What the kernel is

The kernel is the heart of the operating system. The part that talks to the actual hardware (CPU, RAM, network card, disk). When a regular program wants something from the hardware — read a file, allocate memory, open a network connection — it asks the kernel. That ask is called a system call, or syscall for short.

Most of an operating system is utilities, drivers, and apps that sit around the kernel. The kernel itself is small, fast, and trusted with everything dangerous (talking to hardware, deciding who gets memory, enforcing security). When you "talk to Linux" you're really talking to the Linux kernel through syscalls.

Hardware at the bottom. Kernel sits on top of it. The operating system layer (utilities, libraries, services) sits on the kernel. Processes sit on top of all of that. When a process needs something from the hardware, it asks the kernel — that ask is a syscall.
Section 4 · the admin

What "root" means

Root is the administrator account on a Linux machine. Root can do anything — read any file, change any setting, install anything, kill any process. On macOS the equivalent is "admin." On Windows it's "Administrator." Same idea.

By default, most programs should NOT run as root. If a regular app gets compromised — a bug, an exploit, a malicious input — and it's running as root, the attacker has the keys to the whole machine. Modern operating systems are built around the assumption that apps run as a regular user, and only specifically-trusted things run as root (and even those only briefly).

You'll hear "non-root by default" a lot in Module 2. It's exactly this idea applied to containers — a containerized app should run as a regular user, not as root, so a bug in the app can't escalate to controlling the host.

Section 5 · the punchline

Why this matters for containers

Here's the punchline that makes Module 2 click: a container is a Linux process the kernel has been tricked into thinking it's alone on the machine.

That's it. There's no special "container" thing inside Linux. The kernel takes a regular process and gives it a private view of process IDs, files, network, and memory limits. The process can still ask for things — it still makes syscalls — but the kernel makes them look like the process owns the whole computer.

Module 2 is about the five Linux features the kernel uses to do that trick: namespaces (private views), cgroups (resource budgets), capabilities (fine-grained permissions instead of all-or-nothing root), seccomp (syscall filtering), and LSM (mandatory access control like AppArmor or SELinux). They were all added to Linux over years; Docker just made them easy to use together.
