Configuring vSphere Networking

You have hosts (lesson 8) and a vCenter (lesson 9). Inside ESXi, all that physical connectivity is exposed through virtual networks that VMs and ESXi services plug into. Six moves to set up those networks correctly — from primitives to per-host vSwitches to cluster-wide vDS to the VLAN tagging and teaming policies that make it all reliable.

Subtopic 1 of 6

vSphere networking overview

The four primitives every vSphere network is built from.

📖

Words you'll meet in this section

The four building blocks of vSphere networking, plus a few related terms.

Physical NIC hardware
What it is: A real network card on the server. Has a physical port and a cable going to a physical switch. ESXi names them vmnic0, vmnic1, etc.
The boundary between virtual and physical networking. Everything inside ESXi eventually flows through one of these to leave the host.
Uplink hardware
What it is: A physical NIC viewed in the context of a vSwitch — the wire from the vSwitch out to the physical world. Same physical card, different name.
"vmnic0 is uplink 1 on vSwitch0." When NICs are paired into a team, they're called uplinks together.
vSwitch · virtual switch tool
What it is: Software inside ESXi that behaves like a layer-2 network switch. Has uplinks (physical NICs) on one side and ports on the other. Two flavors: vSS (per-host) and vDS (cluster-wide).
VMs and vmkernel adapters plug into a vSwitch through port groups. The vSwitch handles MAC learning, VLAN tagging, traffic shaping.
Port group config
What it is: A named collection of ports on a vSwitch with shared policy: VLAN ID, NIC teaming, security, traffic shaping. VMs and vmkernel adapters attach to a port group, not directly to the vSwitch.
"Production-VLAN-100" is a port group that puts every VM on it onto VLAN 100. Two port groups on the same vSwitch can have different VLANs and policies.
vmkernel adapter config
What it is: A virtual network interface that ESXi itself uses (not a VM) for management, vMotion, vSAN, NFS, etc. Has its own IP. Lives on a vmkernel port group.
When ESXi needs to talk on the network — sending vMotion data, reading from NFS storage, replying to vCenter — it uses a vmkernel adapter.
VM port group config
What it is: A port group designed for VMs to attach to. Carries VM traffic.
When you create a VM and pick "Network: Production-VLAN-100" — that's a VM port group.
vmkernel port group config
What it is: A port group for vmkernel adapters (ESXi's own NICs). Looks the same as a VM port group on the vSwitch — just used by ESXi services rather than VMs.
Same mechanism, different consumer. The distinction matters mainly when you create one — vSphere asks "is this for VMs or for vmkernel services?"
vSS · Standard vSwitch tool
What it is: A vSwitch that lives entirely on a single ESXi host. Configured per-host. Default. Free.
Simple but doesn't scale gracefully across many hosts (drift creeps in). Detailed in subtopic 2.
vDS · Distributed vSwitch tool
What it is: A vSwitch defined in vCenter that's applied identically to many hosts. Eliminates drift. Requires Enterprise Plus license.
The right choice for any environment with more than a few hosts. Detailed in subtopic 3.
Layer 2 (L2) protocol
What it is: The OSI networking layer that handles MAC addresses and Ethernet frames. Switches operate at L2.
vSwitches are L2 devices. They forward by MAC address. They don't route between subnets — that's L3, the physical router's job.
vNIC · virtual NIC concept
What it is: The virtual network card the guest OS sees inside a VM. Each VM can have one or more.
vNIC types matter for performance: VMXNET3 (paravirt, fast) vs E1000 (emulated, slow). Always pick VMXNET3 with VMware Tools installed.
Packet path concept
What it is: The complete journey: VM → vNIC → port group → vSwitch → uplink → physical switch. Reverse on the way in.
Memorise this — it tells you where to look when network troubleshooting. Each step has its own visibility in vSphere Client.

At a glance

A packet leaves a VM through its vNIC → into a port group → onto a vSwitch → out a physical NIC (uplink) → to the physical switch. Same model on every host.
★
Memorise the packet pathVM → vNIC → port group → vSwitch → uplink → physical switch. When the network breaks, walk the path. "Can the VM ping its gateway?" tests the whole chain. Each step has its own visibility in vSphere Client (port group view, vSwitch view, uplink view).
i

Subtopic 2 of 6

Standard vSwitch (vSS)

Per-host networking. Simple, free, drift-prone.

📖

Words you'll meet in this section

Standard vSwitch terms — and the operational hazards that come with per-host config.

vSS · Standard vSwitch tool
What it is: The default virtual switch on every ESXi host. Lives entirely on that one host — no awareness of other hosts. Free.
Easy to set up; hard to keep consistent across hosts. Drift becomes the operational problem.
Drift concept
What it is: Configuration on different hosts gradually diverging — different VLAN IDs, different security settings, different port group names. Often from typos or unsynchronised changes.
Drift is silent until something breaks: vMotion fails because host-3 has a different port-group name; a VM lands on host-7 with the wrong VLAN.
Per-host config concept
What it is: A configuration that has to be set on each host individually. Each host is its own snowflake.
vSS is per-host. To add a port group on 10 hosts you click 10 times — and risk typos. vDS solves this by being defined once at the cluster level.
Free vSphere edition config
What it is: An entry-level vSphere license that's free but limited — no vCenter clustering, no vDS, no DRS, no HA.
If you're on the free edition, you're stuck with vSS — vDS isn't included. Most production customers move past free quickly.
Host profile tool
What it is: A template captured from one "reference" host's config, then applied to other hosts to make them match. Includes vSS configuration. Enterprise Plus feature.
A way to fight drift on vSS without going to vDS. Useful but more setup than just using vDS.
PowerCLI · esxcli scripting tool
What it is: Command-line / scripting tools (PowerShell-based PowerCLI, ESXi-native esxcli) that can apply the same config across many hosts.
Another drift-fighting option for vSS. "Run this script across 10 hosts" beats "click 10 times" — but vDS still beats both for vSwitch config.
Standalone host concept
What it is: An ESXi host that's not joined to vCenter. Managed only via Host Client / DCUI / SSH.
A standalone host MUST use vSS — vDS is a vCenter feature. So vSS is the only option until you join the host to vCenter.
Bootstrapping concept
What it is: The phase before vCenter exists or is reachable. ESXi installed but not yet joined to vCenter.
During bootstrapping, only vSS is available. Many shops keep a small management vSS as a permanent fallback even after migrating to vDS.
Management vSS tool
What it is: A small vSS retained on each host purely for management traffic — even when the rest of networking has moved to vDS.
If vCenter or vDS gets sideways, the management network still works. A safety net pattern in production.
Cluster-wide config concept
What it is: Configuration defined once at the cluster level and applied identically to every member. The opposite of per-host.
vDS does this for networking; permissions and many vCenter settings do it for other domains. Cluster-wide is the modern preference.

At a glance

Same port group added by hand on each host. Host-3 got a typo. vMotion to host-3 fails the moment a VM tries to land. This is what vDS solves.
⚠
vSS doesn't scale gracefullyvSS is fine for 2-3 hosts. By 10 hosts, drift is a constant headache. By 50, it's impossible to manage by hand. If you have Enterprise Plus, use vDS. If not, use host profiles or PowerCLI scripts to enforce consistency.
i
Keep a small management vSS as a fallbackEven when you've moved to vDS for VM and vMotion traffic, many shops keep a small vSS on each host carrying only management traffic. If vCenter or the vDS gets sideways, management still works through the vSS. It's a cheap safety net.

What it is

A Standard vSwitch (vSS) is a vSwitch that lives entirely on one host. It's the default and ships with every ESXi install. The host that owns the vSS configures it; nothing about it is shared with other hosts.

The trade-off: simple and free, but if you have ten hosts in a cluster and want them to all see "Production VLAN 100," you create the same port group on every single host. Drift creeps in — one host has VLAN 110 by typo, vMotion fails, you spend an hour finding it.

vSS — when it's the right choice
  • Free vSphere edition (no vDS license)
  • Small environments where manual consistency is manageable
  • Standalone host before it joins vCenter (vDS impossible off-vCenter)
  • Edge cases: certain DPU/NSX scenarios
  • As a fallback for management traffic alongside a vDS
vSS — what you don't get
  • No central config; every host is its own snowflake
  • No LACP (active-active uplink bundling)
  • No NetFlow, port mirroring, traffic filtering
  • No private VLANs
  • No Network I/O Control (NIOC)

Quiz · vSS (2)

Answer

Subtopic 3 of 6

Distributed vSwitch (vDS)

Cluster-wide networking. No drift. Enterprise Plus.

📖

Words you'll meet in this section

vDS unlocks a lot of features with their own names — many borrowed from physical switching.

vDS · Distributed vSwitch tool
What it is: A virtual switch defined in vCenter and applied identically to every host that joins it. Configuration is centralized; forwarding still happens on each host.
Eliminates drift. One change in vCenter propagates to every host atomically.
Control plane vs data plane concept
What it is: Two halves of any switch. Control plane = "how am I configured?" (lives in vCenter for vDS). Data plane = "actually forward this packet" (lives on every host).
vDS splits these. If vCenter dies, the control plane is offline (you can't change config), but each host's data plane keeps forwarding traffic.
LACP · Link Aggregation Control Protocol protocol
What it is: A standard protocol for bundling multiple physical links between two switches into one logical link. The bundle gets the combined bandwidth and survives single-link failures.
vDS supports LACP; vSS does not. With LACP, two 10 GbE uplinks become a single 20 GbE logical pipe with automatic failover.
NetFlow protocol
What it is: A protocol Cisco invented for exporting flow records ("a conversation between two IPs that lasted 12s and used 3 MB"). Many tools consume it for traffic analysis.
vDS can export NetFlow records. Lets your network team see what VMs are actually talking to. vSS can't do this.
Port mirroring (SPAN) tool
What it is: Copying all traffic on certain ports to another destination port for analysis. The "wiretap" of switching.
When you need to capture VM traffic with Wireshark, vDS port mirroring sends a copy to a sniffer VM. vSS can't do this either.
NIOC · Network I/O Control config
What it is: A vDS feature that allocates uplink bandwidth between traffic types (VM traffic, vMotion, vSAN, NFS, management) using shares and limits.
When the uplink is congested, NIOC ensures vMotion doesn't starve management or vSAN traffic. Critical at scale.
Private VLAN (PVLAN) config
What it is: A way to subdivide a single VLAN into isolated sub-groups. VMs in different sub-groups can't talk to each other even though they share a VLAN.
Used for multi-tenant isolation. vDS supports PVLANs; vSS doesn't.
Distributed port group config
What it is: A port group on a vDS. Same idea as a regular port group, but defined once and applied to every member host.
When you create one in vCenter, it appears on every host in the vDS atomically. No drift possible.
vSphere license tiers tool
What it is: The pricing tiers vSphere comes in: Standard, Enterprise Plus, and bundles like VMware Cloud Foundation. vDS requires Enterprise Plus or higher.
If you have Enterprise Plus (most production), vDS is included. If you have Standard, you're stuck with vSS.
Migration wizard (vSS → vDS) tool
What it is: A built-in vCenter wizard that migrates VM port groups and vmkernel adapters from a vSS to a vDS one host at a time, without breaking running VMs.
Makes the vSS-to-vDS migration safe and incremental. Don't try to do it by hand.

At a glance

One config change in vCenter shows up on every member host atomically. Data plane stays on each host, so forwarding survives vCenter outages.
★

i

What it is

A Distributed vSwitch (vDS) is a vSwitch whose configuration lives in vCenter and is applied identically to every host that's joined to it. Configure once; vCenter pushes the config to all member hosts. Drift goes away.

The trick: the vDS has two halves.

  • Control plane — lives in vCenter. Holds port groups, policies, topology.
  • Data plane — lives on each host. The hidden vSwitch on each host forwards packets locally; it just receives its config from vCenter.

Hosts can keep forwarding traffic even when vCenter is down. vDS doesn't put the data path through vCenter; it just puts the management through vCenter.

vDS-only features
  • LACPLink Aggregation Control Protocol — bundles multiple physical NIC uplinks into one logical link with the upstream physical switch. Provides higher aggregate bandwidth and faster failover. vDS-only feature in vSphere. — bundle multiple uplinks into one logical link with the physical switch.
  • NetFlowFlow-records export protocol — vDS exports per-flow statistics (source/dest IP, port, bytes) to a NetFlow collector for network observability and analytics. — export flow records for network observability.
  • Port Mirroring — mirror traffic to a sniffer for analysis.
  • Network I/O Control (NIOC) — share allocations of host bandwidth between traffic types.
  • Traffic filtering and marking — DSCP, ACLs at the port-group level.
  • Private VLANs.
  • Per-port stats and policies, not just per-port-group.

The cost: vDS requires the Enterprise Plus edition (or specific bundles). Most production environments at any scale have it.

Subtopic 4 of 6

Port groups and VLAN tagging

Named jacks for VMs and ESXi services. VLANs decide isolation.

📖

Words you'll meet in this section

VLAN-tagging modes are abbreviated to three letters that all sound similar. Read these once and the rest is easy.

VLAN · Virtual LAN protocol
What it is: A way to split a physical network into multiple isolated logical networks using a small "tag" (1-4094) on each Ethernet frame. Devices on different VLANs can't see each other unless a router moves traffic between them.
VLANs let many tenants share one physical network safely — Sales VLAN, Engineering VLAN, Production VLAN, all on the same switches.
802.1Q protocol
What it is: The IEEE standard that defines VLAN tagging. Adds a 4-byte tag to each Ethernet frame containing the VLAN ID.
When network people say "trunk port" or "tagged frame," they mean 802.1Q. It's the protocol behind VLANs as you'll see them.
Trunk port protocol
What it is: A physical switch port configured to carry frames from MULTIPLE VLANs (each tagged with its VLAN ID). The opposite of an "access port" which carries one VLAN untagged.
The physical NIC on each ESXi host plugs into a trunk port. The trunk's "allowed VLANs" list must include every VLAN your VMs need.
VLAN tag config
What it is: The actual marker added to a frame to say "this belongs to VLAN 100." Either added or stripped at the boundary between tagged and untagged.
The point where tagging happens (host vSwitch vs physical switch vs guest OS) is what differentiates VST, EST, and VGT below.
EST · External Switch Tagging config
What it is: The PHYSICAL switch handles all VLAN tagging. The vSwitch port group has VLAN ID = 0. The vSwitch sees only untagged frames.
Rare in modern setups. Limits a host to one VLAN. Not flexible.
VST · Virtual Switch Tagging config
What it is: The VSWITCH handles tagging. Port group has VLAN ID = the actual VLAN (1-4094). The vSwitch tags outbound frames; physical switch port is configured as an 802.1Q trunk allowing those VLANs.
The default and most common setup. One trunk port carries many VLANs to the host; the host slices them up via port groups.
VGT · Virtual Guest Tagging config
What it is: The GUEST OS handles tagging. Port group has VLAN ID = 4095 (all-VLAN trunk). The vSwitch passes tagged frames straight through; the guest OS's NIC config decides VLANs.
Used for nested virtualization, virtual firewalls, virtual routers — VMs that legitimately need to see multiple VLANs through one vNIC.
VLAN ID 0 / 4095 config
What it is: Two special VLAN IDs. 0 means "no VLAN tagging" (used by EST). 4095 means "pass all VLANs through, tagged" (used by VGT).
If you see VLAN 0 or 4095 in a port group config, you're in EST or VGT mode respectively. Otherwise it's VST.
Allowed VLANs (trunk) protocol
What it is: The list of VLAN IDs a physical trunk port permits. Frames tagged with VLANs not on the list are dropped silently.
Most "VM has no connectivity" issues come from this list. The vSwitch tags VLAN 100, the trunk doesn't allow 100, the frame disappears. Always coordinate with network team.
Native VLAN protocol
What it is: The VLAN frames go to when they arrive at a trunk port without a tag. Configured per-trunk on the physical switch.
Mostly historical. Best practice is to not put traffic on the native VLAN. ESXi uses tagged frames for everything.
Port group naming config
What it is: The convention you pick for naming port groups. Common: Prod-VLAN-100, Dev-VLAN-110, vMotion-VLAN-30.
On vSS, names must match across hosts for vMotion to work. Bad naming = drift = vMotion failures.
Static binding · ephemeral binding config
What it is: Two ways a vDS port group assigns ports to VMs. Static (default) — VM keeps the same port across reboots. Ephemeral — fresh port every time.
Static is the default and right for most cases. Ephemeral is used when you need to recover via Host Client when vCenter is down.

At a glance

VST: vSwitch adds the VLAN tag. The physical switch's trunk must allow that VLAN, or the frames are dropped silently.

Subtopic 5 of 6

NIC teaming, load balancing, failover

Two or more NICs per vSwitch. One logical pipe.

📖

Words you'll meet in this section

NIC teaming has its own dialect — five different load-balancing modes plus failure-detection vocabulary.

NIC teaming concept
What it is: Grouping two or more physical NICs as uplinks on the same vSwitch. They act as one logical pipe with combined bandwidth and automatic failover.
Single uplink = single point of failure. Two uplinks = redundancy + more bandwidth. Should be standard on every vSwitch.
Load balancing concept
What it is: The rule that decides which NIC carries which traffic. Five modes available; "Originating port ID" is the default.
Different modes have different bandwidth and physical-switch requirements. Most environments use the default and never change it.
Originating virtual port (load mode) config
What it is: The default. Each VM port gets pinned to one uplink based on its port ID. Simple, no switch config needed.
Works with any switch. Each VM uses one NIC at a time, but with many VMs the load distributes naturally.
Source MAC hash (load mode) config
What it is: Hash the VM's MAC address to pick an uplink. Like originating-port but stable across vMotion (because the MAC follows the VM).
Slightly nicer than originating-port for environments that vMotion a lot. No switch-side config needed.
IP hash (load mode) config
What it is: Hash source+destination IP to pick an uplink. A single VM can use multiple uplinks for different conversations. Requires EtherChannel or LACP on the switch.
Better aggregate bandwidth, more switch coordination. Use only if you also configured the switch correctly.
Physical NIC load (load mode) config
What it is: vDS-only. Monitors actual uplink utilisation and rebalances VMs to underused uplinks every 30s.
The "set and forget" mode for vDS. Adjusts to real load over time. Most environments with vDS pick this.
Explicit failover order (load mode) config
What it is: Always use the first listed active uplink; failover to standby only if it dies. No load distribution.
Useful for predictable single-path traffic — management, sometimes vMotion. Easier to troubleshoot.
Active vs standby uplinks config
What it is: Active uplinks are used for traffic; standby uplinks sit idle until all actives fail.
Used to keep traffic on predictable paths or to keep traffic types separate (e.g., management always uses uplink 1, standby on 2).
Failure detection concept
What it is: How vSphere decides an uplink is "down." Two modes: Link status only (cable-level), Beacon probing (deeper).
Link-status alone misses upstream failures (cable up, switch dead). Beacon catches more, at the cost of needing ≥3 uplinks.
Beacon probing protocol
What it is: ESXi sends small "ping" frames between team members through each uplink. If probes don't arrive at the others through a given uplink, that uplink is considered down.
Catches failures the cable-level link-status check misses. Needs at least 3 uplinks (otherwise can't isolate which side is failing).
EtherChannel · port-channel protocol
What it is: Cisco's term for a static link aggregation group on a physical switch. The switch treats multiple links as one logical link.
Required for vSphere's "IP hash" load mode to actually distribute traffic across uplinks correctly.
Notify switches config
What it is: A team setting (yes/no). When yes (default), on failover ESXi sends a RARP frame to make the upstream switch update its MAC table to the new uplink.
Without it, the switch keeps sending traffic to the dead uplink for ~5 minutes (until MAC ages out). Always leave on unless you have a specific reason.

At a glance

Two NICs in a team = bandwidth + resilience. Originating-port pinning means each VM uses one NIC at a time; failover swaps the pin in milliseconds.
★
Two NICs minimum, alwaysSingle uplink = single point of failure. Even on lab hosts, give each vSwitch at least two physical NICs. Cheap insurance against a flapping cable, dead NIC, or upstream switch reboot.
i
Default load mode is fine for most"Route based on originating virtual port" works on any switch and distributes load across uplinks naturally as your VM count grows. Only switch to IP hash if you have EtherChannel configured on the switch side too — otherwise you'll create more problems than you solve.
⚠
Beacon probing needs 3+ uplinksWith only 2 uplinks, beacon probing can't tell which side is failing — both look "silent." Enable beacon only on teams of 3 or more uplinks. Most production teams of 2 stick with "link status only."

What it is

Every host should have at least two physical NICs assigned to a vSwitch. This buys two things: more bandwidth (load balancing) and resilience (failover). The set of physical NICs assigned to a vSwitch is called a NIC teamingGrouping multiple physical NICs as uplinks on the same vSwitch for combined bandwidth and resilience. ESXi distributes traffic across the team using a load-balancing policy and fails over to surviving members if one dies. team.

Each port group inherits the team's policy (or overrides it). Five common load-balancing modes:

  • Route based on originating virtual port (default) — each VM port pinned to an uplink. Simple, no physical-switch config needed.
  • Route based on source MAC hash — hash the VM's MAC. Stable across vMotion.
  • Route based on IP hash — hash source/dest IP. Requires EtherChannel on the physical switch.
  • Route based on physical NIC load — vDS-only. Monitors actual utilization and rebalances. Most "set and forget."
  • Use explicit failover order — only ever use the first listed active uplink; failover to standby. No load balancing.

Active vs standby uplinks: active uplinks are used; standby uplinks sit idle and only take over if all active ones fail.

Network failure detection
  • Link status only — uplink "down" only when physical link drops. Misses the case where the cable is up but the upstream switch is unreachable.
  • Beacon probingFailure-detection mode where ESXi sends probe frames between team members. If probes don't arrive at the others through a given uplink, that uplink is considered down. Catches upstream failures that link-status misses but needs ≥3 uplinks. — ESXi sends probe frames between team members; if probes don't arrive, that uplink is down. Catches more failure modes; needs ≥3 uplinks and only works for some teaming modes.

Notify switches (yes/no): on failover, ESXi sends a RARP so the upstream switch updates its MAC tables. Almost always yes.

Before

→
After

Two-uplink team with active/active load balancing and beacon probing. Same NIC failure: ESXi notices in milliseconds, fails over to surviving uplink, VMs barely notice. No HA event.

Analogy

NIC teaming is having two phone lines into the building, both live, with rules for how calls are split between them and how to fail over if one line dies. The "load balancing" mode is the rule for splitting; the "failure detection" is how you decide a line is really dead.

Your team policy is "Route based on originating virtual port." A single VM generates a sustained 5 Gbps. Both uplinks are 10 Gbps. Does the VM use both uplinks?
Answer

No. Originating-port load balancing pins each VM (per virtual port) to a single uplink. The VM uses one 10 Gbps NIC, not two. Subtopic 6 of 6

vmkernel adapters

ESXi's own NICs — one per service, on its own VLAN.

📖

Words you'll meet in this section

Each vmkernel service has its own purpose — many of which we cover in detail later. Here's the lay of the land.

vmkernel adapter config
What it is: A virtual NIC ESXi itself uses (not a VM). Has its own IP, subnet, MAC. Lives on a vmkernel port group on a vSwitch.
When ESXi needs to send anything itself — heartbeat to vCenter, vMotion data, vSAN replication, NFS reads — it uses a vmkernel adapter.
Management traffic config
What it is: Traffic between vCenter and ESXi (heartbeats, config commands, status). Also the traffic vSphere Client / Host Client / API uses to talk to the host.
Always present. Lives on the management vmkernel adapter (usually vmk0). Should never be saturated by other traffic.
vMotion traffic concept
What it is: The data stream when a running VM is being live-migrated between hosts — primarily memory state being copied.
Bursty and bandwidth-hungry. Can saturate a 10 GbE link during a big VM migration. Must not share with management or storage.
vSAN traffic concept
What it is: Storage traffic between hosts in a vSAN cluster — replication of VM data across hosts (covered in lesson 11).
Sustained, high bandwidth. Latency-sensitive. Strictly requires its own dedicated VLAN — not optional.
NFS / iSCSI traffic concept
What it is: IP-based storage traffic — host reading/writing VM files on a NAS (NFS) or block device on a SAN (iSCSI). Covered in lesson 11.
Storage traffic should be on its own vmkernel and ideally its own NIC pair. Sharing with VM traffic = unpredictable storage latency.
Fault Tolerance (FT) logging concept
What it is: A vSphere feature where a primary VM and a "shadow" secondary VM run in lockstep on different hosts. FT logging is the constant heartbeat / instruction stream between them.
Latency-sensitive — small latency hiccups break FT. Needs its own vmkernel adapter on a low-latency network.
vSphere Replication concept
What it is: A feature that asynchronously replicates VMs to a remote site for DR purposes. Replication traffic is its own thing.
Often crosses WAN links. Worth isolating on its own vmkernel so you can rate-limit it.
Provisioning traffic concept
What it is: The traffic for VM clones, cold migrations, and snapshot/template operations. Often shares with vMotion.
Bursty during deployments. Usually fine to share with vMotion's vmkernel.
Backup NFC traffic concept
What it is: Network File Copy traffic used by backup tools to read VM disk data. A VMware-defined protocol.
Heavy during backup windows. Knowing this traffic exists helps when planning network capacity.
Service checkbox config
What it is: When you create a vmkernel adapter, vSphere shows checkboxes for which services it can carry: Management, vMotion, FT logging, Provisioning, vSAN, NFC Backup, Replication, Replication NFC.
Same vmkernel can carry multiple services. In production, you usually give each big service its own vmkernel.
TCP/IP stack concept
What it is: An ESXi feature where each vmkernel adapter can have its own routing table and gateway, separate from the others. Default stack handles management; vMotion, vSAN, etc., can have separate stacks.
Lets vMotion traffic go through a different gateway from management — useful for cross-site vMotion or air-gapped storage networks.
Isolation concept
What it is: The pattern of giving each big service its own vmkernel + own VLAN + own NIC pair. Each service has independent failure / congestion / security.
Saturation in one (e.g., big vMotion) doesn't ripple into another (e.g., management). Production hosts should always be isolated this way.

Production hosts get one vmkernel per major service, each on its own VLAN and (ideally) its own NIC team. Isolation = predictable performance.
