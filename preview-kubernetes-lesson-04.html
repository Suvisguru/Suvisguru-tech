Lesson 04 · how to design your apps

Twelve-Factor Apps + Microservices vs Monoliths

Two questions: how should an app behave so it lives well in Kubernetes, and how big should each app be? The answers shape every modern codebase.

🚨 The 3 AM Nightmare

You're moving an app to Kubernetes. The app reads its database password from a file at /etc/myapp/db.conf, written by a Chef recipe at server-provisioning time. You bring it up in a Kubernetes Pod. The file isn't there. The app crashes on startup. Pods are temporary; Pod filesystems disappear. Your weekend is gone. This lesson is about the 12 rules that prevent you from ever fighting the platform like this again.

🎯 If you remember nothing else: A 12-factor app is a standard shipping container — drops cleanly into any platform. Microservices vs monoliths is about size, not virtue. Most teams should start with a modular monolith.

Section 1 · Concept

Two design questions, related but different

This lesson covers two ideas that are often taught together because they answer the same overall question: how do I design my apps so they thrive in cloud environments?

The Twelve-Factor App is a 12-rule manifesto written in 2011 by Adam Wiggins (Heroku co-founder) describing how an app should be built and packaged so it runs cleanly on any modern platform — including Kubernetes. The factors cover where config goes, how state is handled, how the app starts and stops, how logs flow, how backing services are connected. Follow them and your app drops into a Pod with almost no work. Violate them and your app fights the platform forever.

Microservices vs monoliths is the architectural choice about how big each app should be. A monolith is one big deployable that does everything. Microservices are many small services that communicate over the network. A modular monolith sits in between: one deployable, but with strict internal module boundaries. Each option has real trade-offs; there's no universal "right answer."

Section 1.5 · The 12 factors at a glance

What are the 12 factors?

Twelve short rules. You don't need to memorize them — you'll absorb them as you write apps. Here they are as a quick reference.

1

Codebase

One codebase tracked in version control, many deploys.

2

Dependencies

Explicitly declare and isolate dependencies (lockfiles, no system reliance).

3

Config

Store config in environment variables, never in code.

4

Backing services

Treat databases, caches, queues as attached resources (swap by URL).

5

Build, release, run

Strictly separate the build, release, and run stages.

6

Processes

Execute as one or more stateless processes (no in-memory state across requests).

7

Port binding

Export services via port binding — be self-contained, no app server.

8

Concurrency

Scale out via the process model (more copies, not bigger ones).

9

Disposability

Maximize robustness with fast startup and graceful shutdown.

10

Dev/prod parity

Keep development, staging, and production as similar as possible.

11

Logs

Treat logs as event streams written to stdout (let the platform collect them).

12

Admin processes

Run admin/management tasks as one-off processes against the same code.

Source: 12factor.net by Adam Wiggins (Heroku, 2011). Almost every cloud-native app today follows most of these.

Section 1.7 · Monolith · Modular monolith · Microservices

The split question · three answers

How big should each app be? Three common answers, each with real trade-offs.

Monolith

One big deployable. All features in one process, one codebase, one image, one deploy.

+ simplest to deploy & debug · − one bug can break everything · − one team can block all

Modular monolith

One deployable, but with strict internal module boundaries. Modules call each other in-process, never bypassing the rules.

+ simple ops + clean boundaries · − boundaries need code-review discipline

Microservices

Many small services, each owning one capability. They communicate over the network. Independent deploy, scale, fail.

+ independent teams & scaling · − network = distributed-systems tax

Most teams should start with a modular monolith and split out microservices only when scale or team boundaries clearly demand it.

Answer: c. Factor 3 (Config) — env vars, never code, never baked-in files. Same image, different config per environment.

Section 2 · Before & After

Custom-built apps vs portable apps

The shift: stop building apps that depend on the specific machine they run on. Start building apps that drop into any container, on any cloud, on any cluster.

Same cargo. Custom crate fights every transfer. Standard container moves smoothly through every step.
Before · custom-built

App reads database password from /etc/myapp/db.conf. Stores user sessions in a sessions/ folder on disk. Writes logs to /var/log/myapp.log rotated by cron. Hardcodes the staging vs prod flag in source.

Moving servers means recreating that exact filesystem layout. Running on Kubernetes means lots of fighting — Pods are temporary, the filesystem disappears, sessions vanish on every restart.

After · 12-factor

App reads database password from $DB_PASSWORD. Stores sessions in Redis (an attached backing service). Writes logs to stdout (Kubernetes captures them). Reads "ENV=prod" from a Kubernetes Secret.

Moving anywhere is a Dockerfile + a Deployment YAML + a Secret. Sessions survive restarts because they're not on the Pod's disk. Logs flow into your platform's pipeline automatically.

What didn't change: 12-factor doesn't make your app correct. A buggy stateless app is still a buggy app. The methodology removes friction with the platform — your application logic is still your problem.

Section 3 · Analogy

Shipping containers + restaurants vs food courts

Two analogies in one lesson because we're answering two questions.

Twelve-factor = the standard shipping container. Before standard containers (1956), every cargo was custom: barrels of oil, bales of cotton, crates of fruit. Loading a ship took a week, and dockworkers handled each piece by hand. The standard ISO container changed everything — same dimensions, same lifting points, same documentation, regardless of what's inside. Now a ship can be loaded in hours, and the same container moves seamlessly between trucks, trains, and ships. A 12-factor app is a standard shipping container for software: same interface to the platform, same way to receive config, same way to emit logs, same way to start and stop.

Monolith vs microservices = restaurant vs food court. A monolith is a single restaurant: one kitchen, one chef, one menu, one team. Easy to coordinate, but if the kitchen's busy, every dish slows down. A food court is many small stalls: pizza, sushi, tacos, salads, each with their own kitchen and team. Customers can mix and match, each stall scales its staffing independently — but you need shared infrastructure (tables, payment, restrooms) and coordination is harder. A modular monolith is a single restaurant with specialty stations: appetizers station, mains station, desserts station — separate stations with their own discipline, but one kitchen, one chef coordinating, one bill.

In the story……in Kubernetes
A standard ISO shipping containerA 12-factor app
A custom crate (barrels, bales)A pre-12-factor app needing custom platform handling
A single restaurantA monolith
A restaurant with specialty stationsA modular monolith
A food court of stallsMicroservices

⚠️ The analogy stops here: standard shipping containers are 20 or 40 feet long. Software containers are whatever size you make them — the "standard" is the interface, not the dimensions.

Section 4 · Two-level explanation

Two ways to hear it

Same ideas, two depths.

ELI5 · explain like I'm 5

Imagine your toys all came in different boxes — some round, some square, some sticky. Hard to put them on shelves! Now imagine all your toys came in identical square boxes that fit perfectly on any shelf. That's a 12-factor app. As for big-vs-small apps: it's like deciding to build one giant Lego castle in one piece (monolith) or making it from many smaller Lego sets that snap together (microservices). One giant piece is easier to build but harder to change; many smaller pieces are easier to change but harder to keep organized.

ELI10 · explain like I'm 10

The 12-factor methodology is twelve rules that turn an application into a portable, platform-friendly unit: read config from environment variables, log to stdout, don't store state on local disk, start and stop quickly, treat external services as swappable URLs. Apps that follow these rules drop into Kubernetes Pods cleanly. The microservices vs monolith decision is about granularity: one big deployable (monolith) is operationally simpler but couples teams together; many small deployables (microservices) give teams independence but introduce a "distributed-systems tax" — network calls, retries, timeouts, distributed tracing, separate dashboards, separate on-call. Most teams should start with a modular monolith and only split when scale or team boundaries clearly justify the cost.

⏸ Pause and check

A 6-engineer startup is debating "should we start with microservices or a monolith?"

Answer: b. Microservices push complexity from the codebase to the network: tracing, retries, timeouts, separate dashboards, separate on-call. A 6-engineer team rarely has the operational maturity for that. Start with a modular monolith; split a service out only when scale or team boundaries clearly demand it.

Section 5 · Real-world scenarios

Where these decisions show up in real companies

Four moments where 12-factor or the split-question shaped a real outcome.

Heroku and the birth of 12-factor (2011)

Heroku ran tens of thousands of customer apps on a shared platform. Some apps deployed cleanly, ran predictably, scaled smoothly. Others fought the platform constantly. Adam Wiggins (Heroku co-founder) and his colleagues distilled what made the "well-behaved" apps work into the twelve-factor manifesto. It became the foundational design guide for cloud-native software and is now de facto required for anything running on Kubernetes.

Netflix · 1,000+ microservices

Netflix famously broke its monolith into hundreds of microservices starting around 2009. Each team owns a few services end-to-end — independent deploy, independent scale, independent on-call. The trade-off: massive investment in observability, service discovery, traffic management, chaos engineering. They built much of this tooling themselves (Hystrix, Eureka, Zuul) because the ecosystem didn't exist yet. For Netflix's scale (thousands of engineers, millions of concurrent streams), the trade-off paid off.

Shopify · the modular monolith champion

Shopify chose to keep its core e-commerce platform as a single Rails monolith — but with rigorous internal module boundaries enforced by code review, custom linters, and architectural fitness functions. They have hundreds of engineers shipping daily into one codebase, deploying it as one image. Their argument: the operational simplicity of a monolith outweighs the team-independence benefits of microservices, as long as you have the discipline to maintain clean module boundaries. Many companies have copied this approach.

The startup that microservices'd too early

A common pattern: a 6-engineer startup splits its product into 12 microservices "because that's the modern way." Six months in, they're spending more time managing service-to-service calls, distributed tracing, separate CI pipelines, and version mismatches than building features. They lacked the operational maturity to absorb the complexity. The fix: collapse most services back into a modular monolith, keep one or two truly independent services where it makes sense. The cautionary tale here: microservices are an answer to organizational scale, not a starting architecture.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: Microservices are always better than monoliths.
Truth: They're an answer to organisational scale, not a starting architecture. Shopify runs hundreds of engineers on one Rails monolith — and ships daily.
Myth: 12-factor is just a coding style.
Truth: It's the contract between your app and any modern platform. Violating it doesn't fail tests — it fails Tuesday afternoons in production.
Myth: Stateless means "the app doesn't store anything."
Truth: Stateless means the process doesn't hold important state in memory. State lives in attached backing services — Postgres, Redis, S3 — that survive restarts.
What is a 12-factor app?
tap to flip
An app written to twelve specific guidelines (config, dependencies, processes, port binding, logs, etc.) that make it run cleanly on dynamic platforms like Kubernetes. Coined by Heroku in 2011.
Where should config live?
tap to flip
In environment variables (or platform-injected config like Kubernetes Secrets/ConfigMaps), never in code or baked-in files. Same image, different config per environment.
Why must processes be stateless?
tap to flip
Because the platform can restart, move, or replace a process at any time. State stored in memory or local disk disappears with it. Put state in attached backing services (DB, Redis, S3).
What is a monolith?
tap to flip
One big deployable that does everything in one process. Simple to deploy and debug, but tightly couples teams and features.
What are microservices?
tap to flip
Many small services, each owning one capability. They communicate over the network. Independent deploy, scale, and fail — at the cost of distributed-systems complexity.
What is a modular monolith?
tap to flip
One deployable, but the codebase is split into modules with strict, code-enforced boundaries. Operational simplicity of a monolith + boundary discipline of microservices.
Microservices trade-off?
tap to flip
+ team independence, independent scaling, blast-radius isolation. − network is now part of your app: retries, timeouts, distributed tracing, service discovery, separate dashboards, separate on-call.
When to start with microservices?
tap to flip
Almost never at the start. Most teams should begin with a modular monolith and split out individual services only when team scale, deployment cadence, or technical scaling clearly demands it.

A team is moving a Python app to Kubernetes. The app currently reads database credentials from /etc/app/db.conf written by a Chef recipe at provisioning time. Which 12-factor rule does this violate, and what's the fix?

answerViolates Factor 3: Config — store config in environment variables, never in code or baked-in files. Fix: store the credentials as a Kubernetes Secret, mount or inject them as environment variables. The app reads os.environ['DB_PASSWORD']. The Chef recipe goes away. The app image becomes generic — same image, different env per environment.

A 6-engineer startup is debating whether to start with microservices or a monolith for their new product. What would you recommend and why?

answerModular monolith. Microservices push complexity from inside the codebase to the network — distributed tracing, retries, idempotency, service discovery, separate dashboards, separate on-call. A 6-engineer team rarely has the operational maturity to absorb that. A modular monolith keeps clear internal boundaries with a fraction of the operational tax. They can split out a single high-traffic module later if scale or team boundaries justify it.

Words from this lesson · open if you want a quick reference
12-factor app
An app following 12 rules from 12factor.net (config, dependencies, processes, etc.) that make it run cleanly on cloud platforms.
Backing service
An external resource the app uses over the network — DB, cache, queue, search index, third-party API. Treat as a swappable URL + credentials.
Stateless process
An app that holds no important state in memory or local disk. State lives in attached backing services so the platform can restart/move freely.
Monolith
One big deployable containing all (or most) of an app's logic. Simple to deploy, can grow tangled if not disciplined.
Modular monolith
One deployable with strict, code-enforced module boundaries. Often the best starting architecture.
Microservice
A small, independently-deployable service responsible for one bounded capability. Communicates with others over the network.
Distributed-systems tax
The complexity microservices introduce: retries, timeouts, distributed tracing, service discovery, separate dashboards, separate on-call.
Bounded context
A clear boundary around a chunk of business logic. Inside the boundary you have one shared model; outside, different models. The unit at which to consider splitting a service.
