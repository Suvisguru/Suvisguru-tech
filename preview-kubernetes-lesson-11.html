Container Security · Rootless · Read-Only · Registries

Two halves: where the image came from (registry + auth) and how it runs (non-root, read-only, no extra powers). A bank vault, a teller, and a glass inspection booth.

🚨 The 3 AM Nightmare

Your web app gets compromised through an unsanitised input. The attacker gets a shell as root inside the container. They write a webshell to disk, escalate via a kernel CVE, pivot to the host, then to other Pods. By morning, your cluster is gone. The same bug, on a hardened Pod, is a Tuesday afternoon ticket. This lesson is the 12 lines of YAML that change which one happens.

🎯 If you remember nothing else: Where the image came from (auth + pull policy) plus how it runs (non-root, read-only, drop caps, seccomp) shrinks blast radius 10×.

Section 1 · Concept

Two halves: where it came from, how it runs

A container image alone is not secure. The runtime configuration around it decides whether a single bug becomes a breached cluster or a contained ticket. There are two halves to this story.

Half one — where the image came from. Public images on Docker Hub pull anonymously. Private images need credentials, supplied via an imagePullSecret. The kubelet uses that secret to authenticate to your private registry (ECR, GCR, ACR, Harbor, etc.). Without the secret, you get ImagePullBackOff. The imagePullPolicy then decides whether the kubelet re-pulls every start (Always — right for mutable tags), uses the cache if present (IfNotPresent — default for tagged images), or refuses to pull and uses only what's already on the node (Never).

Half two — how it runs. The Pod's securityContext decides what the container can do. runAsNonRoot refuses to run as UID 0. runAsUser pins a specific non-root UID. capabilities.drop: [ALL] strips Linux superpowers; add only what you need. readOnlyRootFilesystem makes the container's disk immutable. allowPrivilegeEscalation: false blocks setuid binaries from regaining privilege. seccompProfile: RuntimeDefault filters dangerous syscalls. Apply Pod Security Standards (Privileged / Baseline / Restricted) per namespace and let the API server enforce these defaults at admission, instead of relying on every developer to remember them.

The two halves compose. A signed image (Lesson 10) pulled with credentials (this lesson) running with a hardened securityContext (this lesson) is the production-grade story. Skip any half and you've left a door open.

Section 1.5 · The 5 building blocks, broken down

Each topic, one tile

👤

Default in containers: root inside. One kernel bug → host root. Set runAsNonRoot: true + runAsUser: 1000.

Linux superpowers cut from ~40 to 1. seccompProfile: RuntimeDefault blocks dangerous syscalls. Cheap, big payoff.

Private registry needs creds. Create a docker-registry secret, reference via imagePullSecrets. Without it: ImagePullBackOff.

⏱️

Always · IfNotPresent · Never

Always: re-pulls every start (use for :latest). IfNotPresent: cache OK (default for tags). Never: must already be local.

All five compound: non-root + read-only + drop-caps + private-registry + pinned-tag = production-grade Pod.

Section 1.8 · Pull policies

When does the kubelet phone the registry?

Three options. The default depends on your tag — and that subtlety has caused more outages than people realize.

Always

What: Kubelet checks the registry on every Pod start. If the digest changed, re-pull.

Default for :latest and tags without an explicit version. Right when you genuinely want the freshest image.

IfNotPresent

What: Use the local cached image if any image with that name+tag is on the node. Otherwise, pull.

Default for tagged images (:v2, :1.4.7). Right for pinned, immutable versions. Be careful: re-pushing a tag does NOT re-pull.

Never

What: Don't contact the registry at all. The image must already be on the node.

Air-gapped clusters. Local development with kind load or minikube image load. Bare-metal disconnected sites.

Best practice for production: pin to a digest (image: my-app@sha256:...). Digests are immutable, so the pull-policy question evaporates.

Section 1.9 · Pod Security Standards

Three tiers Kubernetes ships out of the box

Apply per namespace via labels. The API server's PodSecurity admission controller enforces the tier — no third-party admission needed for these defaults.

Privileged

No restrictions

Anything goes. hostNetwork, privileged: true, root user — all allowed. The "I trust this workload completely" tier.

Use for: system Pods (CNI, CSI, node-exporter, kube-proxy). Almost never for app workloads.

Baseline

Block known-bad

No hostNetwork, no hostPID, no hostPath, no privileged: true, no dangerous capabilities. Root user still allowed.

Use for: legacy app namespaces being modernized. The minimal tier any production namespace should be at.

Restricted

Production-grade

Baseline + runAsNonRoot + capabilities.drop: [ALL] + allowPrivilegeEscalation: false + seccompProfile: RuntimeDefault.

Use for: every greenfield app namespace. The default for new clusters.

Apply with kubectl label namespace prod Pod-security.kubernetes.io/enforce=restricted. Use warn and audit levels first to find existing violations before flipping to enforce.

Section 2 · Before & After

Naive Pod vs hardened Pod

Same container. Same workload. Five settings change. The blast radius of a bug shrinks 10×.

Same workload. Five settings change. The same SSRF that takes out the cluster on the left becomes a Tuesday afternoon ticket on the right.
Before · wide open

No securityContext. Container runs as root with all 40 capabilities and a writable root filesystem. Image is my-app:latest from Docker Hub — anonymous pull, mutable tag. Default pullPolicy is Always for :latest — every restart hits the registry, sometimes pulling slightly different content.

An SSRF in the app gives an attacker a root shell. They write a webshell to /var/www, escalate to host via a kernel CVE, and pivot to the cluster.

After · locked down

Six-field securityContext: runAsNonRoot, runAsUser: 1000, drop: [ALL], readOnlyRootFilesystem: true, allowPrivilegeEscalation: false, seccompProfile: RuntimeDefault. Image pinned to digest from a private registry, pulled with an imagePullSecret. Namespace labeled Pod-security.kubernetes.io/enforce: restricted.

The same SSRF lands in a process that's UID 1000, can't write to disk, can't escalate, can't make dangerous syscalls. Pod gets killed; deploy gets hot-fixed.

What didn't change: the app's behaviour. Hardening is operational, not functional. The cost is 12 lines of YAML and a Dockerfile USER 1000 directive.

Section 3 · Analogy

The fortified bank

A container registry is like a bank vault. Images are the safety deposit boxes; the vault door (registry auth) decides who gets to take one out. The teller window is the kubelet asking the registry, "I'm node-3, here are my credentials, give me image X." Without credentials, the teller turns you away — that's ImagePullBackOff.

Once the box is out, the visitor doesn't get to roam the bank. They go to a glass inspection booth — that's the container runtime. The booth has rules: non-root visitor (runAsNonRoot), no tools allowed (drop: [ALL] capabilities), the display case is sealed glass (readOnlyRootFilesystem), and a security camera overhead records every move (seccompProfile: RuntimeDefault). The bank's overall security manager sets which rules apply to which booths — that's Pod Security Standards.

⚠️ The analogy stops here: a bank vault has one combination. A container has dozens of independent locks (each securityContext field). Skipping any one is leaving a door cracked.

Section 4 · Two-level explanation

Two ways to hear it

ELI5 · explain like I'm 5

Imagine a bank. The vault holds boxes — those are the images. You can't just walk in. The teller checks your card before handing one over. Once you have a box, you go to a glass room with a camera. You can't bring tools, can't break the glass, and you're not the manager. The bank decides what you can do.

ELI10 · explain like I'm 10

Container security has two layers. First, where did the image come from? You set imagePullSecrets so the kubelet can authenticate to your private registry, and you set imagePullPolicy so it knows when to re-fetch (Always for mutable tags, IfNotPresent for pinned versions). Second, how does it run? The Pod's securityContext decides: runAsNonRoot, runAsUser: 1000, capabilities.drop: [ALL], readOnlyRootFilesystem, allowPrivilegeEscalation: false, seccompProfile: RuntimeDefault. All together, this matches the Pod Security Standards "Restricted" profile. A bug in your app can no longer become a host-level compromise.

Section 5 · Real-world scenarios

Where security defaults pay off

Container escape blocked by readOnly + non-root

A SaaS company's web app had an SSRF leading to RCE. The attacker landed inside the container as UID 1000 with a read-only filesystem and ALL capabilities dropped. They couldn't write a malicious binary, couldn't escalate to root, couldn't pivot to the host. The Pod was deleted; the deploy was hot-fixed by lunch. Same bug class in 2018 took out their staging cluster for a week.

Private registry pull failed silently — fixed with imagePullSecret

A team migrated images from Docker Hub to their company's private Harbor registry. Pods went into ImagePullBackOff. The kubelet logs said "401 Unauthorized." Fix: create a kubernetes.io/dockerconfigjson secret with Harbor credentials and add imagePullSecrets: [name: harbor-creds] to the Pod spec — or better, attach the secret to the namespace's default ServiceAccount so every Pod inherits it. Pods rolled out cleanly.

Pull policy bug: re-pushing a tag silently kept old code running

A team used image: my-app:v2 and imagePullPolicy: IfNotPresent. They re-pushed v2 with a hotfix instead of cutting v2.1. Most nodes still had the old v2 cached and never re-pulled — Pods kept running the old binary on those nodes and the new one on freshly-scaled nodes. Lesson: never re-push a tag. Cut a fresh version, or pin to @sha256:digest.

Pod Security Standards "Restricted" caught a privileged Pod attempt

A new dev tried to deploy a Pod with securityContext: { privileged: true } for "easier debugging." The namespace was labeled Pod-security.kubernetes.io/enforce: restricted. The API server rejected the Pod at admission with a clear message: "violates PodSecurity 'restricted:latest' — privileged is not allowed." Conversation moved from "let's just allow it" to "let's fix the actual debugging tooling." Defaults won.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: Running as root inside a container is safe because the container is isolated.
Truth: Container root + a kernel exploit = host root. Run as a non-root UID by default.
Myth: imagePullPolicy: Always is overkill.
Truth: For mutable tags it's the only safe policy. For digest-pinned images, the question is moot — same digest, same content, every time.
Myth: Pod Security Standards are something you have to install.
Truth: They're built into the Kubernetes API server (since 1.25). Apply via namespace labels — no third-party admission needed.
What does runAsNonRoot: true do?
tap to flip
Tells the kubelet to refuse the Pod if the image's USER is root (UID 0). Forces non-root execution. Pair with runAsUser: 1000 to pin a UID. The image's Dockerfile must include a USER directive that's non-root.
What does readOnlyRootFilesystem: true do?
tap to flip
Makes the container's root filesystem immutable at runtime. An attacker can't drop binaries or tamper with code. Mount an emptyDir at any path that needs writes (logs, /tmp).
Linux capabilities — best practice in a Pod?
tap to flip
drop: ["ALL"], then add only the specific capabilities your app needs (e.g., NET_BIND_SERVICE for ports under 1024). Default of "inherit all root powers" is dangerous.
What is an imagePullSecret?
tap to flip
A K8s secret of type kubernetes.io/dockerconfigjson holding registry credentials. Referenced from a Pod's imagePullSecrets list (or attached to a ServiceAccount). Without it, kubelet can't pull from private registries — Pod stuck in ImagePullBackOff.
Three imagePullPolicy values?
tap to flip
Always — re-checks registry every start (right for :latest). IfNotPresent — use cache if present (default for tagged images). Never — must already be on the node (air-gapped, dev).
What is seccompProfile RuntimeDefault?
tap to flip
A built-in syscall filter that blocks roughly 40–65 dangerous syscalls (kexec, ptrace certain modes, etc.; varies by runtime version) while allowing what normal apps need. Cheap, effective hardening default.
Three Pod Security Standards levels?
tap to flip
Privileged (no restrictions). Baseline (block known-bad: hostNetwork, hostPath, privileged). Restricted (production-grade: non-root, drop caps, no privilege escalation, seccomp). Apply per namespace via labels.
Why pin images to digest (@sha256:...)?
tap to flip
Tags are mutable — someone can re-push :v2 with different content. Digests are immutable — @sha256:abc always means the exact same image. Pinning defeats supply-chain tag swaps and removes pull-policy guesswork.

You inherit a deployment with no securityContext. The image runs as root and writes a few log files into /var/log/myapp. What are the three highest-leverage hardening changes?

answer(1) runAsNonRoot: true + runAsUser: 1000 — prevents root inside the container, which is one host-kernel-bug away from root on the node. (2) capabilities.drop: [ALL] + allowPrivilegeEscalation: false — strips the ~40 Linux superpowers and prevents setuid binaries from regaining them. (3) readOnlyRootFilesystem: true with an emptyDir mounted at /var/log/myapp so the app can still write logs — prevents an attacker from dropping malware on disk. Add seccompProfile: RuntimeDefault for free syscall hardening. The image's Dockerfile may need a USER 1000 directive added.

Your Pod is stuck in ImagePullBackOff. The image is at private-registry.company.com/team/api:v3.2. You can pull it from your laptop with docker login. Why does the Pod fail and what's the fix?

answerYour laptop has a local docker config with credentials; the cluster does not. The kubelet can't authenticate to your private registry. Fix: create a Kubernetes secret of type kubernetes.io/dockerconfigjson with kubectl create secret docker-registry registry-creds --docker-server=… --docker-username=… --docker-password=…, then reference it from the Pod via imagePullSecrets: [{ name: registry-creds }]. For convenience, attach the secret to the default ServiceAccount in that namespace so every Pod inherits it automatically.

Pod status meaning the kubelet failed to pull the image and is backing off retries. Almost always: missing/wrong imagePullSecret or wrong tag.
image digest
