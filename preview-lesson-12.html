Subtopic 1 of 8

What's inside a virtual machine

A software container that looks like a real computer to the OS inside it.

📖

Words you'll meet in this section

Virtual hardware terms — almost all are "v" prefixed versions of physical concepts.

vCPU · virtual CPU hardware
What it is: A virtualized processor presented to the guest OS. The guest sees it as a real CPU. The hypervisor schedules each vCPU onto a physical core when work is ready.
vCPU count is a hard contract. A VM with 4 vCPUs uses up to 4 cores at a time, no more — even if 60 cores are idle on the host.
vRAM · virtual memory hardware
What it is: The memory amount the guest sees. The hypervisor backs it with real host RAM, can over-commit using ballooning / compression / swap if pressured.
Configured size is the cap from the guest's perspective. Real allocation depends on the host's pressure.
vDisk · vmdk config
What it is: A virtual disk file (.vmdk) stored on a datastore. The guest sees it as a real block device through a virtual SCSI/NVMe controller.
A VM can have many vDisks. Each one is a file — that's what makes vMotion / Storage vMotion possible.
vNIC · virtual NIC config
What it is: A virtual network interface attached to a port group on a vSwitch. The guest OS sees it as a real NIC (e.g., VMXNET3 with VMware Tools, E1000 without).
vNIC type matters for performance — VMXNET3 is paravirtualized and ~10× faster than emulated E1000.
VMXNET3 (vNIC type) driver
What it is: VMware's paravirtualized NIC. Requires VMware Tools to drive it. Hits 10 GbE / 25 GbE line rates.
Always pick this for new VMs. The default and right answer.
Paravirtual SCSI · PVSCSI driver
What it is: The paravirtualized storage controller. Replaces the slow emulated LSI Logic SAS controller. High IOPS, low CPU per I/O.
Default for new VMs. Especially important for database / high-IOPS workloads.
Virtual hardware version (vmx-N) config
What it is: A version number (vmx-19, vmx-20, ...) that defines what virtual devices and features are available. Set at VM creation; can be upgraded later but not downgraded easily.
Newer versions unlock newer features but require newer ESXi to run. Pick the latest your oldest host supports.
BIOS · UEFI / EFI config
What it is: Firmware modes. BIOS is older; EFI/UEFI is modern (faster boot, larger disks, secure boot support).
Set at VM creation. Modern Linux and Windows guests should be EFI. Old guests sometimes need BIOS.
vTPM · virtual TPM security
What it is: A virtual Trusted Platform Module — emulates a real TPM chip. Required for Windows 11, BitLocker, certain crypto operations.
If you're deploying Win11 VMs, vTPM is mandatory. Adds a small bit of complexity at creation but unlocks the modern Windows security stack.
VM files (.vmx, -flat.vmdk, .nvram) config
What it is: The files making up a VM. .vmx = config text file. -flat.vmdk = the actual disk data. .nvram = firmware state. Plus log files, snapshot deltas, etc.
A VM is a folder of files on a datastore. Knowing the file types makes operations like clone / move / restore less mysterious.
Guest OS concept
What it is: The operating system installed inside the VM — Linux, Windows, macOS, etc. From the guest's perspective, the VM is a real computer.
vSphere asks you to specify the guest OS at VM creation so it can pick sensible defaults (controller type, BIOS/EFI, recommended hardware).
vSCSI · vNVMe controller hardware
What it is: Virtual storage controllers. The vDisks "hang off" one of these. PVSCSI is the paravirtualized SCSI option; vNVMe is the modern NVMe option.
Default for new VMs is PVSCSI; for very-high-IOPS workloads, vNVMe is the modern choice.

At a glance

A VM is virtual hardware (vCPU, vRAM, vDisk, vNIC, controller, firmware) plus a hardware version. It lives as a folder of files on a datastore.

Subtopic 2 of 8

Creating a new VM from scratch

The wizard, end to end.

📖

Words you'll meet in this section

New-VM wizard vocabulary. Each term shows up as a wizard step.

New VM wizard tool
What it is: The vSphere Client wizard you walk through to create a fresh VM. Right-click any inventory object → New Virtual Machine.
Seven decisions in order. Each step has consequences — picking right at creation saves rework later.
Compute target concept
What it is: The host or cluster the VM will run on. Pick a cluster and DRS auto-places; pick a host and you've fixed it.
In a DRS-enabled cluster, just pick the cluster — DRS knows current load and picks the right host.
Storage policy (at create) config
What it is: Picking SPBM-style instead of selecting a specific datastore. You say "Gold tier"; vSphere finds eligible datastores.
Better than picking by datastore name. Future-proofs against datastore renames and lets storage layers move things around.
Compatibility (vmx version) config
What it is: Which virtual hardware version the VM uses. vmx-20 = ESXi 8.0; vmx-19 = ESXi 7.0u2; etc.
Pick the latest your oldest expected host supports. Modern features need modern versions.
Guest OS hint config
What it is: The wizard asks "what OS will you install?" — Ubuntu 22.04, Windows Server 2022, etc. vSphere uses this to pick sensible defaults.
Wrong hint = wrong defaults (BIOS/EFI, controller type, recommended hardware). Get it right at creation.
CD/DVD drive (ISO mount) config
What it is: A virtual CD/DVD drive that mounts an ISO file from a datastore or content library. Used during install to boot the OS installer.
After OS install, eject the ISO. Don't leave VMs with an ISO mounted long-term — it interferes with vMotion and Storage vMotion.
Right-sizing concept
What it is: Giving a VM the resources it actually needs (vCPU, vRAM, disk) instead of "as much as possible to be safe."
Over-provisioning hurts. Extra vCPUs add scheduling overhead. Over-allocated RAM blocks other VMs from using it. Right-size, monitor, adjust.
CPU ready time concept
What it is: A metric showing how long a VM's vCPU was ready to run but waiting for the hypervisor to schedule it. Measured in ms or %.
High CPU ready means the VM has too many vCPUs (gang scheduling fights) or the host is over-committed. Counter-intuitively, reducing vCPUs often improves performance.
Power state at end concept
What it is: When the New VM wizard finishes, the VM exists but is powered off and has no OS installed.
First-time learners often expect a "power on and use" experience. Reality: wizard creates the shell; you mount an ISO and walk the OS installer to populate it.
Memory reservation · CPU reservation config
What it is: Optional guarantees that vSphere always provides at least this much CPU/RAM to a VM, regardless of host load.
Use sparingly. Reservations reduce flexibility for the cluster. Reserve only when the application truly can't tolerate any contention.
Hot-add capability (at create) tool
What it is: A VM-level checkbox that enables CPU and memory hot-add. Has to be enabled at VM creation (or before first boot) to work later.
If you forget to enable hot-add, you can't grow vCPU or RAM without a reboot later. Always enable for production VMs.
Folder placement config
What it is: The inventory folder a VM lives in (Production, Development, Lab). Used for permissions and tags.
Place the VM in the right folder at creation. Moving later means re-evaluating permissions.

At a glance

Seven ordered decisions from name to ISO. The "paravirt drivers" choice (VMXNET3 + Paravirtual SCSI) is the one that pays back in performance forever.
!
Don't over-provision vCPUs"Just give it 16 vCPUs to be safe" makes things worse, not better. Extra vCPUs add gang-scheduling overhead — the hypervisor has to find that many cores to schedule the VM at all. CPU ready time goes up. Right-size: give the VM what it actually uses, plus modest headroom.
★

What it is

vSphere Client → right-click any inventory object → New Virtual Machine. The wizard's decisions in order:

  • Creation type — New VM, from template, OVF, clone, etc.
  • Name + location — VM name, datacenter, folder.
  • Compute — host or cluster (DRS picks for you in a cluster).
  • Storage — datastore (or pick by storage policy → SPBM).
  • Compatibility — virtual hardware version. Default to the latest supported by all hosts where the VM might run.
  • Guest OS — vSphere uses this hint to pick sensible defaults.
  • Hardware — vCPU, memory, disk, network port group, CD/DVD.

The wizard creates the VM powered off. To install the OS: mount install ISO → power on → walk OS installer → install VMware Tools → configure.

Sizing rules of thumb
  • Don't over-provision vCPUs (extra vCPUs hurt scheduling, not help).
  • Match memory to workload; don't reserve unless needed.
  • Use VMware Paravirtual SCSI controller (performance).
  • Use VMXNET3 virtual NIC (performance).
  • Enable EFI for new Linux/Windows where supported.

Subtopic 3 of 8

Templates and clones

Build a gold VM once; stamp out copies.

📖

Words you'll meet in this section

Cloning vocabulary. Two clone types, plus the customisation system that makes them unique.

Template tool
What it is: A frozen VM you've decided is your gold reference image. Cannot be powered on (intentional — prevents accidental drift). You clone from it.
Build once carefully (OS, patches, agents); stamp out copies. Eliminates fleet drift.
Clone tool
What it is: A copy of a VM (or template) created as a new VM. The new VM has its own files and identity.
5-minute deploys vs hour-long fresh OS installs. The unit of provisioning at scale.
Full clone concept
What it is: A clone with its own complete copy of the source's disks. Independent of the source — no shared files.
Slower to create, more space, but completely independent. The right choice for production server VMs.
Linked clone concept
What it is: A clone that shares the source's base files and only stores its own changes (deltas). Much smaller and faster to create.
Used for VDI desktops, short-lived test/dev pools. Don't use for production — performance and management overhead at scale.
Customization spec tool
What it is: A recipe applied at clone time. Sets unique hostname, IP, MAC, SID (Windows), domain join, time zone for the new clone.
Without it, every clone starts as an exact bit-copy of the template — same identity. AD chaos. With it, each clone is uniquely identified at first boot.
SID · Security Identifier security
What it is: A unique Windows identifier for a machine. Used by AD, file ACLs, and many Microsoft services. Two machines with the same SID = breakage.
Customization spec generates a new SID per clone. Without it, AD treats all clones as the same machine — disaster.
Sysprep tool
What it is: Microsoft's tool for "generalising" a Windows install — stripping the SID and machine-specific bits so it can be reseeded. Customization spec runs Sysprep under the hood.
If you've ever cloned Windows and seen "this image isn't sysprepped," now you know. Customization spec handles it for you.
Convert to template concept
What it is: The action of turning a regular VM into a template. Right-click the VM → Convert to Template. Reversible (Convert to VM).
Templates can't be powered on. The "convert" step locks in the gold state.
Template lifecycle concept
What it is: Templates need updating — patches, agent versions, baseline changes. Convert to VM, power on, patch, convert back to Template.
"Set and forget" templates rot. Schedule monthly or quarterly updates so newly-cloned VMs aren't deployed pre-vulnerable.
Identity at first boot config
What it is: The phase when a fresh clone first boots and applies the customization (sets hostname, gets IP via DHCP/static, joins domain).
Most operations happen here. If customization fails, this is when. Test customization specs in a lab before bulk deployment.
Customization spec library tool
What it is: A collection of reusable customization specs — one per OS family typically (Linux DHCP, Windows-static-domain-joined, etc.).
Build the spec once, reuse forever. Don't recreate for each clone operation.
Drift (fleet) concept
What it is: Configuration differences across VMs that should be identical — different patch levels, missing agents, slightly different settings.
Templates fight drift on day 0. Without templates, every VM starts as a fresh manual install — and small choices vary by who installed it.

At a glance

Template + customization spec = many uniquely-identified clones. No fresh OS installs, no drift, one minute of clicks.
!
Always apply a customization spec when cloningWithout it, you get N exact bit-copies of the template — same hostname, same MAC, same SID. AD goes haywire; DHCP collisions; certificates get confused. Build a customization spec per OS family, apply every time.
★
Update templates on a schedule"Set and forget" templates rot. Patches don't get applied; agent versions fall behind; baseline changes don't propagate. Convert to VM, patch, convert back to Template — monthly or quarterly. Track in your runbook.
i

Subtopic 4 of 8

Content libraries

Share templates and ISOs across vCenters.

📖

Words you'll meet in this section

Content library terminology — mostly publish/subscribe vocabulary.

Content library tool
What it is: A vCenter-managed catalog of templates, ISOs, OVF appliances, scripts. Defined per-vCenter, shareable across vCenters.
The "share gold images across vCenters" mechanism. Avoids the "every vCenter has its own slightly-different template" drift.
Local library config
What it is: A library that lives on this vCenter only. You upload to it directly. Can be published.
The simplest type. Most environments have one local library at a central vCenter and use it as the source for subscribers.
Subscribed library config
What it is: A library that mirrors a published library elsewhere. Read-only locally; content syncs from the publisher.
Multi-site shops use this. Each vCenter gets a subscribed library; new content published centrally appears at every site.
Published library config
What it is: A local library configured to allow subscribers to pull from it. The "publisher" side of pub-sub.
Toggle a setting: "publish externally." Generates a URL subscribers register against.
Subscription URL protocol
What it is: The HTTPS URL a published library exposes — e.g., https://vc-east/cls/vcsp/lib/abc123/lib.json. Subscribers register this URL.
The "address" of the publisher. Typically authenticated with a token.
Sync mode (on-demand vs scheduled) config
What it is: Two ways subscribers can pull. On-demand — only when needed (saves bandwidth). Scheduled — periodic sync (always current).
On-demand for occasional clones; scheduled for environments that always want the latest gold image ready.
Library item tool
What it is: One thing in a library — a template, an ISO, an OVF appliance. Each has its own metadata.
Versioned independently. You can update the Ubuntu template without touching the ISOs.
Storage backing (library) concept
What it is: A datastore each library uses to store its actual content. Often a dedicated NFS share or small VMFS volume.
Don't put libraries on the same datastore as production VMs — capacity reasons.
Local copy (subscribed) concept
What it is: When a subscriber syncs an item, the actual files come down to the subscriber's storage. Provisioning then uses the local copy.
Avoids cross-WAN traffic at deploy time. The first sync moves the bytes; subsequent deploys are local.
Auto-sync tool
What it is: Subscribed library setting to download items automatically as they appear at the publisher (vs on-demand at first deploy).
Auto-sync = always have content ready locally; uses more bandwidth + storage. On-demand = save bandwidth but slower first deploy.
ISO storage (vs scattered ISOs) config
What it is: Putting ISOs in a content library instead of randomly across datastores. Provides versioning and cross-vCenter availability.
"Where's the Ubuntu 22.04 ISO?" → "Content library." Beats searching across datastores or trying to remember which path.
OVF appliance (in library) tool
What it is: Vendor-shipped OVF/OVA packages stored in the library. Deploy from library = deploy from local copy.
Many enterprises pre-cache vendor appliances (backup proxies, monitoring agents) in the library so they're always ready to deploy.

At a glance

Publish once at the central vCenter. Every other vCenter has a subscribed library and provisioning uses the local copy.
★
One library, many vCentersIf you have more than one vCenter, set up a content library hierarchy. Publish once on the central vCenter; subscribe everywhere else. New gold image gets promoted to all sites in one operation.

VM snapshots

Short-term safety net. Not a backup.

📖

Words you'll meet in this section

Snapshot terminology. Critical to know — this is one of the most-misused vSphere features.

Snapshot tool
What it is: A point-in-time saved state of a VM (disks, optionally memory + settings). Lets you roll back if a change breaks something.
Useful as a short-term safety net. Has real ongoing performance and capacity costs the longer you keep it.
Delta file (-000001.vmdk) config
What it is: The file that holds writes made AFTER a snapshot was taken. The base disk is frozen; new writes go to the delta.
Each new snapshot adds another delta to the chain. Read paths get longer; deletion (consolidate) costs more, the longer it sits.
Snapshot chain concept
What it is: The sequence of base + multiple deltas that make up a VM with snapshots. Reads check the latest delta first, fall back through the chain.
Long chains = slow reads + slow consolidation. Don't stack snapshots.
Memory snapshot config
What it is: An option when taking a snapshot — also save the VM's memory state. Restoring brings the VM back to running mid-process.
Slower to take, restores to live state. Without it, restore = reboot.
Quiesce (snapshot option) config
What it is: An option that asks VMware Tools to pause the guest's filesystem (and VSS-aware apps) before snapshotting. Result is application-consistent.
Without quiesce, the snapshot is "crash-consistent" — like recovering after a power-loss. With it, restored data is clean.
Crash-consistent concept
What it is: A snapshot taken without flushing apps first. On restore, apps see "what if you yanked the power cord" — files may be half-written.
Most apps can recover, but some (databases) may need repair tools. Crash-consistent is the default; quiesce upgrades it to application-consistent.
Application-consistent concept
What it is: A snapshot taken after VMware Tools paused the guest's filesystem and VSS-aware apps flushed their state. On restore, everything is clean.
The right kind of snapshot for production. Requires VMware Tools running.
Revert tool
What it is: The action of rolling back to a snapshot. Discards all changes made after the snapshot was taken.
Use when a change went wrong. The whole point of taking a snapshot is being able to revert.
Delete · Consolidate tool
What it is: Two related actions. Delete commits a snapshot's deltas back into the base. Consolidate handles cases where deltas are still around but the snapshot is gone.
Both reduce chain length. The "I'm done with this snapshot, merge it" operation. Can take a long time on large or stale snapshots.
Snapshot lifetime config
What it is: The time between taking and deleting a snapshot. Best practice: 24-72 hours.
Snapshots older than days accumulate huge deltas. Deleting them later causes long stuns and possibly hours of I/O.
Snapshots are NOT backups security
What it is: A common misconception. Snapshots depend on the VM's base disks; if the datastore fails, both base AND snapshots are lost together.
Real backups go to separate, independent storage (SFTP target, dedicated backup appliance, off-site). Snapshots are only short-term safety, not data protection.
Snapshot alarm config
What it is: A vCenter alarm that fires when a snapshot has existed for too long (configurable threshold).
Catches forgotten snapshots before they grow into a problem. Always have these alarms in production.

VMware Tools

Bridge between guest and host. Every production VM should have it.

📖

Words you'll meet in this section

If any term feels unfamiliar, here's the plain-language version. Each one shows up later in this subtopic.

VMware Tools tool
What it is: A small package of drivers and a background program that you install inside the VM (the guest OS). It runs alongside your applications.
Without it, the VM and the host can't really cooperate — they speak different languages. With it, they're in sync.
Driver driver
What it is: A piece of software that lets the operating system talk to a piece of hardware. Every NIC, disk controller, and graphics card needs one.
If the driver is missing or wrong, the hardware works slowly, doesn't work at all, or only at minimum capability.
Paravirtualized driver driver
What it is: A driver that knows it's running inside a VM. Instead of pretending the virtual NIC or disk is a real piece of hardware (which is slow), it talks to the hypervisor directly using a fast, purpose-built path.
~10× faster than emulated drivers. This is why production VMs need VMware Tools — without it, the guest falls back to slow emulated drivers.
VMXNET3 driver
What it is: The paravirtualized network card driver shipped with VMware Tools. Replaces the slow, emulated "E1000" NIC the guest sees by default.
VMXNET3 hits 10 GbE / 25 GbE line rate; emulated E1000 tops out around 1 Gbps with high CPU cost.
Paravirtual SCSI (PVSCSI) driver
What it is: The paravirtualized storage controller driver. Replaces the emulated "LSI Logic SAS" controller for disk I/O.
Higher IOPS, lower CPU per I/O. Required for high-performance database VMs and modern storage.
Daemon tool
What it is: A program that runs continuously in the background, waiting for requests or events to handle. On Linux it shows up as a service; on Windows as a Windows Service.
VMware Tools includes a daemon (vmtoolsd) that's always listening for messages from the hypervisor — "shut down please," "what's your IP?", "freeze the disk for a snapshot."
Quiesce config
What it is: "Quiet down" the file system before a snapshot. Tools tells running apps to flush in-memory data to disk and pause writes for a moment so the snapshot captures a consistent state.
Without quiesce, a snapshot is like a power-loss — half-written files, partial database transactions. With it, the snapshot is clean and recoverable.
VSS (Volume Shadow Copy Service) tool
What it is: Windows's built-in coordination service for taking consistent backups. Apps like SQL Server register as "VSS-aware" so they can flush properly when asked.
VMware Tools talks to VSS so a snapshot pauses SQL Server / Exchange / IIS at a clean point. Without Tools, snapshot = "what if you yanked the power cord."
Heartbeat config
What it is: A regular "I'm alive" signal Tools sends from inside the guest to ESXi. Default: every few seconds.
vSphere HA's "VM Monitoring" can restart a VM if the heartbeat stops — a guest OS hang where the VM is "running" but actually frozen. Without Tools = no heartbeat = no automatic recovery from a hung guest.
Crash-consistent vs application-consistent config
What it is: Two qualities of snapshots/backups. Crash-consistent = like recovering after a power loss; the disk is whole but apps may have lost in-flight work. Application-consistent = apps were told to flush first; restore is clean.
Application-consistent only happens with VMware Tools quiescing the guest first. The difference can mean "5 minutes of restore" vs "8 hours of database repair."
open-vm-tools tool
What it is: The open-source version of VMware Tools, packaged inside Linux distributions (Ubuntu, RHEL, Debian, ...). Same supported codebase, but installed via the distro's package manager.
For Linux VMs, this is the modern preference: apt install open-vm-tools and the distro keeps it updated. No mounting an ISO from vSphere.
Hot-plug concept
What it is: Adding or removing hardware while a system is running, without powering off. Modern OSes support hot-plug for CPU, memory, disk, and NIC.
VMware Tools makes hot-plug events visible to the guest properly — without it, the OS may not see the new RAM or NIC until reboot.

How VMware Tools sits inside a VM

VMware Tools is split into two parts inside the guest: drivers (purple — VMXNET3, PVSCSI, balloon) that handle fast I/O, and the daemon (green — vmtoolsd) that listens for messages from the host over a built-in channel called VMCI. The drivers replace slow emulated hardware; the daemon makes the host and guest cooperate.

Operational vocabulary — what you'll do to running VMs day in, day out.

Day-2 ops concept
What it is: Everything you do to a VM AFTER it's deployed and running. Day 1 = create. Day 2+ = operate.
Most of a VM's life is day-2. Modern vSphere lets most growth and relocation happen without downtime — that's what hot-add and vMotion are for.
Power Off (hard) tool
What it is: Hard pull-the-plug. Equivalent to yanking the power cord. Apps die mid-write; FS may need recovery.

Shut Down Guest OS (graceful) tool

Suspend / Resume tool
What it is: Pauses the VM and writes its memory to disk. Resume picks up where it left off — no reboot.
Like closing a laptop lid. Useful for temporarily freeing host resources without a reboot. Less common in production servers.
Hot-add concept
What it is: Adding CPU, memory, disk, or NIC to a running VM without shutting it down. Requires the VM's hot-add capability to be enabled AND the guest OS to support hot-plug.
Modern Linux and Windows Server support most hot-add cases. Saves a reboot for routine grow operations.
Hot-extend (disk) concept
What it is: Growing an existing virtual disk while the VM is running. The guest OS still has to extend the partition / filesystem.
First the vSphere side (disk size up); then the guest side (LVM extend, Disk Management extend, etc.). Two steps.
vMotion (recap) tool
What it is: Live-migrate a running VM's compute (where it executes) to a different host. Files stay on the same datastore.
No downtime; one missed packet at most. Used for hardware maintenance, load balance.
Storage vMotion tool
What it is: Live-migrate a running VM's files (its disks) to a different datastore. Compute stays on the same host.
Used for storage maintenance, datastore evacuation, or moving to faster storage. No downtime.
Combined vMotion tool
What it is: Doing vMotion + Storage vMotion at once — moving both compute and storage in a single operation. Used for full cluster / datacenter / vCenter relocation.
Modern vSphere can do this seamlessly. Cross-cluster, cross-datacenter migrations work the same as a basic vMotion.
Edit Settings config
What it is: The vSphere Client menu where you change a VM's hardware: resize CPU/RAM/disk, add/remove devices, change port group, change firmware.
Where most day-2 hardware changes happen. Right-click VM → Edit Settings.
Remove · Delete from disk tool
What it is: Two cleanup options. Remove from inventory — vCenter forgets the VM, files stay on datastore. Delete from disk — vCenter forgets AND files are deleted.
Be careful — Delete from disk is irreversible. Always double-check before clicking.

At a glance

Two everyday day-2 ops on a running VM: hot-add memory (no reboot) and vMotion (live-migrate). Modern vSphere lets most growth and relocation happen without downtime.

Answer
