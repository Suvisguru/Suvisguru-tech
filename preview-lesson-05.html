Words you'll meet in this lesson

vSphere touches CPU, memory, storage, network, and GPU — each has its own technical vocabulary. Plain language for each.

vCPU · pCPU hardware
What it is: Virtual CPU (what the guest sees) vs Physical CPU core (the real hardware). The hypervisor schedules each vCPU onto a pCPU when work is ready.
A VM with 4 vCPUs needs 4 pCPUs available simultaneously to actually run. Over-allocating vCPUs makes scheduling worse, not better.
NUMA hardware
What it is: Non-Uniform Memory Access. On multi-socket servers, each CPU has its own attached memory. Accessing the local memory is fast; the other socket's memory is slower.
vSphere is NUMA-aware: it tries to keep a VM's vCPUs and memory on the same NUMA node. Cross-NUMA traffic is the silent performance killer.
Hyperthreading · SMT hardware
What it is: Intel's term (AMD calls it SMT) for "two logical CPUs per physical core." Most x86 CPUs run with hyperthreading enabled.
Doubles the apparent CPU count. ESXi sees 2× cores in vSphere — useful for scheduling, but the second thread isn't a full extra core in practice.
Memory overcommit concept
What it is: Configuring more total VM memory than the host has physically. Works because most VMs don't use all their RAM all the time.
Lets you run more VMs per host. Risky if heavy: vSphere has tools (TPS, ballooning, swap) to recover memory under pressure, but performance suffers.
TPS · Transparent Page Sharing concept
What it is: ESXi finds identical memory pages across VMs (e.g., same OS kernel) and stores only one copy. Saves RAM with no app changes.
First line of defence under memory pressure. Mostly silent; sometimes disabled for security reasons (it's a side-channel risk).
Ballooning concept
What it is: Under memory pressure, ESXi asks VMware Tools inside each guest to "balloon" — allocate memory inside the guest so the guest gives up unused pages back to the host.
A graceful recovery — apps that don't need RAM let it go, apps that do keep it. Requires VMware Tools running.
Memory compression · swap concept
What it is: Last-resort memory recovery. Compress memory pages (still in RAM, just smaller); finally, swap pages out to disk.
By the time you're swapping, performance is bad. The presence of swap activity in vSphere metrics is a strong "add memory" signal.
vGPU hardware
What it is: Virtual GPU. Lets multiple VMs share one physical GPU's processing power. Used for VDI graphics, ML training, professional graphics workstations.
Lets a single $$$ GPU serve many users. Requires NVIDIA AI Enterprise or similar licensing.
DirectPath I/O · PCI passthrough hardware
What it is: Giving one VM exclusive access to a physical PCI device (a GPU, NIC, FPGA). The device disappears from ESXi's pool and is owned by the VM.
Used when a VM needs direct hardware (CUDA, certain NIC features). Costs flexibility — that VM can no longer vMotion.
vSphere Client · Host Client · DCUI · CLI tool
What it is: The four management surfaces. vSphere Client (via vCenter) for fleet ops; Host Client (per-host browser UI); DCUI (yellow console) for recovery; CLI (esxcli, PowerCLI) for scripting.
Different tool for different moment. vSphere Client is daily life; the others are when something's broken or you're automating.
Resource scheduler concept
What it is: The part of ESXi that decides which vCPU runs on which pCPU at any moment. Aware of NUMA, hyperthreading, current load.
Your VMs share hardware. The scheduler is what makes that sharing fair and efficient. Most of the time it just works.
SDDC context (recap) concept
What it is: vSphere is the COMPUTE layer of VMware's Software-Defined Data Center. vSAN adds storage; NSX adds networking. Detailed in lesson 7.
Knowing where vSphere fits in the bigger picture helps when you start touching vSAN and NSX later.

Section 1 · Concept

Three angles on vSphere

This lesson zooms in on vSphere from three angles: where it fits in the bigger picture, how human operators talk to it, and what it actually does with hardware.

Where it fits

vSphere is the compute layer of the SDDC. Lesson 7 introduced the four-layer SDDC: compute, storage, network, management. vSphere is the compute layer. Storage virtualization (vSAN), network virtualization (NSX), and the management plane (vCenter, vRealize) sit alongside it. In public cloud, vSphere is also the foundation of services like VMware Cloud on AWS, Azure VMware Solution, and Google Cloud VMware Engine — each is a vSphere environment running on someone else's hardware.

How you access it (four interfaces)

  • vSphere ClientThe vSphere Client is VMware's modern web-based UI for managing a vSphere environment. It's served by vCenter Server and accessed through a browser. This is where day-to-day vSphere work happens.

    The modern web UI, served by vCenter. Browser-based. This is where day-to-day work happens.

  • VMware Host Client

    A per-host web UI served by ESXi itself. Used when vCenter is unavailable or when you need direct host management.

  • DCUIDirect Console User Interface — the keyboard-and-monitor console you see when you stand in front of an ESXi host. Yellow background, F2 to log in. Used for initial network configuration and emergency recovery.

    The yellow-on-black console you see standing in front of an ESXi host with a keyboard. F2 to log in. Initial setup and emergencies.

  • CLI & API

    PowerCLIPowerCLI is a PowerShell module from VMware. It lets you script vSphere operations — provision VMs, automate snapshots, integrate with CI/CD. The standard tool for managing vSphere at scale. (PowerShell), esxcli (on ESXi), govc (Go), and the REST API. For automation and scripting.

What it touches (five hardware resources)

vSphere coordinates CPU, memory, network, storage, and GPU. Each is a different kind of hardware with its own quirks, and vSphere abstracts each one so VMs can use it transparently. The hero widget above lets you click through each.

Section 2 · Before and after

Three vendor tools or one pane of glass

Before vSphere (per-resource silos)

Three (or four) tools, three vocabularies

CPU and memory tracked through OS tools on each individual server. Network in the network team's switch console. Storage in EMC or NetApp's SAN tool. GPU often ignored or required physical access.

Adding a new application meant logging into three or four separate tools and wiring them together by hand.

After vSphere

One inventory, one vocabulary

The vSphere Client shows every host, VM, datastore, virtual network, and GPU allocation in one inventory. The same UI that creates a VM picks the storage, picks the network, attaches a GPU.

Honest caveat: when something specifically goes wrong at the storage or network layer, you still need the deeper tooling for that domain.

Section 3 · The analogy, extended

Control rooms and building services

vSphere is the building's full operations system.

Control rooms (the UIs): a modern apartment building has a front desk (vSphere Client), an individual unit panel in each apartment (Host Client), a basement maintenance terminal (DCUI), and a master control system the engineers script against (CLI / API). Different audiences, different access levels, all controlling the same building.

Building services (the hardware): heating and cooling (CPU and memory), water and waste (storage), electricity and internet (network), and specialized hookups for residents who need them (GPU). The superintendent (vSphere) manages all of them as one coordinated system.

The coordination is what's valuable. Five separately-managed services is exhausting. Five managed as one is calm.

vCenter has just crashed. An admin needs to verify that one critical ESXi host's VMs are healthy. Which UI should they use?

Section 4 · At your level

The same idea, two depths

vSphere is the brain of the apartment building. There are four ways to talk to it — walk into the lobby, knock on a single apartment door, go to the basement and read the boiler, or text the engineer who controls everything from a distance. The brain takes care of five things in every apartment: heat (CPU), water tank (memory), water taps (storage), wires (network), and special hookups for some apartments (GPU).

vSphere is VMware's compute platform — the part of the SDDC that virtualizes servers. Storage virtualization (vSAN), network virtualization (NSX), and the management plane (vCenter) sit alongside it.

You access vSphere through one of four interfaces: the vSphere Client for day-to-day management, the VMware Host Client for per-host management, the DCUI for emergencies, and the CLI / API layer for automation.

vSphere manages five hardware resources. CPU scheduling uses Intel VT-x or AMD-V and respects NUMANUMA (Non-Uniform Memory Access) means a CPU can access its own local memory faster than memory attached to another CPU. vSphere is NUMA-aware: it tries to keep each VM's CPU and memory on the same NUMA node. topology. Memory is sliced per VM with techniques like transparent page sharing and ballooning. Network presents NICs through virtual switches and port groups. Storage abstracts physical storage into datastores. GPU support comes in two flavors: pass a whole GPU to one VM (DirectPath I/O), or slice a GPU into vGPUsvGPU (NVIDIA's term; AMD has similar) is the technology that slices one physical GPU into multiple virtual GPUs that many VMs share. Used for VDI environments where each user needs some GPU but not the whole card. shared by many VMs.
