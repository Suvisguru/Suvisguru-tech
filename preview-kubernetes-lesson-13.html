Module 3 · Lesson 13 · how a cluster is built

Cluster Architecture · Control Plane & Nodes

A cluster is two halves: a control tower (control plane — decides) and several terminals (worker nodes — execute). One radio between them: the API server.

🚨 The 3 AM Nightmare

A network partition isolates 2 of your 3 etcd nodes from each other. Quorum is lost. The cluster goes read-only — no new Pods scheduled, no failover, no Deployments updated. Anything currently running keeps running, but the cluster is frozen in amber. You can't even apply the fix. This lesson is about the brain–muscle split that makes Kubernetes work, and where it breaks if you're not careful.

🎯 If you remember nothing else: Control plane decides; nodes execute. Five control-plane components, three node components, one API door between them. Memorise the 6-hop Pod-creation flow.

Section 1 · Concept

Two halves: brain and muscle

A Kubernetes cluster is two distinct halves: a control plane (the brain — decides what should run) and one or more worker nodes (the muscle — actually runs the containers). They are different machines with different jobs, talking over a network.

The control plane. Five processes, usually on dedicated control-plane nodes (and never running your app workloads):

• API server (kube-apiserver) — the only door into the cluster. Every read, write, watch, authentication request goes here. Stateless, horizontally scalable.
• etcd — strongly-consistent key-value store holding the entire cluster state. Source of truth.
• Controller manager (kube-controller-manager) — runs dozens of reconcile loops (Deployment, ReplicaSet, Node, Endpoints, ServiceAccount, Job…). Each watches the API for desired state, compares to actual, takes action.
• Scheduler (kube-scheduler) — picks which node runs each newly-created Pod, based on resource requests, taints, affinity, topology.
• Cloud controller manager (cloud-controller-manager) — bridges K8s to the cloud (load balancers, persistent disks, node lifecycle). Optional on bare metal.

The worker nodes. Every node — control plane or worker — runs three things:

• kubelet — the node agent. Watches the API server for Pods assigned to its node, tells the runtime to start/stop them, reports node status.
• kube-proxy — manages the local network rules (iptables, IPVS, eBPF) that implement Services on this node.
• Container runtime — containerd, CRI-O, or runc-based — actually starts and stops the containers (Lesson 09).

The constraint. Nothing talks directly to etcd. Nothing talks directly between controllers. All communication goes through the API server. That single-door discipline is the architectural constraint that makes the rest of K8s possible: scaling, watching, RBAC, audit, admission — every mechanism is built on top of "everything is a typed object you read/write through the API server."

Section 1.5 · The 5 control-plane components

Each one, one tile

📡

1 · API server

the only door

REST + watch endpoints. Validates, authenticates, persists to etcd, fans out watches. Stateless · scales horizontally.

🗄️

source of truth

Strongly-consistent KV store. Holds every K8s object. Odd-member quorum (3 or 5). Lose majority → cluster read-only.

🔄

3 · Controller manager

reconcile loops

Bundle of loops: Deployment, ReplicaSet, Node, Endpoints, Job, etc. Each watches API for desired vs actual, takes action.

🎯

4 · Scheduler

picks the node

For each unscheduled Pod: filter feasible nodes, score them, pick highest. Resource fit · taints · affinity · topology.

☁️

5 · Cloud controller mgr

bridge to provider

Translates K8s objects into AWS/GCP/Azure API calls (LBs, disks, node info). Optional on bare metal.

All five compound: API + etcd + controllers + scheduler + cloud-cm = the brain that survives any single failure.

Section 1.7 · Worker node components

Three things on every node

The control plane decides; these three actually do the work on each node.

🛠️

kubelet

Node agent. Watches API server for Pods bound to this node, asks the runtime to start them, reports status (CPU, memory, disk, conditions) back. Also runs liveness/readiness probes.

🚦

kube-proxy

Implements Kubernetes Services on this node — programs iptables, IPVS, or eBPF rules so traffic to a Service's virtual IP gets load-balanced across the right Pod IPs.

🐳

container runtime

containerd, CRI-O, or runc-based. Pulls images, manages container lifecycle, sets up namespaces/cgroups. Speaks the CRI (Container Runtime Interface) to the kubelet.

Together they form the "node runtime" — the per-machine subsystem that turns control-plane decisions into actually running containers.

Section 1.8 · How a Pod gets created

The end-to-end flow, in 6 hops

This is the request path that every Kubernetes object follows. Memorize it; it explains 90% of the questions you'll have about K8s behavior.

1kubectl→API serverPOST /api/v1/namespaces/default/Pods (your YAML)
2API server→etcdValidate, authorize, run admission, persist Pod object
3scheduler→API serverWatches new Pods · picks node-3 · POSTs Binding
4kubelet (n3)→API serverWatches Pods bound to node-3 · sees the new Pod
5kubelet→runtimeCRI: pull image, create container, start container
6kubelet→API serverPATCH Pod status: phase=Running, containerStatuses

Notice every step routes through the API server. The scheduler never tells the kubelet anything directly — it writes a Binding, the kubelet's watch picks it up. That's the design.

Section 1.9 · HA control plane

What "production-ready" means

One control-plane node is fine for learning. Production needs replication. Two pieces matter: API server replication and etcd quorum.

API server: 3+ replicas behind a LB

API server is stateless — etcd holds all state. Run 3 (or 5) instances behind a load-balancer. Any can serve any request. Lose one, the LB routes around it. Controller manager and scheduler also run with leader election: only one of the three is active at a time, others stand by; if the leader dies, a new one wins the lease in etcd within ~15s.

etcd: odd-member quorum

[ deep dive — skip if new ]etcd uses Raft. Writes need a majority (quorum). 3 members tolerate 1 failure; 5 members tolerate 2. Always odd (4 members tolerate the same as 3 — 1 — and you've added latency). Co-locate one etcd per CP node, or run them as a separate cluster (stacked vs external topology). Leader-election internals (term numbers, vote requests, log replication) live in the Raft paper if you want them; for operating Kubernetes, "lose majority → read-only" is the load-bearing fact.

Lose quorum (e.g., 2 of 3 etcd dead) and the cluster goes read-only — gets work, no writes, no failover, no scheduling. This is why etcd backups and quorum-zone planning matter.

Section 1.10 · Managed vs self-managed

Who runs the control plane?

Managed (EKS · GKE · AKS · DO · Linode)

Cloud provider runs the control plane for you. You manage worker nodes (or use managed node pools too). Pay ~$73/cluster/mo (GKE / EKS) or sometimes free (AKS, GKE Autopilot). Patching, etcd backups, HA — all theirs. Best for teams that aren't full-time cluster operators.

Self-managed (kubeadm · kops · k3s · RKE)

kubeadm sets up a vanilla CP on your VMs. k3s is a tiny K8s for edge/IoT. RKE automates the install. You own everything: etcd, certs, HA, upgrades. More control, more ops time. Bare-metal, edge, regulated environments, or 100+ clusters where managed-per-cluster cost adds up.

For most teams under 50 engineers: managed. The control plane is undifferentiated heavy lifting and the hourly rate is silly cheap compared to engineering time spent on it.

Section 2 · Before & After

Single-machine app vs cluster-native app

The same workload, before and after the control-plane / nodes split.

Left: one process is your scheduler, your state store, and your app — and your single point of failure. Right: those concerns are factored apart, and any node can disappear without the cluster noticing for more than a few seconds.
Before · single machine

One process running your app + a "scheduler" + a "config store" all together. Process dies → everything dies. Adding a second machine? Now they need to coordinate state — and you've invented half of K8s, badly.

After · CP + nodes

Control plane is one logical unit (HA-replicated underneath). Worker nodes are interchangeable cattle. The Pod you deployed yesterday survives a node failure tonight, because the control plane noticed and recreated it elsewhere.

What changed: separation of concerns. The control plane decides; the nodes do. Every K8s feature builds on top of this split.

Section 3 · Analogy

The airport: tower + terminals

A Kubernetes cluster is an airport with one control tower and several terminal buildings. The tower (the control plane) is where all the decisions happen. The terminals (worker nodes) is where aircraft (your Pods) actually park, board, and fly.

Inside the tower, five rooms: radio dispatch at the top window (API server — every plane radios here, nobody else has authority to broadcast); flight plan archive in the basement (etcd — every plane that ever existed, every assignment, written down); specialist controllers (controller manager — "always-3-planes-on-this-route", "this-plane-needs-a-gate", "node-just-went-dark"); runway assignment desk (scheduler — picks which terminal/runway each plane goes to); and a cloud liaison (cloud-controller-manager — radio link to outside power, fuel, navigation aids).

Inside each terminal, three roles: gate manager (kubelet — receives assignments, executes them, reports back); ground crew router (kube-proxy — directs taxiing planes via the local taxiway map); tug operator (container runtime — actually moves the planes).

The tower never directly grabs a plane. It writes the assignment to the archive; the gate manager picks it up. The gate manager never calls another terminal directly; it goes through the tower's radio. That single-door discipline is what lets the airport scale to hundreds of terminals.

Section 4 · Two-level explanation

Two ways to hear it

ELI5 · explain like I'm 5

A control tower talks to terminals. Terminals park planes. The tower decides where each plane goes, the terminal makes it happen. If a plane crashes, the tower puts up a new one somewhere else. If a terminal closes, the tower moves all its planes to another terminal.

ELI10 · explain like I'm 10

A K8s cluster has a brain (control plane) and muscles (worker nodes). The brain has 5 parts: an API server (the only door in), etcd (where state is stored), a controller manager (lots of small loops keeping desired state matching reality), a scheduler (picks which node runs which Pod), and a cloud-controller-manager (bridges to AWS/GCP/Azure). Each worker node runs kubelet (the node agent that runs your Pods), kube-proxy (network rules for Services), and a container runtime (containerd or CRI-O). When you kubectl apply, your YAML goes to the API server, which writes it to etcd; the scheduler picks a node and writes the binding back; the kubelet on that node reads its assignment and tells the runtime to start the container. Everything talks through the API server, nothing talks directly.

Section 5 · Real-world scenarios

Where architecture decisions show up in production

etcd lost quorum during a network partition — cluster went read-only

A 3-member etcd cluster across three AZs hit a network partition that isolated 2 of 3 nodes from each other. Quorum couldn't be achieved → etcd switched to read-only → API server could serve gets but not writes → no new Pods scheduled, no failover, no Deployments updated. Lesson: 5-member etcd survives 2 failures; 3-member only survives 1. Cross-AZ etcd needs careful quorum-zone planning.

API server overloaded by an aggressive controller

A custom controller was watching all Pods and re-LISTing every second. At 8000 Pods, that hammered the API server with hundreds of MB/s of JSON. API server fell behind on writes, kubectl timed out cluster-wide. Fix: use proper informers (LIST once + WATCH for diffs) and respect server-side rate limits. Standard controller-runtime libraries do this for you.

Managed CP (GKE) saved a startup from etcd backup nightmares

A 12-engineer startup ran self-managed K8s on EC2 with kubeadm. They spent ~30 hours/month on etcd backups, version upgrades, and CP patching. Migrated to GKE. Engineering time on cluster ops dropped to zero. They paid $73/month per cluster; they paid one engineer ~$14k/month before. Math is what it is.

Pricing as of 2024: GKE Standard ~$73/cluster/mo; AKS free; EKS $73/mo; GKE Autopilot bills per-Pod. Check your provider's current pricing — the conclusion (managed beats self-managed at this team size) doesn't change.

kube-scheduler bug pinned all Pods to one node

A scheduler bug in a specific minor version assigned all newly-created Pods to the same node, ignoring resource constraints. That node OOM'd and got cordoned. Subsequent Pods went to the next node, also OOM'd. Cascading failure. Workaround: deploy a second scheduler (multi-scheduler is supported via schedulerName on the Pod spec) until the upstream fix landed. Real fix: upgrade.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: The control plane runs your apps.
Truth: Control plane decides; worker nodes run. In production they're usually different machines, often with taints to keep workloads off the CP.
Myth: Components talk directly to each other.
Truth: Everything goes through the API server. The scheduler doesn't tell the kubelet anything — it writes a Binding; the kubelet's watch picks it up.
Myth: Self-managed Kubernetes is "more flexible."
Truth: For most teams under 50 engineers, managed (EKS / GKE / AKS) is a no-brainer. Self-managed wins for specific cases: bare metal, regulated environments, or 100+ clusters.

You run kubectl apply -f Pod.yaml. Trace the request end-to-end: which components touch it, in what order, and what does each do?

answer(1) kubectl POSTs the YAML to the API server, authenticated with your kubeconfig. (2) The API server runs admission, validates, applies defaults, and writes the Pod object to etcd. (3) etcd persists across its quorum. The API server returns 201 Created. (4) The scheduler is watching the API server for unscheduled Pods. It evaluates fit, picks node-3, POSTs a Binding to the API server. (5) The kubelet on node-3 is watching the API server for Pods bound to itself. It sees the new Pod, reads the spec, and asks the container runtime (containerd) to pull the image and start the container. (6) The kubelet PATCHes Pod status (Running) back to the API server, which writes it to etcd. kubectl get Pod sees Running. Notice: nothing talks directly to etcd; nothing talks between scheduler and kubelet directly. The API server is the only door.

Your 3-member etcd is split across three AZs. A network partition isolates AZ-1 from AZ-2 and AZ-3 (but AZ-2 and AZ-3 still talk). What happens? How would 5 members have changed it?

answerAZ-2 and AZ-3 still have 2 of 3 etcd members → quorum (majority) is achieved → those 2 form the active etcd group, accept writes, cluster keeps running. AZ-1's etcd is isolated and goes read-only. So a 3-member etcd survives this partition fine, as long as 2 members remain together. But: if the partition isolated EACH AZ separately, no group has 2 members and the whole cluster goes read-only. With 5 members spread 1-2-2 across 3 AZs, two AZs together have quorum. Bottom line: 3-member tolerates 1 failure, 5-member tolerates 2, but topology of the partition matters as much as the count.
