Subtopic 1 of 6

Installing the ESXi host

Putting the hypervisor on bare-metal so the box becomes a host that can run virtual machines.

📖

Words you'll meet in this section

Server-installation jargon piles up fast. Here's the plain-language version of every term that appears later.

Hypervisor concept
What it is: The software layer that creates and runs virtual machines. It slices a physical server's CPU, memory, network, and storage and gives each VM its own share.
ESXi is VMware's hypervisor. Without one, you have no VMs.
Bare-metal concept
What it is: A hypervisor that installs directly on hardware — no operating system underneath. The hypervisor is the OS for that box. Type 1.
ESXi is bare-metal. That's why it boots straight from the install ISO with nothing else on the server. Contrast: Type 2 (Workstation, Fusion) sits on Windows / macOS / Linux.
HCL · Hardware Compatibility List hardware
What it is: VMware's official catalog of supported server, NIC, storage controller, and GPU hardware. Find it at vmware.com/go/hcl.
Unsupported hardware may install fine and then crash mysteriously under load (driver bugs, dropped packets, storage timeouts). Always check before buying or reusing servers.
ISO · ISO image config
What it is: A single file (.iso) containing the contents of a bootable installation CD/DVD. You can mount it as a virtual disc through iLO/iDRAC.
VMware ships ESXi as an ISO. You boot the server from it and the installer runs.
Kickstart · ks.cfg config
What it is: A configuration file that pre-answers every installer prompt — disk choice, root password, network, post-install commands. Borrowed from Red Hat's installer style.
Lets you install ESXi unattended. Critical at scale: 50 hosts × manual install = 50 chances to make different choices. Kickstart guarantees identical installs.
Auto Deploy tool
What it is: A vCenter feature where ESXi hosts boot from the network (PXE) and pull their image from a central server. Stateless (image lives on network) or stateful (cached locally).
At scale (hundreds of hosts), Auto Deploy means new hardware just shows up in your fleet. No ISO mounting per server.
OEM image config
What it is: A custom ESXi installer ISO from a hardware vendor (Dell, HPE, Cisco, Lenovo) that bundles drivers and firmware tools tuned for their servers.
On vendor hardware, OEM image is recommended over the generic VMware ISO — fewer driver mismatches, vendor health agents pre-installed.
Boot device hardware
What it is: The small disk where ESXi itself is written. Common choices: USB stick, SD card, M.2 SSD, or a small SAN boot LUN. ESXi is small (a few GB).
VMs and their disks live on bigger, separate storage (datastores). Keeping the boot device separate means a failed boot device doesn't lose VM data.
SAN boot LUN hardware
What it is: A small dedicated chunk of storage on a SAN array that the server is configured to boot from over Fibre Channel or iSCSI — instead of from a local disk.
Diskless servers boot from SAN. The hardware itself has no local drive; the boot device lives on the array.
iLO / iDRAC / IPMI tool
What it is: A separate management chip on enterprise servers (HPE iLO, Dell iDRAC, generic IPMI) with its own network port. Lets you reach the server's console and power it on/off remotely, even if the OS is dead.
You install ESXi by mounting the ISO through iLO/iDRAC's virtual media — you never need to walk to the data center.
Root password security
What it is: The super-admin password set during install. The "root" user can do anything on the host; if you forget this password, the only fix is reinstalling.
Set strong, store in a vault. Don't pick something easy "to test with" — production hosts will keep that password until someone changes it.
DCUI · Direct Console UI config
What it is: The yellow-and-black text screen ESXi shows on the server's console after install. Press F2, log in as root, and you can change the IP, restart agents, view logs.
When everything else is broken (vCenter down, network half-up), DCUI on the iLO console is how you fix the host. Always reachable.

At a glance

Bare server → installer prompts → progress bar → yellow DCUI live. ~10 minutes on real hardware.

Two ways to think about it

ELI5

A regular computer needs Windows or macOS to do anything. ESXi is a special kind of computer brain that doesn't need any of that — you put ESXi straight on the computer and now the computer can pretend to be many smaller computers at the same time.

ELI10

Quiz · install (2)

⏸ pause animation in 'install' mode
The diagram shows the ESXi installer running against a server with a 256 GB SSD and a 64 GB USB stick plugged in. Which device should you choose as the install target, and why does this even come up?
Answer

Choose the USB stick (or a small dedicated boot device like an SD card or M.2 SSD). ESXi only needs a small device for itself; you keep the 256 GB SSD free for use as a local datastore that holds VMs and their disks. This separation means a boot-device failure doesn't lose VM data, and a SSD failure doesn't lose the hypervisor. On larger production hosts you'd skip the local SSD entirely and put VMs on shared storage (vSAN, FC, iSCSI, NFS).

The install target is one of the four important inputs the installer asks about, and "the biggest disk" is the wrong default. ESXi assumes a small dedicated boot device and separate (often shared) storage for VMs.

A teammate claims they 'installed ESXi on top of Windows.' What's wrong with that claim?
Answer

ESXi is a bare-metal (Type 1) hypervisor — it doesn't install on top of an OS, it is the OS. What that teammate probably did is run ESXi nested inside VMware Workstation on Windows (a Type 2 setup). That's a valid lab pattern but not how production ESXi runs.

Distinguishing bare-metal from hosted hypervisors is foundational. Common point of confusion for first-time learners coming from desktop virtualization.

Subtopic 2 of 6

ESXi user account best practices

Who can log in, how, and what they can do once they're in.

📖

Words you'll meet in this section

Accounts and access-control vocabulary explained before they show up in the prose.

root security
What it is: The super-admin account on a Unix-like system (including ESXi). Can do anything: read any file, change any setting, kill any process.
ESXi ships with one root account. If you only ever use root, the audit log just shows "root did everything" — useless during incident review.
Glass-break account config
What it is: An emergency-only account whose password lives in a vault. The "break the glass in case of fire" pattern. You log this checkout and check-in.
root should be glass-break. Daily admin happens through named accounts. You only touch root when nothing else works.
Password vault tool
What it is: A secure, audited system that stores passwords (HashiCorp Vault, CyberArk, 1Password Business, AWS Secrets Manager). Access is logged.
Glass-break accounts only work if they're stored somewhere that records who accessed the password and when.
Named account security
What it is: An account tied to one specific human or service — Alice's account, Bob's account, the backup tool's account. Not shared.
When the audit log says "alice@corp restarted hostd," you know exactly who and can ask why. Shared accounts make that impossible.
Audit log security
What it is: A append-only record of every authentication and privileged action. Lives at /var/log/auth.log and similar files on ESXi, plus shipped to a central syslog server.
The whole point of named accounts is that the audit log becomes meaningful. Post-mortems read this file.
vCenter SSO tool
What it is: vCenter's identity layer (covered in lesson 9). Authenticates users against AD/LDAP/SAML and issues tokens. Daily admin should go through vCenter SSO, not directly to host accounts.
Centralizes identity: one place to add/remove people, one audit trail, one set of role assignments. Direct host login becomes the rare exception.
Active Directory (AD) tool
What it is: Microsoft's directory service — the master list of users, groups, and computers in a Windows-centric organization.
When AD is added to vCenter SSO, every AD user can log in to vSphere with their normal corporate credentials. Joiners/leavers in AD propagate automatically.
RBAC · Role-Based Access Control concept
What it is: A model where you give people roles (Read-Only, Administrator) instead of granting individual permissions. Roles bundle privileges; permissions assign roles to users at specific places.
Without RBAC, you'd grant or revoke hundreds of privileges per user. With RBAC, "Alice gets Read-Only on Production" is one click.
Predefined roles concept
What it is: Three default roles ESXi ships with. No Access (cannot log in), Read-Only (browse only), Administrator (full control).
Most environments live with the defaults plus vCenter-delegated roles. Custom roles are rare at the host level.
Least privilege concept
What it is: The principle of giving each account exactly the permissions it needs and no more. A junior engineer who only watches metrics gets Read-Only, not Administrator.
Reduces blast radius. A compromised Read-Only account can only see; a compromised Administrator account can destroy.
Domain join (AD) tool
What it is: The act of adding a server to an Active Directory domain so it trusts AD for authentication. Possible on ESXi but operationally heavy.
Most shops don't domain-join hosts directly — they let vCenter SSO talk to AD and the host trusts vCenter. One less moving part on each host.
Shared password security
What it is: A single password used by multiple people. Anti-pattern in modern security.
When 6 people share root, the audit log can't tell them apart. Anyone leaving the team means everyone has to remember a new password. One leaked password = breach.

Quiz · accounts (2)

⏸ pause animation in 'accounts' mode
The diagram shows six admins all logged in as root. The audit log is being highlighted. What does the audit log show after each admin completes an action, and what's the operational consequence?
Answer

The audit log shows 'root' as the actor for every action — there's no way to tell who actually did what. The operational consequence bites during incident review: when something goes wrong, no one can prove they didn't do it, and no one can prove who did. Answer

Give them a named identity in your IdP (AD/LDAP/SAML), let vCenter SSO authenticate them, and assign the appropriate vCenter role. They almost never log in to a host directly, and they definitely should not use root.

The "vCenter SSO with role assignment" pattern is what production vSphere is designed around. Direct host login is the exception.

Subtopic 3 of 6

Configuring host settings via DCUI and Host Client

Two front doors to one host, used for two different moments.

📖

Words you'll meet in this section

Three different management surfaces with overlapping names. Here's what each one is.

DCUI · Direct Console UI tool
What it is: The yellow-and-black text screen you see on a monitor plugged directly into the server (or via iLO/iDRAC remote console). Function-key driven. Press F2 to log in.
Runs locally on the host. Needs no network and no vCenter. The "fix it when everything else is broken" surface.
Host Client tool
What it is: A web UI served by the ESXi host itself. Browse to https://<host-ip>. Shows that one host's VMs, networks, datastores, services, and hardware.
When vCenter is down, this is the friendliest way into a host — provided the host's network is up.
vSphere Client tool
What it is: The central web UI served by vCenter. Browse to https://<vcenter-fqdn>/ui/. Shows every host vCenter manages, plus clusters, vMotion, HA, DRS.
This is your normal workplace. You only fall back to Host Client / DCUI when vCenter itself is unreachable.
iLO / iDRAC hardware
What it is: A separate management chip on enterprise servers (HPE iLO, Dell iDRAC) with its own NIC. Lets you remote-control the server's console and power, even if the OS is dead.
When the host is unreachable over the network, iLO/iDRAC is how you get to the DCUI. It's a backup management plane on the hardware itself.
Management agents · hostd, vpxa config
What it is: Two background processes on every ESXi host. hostd handles host-side management (the API the Host Client uses); vpxa is vCenter's agent on the host (talks back to vCenter).
"Restart Management Agents" in the DCUI restarts these two. Fixes "host not responding to vCenter" more often than anything else, no VM downtime.
Web UI / HTTPS protocol
What it is: A browser-based interface served over encrypted HTTPS (TLS). Both Host Client and vSphere Client are web UIs.
No software to install on your laptop. Works from any modern browser. Requires the host's management network to be up.
Management network config
What it is: The IP-network interface ESXi uses for management traffic — the IP you typed into the installer. Lives on a vmkernel adapter (covered in lesson 10).
When this network is misconfigured, vSphere Client and Host Client are unreachable. DCUI through iLO is your last resort.
Restart Management Agents concept
What it is: A DCUI menu option (F2 → Troubleshooting Options → Restart Management Agents). Restarts hostd and vpxa.
The single most useful one-line fix on the DCUI. Takes ~30 seconds. Doesn't restart the host or VMs.
esxcli tool
What it is: ESXi's built-in command-line tool. Available from the local ESXi Shell or remote SSH, also via the vSphere Client's API.
A fourth management surface for advanced ops — scripting, deep configuration, troubleshooting. Lives alongside DCUI / Host Client / vSphere Client.
PowerCLI tool
What it is: VMware's PowerShell-based command-line / scripting tool. Talks to vCenter or directly to a host. Used for automation.
When you need to do the same operation across 50 VMs, PowerCLI is one script vs 50 clicks.
F2 / F12 keys config
What it is: The two functional shortcuts the DCUI uses. F2 opens settings (after login); F12 shuts down or restarts.
Worth memorising: F2 is where you change IP / hostname / DNS / agent restart; F12 is the polite shutdown.
Out-of-band management concept
What it is: A management path that doesn't use the same network as production traffic. iLO/iDRAC (with its own NIC) is out-of-band; vSphere Client (over the same management VLAN) is in-band.
When the production network is down, out-of-band management still works. Worth the cabling.

At a glance

Same host, three management surfaces. vSphere Client for daily ops; Host Client when vCenter is down; DCUI for emergency recovery on the console.

What it is

ESXi gives you two consoles to talk to a single host directly, without vCenter in the loop. They look different, run on different surfaces, and exist for different moments.

DCUI · Direct Console UI

Yellow-and-black text screen on a monitor plugged into the server (or via iLO/iDRAC remote console). Press F2, log in as root, and you can change management IP / VLAN / NIC, hostname, DNS, restart management agents, view logs, reset to defaults, shut down. Runs locally; needs no network and no vCenter to be useful.

Host Client

Web UI served by the host itself on port 443. Browse to https://<host-ip>, log in as root or a local user, and you get VMs, networks, datastores, hardware health, services, alarms, events. Needs the host's management network to work, but is much friendlier than DCUI.

Rule of thumb: DCUIDirect Console User Interface — the yellow-and-black local console UI on an ESXi host. F2 to log in. Works without network or vCenter; ideal for bootstrapping and recovery. for bootstrapping and recovery, Host Client for everything else when vCenter is unreachable. When vCenter is reachable, you live in the vSphere Client instead — Host Client and DCUI become the friends you remember you have only when vCenter is down.

The single most useful DCUI move to remember: Restart Management Agents (Troubleshooting Options → Restart Management Agents). When a host stops responding to vCenter but is otherwise healthy, restarting the agents fixes it more often than any other single action.

Quick check

vCenter is down. A host has stopped showing up in vSphere Client (because vCenter is down). Network looks fine. What's your first move?

Configuring NTP, DNS, and routing

Three small settings, three big failure modes if you skip them.

📖

Words you'll meet in this section

Network plumbing terms. Get any one wrong and vCenter join fails with confusing errors.

NTP · Network Time Protocol protocol
What it is: A protocol for synchronising computer clocks over the network. ESXi runs an NTP client (ntpd) that polls one or more NTP servers and adjusts its clock.
Time is silently load-bearing. Wrong time → certs fail, AD fails, log correlation fails, vMotion fails. ~80% of "weird vCenter" issues trace back to clock drift.
DNS · Domain Name System protocol
What it is: The phone book of the internet. Maps names like vc-01.lab.local to IPs like 10.20.30.50.
vSphere uses hostnames everywhere — in certificates, in cluster config, in AD integration. Without DNS, those break in confusing ways.
A record (forward DNS) protocol
What it is: A DNS entry that maps a name to an IP. Direction: name → IP. Example: vc-01.lab.local → 10.20.30.50.
When vSphere Client looks up vc-01.lab.local, this is what it follows. Most DNS configurations have these.
PTR record (reverse DNS) protocol
What it is: A DNS entry that maps an IP back to a name. Direction: IP → name. Example: 10.20.30.50 → vc-01.lab.local.
vCenter join checks both forward AND reverse DNS. Missing PTR = "certificate verification failed" with a confusing message. Most vCenter join failures are missing PTRs.
TLS / SSL certificate security
What it is: A signed file that proves a server's identity to clients connecting over HTTPS. Includes a "valid from / valid to" time range.
If the host's clock is outside the cert's valid range (drift > few minutes), TLS rejects the cert as expired even though it isn't. This is why time matters so much.
Kerberos security
What it is: A network authentication protocol AD uses. Issues short-lived "tickets" instead of passing passwords. Needs all parties to agree on the time.
If your clock is more than ~5 minutes off the AD server, Kerberos refuses tickets and AD authentication breaks across the board.
vmkernel adapter config
What it is: A virtual network interface ESXi itself uses (not a VM) — for management, vMotion, vSAN, etc. Each one has an IP and lives on a port group. Covered fully in lesson 10.
Routing settings here decide how ESXi traffic reaches the rest of the network. Wrong gateway = host can't talk to anyone.
Default gateway config
What it is: The router IP that ESXi sends traffic to when the destination is on a different subnet. The "default route" in the routing table.
Every host needs one for its management vmkernel. Wrong gateway = host can't reach DNS, NTP, vCenter, or anything else off-subnet.
Static route config
What it is: A manually-configured rule: "to reach network X, go through gateway Y." Used when a vmkernel needs to talk to a specific subnet that isn't reachable through the default gateway.
When you have separate vMotion / vSAN / iSCSI vmkernels on isolated subnets, each may need its own static route to reach its peers. Without them: long mystery timeouts.
FQDN · Fully-Qualified Domain Name protocol
What it is: The complete name of a host including domain — e.g., esx-01.lab.local (not just esx-01).
vSphere uses FQDNs in certificates and AD binding. Short names lead to ambiguity and broken integrations.
NTP stratum protocol
What it is: A measure of how close a clock is to a reference time source. Stratum 0 = atomic clock; stratum 1 = directly attached; higher = farther. Most internal NTP servers are stratum 2 or 3.
Lower stratum = more accurate. ESXi prefers lower-stratum NTP sources when given multiple options.
DNS suffix · search domain config
What it is: The domain ESXi automatically appends when you type a short name. If suffix is lab.local and you type vc-01, it tries vc-01.lab.local.
Convenience setting, but worth getting right — wrong suffix means short-name lookups fail.

Configuring ESXi for lockdown mode

Closing the side and basement doors so the only way in is the lobby.

📖

Words you'll meet in this section

Security posture vocabulary. Most are concepts, not technologies — pay attention to the differences.

Lockdown mode security
What it is: A host-level switch that restricts who can talk to ESXi directly — bypassing vCenter. Three settings: Disabled, Normal, Strict.
Funnels admin access through vCenter. The audit log becomes meaningful when there's only one path in. Without lockdown, anyone with creds can sneak around vCenter.
Disabled (mode) security
What it is: The default. No restriction — anyone with valid host credentials can log in directly via Host Client, DCUI, SSH, or PowerCLI/API.
Fine for labs and isolated test environments. Too open for any production environment that cares about audit trails.
Normal (mode) security
What it is: Direct host access is restricted EXCEPT through DCUI (still works at the local console / iLO) and for configured exception users. All daily admin must go through vCenter.
The right default for most production environments. DCUI stays as a recovery path for when vCenter is down. Everything else funnels through vCenter.
Strict (mode) security
What it is: Direct host access is disabled EVEN at the DCUI. Only vCenter and configured exception users can reach the host.
Required for some regulated environments (PCI, HIPAA, FedRAMP). Costs you the DCUI safety net, so you'd better have tested recovery via exception users before flipping the switch.
Exception user security
What it is: A local account that bypasses lockdown — they can still reach the host directly even when lockdown is on. Configured per-host.
Some service accounts legitimately need direct host access (a backup tool, a hardware vendor's diagnostic agent). Exception users are how you allow that without disabling lockdown.
Security posture concept
What it is: Your overall stance on security. The collection of choices (lockdown mode, account practices, services-off-by-default) that together define how locked-down your environment is.
Lockdown mode is one piece. Setting it to Strict on a host where six people share root doesn't actually improve security — the posture is still poor.
Recovery path concept
What it is: A way to get back into a host or fix a problem when the normal management path is broken. DCUI is the classic recovery path; exception users are the recovery path under Strict lockdown.
Test recovery paths before you need them. Discovering at 3 a.m. that you have no way in = a very long night.
Compliance / regulated config
What it is: Workloads that fall under a regulatory standard (PCI for payment cards, HIPAA for health, FedRAMP for US government). Each standard has explicit security requirements.
"No direct admin access to systems handling card data" is a common requirement. Strict lockdown is one way to demonstrate compliance.
Host profile tool
What it is: A template that captures one host's full configuration (lockdown setting included) and applies it to other hosts in a cluster. An Enterprise Plus feature.
Apply lockdown across a 50-host cluster in one operation, with drift detection. Beats clicking through 50 hosts.
Audit finding config
What it is: A specific item flagged by an auditor as deviating from the security standard. "User X is an exception user but no business reason is documented" is a common one.
Document every exception user with a one-line reason. Auditors find them anyway; you want them to find your documented justification right next door.
Service account (host-level) tool
What it is: An account a piece of software (not a human) uses to log in. Backup agents, monitoring tools, hardware health pollers all run as service accounts.
When lockdown is enabled, service accounts that genuinely need direct host access become exception users. Document why each one needs it.
Audit log centralization concept
What it is: Forwarding all hosts' audit logs to a single place (a SIEM, a syslog server). Combined with lockdown, you get one funnel through vCenter and one log destination.
Funnel + central logs = a clean trail of every admin action. Without lockdown, the trail is fragmented across many sources.

Two ways to think about it

ELI5

You can lock the building so the only way in is the front door. You can lock it harder so even the basement door is closed and only people with a special pass can get in.

ELI10

Lockdown mode controls who can talk to an ESXi host directly without going through vCenter. Disabled means anyone with valid credentials can log in directly. Normal means direct logins are restricted but the console (DCUI) still works for emergencies. Strict means even the console is restricted; the only way in is vCenter or a pre-approved exception user. You pick the mode based on how locked-down you need to be — production usually picks Normal; regulated environments pick Strict.

A compliance audit is coming. The team flips every production host to Normal lockdown, adds the backup service account to the exception list on the cluster's hosts, and writes a one-line entry in the runbook explaining what that exception user is for. The audit asks one question: "why is account svc-backup an exception user?" The team points at the runbook entry. Audit moves on. That's the whole point of documenting exceptions.

Flashcards · lockdown (4)

Name the three lockdown modes and what each restricts.
tap to flip
Disabled (default) — anyone with valid host credentials can log in directly. Normal — direct host access restricted to DCUI and exception users; daily admin must go through vCenter. Strict — direct access disabled even at the DCUI; only vCenter or exception users.
What are 'exception users' and when do you add them?
tap to flip
Local accounts that bypass lockdown — they can still reach the host directly. Add them for service accounts that legitimately need direct host access (e.g., a backup agent), or for break-glass accounts compliance requires. Document why each exception exists.
Practical difference between Normal and Strict lockdown?
tap to flip
Normal still leaves the DCUI usable for recovery if vCenter dies. Strict disables the DCUI as well; recovery without vCenter requires an exception user. Strict is more locked down but leaves less safety net.
Why test recovery before flipping Strict on a remote host?
tap to flip
Strict disables the DCUI as a recovery path. If vCenter dies AND you have no exception user, you may end up reinstalling the host. Test in a lab: enable Strict, simulate vCenter loss, confirm exception-user recovery before doing the same in production.

Quiz · lockdown (2)

⏸ pause animation in 'lockdown' mode
A regulated workload requires that no admin connects directly to a host. Which lockdown mode and what supporting setup do you choose?
Answer

Strict mode, with one or two carefully documented exception users for service accounts that genuinely need direct host access (e.g., a backup agent). All human admin goes through vCenter. Test recovery in a lab first — Strict disables the DCUI, so if vCenter goes down and you have no exception user, recovery is hard.

Strict is right for regulated environments, but only with tested recovery and documented exceptions. The mode choice is one decision; the exception-user list and the recovery test are the choices that make Strict survivable.

Your team is enabling Normal lockdown across production. A backup service account currently logs in to each host directly to take VM-level backups. What do you do?
Answer

Add the backup service account to the exception users list on each host (or via host profile applied to the cluster). Document why it's an exception, ideally in a runbook entry the audit team can find. The backup keeps working because the service account bypasses lockdown; the rest of the world still has to go through vCenter.

Real-world lockdown rollouts always have legitimate service accounts that need exceptions. The discipline is identifying them, adding them, and documenting them — not pretending no exceptions exist.

The whole lesson in one breath

One host. Six configuration moves. The order matters — each step assumes the previous one is done.

1 · Install
Bare-metal hypervisor onto a small boot device. Set root password. Pick the right install method (ISO / kickstart / Auto Deploy / OEM).
2 · Accounts
Root = glass-break. Named accounts via vCenter SSO. Roles: No Access, Read-Only, Administrator.
3 · DCUI / Host Client
Two front doors when vCenter is down. DCUI for recovery; Host Client for richer admin via browser.
4 · Services
SSH off by default; "Start manually" so reboots stay safe. NTP, syslog on. Audit-driven defaults.
5 · NTP / DNS / Routing
Time, names, paths. Most "cert" errors at vCenter join are clock drift in disguise.
6 · Lockdown
Disabled / Normal / Strict. Funnel admin through vCenter; document every exception user.
