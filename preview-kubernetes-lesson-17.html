Services & Networking · ClusterIP, NodePort, LoadBalancer, Ingress, NetworkPolicy

Pods come and go; their IPs come with them. Services give your Pods a stable name. Ingress lets the outside world reach them. NetworkPolicy controls who can talk to whom.

🚨 The 3 AM Nightmare

Your front-end app's database connection string was hardcoded to a Pod IP. 10.0.5.7. It worked yesterday. Tonight, the database Pod restarts (a normal event — node maintenance). It comes back up with a fresh IP, 10.0.6.2. Your front-end keeps trying 10.0.5.7. Every API call returns 500. By morning the support queue is at 200 and you're explaining to your boss what happened. This lesson is about why Pods don't have stable IPs — and the four flavours of Service that do.

🎯 If you remember nothing else: Pods come and go; their IPs come with them. Services give your Pods a stable name. Four flavours of Service plus Ingress for HTTP plus NetworkPolicy for who-can-talk-to-whom.

Section 1 · Concept

Why Pods don't have stable IPs (and what does)

Each Pod gets its own IP address from the cluster's internal network. That IP works — Pods can reach each other directly by IP. The problem is that Pod IPs are ephemeral. When a Pod restarts, dies, or gets rescheduled to another node, it gets a fresh IP. There's no way to predict in advance what IP the new Pod will have. Hardcoding a Pod IP is hardcoding tomorrow's outage.

A Service is a Kubernetes object that gives a group of Pods a stable name and a stable network identity. You declare a Service that selects Pods by labels (app: my-api). The Service tracks which Pods match — when Pods come and go, the Service's backend list updates automatically. Anything inside the cluster reaches the Service by its DNS name (my-api.default.svc.cluster.local) and the kubelet's networking layer routes the request to one of the matching Pods.

Four Service types exist for four scopes of access:

Section 1.5 · The four Service types

Each type is a different scope of "reachable from"

The default is ClusterIP. The other three add increasingly external reach. Pick the smallest scope that fits.

📞

ClusterIP

internal directory

A virtual IP reachable only from inside the cluster. Other Pods reach the Service by DNS name. The default and most common type — every internal microservice talks to others through ClusterIPs.

When: Pod-to-Pod traffic inside the cluster.

📱

NodePort

phone booth on every block

Exposes the Service on a high port (30000–32767) on every node. External traffic to any-node-IP:30080 reaches the Service. Useful for development, custom load balancers, or specific traffic-shaping needs.

When: on-prem clusters without a cloud LB; bare-metal setups.

📡

LoadBalancer

dispatch operator

On a cloud cluster, asks the cloud-controller-manager to provision an external load balancer (an ELB on AWS, an NLB on GCP, an Azure LB) with a public IP. Traffic to that IP routes to all backing Pods.

When: exposing a non-HTTP service on a managed cloud cluster.

🪧

ExternalName

phone-book alias

Doesn't have any Pods or selectors. Just a CNAME-style DNS alias mapping the Service name to an external DNS name (db.example.com). Lets in-cluster code reach external services through the same naming convention.

When: integrating with managed databases, third-party APIs, or migrating workloads in.

Section 1.7 · Ingress in detail

HTTP / HTTPS routing: when LoadBalancer per service is too much

If every public-facing service had its own LoadBalancer, a 30-microservice product would burn 30 cloud LBs (and 30× the bill). Ingress solves this: one LoadBalancer in front of an Ingress controller, and an Ingress resource describes the routing rules. api.example.com/users goes to the user-service; api.example.com/orders goes to the order-service; app.example.com goes to the front-end.

Ingress is layer 7 — it speaks HTTP / HTTPS. It can do host-based routing (app.example.com vs api.example.com), path-based routing (/users vs /orders), TLS termination (Ingress holds the cert; backend services see plain HTTP), redirects, rewrites, basic auth, and rate-limiting if the controller supports it.

The Ingress resource

  • A YAML object describing routing rules
  • Doesn't run any traffic itself — it's just configuration
  • Specifies host, path, backing Service, TLS
  • Lives in the namespace that owns the routes

The Ingress controller

  • The actual proxy that processes traffic
  • Watches Ingress objects and configures itself
  • Examples: nginx-ingress, Traefik, HAProxy, Contour
  • Cluster has at least one (sometimes several, by class)

Gateway API is the next-generation replacement for Ingress — same job, more expressive, with proper role-separated APIs (cluster operators define gateways; app teams define routes). New clusters often ship Gateway API alongside Ingress; both work.

Section 1.9 · NetworkPolicy in detail

Who can talk to whom — by default, everyone

By default in Kubernetes, every Pod can reach every other Pod. Your front-end can talk to the database. Your database can talk to the public internet. A compromised Pod has full network reach. That's the default; it's not necessarily what you want.

NetworkPolicy is the firewall. You write rules saying "Pods labelled tier: frontend can talk to Pods labelled tier: backend on port 8080, and nothing else." Once a Pod is selected by any NetworkPolicy, it becomes default-deny — only the rules you've written allow traffic. Pods not selected by any policy stay default-allow.

Default-deny ingress

Empty ingress: [] rule selecting all Pods. After applying: nothing inbound reaches any Pod unless you write an explicit allow rule. Standard hardening pattern.

Allow frontend → backend

Selector: tier: backend. Rule: from: { podSelector: { tier: frontend } }, ports: [{port: 8080}]. Frontend Pods can reach backend on port 8080; nothing else can.

Default-deny egress

Empty egress: [] rule selecting compromise-prone Pods. They can't reach the internet, the API server, or other namespaces unless explicitly allowed. Containment by default.

Allow DNS

Almost every Pod needs DNS. Allow egress to kube-system namespace's DNS Pods on UDP 53. Without this, name resolution breaks.

[ deep dive — skip if new ]NetworkPolicy enforcement is done by the CNI plugin — Calico, Cilium, Weave Net, etc. Vanilla kube-proxy doesn't enforce NetworkPolicy on its own; you need a CNI that supports it. Check your cluster's CNI before assuming policies work — applying a NetworkPolicy on a CNI that ignores them silently leaves you wide open.

⏸ Pause and check

A Pod restarts. Its IP changes. What still works to find it?

Answer: b. Services use label selectors to find their backing Pods, then resolve a stable DNS name to whatever Pods currently match. Pod restart → new IP → Service notices and updates its backend list automatically.

Section 2 · Before & After

Raw Pod IPs vs Service abstraction

Before · raw Pod IPs

You connect your front-end to the database by IP. Every Pod restart breaks something. You write scripts to scrape kubectl get pods -o wide for current IPs. You hardcode IPs in config files; you update those config files manually after every deploy.

Cross-cluster portability is impossible — your IPs are valid on this cluster only. You can't load-balance across multiple Pod replicas because there's no abstraction over them. Every change is brittle.

After · Services + DNS

You declare apiVersion: v1, kind: Service with a label selector and a port. Inside the cluster, anything reaches the Service via DNS: postgres.default.svc.cluster.local (or just postgres in the same namespace). The Service tracks which Pods match and load-balances across them.

Pod restarts — invisible. Pods scale up — Service notices and includes them. Cluster moves to a new namespace or environment — names stay the same, IPs change underneath. Add an Ingress and the outside world reaches your services through one cloud LB instead of one per service.

What didn't change: the Pods themselves. They still run on ephemeral IPs. The Service is a layer of indirection between callers and Pods — and that indirection is exactly what makes the system survive Pod churn.

Section 3 · Analogy

The K-Town Switchboard

K-Town's office buildings used to have direct phone lines — call the bakery's specific instrument and you got the bakery. The problem: instruments moved. The bakery rearranged its kitchen, the hallways were renumbered, the line that used to be 5567 became 5824. Anyone with the old number got a wrong number; mail got lost.

So the city built the Switchboard — a central exchange. Now you don't dial the bakery's instrument. You ask for "Bakery District" and the operator patches your call through to whatever instrument the bakery is currently using. The bakery moves its phones around all it likes; the operator's directory is updated continuously, and your call still lands.

Different kinds of calls need different switchboard treatments. Calls between K-Town businesses go through the internal directory (anyone can dial Bakery and reach it). Calls from out-of-town that arrive on a public phone booth on each block go through NodePort. Calls that need a single famous public number routed to the right department use the dispatch operator (LoadBalancer). And HTTP-shaped calls — traffic that arrives at the city's main turnstile — get routed by Ingress based on which website the caller asked for. Plus the city's traffic rules — NetworkPolicy — decide who's allowed to call whom in the first place.

In the story……in Kubernetes
The SwitchboardThe Service abstraction
The instrument's number on a deskA Pod IP (ephemeral)
The name in the directoryThe Service DNS name
An internal phone directoryA ClusterIP Service
A public phone booth on each blockA NodePort Service

⚠️ The analogy stops here: real telephone exchanges have one switchboard at the centre. Kubernetes runs many networking components in parallel — the CNI plugin, kube-proxy on every node, the cluster DNS, the cloud LB. The "switchboard" is distributed.

Section 4 · Two-level explanation

Two ways to hear it

ELI5 · explain like I'm 5

Imagine your friends keep moving to new houses, but you still want to call them. Instead of remembering everyone's new address every time, you just call the operator and say "I want to talk to Sam!" The operator knows where Sam lives right now and connects you. That's a Kubernetes Service. The Pods (your friends) move around. The Service (the operator) always knows where to find them.

ELI10 · explain like I'm 10

Pods get IPs that change every restart, so you can't talk to a Pod by IP. A Service gives a group of Pods (selected by labels) a stable name and a stable virtual IP. Other Pods reach the Service by DNS — my-api.default.svc.cluster.local — and the cluster's networking layer routes the request to one of the matching Pods. Four Service types: ClusterIP (internal), NodePort (exposed on each node), LoadBalancer (cloud LB), ExternalName (DNS alias). For HTTP traffic, an Ingress routes by hostname and path, terminating TLS — one cloud LB shared across many services. NetworkPolicy firewalls Pod-to-Pod traffic.

⏸ Pause and check

You want to expose a web app to the internet on a managed cluster (EKS, GKE, AKS). Which Service type is the natural fit?

Answer: c. LoadBalancer triggers the cloud-controller-manager to provision a managed LB. For multiple HTTP services, share one LB by putting an Ingress controller in front of it.

Section 5 · Real-world scenarios

Where each piece shows up

A SaaS company with 40 microservices

Public traffic enters one nginx Ingress controller (one cloud LB). The Ingress resource defines routes: api.acme.com/users → user-service, api.acme.com/billing → billing-service, app.acme.com → frontend. TLS terminated at the Ingress; backends see plain HTTP. One cert, one LB, 40 services routed by URL. Saves ~$3,000/month vs LoadBalancer-per-service.

A trading firm running internal-only services

Every internal service is ClusterIP. NetworkPolicy enforces default-deny on every namespace. Specific allow rules let trading apps reach Postgres + Redis on specific ports; everything else is blocked, including the public internet. A compromised app can't even DNS-lookup an external host.

A logging platform on a DaemonSet

Fluent Bit runs as a DaemonSet — one Pod per node. App Pods write logs to localhost:24224; the local Fluent Bit batches and ships them to a central log store. No Service needed for that internal communication — the agent is on the same node, reachable via hostNetwork. Cuts cross-node traffic for logs to zero.

A multi-tenant cluster with namespace isolation

Each tenant gets a namespace. NetworkPolicy uses namespace selectors to enforce: tenant Pods can talk inside their own namespace + to kube-system (DNS) + to monitoring (Prometheus scraping) — and that's it. Cross-tenant traffic is blocked at the network layer. RBAC controls who can create resources; NetworkPolicy controls how the resources communicate.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: Services have IPs, so the IP is the right thing to use in app config.
Truth: Use the Service name (DNS), not the Service IP. The Service IP is stable but coupling to it ties you to one cluster's CIDR. Names work everywhere.
Myth: A LoadBalancer Service is the only way to expose HTTP traffic.
Truth: For HTTP, an Ingress controller (or Gateway API) is more flexible — host and path routing, TLS termination, one cloud LB shared across many services. LoadBalancer is for non-HTTP or single-service setups.
Myth: NetworkPolicy is on by default; my Pods are isolated.
Truth: By default every Pod can talk to every other Pod in the cluster. NetworkPolicy is opt-in — you write rules to restrict, otherwise everything is open. Default-deny is a posture you have to configure.

A team has 30 microservices behind one Ingress controller. They want to add a new service available at api.acme.com/inventory. Walk through what they create.

answerThree things. (1) A Deployment running the inventory service Pods (with labels app=inventory). (2) A ClusterIP Service named inventory that selects those Pods on the right port. (3) An Ingress rule (either appended to the existing Ingress resource or a new one) saying "traffic to api.acme.com/inventory goes to the inventory Service." The shared Ingress controller already terminates TLS at api.acme.com; this is just one more route. No new cloud LB; same external IP serves the new endpoint.

A team's kubectl get pods shows three replicas of the front-end with three different IPs. They want to load-balance traffic across all three. They write the front-end's URL into the back-end's config as http://10.0.4.7:80. Why is this wrong, and what's the right fix?

answerTwo problems. (1) 10.0.4.7 is one specific Pod's IP. The other two Pods get no traffic; if that Pod dies, the back-end breaks. (2) That IP changes on every restart. Fix: create a ClusterIP Service named frontend that selects all three Pods by label. Reference it as http://frontend (or frontend.default.svc.cluster.local) from the back-end. The Service load-balances across all matching Pods, and survives Pod churn. This is the Service abstraction's whole point.

Words from this lesson · open if you want a quick reference
Service
A K8s object giving a group of Pods a stable name and virtual IP. Selects Pods by labels; tracks them as they come and go; load-balances requests across healthy ones.
ClusterIP
Default Service type. Reachable only inside the cluster. The natural choice for Pod-to-Pod traffic.
NodePort
Service type exposing a high port (30000–32767) on every cluster node. External traffic to any-node-IP:port reaches the Service.
LoadBalancer
Service type that triggers the cloud-controller-manager to provision an external LB. Public IP, cloud-managed health checks, billing per LB.
ExternalName
Service type that's just a DNS alias to an external hostname. No Pods, no selectors. Used to integrate external dependencies into in-cluster naming.
Endpoints / EndpointSlice
The K8s objects holding the live list of Pod IPs that back a Service. Updated continuously as Pods come and go.
Ingress
L7 routing object: one cloud LB in front, host- and path-based routing rules, TLS termination, optional rewrites and auth.
Ingress controller
The proxy (nginx, Traefik, HAProxy, Contour) that watches Ingress objects and applies the routing rules to live traffic.
Gateway API
Next-generation replacement for Ingress. More expressive, role-separated (Gateway, HTTPRoute, TCPRoute). Available alongside Ingress in newer clusters.
NetworkPolicy
Firewall rules for Pod-to-Pod traffic. Default-allow until a Pod is selected by any policy; then default-deny except for explicit allow rules.
CNI
Container Network Interface. The plugin that gives Pods their network — Calico, Cilium, Weave Net, AWS VPC CNI, Azure CNI, etc. Also enforces NetworkPolicy on most plugins.
kube-proxy
Per-node networking component. Implements Service IPs by writing iptables/IPVS/eBPF rules so traffic to a Service IP gets DNAT'd to a healthy Pod.
Cluster DNS
CoreDNS Pods running in kube-system. Resolves service-name.namespace.svc.cluster.local to the Service's ClusterIP. Every Pod's /etc/resolv.conf points at it.
