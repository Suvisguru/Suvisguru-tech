Subtopic 1 of 8

Types of VM migrations — within and across vCenter instances

Cold vs hot · same vCenter vs cross-vCenter · cross-cloud. Same VM, different journeys.

📖

Words you'll meet in this section

Migration vocabulary. Each term is a button or a checkbox in the Migrate wizard.

Cold migration concept
What it is: Moving a VM that is powered off. The wizard relocates files (and optionally compute) without worrying about live state.
Cold migrations have the fewest constraints. If hot fails, power the VM off and try cold.
Hot migration concept
What it is: Moving a powered-on VM with no service interruption. vMotion (compute) and Storage vMotion (storage) are the two flavors.
Zero-downtime moves are the headline feature of vSphere. They make patching, balancing, and decommissioning routine instead of dramatic.
vMotion tool
What it is: Moves a running VM's compute (where it executes) to another host. Memory and CPU state transfer over a vMotion VMkernel network.
The most-used migration. Patch a host? vMotion its VMs off first.
Storage vMotion (svMotion) tool
What it is: Moves a running VM's files (vmdks, swap, config) to a different datastore. Compute stays on the same host.
Live storage maintenance — drain a datastore for retirement without taking VMs down.
X-vMotion (compute + storage) tool
What it is: A single migration that moves both compute and storage at once. Pick "Change compute resource and storage" in the wizard.
Common when moving VMs between sites or clusters that don't share storage.
Cross-vCenter vMotion tool
What it is: A vMotion that crosses vCenter boundaries — source vCenter and target vCenter are different. Live, no shared storage required.
Lets you move VMs between datacenters, between business units' vCenters, or to a new vCenter during migration projects.
Cross-cloud vMotion tool
What it is: Cross-vCenter vMotion that crosses to a public-cloud-hosted vSphere (VMware Cloud on AWS, Azure VMware Solution, GCVE).
Lift-and-shift to cloud without changing the VM. The VM thinks it's still in the same place.
vMotion VMkernel port config
What it is: A VMkernel network adapter on each host with the "vMotion" service ticked. Carries the memory/CPU state across.
No vMotion VMkernel = no vMotion. Always configure on every host that participates in vMotion. 10 GbE recommended.
Encrypted vMotion protocol
What it is: An option (Required / Opportunistic / Disabled) that encrypts the vMotion data in flight.
Required for compliance environments and for cross-cloud over public networks. Defaults to Opportunistic.
Compute resource concept
What it is: The host or cluster the VM will run on after migration. The wizard's first big choice.
Pick a cluster (DRS auto-places) instead of a specific host whenever possible.

The four migration journeys, one map

Four migration paths between source and target. Cold powers the VM off. vMotion moves only compute. Storage vMotion moves only files. X-vMotion moves both — and across vCenters or clouds.
★
The big ideaThe Migrate wizard offers three radio choices: "Change compute resource only" (vMotion), "Change storage only" (svMotion), "Change both" (X-vMotion). Cold uses the same UI but is allowed when the VM is off.
i
Cross-vCenter and cross-cloudBoth are X-vMotions where the target compute resource is in a different vCenter. The vCenter must be linked via Enhanced Linked Mode (same SSO) or connected via the Hybrid Cloud Extension (HCX) for cloud targets.

What it is

A VM migration moves a VM from one place to another. "Place" splits two ways: compute (the host the VM runs on) and storage (the datastore the VM's files live on). vSphere lets you change either or both, with the VM either off (cold) or on (hot).

The migration matrix
Cold migrationVM off · move compute, storage, or both · fewest constraints
vMotion (hot)VM on · compute moves · storage stays · <1 s switchover
Storage vMotion (hot)VM on · storage moves · compute stays · live mirror
X-vMotion (hot)VM on · compute + storage move · cross-host, cross-cluster, cross-vCenter, cross-cloud

Plus the cross-instance variants: cross-vCenter vMotion spans two vCenters; cross-cloud vMotion reaches into VMware Cloud on AWS, Azure VMware Solution, or Google Cloud VMware Engine. Both are still vMotions — same wizard, same live behavior, just a different target picker.

Before / After

Before

"Migration" is one fuzzy word. You'd schedule a maintenance window any time you needed to touch a host, datastore, or move workload to another site.

→
After

You see four crisp paths in the Migrate wizard. Hot migrations run in the background while users keep working. Maintenance windows shrink — or disappear.

Analogy

A migration is moving a tenant. Cold is moving while they're at work — easy, no fuss. vMotion is teleporting just the person to a new identical apartment, leaving their furniture in place. Storage vMotion is having movers swap all the furniture under them while they're sitting on the couch — they don't notice. X-vMotion does both: new building, new furniture, mid-conversation.

Two ways to think about it

ELI5

Imagine moving from one apartment to another. Sometimes you turn the lights off first. Sometimes you teleport while still sipping your coffee. vSphere has a wizard that lets you pick which kind of move.

ELI5

vSphere classifies migrations along two axes: state (cold vs hot) and what's moving (compute, storage, both). The hot variants — vMotion, Storage vMotion, X-vMotion — keep the VM running with no interruption. Cross-vCenter and cross-cloud are X-vMotions where the target compute lives under a different vCenter, including a public-cloud-hosted one.

Flashcards · migration types (3)

Cold vs hot migration?
tap to flip
Cold = VM is powered off; fewest constraints; moves files (and optionally compute). Hot = VM is powered on, no service interruption — vMotion (compute), Storage vMotion (storage), X-vMotion (both).
vMotion vs Storage vMotion vs X-vMotion?
tap to flip
vMotion moves only compute (host changes, datastore unchanged). Storage vMotion moves only files (datastore changes, host unchanged). X-vMotion moves both at once — used across hosts/clusters/vCenters/clouds without shared storage.
Cross-vCenter and cross-cloud vMotion — what makes them work?
tap to flip
They're X-vMotions where the target compute is under a different vCenter — either linked via Enhanced Linked Mode (same SSO) or connected via HCX (for VMC on AWS, AVS, GCVE). The VM keeps running through the move.

Quiz · migration types (2)

⏸ pause animation in 'migration types' mode
A VM has 80 GB of vmdks on ds-east-01 (about to be retired) and runs on esxi-a1 (healthy). The VM cannot have downtime. Which migration?
Answer

Storage vMotion. Compute is fine; only the storage needs to change. Pick "Change storage only" in the Migrate wizard, target a healthy datastore, hit Finish. The VM never stops.

When would cold migration still be the right choice today?
Answer

(1) The VM is already powered off. (2) Hot migration is blocked by an unsupported configuration (e.g., raw-device-mapping in physical mode, or a USB device pinned to the source host). (3) You need to change a property that requires a power-cycle anyway (e.g., guest OS type). Otherwise hot wins.

Subtopic 2 of 8

Migrate VMs using vSphere vMotion

Move a running VM from one host to another with zero downtime. The signature feature.

📖

Words you'll meet in this section

The mechanics of how a live VM teleports without users noticing.

Pre-copy phase concept
What it is: The first stage of vMotion. The VM keeps running on the source while its memory pages are copied page-by-page to the target host over the vMotion network.
Most of vMotion's wall-clock time is here. Bigger memory = longer pre-copy.
Dirty pages concept
What it is: Memory pages the VM modified after they were copied. Each pre-copy iteration captures dirty pages and re-sends them.
A very busy VM keeps dirtying pages faster than they can be sent. vMotion handles this with iterative passes plus stun-and-switch.
Stun phase (switchover) concept
What it is: The very brief (typically <1 s) freeze when the VM is paused on the source, the last dirty pages and CPU state ship over, and the VM resumes on the target.
This is the "downtime." It's so short that TCP doesn't drop and users don't notice. The whole operation is what learners call "live migration."
vMotion VMkernel network config
What it is: A dedicated VMkernel adapter on each host with the "vMotion" service enabled. Carries memory pages and state during the migration.
Sized for throughput — 10 GbE minimum, 25 GbE preferred. Slow vMotion network = long pre-copy = stressed VMs.
Multi-NIC vMotion config
What it is: Configuring more than one VMkernel for vMotion to parallelize copies across multiple NICs.
Doubles or triples vMotion throughput, which lets you move large or busy VMs faster.
Encrypted vMotion protocol
What it is: Required (always encrypts), Opportunistic (encrypts when both ends support it), or Disabled. Per-VM setting.
Default Opportunistic is fine in trusted networks. Required is the right call for cross-cloud or compliance environments.
CPU compatibility concept
What it is: The target host's CPU must expose at least the features the VM is using. If the target lacks AVX2 and the VM uses AVX2, vMotion blocks.
This is exactly what EVC (next subtopic) solves at the cluster level — it presents a uniform CPU baseline so any VM can move to any host.
Shared storage (or not) config
What it is: Classic vMotion requires source and target hosts to see the same datastore. vMotion-without-shared-storage (a.k.a. unified vMotion / X-vMotion) copies the disks too.
In modern vSphere you don't need shared storage to vMotion. Storage and compute can move together when needed.
Maintenance Mode tool
What it is: A host state that says "drain VMs off me." DRS or you manually vMotion every VM elsewhere; the host then enters Maintenance Mode and is safe to patch / reboot / replace.
vMotion's #1 production use case. You'll click "Enter Maintenance Mode" hundreds of times in a vSphere career.
vMotion priority concept
What it is: High vs Normal. High guarantees CPU on source/target to finish quickly even under contention.
For evacuating a host fast (e.g., before a known reboot window), pick High.

How vMotion actually works

vMotion pre-copies memory (green = clean, orange = dirty), then a sub-second stun ships final state. The VM resumes on the target with users none the wiser.
★
Always vMotion before you patch a hostThe Maintenance Mode workflow is: right-click host → Maintenance Mode → DRS auto-vMotions every VM off → host enters maintenance → patch / reboot → exit Maintenance. No service interruption, no scheduled downtime.
⚠
Slow vMotion network ≈ stressed VMIf pre-copy can't keep up with dirty rate, the VM gets throttled (Stun During Page Send) so vMotion can complete. Symptom: brief CPU stalls in the guest. Fix: bigger vMotion network (10 → 25 GbE, multi-NIC).

What it is

vMotion moves a running VM from one host to another with no perceived downtime. The hypervisor copies the VM's memory in the background (pre-copy), then briefly pauses the VM (stun, <1 s) to ship the last dirty pages and CPU register state, then resumes execution on the target host.

From the user's perspective, nothing happens — TCP sessions don't drop, applications don't notice. From the admin's perspective, "esxi-a1" disappears from the VM's host column and "esxi-a2" appears in its place.

Before / After

Before vMotion

Patching a server meant an outage notice, a Saturday at 2 a.m., and a relocation of services.

→
After vMotion

Patching a host is a Tuesday afternoon click. Workloads slide off, host gets patched, workloads slide back. Users never knew.

Analogy

vMotion is teleporting a person between two identical apartments while they're mid-conversation. You first build them a copy of their stuff in the new apartment (pre-copy), then for a microsecond freeze them, swap the doorways, and unfreeze. They keep talking; the building changed.

Quick check

A 256 GB VM is being vMotioned over 10 GbE. The vMotion network is fully utilized but the VM is heavily writing to memory. What happens?

Two ways to think about it

ELI5

Imagine copying a Lego castle to another table. You build a copy piece-by-piece while the kid keeps playing. Then for a blink you take the toy out of their hand and put the copy in. They keep playing — they didn't notice the table changed.

ELI10

vMotion does iterative live memory pre-copy: copy all pages, capture pages dirtied during the copy, re-copy those, repeat. When the working set converges (or under throttling), it stuns the VM, ships the final delta plus CPU registers, and resumes on the target. Single-digit milliseconds of pause; TCP doesn't notice.

Tuesday 2 p.m. You need to apply ESXi 8.0 U2c to esxi-cluster-prod1's twelve hosts. Workflow: right-click each host → Maintenance Mode → DRS vMotions all VMs off → patch via vLCM → reboot → Exit Maintenance Mode → DRS rebalances. Twelve hosts in an afternoon, the on-call doesn't even page.

Flashcards · vMotion (3)

What does vMotion actually move?
tap to flip
A running VM's compute (where it executes) — its memory contents and CPU register state — from one ESXi host to another. The VM's files (vmdks) stay on the same shared datastore (or are also copied, in the no-shared-storage variant).
Pre-copy vs stun phase?
tap to flip
Pre-copy: memory pages stream from source to target while the VM keeps running. Iterative — re-sending pages dirtied during prior passes. Stun: brief (<1 s) pause on source while final dirty pages and CPU state ship; VM then resumes on target.
What does vMotion need to work?
tap to flip
VMkernel adapters with the vMotion service on both source and target hosts; a routable vMotion network (10 GbE+); CPU compatibility (or EVC); shared storage (or X-vMotion which copies disks too); enough free CPU/RAM on target.

Quiz · vMotion (2)

⏸ pause animation in 'vMotion' mode
An admin clicks "Enter Maintenance Mode" on a DRS-enabled cluster host with 24 running VMs. What happens?
Answer

DRS evaluates each VM and vMotions it to the best-fit host. The host stays online but stops accepting new VMs. Once all 24 VMs are off, the host enters Maintenance Mode (a status, not a power state). Now the admin can patch / reboot safely.

Why does vSphere need a dedicated vMotion VMkernel network?
Answer

vMotion ships gigabytes of memory in seconds. Mixing it with management or VM traffic causes contention and slow / failed migrations. The dedicated VMkernel (10 GbE+, ideally 25 GbE multi-NIC) isolates vMotion bandwidth and keeps stun phases short.

Subtopic 3 of 8

Enhanced vMotion Compatibility (EVC)

Hide newer CPU features so VMs can vMotion across mixed CPU generations in the same cluster.

📖

Words you'll meet in this section

EVC is a cluster setting that solves one specific problem: mixed CPU generations.

CPU feature flag concept
What it is: A capability the CPU exposes to software — AVX, AVX2, AVX-512, AES-NI, BMI1, etc. Applications use feature flags to enable faster code paths.
When a guest detects a feature at boot, it remembers it. vMotion to a host that lacks the feature would crash the guest mid-instruction.
EVC mode tool
What it is: A cluster-wide CPU baseline. The cluster exposes only feature flags supported by the chosen baseline (e.g., "Intel Cascade Lake"), even if newer hosts have more.
All hosts in the cluster look identical to VMs. A VM doesn't see a feature it can't rely on after vMotion.
EVC baseline config
What it is: The named CPU generation the cluster pretends to be — Intel Merom / Penryn / Nehalem / ... / Sapphire Rapids; AMD Bulldozer / Piledriver / ... / Zen 4.
Pick the lowest-common-denominator across cluster hosts. Higher baselines unlock more features but exclude older hosts.
Per-VM EVC concept
What it is: A per-VM EVC mode (introduced in vSphere 6.7) that travels with the VM across clusters and even clouds.
Lets a single VM safely cross-vCenter / cross-cloud vMotion to a target whose cluster baseline is at least as high.
Power-cycle requirement concept
What it is: Raising EVC level for an existing VM requires a guest reboot so it re-detects features. Lowering generally requires a full power-off.
Plan EVC raises into a maintenance window. Once set, the VM remains compatible across all hosts in the cluster.
CPU mismatch error config
What it is: The validation error that vMotion throws when target CPU lacks a feature the VM is using. EVC prevents this proactively.
"VM uses CPU features that are not supported by host" — the most common reason a vMotion is blocked. EVC kills this class of failures cluster-wide.
CPU vs OS compat concept
What it is: EVC handles CPU features only. It doesn't help with vmdk format mismatches, RDM mode, NSX overlay, or guest-specific issues.
If a vMotion still fails with EVC enabled, look at storage / network / VM-config compat — not CPU.
When NOT to enable EVC concept
What it is: If a workload (HPC, GPU/AI, certain databases) genuinely needs the newest CPU features and the cluster is uniform, EVC may hide useful features.
EVC is a trade-off — uniform compatibility vs. potentially-masked performance features. For uniform clusters with cutting-edge workloads, you may keep it off.

Without EVC vs With EVC

Without EVC, mixed-generation hosts block vMotion. With EVC at the lowest-common baseline, every host looks identical and any VM can move to any host.
★
Always enable EVC at cluster creationIt's free, it costs almost nothing, and it future-proofs the cluster against the day you add a newer-generation host. You'll thank yourself.
⚠
Per-VM EVC for cross-cluster movesFor VMs that may cross-vCenter / cross-cloud vMotion, set per-VM EVC instead of (or in addition to) cluster EVC. The setting travels with the VM and stops surprises at the destination.

What it is

EVC is a cluster setting that pins the visible CPU feature set to a chosen baseline. Each host's CPUID gets filtered: if the baseline is "Intel Haswell," even a Sapphire Rapids host hides AVX-512 and AMX from VMs. Every host in the cluster looks like the same CPU generation.

Why this matters: a VM detects features at boot and uses them throughout its lifetime. If those features disappeared during a vMotion, the guest would crash. EVC's job is to ensure features can never disappear.

Before / After

Before

Cluster has hosts from 2018, 2021, and 2024. vMotion sometimes fails with "CPU features not supported." DRS won't balance. Maintenance windows hurt.

→
After

Enable EVC at the lowest baseline. Every VM sees one CPU generation. vMotion works between every pair of hosts. DRS balances freely.

Analogy

EVC is a uniform. The team's tallest player is 6'8" and the shortest is 5'9", but everyone wears the same standard-issue uniform. From a distance — and from the VM's perspective — they all look the same height. Trades between teams are simpler because no one's wearing custom gear.

Quick check

A cluster has Haswell, Cascade Lake, and Sapphire Rapids hosts. EVC baseline is set to Cascade Lake. What's the consequence?

Two ways to think about it

ELI5

Imagine a kid moving between rooms with different toys. EVC says "everyone only plays with the toys all rooms have." Now the kid can move anywhere without missing a toy.

ELI10

EVC is per-cluster CPUID masking. The cluster declares a baseline (a named CPU generation); each host's CPUID is filtered down to that baseline. VMs see a uniform feature set across all hosts, eliminating CPU-mismatch vMotion failures. The trade is hidden bleeding-edge features (AVX-512, AMX) that the newest hosts could otherwise expose.

A retailer's cluster grew over six years. Hosts: 2 × Haswell (2018), 4 × Cascade Lake (2021), 4 × Sapphire Rapids (2024). With no EVC, DRS would shuffle VMs onto Sapphire Rapids hosts where they'd "learn" AMX, then refuse to move back. EVC at the Haswell baseline costs no peak performance for these workloads (web/app, no AVX-512), and lets DRS treat the whole cluster as one pool. Saturday patching is a slack message away.

Flashcards · EVC (3)

What does EVC do?
tap to flip
EVC (Enhanced vMotion Compatibility) is a cluster setting that filters each host's CPUID to a chosen baseline. Every host in the cluster appears as the same CPU generation to VMs, so vMotion always works regardless of physical CPU.
EVC trade-off?
tap to flip
VMs can't use CPU features above the baseline — even on hosts that physically have them. Trade is uniform vMotion compatibility (huge win) vs. masked bleeding-edge features (rarely used by typical workloads).
Per-VM EVC vs cluster EVC?
tap to flip
Cluster EVC normalizes a single cluster. Per-VM EVC (a property of the VM) makes a VM portable across clusters and clouds — it carries its own baseline so cross-vCenter and cross-cloud vMotion can validate target compatibility.

Quiz · EVC (2)

⏸ pause animation in 'EVC' mode
Cluster has Haswell + Sapphire Rapids hosts. EVC is off. A VM was created on a Sapphire Rapids host. Why might vMotioning it to a Haswell host fail?
Answer

The VM may have detected and used CPU features (AMX, AVX-512) that exist on Sapphire Rapids but not on Haswell. vMotion validation rejects the move because guest software running today might use a feature instruction tomorrow that Haswell can't execute.

When would you NOT want EVC enabled?
Answer

When the cluster is uniform (all hosts identical) and runs workloads that genuinely benefit from the latest features (HPC with AVX-512, ML inference with AMX, certain DBs). In that narrow case, EVC just hides useful capabilities. For mixed clusters or general-purpose workloads, leave it on.

Subtopic 4 of 8

Migrate VMs using vSphere Storage vMotion

Move a running VM's disks to a different datastore — live. Compute stays put.

📖

Words you'll meet in this section

Storage vMotion is the storage-side twin of vMotion. Same wizard, different button.

Storage vMotion (svMotion) tool
What it is: A live migration of a VM's files (vmdks, swap, config) from one datastore to another while the VM keeps running on the same host.
Drain a datastore (retirement, full disk, slow array) without scheduling downtime.
I/O mirror driver concept
What it is: The mechanism that intercepts the VM's writes and sends them to both the old and new datastore during the copy phase.
Keeps the destination consistent with live writes. When the bulk copy finishes, the destination is identical and the source can be deleted.
Bulk copy phase concept
What it is: The first stage — vmdk blocks stream from source to destination datastore in the background while the VM keeps running.
Most of the wall-clock time is here. Bigger disks = longer bulk copy. I/O during this phase is mirrored to both ends.
Switchover concept
What it is: A moment of atomicity at the end: the source disks are unmounted and the VM points to the destination disks. Microsecond-scale.
No stun, no perceived pause. Storage vMotion's switchover is even quieter than vMotion's.
Disk format change config
What it is: The wizard lets you change disk provisioning during the move — Thin, Thick Lazy-Zeroed, Thick Eager-Zeroed.
A handy "two birds" — relocate AND reformat. Useful when consolidating onto a thin-provisioning-friendly array.
Storage policy change config
What it is: Pick a new SPBM policy at migration time. The new datastore gets the new policy applied.
Tier-up or tier-down a VM (Bronze → Gold) with one operation.
Storage DRS (sDRS) tool
What it is: Cluster-level automation built on Storage vMotion. Watches datastore latency and free space; auto-rebalances VMs across a datastore cluster.
Operationally similar to compute DRS. You set thresholds, sDRS handles the moves.
Datastore cluster concept
What it is: A logical group of similar datastores that sDRS treats as one pool.
Pre-requisite for sDRS. VMs are placed into the cluster; sDRS picks the best datastore.
VM affinity rules (storage) concept
What it is: Rules to keep / separate a VM's disks across datastores within the cluster.
Default is "keep all disks together." For high-I/O DB VMs you might separate redo logs onto a different datastore for parallel I/O.

How Storage vMotion mirrors writes

Bulk copy streams blocks; an I/O mirror keeps live writes consistent at both ends; an atomic switchover redirects the VM to the destination. Compute never moves.
★
Storage vMotion vs vMotionvMotion changes the host. Storage vMotion changes the datastore. X-vMotion changes both. Same Migrate wizard, different radio choice.
⚠
Don't run during peak hours on stretched linksStorage vMotion generates real I/O load on both source and destination arrays plus the storage network. On busy production storage, schedule it for off-peak. The UI lets you cap concurrent migrations.

What it is

Storage vMotion moves a running VM's files — vmdk disks, swap file, configuration, snapshots — from one datastore to a different one without taking the VM down. The VM continues to execute on the same host throughout. The host writes blocks to the new location in the background and uses an in-kernel I/O mirror to keep live writes consistent. At the end, a tiny atomic switchover redirects the VM to the destination, and the source files are freed.

Before / After

Before

Decommissioning a SAN meant waiting for an outage window per VM. Datastore at 95% full was a panic.

→
After

Click "Migrate → Change storage only," pick the new datastore, walk away. The VM never knew. Storage retirements are now Tuesday-afternoon work.

Analogy

Storage vMotion is swapping the bookshelves under someone reading a book. Movers carry shelves of books from the old wall to the new wall. While they work, any new book the reader picks up gets placed on both walls (the I/O mirror). When the move is done, the old wall is removed; the reader never paused.

Quick check

A VM with 4 TB of vmdks on a slow datastore is being Storage vMotioned to a fast one. The VM is doing heavy writes. What ensures the destination is consistent?

Two ways to think about it

ELI5

Imagine carrying every toy from one toy box to another while the kid is still playing. Anything they grab during the move, you put in both boxes. When the move is done, you throw the old box away.

ELI10

Storage vMotion does live block-copy of vmdk extents while an I/O mirror funnels new writes to both source and destination. The bulk-copy pass finishes before live changes outpace it; an atomic switchover swaps the VM's pointers to the destination disks. Compute never relocates; only the file layer changes.

Datastore ds-old-fc01 hits 92% full. Instead of paging the storage team, you select 12 fattest VMs in the vSphere Client, right-click → Migrate → Change storage only → target ds-vsan-prod. Twelve concurrent Storage vMotions kick off; the source datastore drops to 41% in three hours. Nobody filed a ticket because nobody noticed.

Flashcards · Storage vMotion (3)

What does Storage vMotion move?
tap to flip
A running VM's files (vmdks, swap, config, snapshots) from one datastore to another. Compute stays on the same host. Live, no perceived downtime.
How does it stay consistent during the copy?
tap to flip
In-kernel I/O mirror driver writes every live block change to both source and destination during the bulk copy. After bulk finishes, destination is byte-identical, an atomic switchover repoints the VM, source is freed.
Two production use cases?
tap to flip
(1) Drain a datastore for retirement / full / latency. (2) Tier a VM up or down (Bronze → Gold) by moving it onto a higher-performance datastore — also lets you change disk format and storage policy in the same operation.

Answer

Subtopic 5 of 8

Take a snapshot of a VM

A 30-second safety net before a risky change. Preserves a known-good state.

📖

Words you'll meet in this section

Snapshots are simple in concept, dangerous when misused. Vocabulary first.

VM snapshot tool
What it is: A point-in-time capture of a VM's state — disks (frozen), optionally memory (running RAM dump), and CPU state. The base disk is sealed read-only; new writes go to a delta file.
A safety net for short-term changes. Easy to roll back from a bad upgrade or change in seconds.
Delta disk (-000001.vmdk) config
What it is: A new vmdk created when a snapshot is taken. The base is frozen; subsequent writes go here. Each delta is a redo log.
The hidden cost of a snapshot. Every write traverses the delta chain — performance degrades as the chain grows.
Snapshot memory config
What it is: Optional checkbox at snapshot time. Captures the VM's running RAM into a file so you can revert to a powered-on state.
Adds time and space (a copy of the entire vRAM) but lets you revert without rebooting the guest. Off by default.
Quiesce file system config
What it is: Optional checkbox. Calls VMware Tools to flush guest writes and make the snapshot crash-consistent at the file-system level (Linux fs-freeze, Windows VSS).
Required if a backup tool will use the snapshot to image the disks. Otherwise unnecessary for short admin snapshots.
Snapshot tree concept
What it is: The hierarchy of snapshots, including branches if you reverted then snapped a different timeline. Visible in Snapshot Manager.
A tree, not a list. You can revert to any node and snapshot a new branch from there.
Snapshots are NOT backups concept
What it is: A snapshot lives in the same datastore as the VM. If the datastore fails, the snapshot dies with it. Snapshots also depend on the running VM file structure.
The single most important rule in this lesson. Use snapshots for short-term safety nets; use real backup software for protection.
.vmsd / .vmsn files config
What it is: The snapshot manifest (.vmsd) and per-snapshot state (.vmsn) files that track the chain.
Knowing they exist helps when troubleshooting orphaned snapshots or weird Snapshot Manager state.
Maximum chain depth concept
What it is: vSphere supports up to 32 snapshots per VM, but VMware recommends <3 for any duration and <72 h lifetime.
Long chains kill performance and bloat disk. Treat 3 deep / 3 days as the practical max.

What happens when you take a snapshot

Taking a snapshot freezes the base vmdk and creates a delta. The VM keeps writing — but to the delta. Reverting just discards the delta.
!
Snapshots are NOT backupsThe snapshot lives next to the VM on the same datastore. Datastore lost = snapshot lost. Use snapshots only for short-term safety nets (an upgrade, a config change). For real protection, use Veeam, NetBackup, or another proper backup product.
⚠
Do NOT skip "snapshot the memory" without thoughtIf unchecked, you can revert only to a powered-off state. If checked, you revert to the live in-memory state but the snapshot itself takes longer (memory dump) and uses more disk. Pick deliberately.

What it is

Right-click VM → Snapshots → Take Snapshot. Give it a name and description, optionally check "Snapshot the virtual machine's memory" and/or "Quiesce guest file system." Click OK.

Under the hood, vSphere freezes the current vmdk(s), creates a delta vmdk per disk, and redirects all subsequent writes to the delta. Reads check the delta first; if a block isn't there, fall through to the base. The base hasn't changed, so revert = throw away delta.

Before / After

Before

Major OS patch on a production VM at 2 a.m. with a backup tape and a prayer.

→
After

Snapshot at 1:55 a.m. Apply patch. If anything goes wrong, revert in 60 seconds. Patch confirmed working at 2:30 a.m.? Delete the snapshot.

Analogy

A snapshot is bookmarking your place in a video game before fighting a boss. If the boss kills you, load the save. If you win, delete the save. Saves are useful but they pile up — a folder of 30 saves slows the whole machine down.

Quick check

Two ways to think about it

ELI5

Imagine taping a piece of paper over a drawing before you change it. You only draw on the paper now. If you don't like the change, throw the paper away — your original drawing is fine.

ELI10

A snapshot creates a redo-log delta on top of each vmdk. The base is sealed read-only and writes redirect to the delta. Reads check delta first then fall through. Reverting throws the delta away. Optional memory snapshot and quiesce add capabilities (live revert / fs-consistent) at extra time and space cost.

Friday 4 p.m. You're upgrading a Windows DC from 2019 to 2022. Snapshot the VM with memory off, quiesce on. Run the upgrade. If the in-place upgrade fails, revert in 60 s and call Microsoft. If the upgrade succeeds, validate for an hour, then DELETE the snapshot. Don't go home with the snapshot still attached.

⏸ pause animation in 'snapshot' mode

Subtopic 6 of 8

Manage, consolidate, and delete snapshots

Snapshots accumulate. Knowing how to clean them up safely is more important than how to take them.

📖

Words you'll meet in this section

Five buttons in Snapshot Manager. Each has very different consequences.

Snapshot Manager tool
What it is: The vSphere Client window that shows the snapshot tree for a VM with buttons: Revert · Delete · Delete All · Consolidate · Take.
Right-click VM → Snapshots → Manage Snapshots. Spend a few minutes learning each button's effect.
Revert concept
What it is: Throws away the current state and returns the VM to the snapshot point. Any writes since the snapshot are lost.
The whole point of snapshotting. Use after a failed change to undo cleanly.
Delete (one) concept
What it is: Merges this snapshot's delta into its parent (base or higher delta) and removes this node from the tree. Keeps current state.
Counter-intuitive name — "Delete" really means "merge and forget." Your data is preserved.
Delete All concept
What it is: Merges every delta into the base in tree order, leaving only the live VM with no snapshots. Most common cleanup.
After confirming a change is good, Delete All. Don't leave snapshots lingering — they cost performance and disk.
Consolidate concept
What it is: Cleans up orphaned delta files left behind when a Delete failed mid-way (often after a backup-job crash). Forces vSphere to merge any deltas not in the manifest.
If you see "Configuration Issues: Virtual machine disks consolidation is needed," click Consolidate. Don't ignore — disk usage grows silently.
Hidden snapshots concept
What it is: Backup software (Veeam, VBR) takes vSphere snapshots, then deletes them when done. Sometimes a delete fails and the delta lingers without showing in Snapshot Manager.
Run the "Needs Consolidation" alarm or check the VM's datastore folder for stray -000001.vmdk files.
Snapshot lifetime concept
What it is: How long a snapshot has existed. VMware best practice: under 72 hours.
Older = bigger delta = longer to merge = more performance impact. A snapshot from last month can take hours to consolidate.
"Needs Consolidation" alarm config
What it is: A pre-canned vCenter alarm that fires when a VM has stray deltas not tracked by Snapshot Manager.
The first place to look for "I have full datastores but no snapshots in Manager — what's eating space?"

Snapshot tree and what each button does

Revert throws away changes; Delete merges (preserves data); Delete All cleans up everything; Consolidate fixes orphaned deltas. "Delete" is misleadingly named — it does NOT delete your data.
!
"Delete" preserves your data — "Revert" destroys itThe most-misread word in vSphere. Delete = merge and remove the snapshot node (current data preserved). Revert = throw away current changes and roll back to the snapshot. Always re-read the dialog before clicking.
★
Run Delete All within 24-72 hours of taking a snapshotSet yourself a calendar reminder. Snapshots that survive a week become operational debt — bigger every day, slower to merge, blocking other operations.

What it is

Once you've taken a snapshot, the VM has accumulated state in deltas. Cleaning up means folding deltas back into the base. Two paths: Delete a single snapshot (merge that one), or Delete All (merge everything, leave only the base). Both keep your current data — the only thing they remove is the safety net.

If a Delete failed (often during a backup job), the deltas may linger as orphans. Consolidate is the explicit cleanup. The "Needs Consolidation" alarm tells you when.

Before / After

Before

Datastore at 95% full. Backup keeps failing. You can't tell why because Snapshot Manager shows no snapshots.

→
After

vCenter alarm: "Needs Consolidation." Click Consolidate. 11 orphaned deltas merge. Datastore drops to 60%. Backup succeeds tonight.

Analogy

Snapshots are saved games and the buttons are very differently named. Revert = "Load Save" (throw current play away). Delete = "Forget Save" (keep current play, just remove the bookmark). Delete All = "Forget All Saves" (clean slate, just current play). Consolidate = "Defragment your save folder" — find the saves the launcher couldn't see and tidy them up.

Quick check

A VM has 4 snapshots accumulated over 3 weeks. The latest is current. The owner says "we're done with all of them." What's the right click?

Two ways to think about it

ELI5

Imagine a stack of sticky notes covering a drawing. Delete a note = press it down so the drawing absorbs it; the drawing now includes the change. Revert a note = peel it off and throw it away.

ELI10

Delete is a destructive merge from the perspective of the snapshot tree, but it's nondestructive for VM data — the delta's blocks fold into the parent vmdk. Revert is the opposite: keep the parent, throw the delta away. Consolidate is a recovery operation when the manifest disagrees with on-disk delta state.

Veeam runs nightly backups by snapshotting each VM, reading the delta + base, then deleting the snapshot. One night the storage was full and the delete failed silently. Two weeks later, datastore is at 99% and an alarm fires: "Needs Consolidation" on 23 VMs. Right-click → Snapshots → Consolidate. Each VM merges its orphaned delta into base. Datastore frees 14 TB. Now you also schedule the orphaned-snapshot alarm to email you instead of just sitting in vCenter.

"Delete" vs "Revert" in Snapshot Manager?
tap to flip
Delete merges the snapshot's delta into its parent and removes the snapshot from the tree — current data preserved. Revert throws current changes away and rolls back to the snapshot point — current data lost. Different actions, easily confused.
When to run Consolidate?
tap to flip
When the "Needs Consolidation" alarm fires (usually after a failed Delete, often by backup software). Consolidate finds orphaned delta files that aren't in the manifest and merges them into the base. Free disk usually returns dramatically.
Best practice for snapshot lifetime?
tap to flip
Under 72 hours. The longer a snapshot lives, the bigger its delta and the slower the eventual merge. Older = more performance impact, more disk consumed, riskier consolidation. Use them as short-term safety nets, then Delete All.

Quiz · manage snapshots (2)

⏸ pause animation in 'manage snapshots' mode
An OS upgrade succeeded a week ago. The snapshot is still attached. What should you do, and why is it urgent?
Answer

Delete All. Urgent because: (1) the delta has been growing for a week — every guest write since the snapshot lives there; (2) read I/O traverses the chain; (3) at some point the merge will become long and disruptive. Delete now while it's still cheap.

Subtopic 7 of 8

CPU and memory in a virtualized environment

How vSphere shares physical CPUs and RAM among many VMs — and what to monitor.

📖

Words you'll meet in this section

Two layers of vocabulary: how the hypervisor shares CPU, and how it shares memory.

vCPU scheduling concept
What it is: The hypervisor's scheduler picks ready vCPUs and runs them on physical cores in time slices. A VM's vCPUs share cores with other VMs' vCPUs.
When the host is calm, every vCPU runs immediately. When it's busy, vCPUs queue. That queuing is what CPU ready measures.
CPU ready (%RDY) concept
What it is: Time a vCPU was ready to run but the scheduler couldn't give it a core. Measured in ms or %.
The #1 CPU metric. Under 5% per vCPU = healthy. Over 10% = guest is stuttering. Over 20% = serious contention.
Co-stop concept
What it is: Time a multi-vCPU VM was paused because not all of its vCPUs could be scheduled simultaneously.
A VM with too many vCPUs creates gang-scheduling contention. Reducing vCPUs often improves performance.
NUMA concept
What it is: Non-Uniform Memory Access — modern CPUs are split into nodes, each with its own RAM bank. Cross-node memory access is slower.
vSphere's NUMA scheduler tries to keep a VM's vCPUs and vRAM on the same node. Stretching across nodes hurts latency. Match VM size to NUMA node size.
TPS · Transparent Page Sharing concept
What it is: Memory dedup. Identical pages across VMs are kept once and shared by reference.
Free memory savings. Disabled by default across-VMs since 2014 for security. Within-VM TPS still helps.
Ballooning concept
What it is: The first-line memory reclamation. The balloon driver (in VMware Tools) inflates inside a guest, asking the guest to release memory it isn't using. The hypervisor reclaims it.
Most graceful reclamation. Works only if VMware Tools is installed — another reason to always install Tools.
Memory compression concept
What it is: Second-line reclamation. Hypervisor compresses cold memory pages into a small in-memory cache.
Faster than swap (no disk I/O), gives some breathing room when ballooning isn't enough.
Hypervisor swap concept
What it is: Last-resort reclamation. Hypervisor swaps VM memory pages to a swap file on disk.
Catastrophic for performance. If you see hypervisor swap in use, the host is severely over-committed — fix it now.
Memory over-commit concept
What it is: Sum of configured vRAM exceeds physical RAM. Common because most VMs don't use all their configured memory.
vSphere handles light over-commit gracefully via TPS / balloon / compress. Heavy over-commit triggers swap.
Active memory · Consumed memory tool
What it is: Active = pages the guest has touched recently. Consumed = total host RAM allocated to the VM.
Use Active to right-size. A VM with 32 GB configured but 4 GB active for weeks can probably shrink to 8 GB.

How the hypervisor shares CPU and memory

CPU scheduling time-slices vCPUs across cores — gaps are CPU ready time. Memory reclamation walks four tiers from cheap dedup to last-resort swap.
⚠
The fewer vCPUs the better — usuallyCounter-intuitive but true: a 4-vCPU VM often outperforms an 8-vCPU VM running the same workload, because the scheduler can place 4 vCPUs simultaneously far more often than 8. Right-size against actual CPU usage.
★
VMware Tools enables ballooningWithout Tools, the host can't ask the guest to release memory and falls straight to compression / swap. Always install Tools — for snapshots, Tools, and ballooning.

What it is

vSphere abstracts physical CPU and memory into a shared pool. The CPU scheduler time-slices physical cores among ready vCPUs. The memory subsystem hands out RAM, and when supply tightens, walks four reclamation tiers: TPS dedup → guest-driven balloon → in-memory compression → swap to disk.

For most workloads, vSphere does this invisibly. Performance issues appear when a VM has too many vCPUs (high ready / co-stop) or the host is over-committed (active memory exceeds physical, swap kicks in).

Before / After

Before

Each app gets a dedicated server. Servers run at 5% utilization. CPU and memory are mostly wasted.

→
After

One host runs 30 VMs sharing the same 32 cores and 384 GB RAM. The hypervisor schedules and reclaims so workloads see what they need, when they need it.

Analogy

The host is a coffee shop with 4 baristas (cores). Customers (vCPUs) line up; each barista serves whoever's next. If the line is short, no waiting. If everyone shows up at once, the line grows — that wait is CPU ready time. Memory works like a counter with limited cup space — first you reuse cups (TPS), then you politely ask people to give back unused cups (balloon), then you crush cups together (compress), and only as last resort you tell people to bring their own cups from home (swap).

Quick check

A VM is sluggish despite low guest CPU usage. esxtop shows %RDY at 35% per vCPU and high co-stop. The VM has 8 vCPU on a host with 16 cores. Most likely fix?

Two ways to think about it

ELI5

The big computer is a kitchen with 4 stoves. Many cooks (VMs) want to cook. The hypervisor is the kitchen manager — gives each cook a stove for a few seconds, then rotates. Memory is the pantry — when stuff runs low, the manager first reuses what's already opened, then asks cooks to share, then squishes things together, and only as a very last step actually goes to the basement to bring more (which is slow).

ELI10

Two distinct subsystems. CPU: a hierarchical scheduler with NUMA-aware placement. vCPUs are co-scheduled when running multi-vCPU VMs. Metrics: %RDY (queuing for a core), %CSTP (co-stop), DEMAND vs USED. Memory: 4-tier reclamation from TPS to swap. Metrics: Active, Consumed, Balloon, Compressed, Swapped. Healthy hosts run with light over-commit and zero swap.

An app team complains "the database VM is slow" — sees high CPU in their guest tools. esxtop shows %RDY at 28% on each of its 16 vCPUs. The host has 32 cores but is hosting 5 other 16-vCPU VMs. Diagnosis: gang-scheduling contention. Fix: trim the DB VM to 8 vCPU; trim the others to 4-8 each. Performance jumps despite the smaller "size" because each VM's vCPUs now schedule simultaneously almost every time.

Flashcards · CPU and memory (3)

CPU ready (%RDY) — what is it and what's healthy?
tap to flip
Time a vCPU was ready to run but couldn't get a core. Per vCPU: under 5% healthy, 10-20% stutter, over 20% serious pain. Sign of host CPU contention or oversized VM (too many vCPUs).
Memory reclamation order?
tap to flip
(1) TPS dedup → (2) Balloon (asks guest to release via Tools) → (3) Compression (compress cold pages) → (4) Hypervisor swap (last resort, disk-backed). Healthy hosts never reach step 4.
Why does adding vCPUs sometimes hurt performance?
tap to flip
Multi-vCPU VMs need their vCPUs co-scheduled. More vCPUs means harder to find that many simultaneously-free cores, which raises co-stop. The result: VM is paused waiting for a coherent timeslice. Often, fewer vCPUs = better.

Quiz · CPU and memory (2)

⏸ pause animation in 'CPU and memory' mode
Why is hypervisor swap considered a red alert?
Answer

Because it means physical RAM is exhausted and the host is moving VM memory to disk. Disk is ~1000× slower than RAM. Any VM whose pages are swapped will stall hard. Fix immediately by adding RAM, evicting VMs (vMotion), or reducing memory reservations.

An admin sees Active memory of 4 GB on a VM configured with 32 GB. Should they shrink the VM?
Answer

Probably yes — observe over a typical week. If active stays under 8 GB consistently, drop configured to 8-12 GB (with headroom). Releases the unused 20+ GB back to the pool. Right-sizing is a high-leverage operational habit.

Subtopic 8 of 8

How VMs compete for resources

Reservations · Limits · Shares — the three knobs that decide who wins under contention.

📖

Words you'll meet in this section

Three knobs per VM, per resource (CPU and memory). Each does a different job.

Reservation config
What it is: A guaranteed minimum of CPU (MHz) or memory (MB). The hypervisor will ALWAYS deliver this much, no matter how busy the host gets.
Use sparingly. A VM with a high reservation can prevent other VMs from powering on (admission control). For latency-sensitive workloads only.
Limit config
What it is: A maximum cap on CPU or memory the VM can consume — even if more is available. Default: unlimited.
Use very sparingly. Limits can starve a VM (memory limit forces ballooning even when host has free RAM). Almost always wrong as a tuning knob.
Shares config
What it is: A relative weight (Low / Normal / High / Custom number). When VMs compete, contested resource is split in proportion to shares.
The right knob for prioritization. Normal = 1000, High = 2000, Low = 500. Affects only contended resources.
Resource pool concept
What it is: A logical grouping of VMs (e.g., "Production," "Test") with its own reservations / limits / shares. Pools nest into a hierarchy.
Apply controls at the team / app level, not per-VM. Production pool gets higher shares; Test pool gets lower.
Admission control concept
What it is: The check that runs at VM power-on: "Is there enough unreserved capacity to satisfy this VM's reservation?" If no, power-on fails.
Prevents over-promising. Heavy reservations can cause power-on failures even on apparently-empty hosts.
Expandable reservations concept
What it is: A resource pool flag — child reservations can borrow from the parent pool's unreserved capacity.
Default ON. Lets nested pools share parent capacity gracefully.
Contention concept
What it is: The state where total demand for a resource exceeds supply. Shares only matter when contention exists.
When the host is calm, every VM gets what it asks for. Shares only kick in to mediate a fight.
When in doubt — Shares concept
What it is: Pragmatic rule. To prioritize a VM under contention, raise its shares. Don't reach for reservations or limits.
Shares is the proportional, well-behaved knob. Reservations and limits are blunt and create traps.

Reservations · Limits · Shares · what each one does

Reservation = floor. Limit = ceiling. Shares = relative weight when there's a fight. Reach for Shares first, Reservations only when latency really matters, Limits almost never.
★
Reach for Shares firstIf a VM needs to win under contention, raise its shares to High (2000). No traps. No admission-control surprises. Proportional. Done.
!
Memory Limits are almost always wrongA 16 GB VM with an 8 GB memory limit will balloon and swap inside the guest even on an empty host. The "savings" are paid in performance you'll never get back. Leave Limits at Unlimited unless you have a very specific reason.

What it is

Per VM (or per resource pool), vSphere exposes three knobs for each of CPU and memory. Reservation is a floor — the host guarantees this much. Limit is a ceiling — never more, even if available. Shares is a relative weight that decides how a contested resource is divided when total demand exceeds supply.

The vast majority of VMs should keep defaults (reservation 0, limit unlimited, shares normal). Reach for shares to prioritize, reservations sparingly for latency-sensitive workloads, limits almost never.

Before / After

Before

"How do I make this VM go faster?" Admin sets a Memory Limit thinking it caps consumption helpfully. The VM degrades anyway because limits force in-guest swap.

→
After

Admin sets Shares: High on the priority VM. Under contention, that VM gets 2× the resource. No traps, no in-guest swap, no admission-control surprises.

Analogy

A restaurant with limited tables. Reservation = "this booth is always saved for the regulars." Limit = "this party may never get more than four seats no matter how empty the place is." Shares = "if there's a wait, VIPs go in twice as fast." Most diners just walk in. Use Shares for VIPs. Use Reservations carefully for the booth. Avoid Limits — they cost goodwill without saving seats.

Quick check

A production VM should "always win" if there's CPU contention. Three options: reservation, limit, or shares. Which is the right knob?

Two ways to think about it

ELI5

Imagine a pizza shared between four kids. If everyone shows up hungry, you give the biggest kid two slices and the others one each — that's Shares. Saying "Mike always gets at least one slice" is Reservation. Saying "Sarah may never get more than one slice" is Limit. Most days nobody is starving and these don't matter.

ELI10

Three orthogonal controls per resource. Reservation: admission-control floor (host commits unreserved capacity). Limit: hard cap (forces in-VM reclamation if exceeded). Shares: proportional weight applied during contention. Resource pools wrap them at a hierarchical level. Default knob for prioritization is Shares; reservations are powerful but invasive (they pin RAM, block power-on); limits are almost always counterproductive.

Trading firm has a latency-sensitive matching engine. The CPU reservation guarantees 8 cores' worth of MHz so it never queues.
