Module 3 · Lesson 14 · how YAML becomes truth

The K8s API & YAML · Objects & CRDs

Every object is the same 4-section permit form: header (apiVersion+kind), identity (metadata), what-you-want (spec), and a blank for the inspector's stamp (status).

🚨 The 3 AM Nightmare

An engineer ran kubectl scale deployment api --replicas=10 to handle a slow service. It worked. Three days later, a CI deploy from the git-tracked YAML (which still says replicas: 3) ran. apply diffed, scaled DOWN to 3 mid-traffic-spike. 5xx surge. This lesson is about why imperative kubectl is dangerous, and why every K8s object has the same 4-section permit form.

🎯 If you remember nothing else: Every K8s object is the same 4-block form (apiVersion + kind + metadata + spec). kubectl explain is the manual. Never use imperative kubectl in production. CRDs make custom types first-class.

Section 1 · Concept

Every K8s object has the same 4 blocks

Everything in Kubernetes is a typed object: a Pod, a Service, a Deployment, a ConfigMap, your custom Foo. Every object — built-in or custom — has the same four-block structure and lives in the same etcd-backed object store. Once you've internalized this, every kubectl apply feels familiar.

apiVersion: apps/v1          # 1. Which form template (group/version)
kind: Deployment             # 2. Which form
metadata:                    # 3. Identity (name, namespace, labels, annotations)
  name: my-app
  namespace: default
  labels: { app: my-app, env: prod }
spec:                        # 4. What you want (desired state)
  replicas: 3
  ...
# status:                    # 5. What's actually true (filled in by controllers, not you)

apiVersion + kind = "which API endpoint, which schema." metadata identifies the object. spec is what you declared. status is what controllers later wrote back — don't put it in your YAML, you don't own it.

API groups. The API is partitioned into groups so it can evolve independently. core (v1): Pods, Services, ConfigMaps. apps: Deployments, StatefulSets. batch: Jobs. networking.k8s.io: Ingress, NetworkPolicy. rbac.authorization.k8s.io: Roles, ClusterRoles. Each has its own version lifecycle (v1alpha1 → v1beta1 → v1).

kubectl explain is the manual. kubectl explain Pod.spec.containers shows the documented schema for any field. It reflects exactly what your cluster's API server accepts (including CRDs), not the docs site for some other version. Treat it as the source of truth.

Server-side validation. The API server validates every submission against its OpenAPI schema. Typos like replicaCount: 3 (should be replicas) are rejected before persistence. In CI, always kubectl apply --server-side --dry-run=server — that's the cluster's actual schema, not a client-side guess.

CRDs (Custom Resource Definitions) are the most powerful K8s feature. Define your own object type, apply the CRD, and the API server immediately serves it with the same RBAC, audit, validation, and storage as built-ins. Pair with a controller and you've extended K8s with a first-class new abstraction. cert-manager, Argo CD, Crossplane, Istio, Prometheus Operator — all CRDs + controllers.

Section 1.5 · The 4 blocks + CRDs

Each one, one tile

📑

1 · apiVersion + kind

which form, which version

Tells the API server which group/version to dispatch to and which schema to validate against. v1 alone = core group.

🏷️

2 · metadata

name · namespace · labels · annotations

Identity + selection. Labels are indexed and used for selection. Annotations are free-form metadata, not selectable.

✏️

3 · spec

desired state · you write

Your declared intent. Each kind has its own spec schema — see kubectl explain <kind>.spec for the fields.

🔖

4 · status

actual state · system writes

What's actually true. Read-only from your perspective. Controllers update it as they reconcile.

🧩

5 · CRDs

custom form types

Define your own object type. Same RBAC · audit · validation · storage as built-ins. Pair with a controller (operator).

All five compound: apiVersion + kind + metadata + spec + status + CRDs = the whole declarative API surface of K8s.

Section 1.8 · kubectl explain · the manual

Stop guessing field names

Your cluster ships with the manual built in. kubectl explain <kind>.<path> shows the documented schema of any field, with types and descriptions. Reflects your cluster exactly — including CRDs.

$ kubectl explain Pod.spec.containers KIND: Pod VERSION: v1 RESOURCE: containers <[]Object> DESCRIPTION: List of containers belonging to the Pod. Containers cannot currently be added or removed. There must be at least one container in a Pod. FIELDS: name <string> -required- Name of the container. Must be a DNS_LABEL. image <string> Container image name. ports <[]Object> List of ports to expose from the container. resources <Object> Compute Resources required by this container. securityContext <Object> Security options the container should run with. ... # --recursive prints the entire tree, all the way down.

Pro: kubectl explain certificate.spec --recursive works for CRDs the moment they're installed. No "is this in the docs yet?" guesswork.

Section 1.9 · Declarative vs imperative

Two ways to make changes — only one works in production

✓ Declarative

kubectl apply -f deployment.yaml

You write a YAML describing what should exist; the cluster diffs against current state and converges. GitOps-friendly: the YAML in git is the truth. Audit trail = git history.

✗ Imperative

kubectl create deployment … kubectl scale --replicas=10 … kubectl run …

Direct mutation. No diff. No audit. No git history of what changed. Fine for ad-hoc poking and learning. Disastrous in production: out-of-band changes silently divert from your YAML in git.

The strongest version of declarative is GitOps (Lesson 06): an Argo CD or Flux controller continuously syncs your git repo to the cluster, reverting any out-of-band kubectl changes automatically.

Section 1.10 · Labels vs annotations

Two metadata fields that look the same and aren't

labels

Indexed. Used for selection: Services find Pods by labels, Deployments manage Pods by labels, kubectl filters by labels. Keep them short, structured, and meaningful.

app: my-app · env: prod · tier: frontend

annotations

Free-form. Not indexed, not used for selection. Hold metadata humans or tools read: build SHA, deploy timestamp, on-call team, ingress controller hints, prometheus scrape config.

git-sha: a3f9c2 · deployed-by: alice · prometheus.io/scrape: "true"

Rule of thumb: if you'd ever write -l key=value to find it, it's a label. Otherwise it's an annotation.

⏸ Pause and check

Which of these is not a mandatory block in every Kubernetes YAML?

Answer: b. status is filled in by controllers, not by you. The four mandatory blocks are apiVersion, kind, metadata, spec.

Section 2 · Before & After

Five tools, five formats vs one form everywhere

The K8s API isn't just YAML; it's a uniform, validated object model that subsumes config formats from a dozen older tools.

Same 4 sections, different kind. Validated by one schema, stored in one etcd, audited by one log, secured by one RBAC. CRDs (last row) are first-class — same shape.
Before · 5 tools, 5 formats

Nginx config, Terraform HCL, systemd unit, Ansible YAML, crontab. Each validated by its own tool, stored separately, audited by different mechanisms. "What's running in prod?" requires inspecting 5 systems.

After · K8s objects

Same 4-block form for Deployment, Service, ConfigMap, CronJob, and your custom Certificate (CRD). Validated by one OpenAPI schema. Stored in one etcd. kubectl get all,certificates = the whole inventory.

What changed: a uniform object model. Every operational concern is now "just another typed object you read/write through one API."

Section 3 · Analogy

The permit office

Kubernetes is a permit office for your software. Every change requires a permit, and every permit is the same standard 4-section form.

You walk in with a filled form (your YAML). The form has four sections: at the top, the type-of-permit declaration (apiVersion: apps/v1, kind: Deployment — which form template, which version). Below that, the identity section (metadata: name, namespace, labels, annotations). Then the what-you-want section (spec — the only part you really write). The fourth section, status, is left blank — that gets stamped in later by the inspector (a controller) when they verify the work.

The clerk at the counter (the API server) looks up the form template, checks every field is filled correctly (server-side validation against the OpenAPI schema), and either accepts the form or points to the typo. If accepted, the form goes into the filing cabinet (etcd) and a watch event fires.

Different permit types are filed at different counters (API groups). core/v1 for Pods and Services, apps/v1 for Deployments, batch/v1 for Jobs. The handbook on the desk that lists every field of every form is kubectl explain.

If the office doesn't have a form for what you need, you can register your own form template (a CRD). Once registered, the office accepts your custom permits with the same validation and storage as the built-in ones. Pair it with your own inspector (controller / operator) and you've extended the office without forking the codebase.

Section 4 · Two-level explanation

Two ways to hear it

ELI5 · explain like I'm 5

A permit office. Every form has the same four parts: what kind of form, who's asking, what you want, and a blank space for the inspector to stamp. The clerk checks your form for typos. If it's good, it goes in the filing cabinet. If you need a kind of form they don't have, you can invent one and they'll start accepting it.

ELI10 · explain like I'm 10

Every Kubernetes object is YAML with four blocks: apiVersion (which group/version), kind (which type), metadata (name, namespace, labels, annotations), and spec (what you want). A fifth block, status, is filled in by controllers. Resources are partitioned into API groups (core, apps, batch, networking.k8s.io...) each with versions (v1alpha1 / v1beta1 / v1). The API server validates against an OpenAPI schema — kubectl explain Pod.spec.containers shows that schema. Use kubectl apply -f (declarative, GitOps-friendly) instead of kubectl create / run (imperative, no audit). When K8s doesn't ship a type you need, write a CRD. The API server immediately serves it with the same RBAC, audit, and validation as built-ins. cert-manager, Argo CD, every operator — all CRDs.

Section 5 · Real-world scenarios

Where the API model shows up daily

kubectl explain saved hours of guessing the field name

A team integrating with cert-manager guessed spec.commonNames. Got "field not declared in schema." Tried spec.dnsName. Same error. Finally ran kubectl explain certificate.spec --recursive. Saw spec.dnsNames (plural). Five-second fix once they knew. Lesson: the cluster's own schema is authoritative; don't guess or copy from blog posts that may be from a different version.

Imperative kubectl scale disaster in production

An engineer ran kubectl scale deployment api --replicas=10 to handle a slow service. It worked. Three days later a CI deploy from the git-tracked YAML (which had replicas: 3) ran. apply diffed and scaled DOWN to 3 mid-traffic-spike. 5xx surge. Lesson: never use imperative commands on prod state — they don't show up in git.

cert-manager CRDs let teams declare TLS as K8s objects

Before cert-manager, getting a Let's Encrypt cert involved certbot, copying PEM files into a Secret, scheduling renewals. With cert-manager (a Certificate CRD + controller), you write 10 lines of YAML: "I want a cert for app.example.com, store in Secret app-tls." The controller talks to Let's Encrypt, completes ACME, fetches, writes the Secret, renews 30 days before expiry. Same kubectl apply workflow, same RBAC, same audit.

Server-side validation caught a typo before it shipped

A PR added replicaCount: 5 to a Deployment YAML. Local kubectl apply --dry-run=client did NOT catch it (client-side validation is shallow). CI used kubectl apply --server-side --dry-run=server. The cluster's schema rejected: unknown field "spec.replicaCount". Caught at PR review, never hit prod. Lesson: always --server-side --dry-run=server in CI.

Section 7 · Misconceptions, flashcards & quiz

Lock it in

Three common misconceptions to clear up first, then the flashcards and quiz.

Common Misconceptions

Myth: CRDs are second-class objects.
Truth: Once registered, they get the same RBAC, audit, validation, and storage as built-ins. cert-manager's Certificate and a Pod are equivalent in the API server's eyes.
Myth: kubectl run and kubectl create are the right way to deploy.
Truth: Those are imperative — direct mutations with no audit trail. Production should always use kubectl apply -f (and ideally GitOps on top).
Myth: Labels and annotations are interchangeable.
Truth: Labels are indexed and used for selection (-l app=foo). Annotations are free-form metadata, not selectable. If you'd ever filter by it, it's a label.

You're staring at this YAML and don't know if it's valid. What's wrong, and what's the fastest way to discover it?

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicaCount: 3
  template: ...
answerThe field is replicas, not replicaCount. Fastest discovery: kubectl explain deployment.spec — lists every valid top-level field of a Deployment spec with the type. You'll see replicas <integer> and immediately know what to use. --recursive for the full tree. To catch this in CI: kubectl apply --server-side --dry-run=server talks to the actual API server and validates against the OpenAPI schema. Returns error: unknown field "spec.replicaCount". Client-side dry-run won't catch it because client-side validation is shallow.

Words from this lesson · open if you want a quick reference
apiVersion
Group/version string at the top of every YAML. Tells the API server which schema to dispatch to. v1 = core; apps/v1, batch/v1, etc. = grouped.
kind
The object type within the apiVersion. Pod, Deployment, Service, ConfigMap, Certificate, etc.
metadata
Identity block: name, namespace, labels (indexed, used for selection), annotations (free-form metadata).
spec
Desired state — the only block you really write. Each kind has its own spec schema visible via kubectl explain.
status
Actual state, written by controllers. Read-only for you. Don't include it in your applied YAML.
API group
Partition of the API allowing independent versioning. Each group has its own URL prefix and lifecycle (alpha → beta → stable).
kubectl explain
Built-in schema browser. kubectl explain <kind>.<path> shows fields, types, and descriptions. Add --recursive for the tree.
OpenAPI schema
Machine-readable definition of every kind's structure. The API server validates submissions against it. kubectl explain reads it.
declarative
kubectl apply -f. You declare desired state; the cluster diffs and converges. GitOps-friendly. Default in production.
imperative
kubectl create / run / scale. Direct mutation. Fine for ad-hoc; banned in production.
CRD
Custom Resource Definition. Registers a new typed object with the API server. Validated, stored, RBAC'd, audited like built-ins.
operator
CRD + controller. Encodes domain knowledge (provision DB, rotate cert, manage cluster) as a first-class K8s primitive.
