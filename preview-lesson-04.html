Words you'll meet in this lesson

Sixty years of computing history brings a few terms you may not have seen. Plain language for each.

Mainframe hardware
What it is: A large, very expensive, highly reliable computer system designed for high-volume data processing. IBM's System/360 and z/Series are the classic examples.
The first machines to support virtualization (1960s). Modern x86 virtualization borrowed the patterns IBM invented decades earlier.
CP/CMS · CP-67 tool
What it is: IBM's pioneering virtualization software (1968). CP = Control Program (the hypervisor); CMS = Conversational Monitor System (the small OS run inside each VM).
The first true hypervisor. Established the patterns we still use — partitioning hardware, isolating workloads, sharing one machine across many users.
x86 hardware
What it is: The instruction-set architecture used by Intel and AMD chips — the processors in almost every PC and commodity server.
VMware's original breakthrough was making virtualization work on x86, which (unlike mainframes) wasn't designed for it. That breakthrough enabled the rest of the modern stack.
Binary translation concept
What it is: A clever software technique VMware used to virtualize x86 before the hardware supported it. Rewrote certain CPU instructions on the fly so they could run safely inside a VM.
The technical insight that made VMware possible in 1998. Today's hardware-assisted virtualization (Intel VT-x, AMD-V) replaced binary translation, but it was the original magic.
VT-x · AMD-V hardware
What it is: CPU extensions Intel and AMD added (~2005-2006) specifically to support virtualization in hardware. Replaced VMware's binary-translation trick with native chip support.
Made virtualization fast and ubiquitous. Modern hypervisors require these extensions; old binary translation is gone.
Hardware-assisted virtualization concept
What it is: Virtualization where the CPU itself provides instructions for the hypervisor to use, instead of requiring software trickery. Built on VT-x / AMD-V.
Faster, simpler, more secure than the original software-only approach. The era we're in now.
Cloud computing (origin) tool
What it is: AWS launched EC2 in 2006 — virtual machines as an on-demand service. The technology that made this possible: virtualization.
Cloud is just virtualization at scale, with self-service APIs. Without VMware-style virtualization, AWS / Azure / GCP couldn't exist.
Containers · Docker tool
What it is: A lighter-weight form of isolation than full VMs. Containers share the host OS kernel; VMs have their own. Docker (2013) popularised them.
Containers run on top of VMs in most production environments — extending the virtualization stack one level deeper.
VMware Workstation tool
What it is: VMware's first product (1999). Type-2 hypervisor for desktop OSes — let developers run multiple operating systems on a single PC.
Came before vSphere. Showed the technical approach worked. ESXi (server-class) followed in 2001.
ESX · ESXi tool
What it is: VMware's bare-metal hypervisor. ESX (2001) had a Linux-based service console; ESXi (2007) is leaner, "thin" — only the hypervisor.
ESXi is what runs in modern data centers. ESX was retired around 2010.
Server consolidation era concept
What it is: The 2005-2010 wave when enterprises massively replaced racks of single-purpose servers with consolidated VMs.
The economics were impossible to ignore — one host running 20 VMs cost a fraction of 20 hosts. This wave is when VMware became dominant.
SDDC (origin) concept
What it is: Software-Defined Data Center. The vision (mid-2010s) where compute, storage, and networking are all virtualized and managed as software. Detailed in lesson 7.
vSphere virtualized compute. vSAN added storage. NSX added networking. Together: the full SDDC.

Section 1 · Concept

Six moments, sixty years

Virtualization isn't new. It's older than personal computers. The technology you use today — vSphere, AWS EC2, Azure VMs — descends from work IBM did on mainframesA mainframe is a large, very expensive, highly reliable computer system designed for high-volume data processing. IBM's System/360 and z/Series are the classic examples. Mainframes were the first machines to support virtualization. in the 1960s.

The story has six big moments.

1960s. IBM builds CP/CMS, the first hypervisor, on its System/360 mainframes. The goal: let one $20-million machine support many users running independently. The term "virtual machine" is born here.

1998. Five Stanford researchers found VMware. Their insight: the techniques that worked on mainframes can work on commodity x86x86 is the instruction-set architecture used by Intel and AMD processors — the chips in nearly every PC and commodity server. Mainframe instruction sets were different and supported virtualization more easily; making virtualization work on x86 was VMware's original technical breakthrough. hardware too. This was not obvious at the time.

2001. VMware ESX Server ships — the first Type-1 hypervisor for x86 servers. Virtualization moves from curiosity to enterprise infrastructure.

2003. vMotion ships. VMs move between hosts while running. The "magical" capability becomes ordinary.

2006. AWS launches EC2. Public cloudPublic cloud refers to compute, storage, and networking services rented over the internet from providers like AWS, Microsoft Azure, and Google Cloud. Every public cloud is built on virtualization — the ability to run many independent VMs on shared physical hardware. is virtualization at hyperscale, rented by the hour.

2013. Docker popularizes containersContainers are a lighter-weight alternative to VMs. They package an application with its dependencies but share the host operating system. Most production container workloads run inside VMs, not instead of them — containers complement virtualization rather than replace it.. They run alongside VMs, not as a replacement.

Each era took an idea that worked at one scale and made it work at the next scale up.

i
The recurring patternEvery era took an existing idea (partitioning, sharing, isolation) and made it work at a bigger scale. Mainframes → x86 servers → public cloud → containers. Each generation thought "this is the end of the road"; each was wrong. The next abstraction is always coming.
★
Why the history mattersModern vSphere features have names that hint at their origins. CP/CMS lineage shows up in vCenter's clustering. Mainframe-era partitioning reappears in vSAN. Knowing the lineage makes the present make sense.

Section 2 · Before and after

A mid-sized company, 1996 vs 2006

Before VMware (1996)

One server, one workload

Windows NT, Linux, NetWare on dedicated physical servers. Buying a server is a capital purchase. Refresh cycles are three years. Disaster recovery is warm-spare hardware in another room.

The only people getting many OSes on one box are the ones running mainframes — and those cost millions.

After (2006)

Many workloads, fewer machines

Same workloads as VMs on a fraction of the hardware. New workloads come up in minutes. DR is replicating VM files. Mainframe-class capability is now available on commodity x86 servers at a tenth of the cost.

Twenty years later, public cloud rents this same model by the second.

Section 3 · The analogy, extended

The city, over time

The apartment-building analogy from Lesson 1 is itself a piece of history.

In the 1960s, only a few cities had skyscrapers — and they were the financial capitals. Mainframes were the same: a few institutions had them, and only big ones could afford the elevators, the structural engineering, the maintenance.

In 1998 VMware figured out how to build a small apartment building on a regular street, using ordinary materials. Suddenly any city could have one.

By 2010, the cloud era was every-skyscraper-everywhere. The bones of every skyscraper are still virtualization.

Quick check

A developer in 2024 deploys their app to AWS Fargate using only containers. They never log into a hypervisor. Is virtualization still involved?

Section 4 · At your level

The same idea, two depths

Long ago, only really big computers (mainframes) could pretend to be lots of computers. Then in 1998, a company called VMware figured out how to make small computers do the same trick. Soon every company was using it. Now the cloud — Amazon, Google, Microsoft — runs the same trick at a giant scale. So the cloud is built on virtualization that is built on a 60-year-old idea.

Virtualization originated on IBM mainframes in the 1960s, where a hypervisor called CP/CMS let one expensive mainframe support many independent users. The technology stayed mainframe-only for thirty years because nobody knew how to apply it to commodity x86 hardware.

In 1998 VMware solved the x86 problem. ESX Server (2001) brought hypervisors to commodity datacenter servers. By the late 2000s most enterprises were virtualized. AWS (2006), Azure, and GCP all built their compute services on virtualization at hyperscale. Containers (Docker, 2013) added a lighter abstraction that mostly runs on top of VMs, not instead of them. Today virtualization is invisible infrastructure under nearly every internet-facing service.
