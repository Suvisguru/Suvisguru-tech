Configuring vSphere Storage

Hosts and a vCenter without storage hold no VMs. vSphere consumes storage as datastores — logical containers that the hypervisor mounts. Six moves to understand the four datastore types and the policies that govern them.

Subtopic 1 of 6

Storage overview — datastores and the four types

The unit of storage vSphere consumes, and the four backings.

📖

Words you'll meet in this section

Storage in vSphere comes in four flavours each with their own acronyms. Here's each one in plain language.

Datastore concept
What it is: The logical container vSphere uses for storage. Hosts mount datastores; VMs are stored on them as files (.vmdk, .vmx, snapshots, ISOs).
"Where do VMs live?" → on a datastore. Every operation that creates a VM asks which datastore.
Block storage concept
What it is: Storage that exposes a raw chunk of bytes (a "block device") that the host puts a file system on top of. Like a USB drive your OS formats.
VMFS sits on top of block storage. The host owns the filesystem, the array owns the bytes.
File storage concept
What it is: Storage that exposes a network filesystem (NFS, SMB) — the array owns the filesystem; the host just mounts it.
NFS is file storage. Simpler to provision than block; no LUN dance, no multipathing.
VMFS · Virtual Machine File System tool
What it is: VMware's clustered file system that lives on top of a block LUN. Multiple ESXi hosts can mount the same VMFS volume simultaneously and share VMs.
VMFS is the magic that lets a SAN LUN be shared across many hosts safely. Without VMFS, only one host could write to the LUN at a time.
NFS · Network File System tool
What it is: A Unix-world file-sharing protocol. The NAS exports a directory tree; clients mount it as if it were local.
In vSphere, NFS is one of the four datastore types. Simpler to provision than VMFS.
vSAN tool
What it is: VMware's software-defined storage. Pools the local disks (SSDs, NVMe) on every host in a cluster into one shared datastore. No external array.
"Hyperconverged" pattern — compute and storage in the same boxes. One vendor relationship instead of two.
vVols · Virtual Volumes tool
What it is: A storage model where the array tracks each VM's storage as separate objects (config, disk, swap, snapshot). Requires a vVol-capable array.
Per-VM array-native operations: snapshots, replication, encryption all per-VM, not per-LUN. The most modern option.
SAN · Storage Area Network concept
What it is: A dedicated network for block storage traffic between servers and arrays. Usually Fibre Channel, sometimes iSCSI over Ethernet.
SAN is the wire/fabric; LUN is the chunk of storage; VMFS is the filesystem. All three together = the classic block-storage pattern.
NAS · Network-Attached Storage concept
What it is: A storage appliance you talk to over IP using a file protocol (NFS, SMB). The opposite of SAN — file vs block.
NAS is what you mount NFS datastores from. Easier to provision, fewer moving parts than SAN.
HCI · Hyperconverged Infrastructure concept
What it is: An architecture pattern where compute, storage, and (sometimes) networking live in the same boxes — pooled and managed as one. vSAN is the storage half of HCI.
Avoids the separate-storage-array purchase and skill set. One vendor, one operating model.
VM files (.vmdk, .vmx) concept
What it is: The files that make up a VM. .vmx = config; -flat.vmdk = disk data; .nvram = firmware state; plus log files.
A VM is a folder of files on a datastore. Knowing the file types makes the rest of vSphere less mysterious.
Datastore choice concept
What it is: The decision: VMFS, NFS, vSAN, or vVols. Driven by what storage you already have and what features you need.
No "right" answer — depends on existing infrastructure. New environments lean toward vSAN; existing FC shops lean toward VMFS.

At a glance

A VM is stored on a datastore. Four ways the datastore can be backed: VMFS (shared block), NFS (file from a NAS), vSAN (cluster-pooled local disks), vVols (per-VM objects on a smart array).
★
Pick by what you already haveAlready have an FC SAN? VMFS. Have a NAS? NFS. Greenfield with capable servers? vSAN. Have a vVol-capable array? vVols. There's no single right answer — the decision is driven by existing infrastructure and team skills.
i

Subtopic 2 of 6

VMFS datastores on block storage

VMware's clustered file system on shared block storage.

📖

Words you'll meet in this section

Block-storage vocabulary. Mostly hardware-and-protocol terms.

LUN · Logical Unit Number hardware
What it is: A chunk of block storage from an array, presented to one or more hosts as a "disk." Each LUN has a unique identifier.
When the storage team "carves a 2 TB LUN," they're creating a chunk for the host to put VMFS on. The unit of block-storage allocation.
HBA · Host Bus Adapter hardware
What it is: A specialized card on the host (Qlogic, Emulex) that talks to a storage fabric. FC HBAs talk Fibre Channel; iSCSI initiators talk iSCSI over Ethernet (can be hardware or software).
The bridge between the host's PCIe bus and the storage network. ESXi names them vmhba0, vmhba1, etc.
FC · Fibre Channel protocol
What it is: A purpose-built fabric for storage traffic. Very low latency, high cost. Used in big enterprise SAN environments.
FC was the dominant SAN protocol for decades. Newer environments often use iSCSI or NVMe over Fabrics instead.
iSCSI protocol
What it is: Block storage over TCP/IP. Cheaper than FC because it rides Ethernet. Common in mid-market and labs.
"SCSI commands wrapped in TCP/IP packets." Same VMFS on top — the fabric is just IP networking.
VMFS-6 · VMFS-5 tool
What it is: Two versions of VMFS. VMFS-6 is the latest (since vSphere 6.5); VMFS-5 was the prior major version. Each has different on-disk layouts and features.
New datastores should always be VMFS-6. Older VMFS-5 datastores can be upgraded but not in-place — you'd Storage vMotion off, recreate, Storage vMotion back.
Distributed locking concept
What it is: A coordination mechanism that ensures only one host writes to a given block at a time, even when many hosts share the same VMFS volume.
VMFS's killer feature. It's what makes shared LUNs safe — without it, two hosts writing the same block would corrupt the data.
Multipathing concept
What it is: Configuring multiple physical paths from host to storage array — through different HBAs, different fabric switches, different array controllers.
A single cable / HBA / switch / controller failure shouldn't lose the storage. Multipathing distributes I/O across paths and fails over automatically.
PSA · Pluggable Storage Architecture tool
What it is: ESXi's framework for handling multipathing. Vendors plug their own multipathing modules into PSA.
Lets array vendors ship optimised drivers (Dell PowerPath, EMC, NetApp). Most environments stick with VMware's built-in modules.
MRU · Most Recently Used (path policy) config
What it is: A multipathing policy: keep using whichever path was last working. Default for active/passive arrays.
For arrays where only one controller serves a LUN at a time, MRU avoids "bouncing" the LUN between controllers.
Round Robin (path policy) config
What it is: A multipathing policy: distribute I/O across all available paths in turn.
Best for active/active arrays (most modern arrays). Maximises aggregate bandwidth and reduces single-path bottlenecks.
WWN · World Wide Name protocol
What it is: A 16-character hex identifier baked into FC HBAs (like a MAC address but for FC). Each port has its own WWN.
When you're "presenting a LUN to a host," the storage team adds the host's WWN to the LUN's access list. Wrong WWN = host can't see LUN.
IQN · iSCSI Qualified Name protocol
What it is: An iSCSI identifier — the iSCSI equivalent of a WWN. Looks like iqn.1998-01.com.vmware:esx-01-12345.
Same role as WWN: storage uses IQNs to control which hosts can access which LUNs.

At a glance

VMFS on a shared LUN. Multipathing means a cable, switch, or controller can fail without taking the storage down.
⚠
Coordinate WWN/IQN with the storage teamFor a host to see a LUN, the storage team must add the host's WWN (FC) or IQN (iSCSI) to the LUN's access list. New host not seeing a LUN? Almost always missing from the access list. The fix is on the array side, not the host side.
★

Subtopic 3 of 6

NFS datastores

File storage over IP. Simpler to provision than block.

📖

Words you'll meet in this section

NFS terms, plus a few that distinguish it from VMFS.

NFS · Network File System protocol
What it is: A Unix-world protocol where a server (NAS) exports a directory tree, and clients mount it over IP as if it were local.
In vSphere, NFS is one of the four datastore types. Storage admin owns the filesystem; vSphere just stores files in it.
NFSv3 · NFSv4.1 protocol
What it is: Two NFS protocol versions. v3 is older (no Kerberos, single-path). v4.1 is modern (Kerberos auth, multipath, better locking).
New deployments should use NFSv4.1 if the NAS supports it. Many existing environments still run v3.
NAS appliance tool
What it is: A purpose-built file server (NetApp, Synology, Pure, Dell EMC Unity) that exports NFS and/or SMB shares.
The NAS owns the storage and the filesystem. vSphere just mounts what the NAS exports.
Export · share config
What it is: A directory the NAS makes available to clients over NFS. Has an access list (which IPs can mount it) and read/write permissions.
"Mount the export" is what the host does. Each NFS datastore in vSphere = one export from the NAS.
Mount point config
What it is: The local name a host uses for an NFS export. In vSphere this is the "datastore name" you pick.
When you see a path like /vmfs/volumes/ds-nfs-prod on the host, the part after volumes/ is the mount point.
Kerberos (NFS auth) security
What it is: Authentication via tickets, used by NFSv4.1 to verify the host is allowed to mount an export. Eliminates "trust the IP."
More secure than IP-based access lists. Required in some compliance environments. NFSv3 doesn't support it.
Service account (NFS) config
What it is: An account on the NAS / AD used by ESXi to authenticate when NFSv4.1 with Kerberos is enabled.
Each host needs the credentials. Not human-driven — set once during datastore mount.
Read-only export config
What it is: An NFS export that hosts can read but not write. Used for ISO repositories, template stores.
Don't try to host VMs on a read-only export — they need to write. Read-only is for reference data.
File-level visibility concept
What it is: The ability for storage admins to see individual VM files (.vmdk, .vmx) on the array. NFS gives this; VMFS doesn't (VMFS volumes are opaque to the array).
Useful for storage-side ops: backups, replication, analytics. The array can act on files, not just blocks.
Single-path vs multipath (NFS) concept
What it is: NFSv3 supports only single-path; NFSv4.1 introduces multipath (called nconnect / pNFS depending on flavour).
A reason to prefer NFSv4.1 — better resilience and aggregate bandwidth.
RDM · Raw Device Mapping concept
What it is: A vSphere feature that lets a VM see a SAN LUN directly (bypassing VMFS). NFS doesn't support RDM.
Used for shared-disk clusters (Microsoft Cluster Service, certain Oracle setups). One reason a shop might still pick VMFS over NFS.
Provisioning speed config
What it is: How long it takes to make a new datastore available. NFS = 5 minutes (NAS team carves an export, vSphere mounts). VMFS = longer (LUN, zoning, multipathing, format).
If "speed of getting new storage" is a pain point, NFS often wins.

At a glance

Host mounts a share from a NAS appliance over IP. The NAS owns the file system; vSphere stores VM files in it.
★
NFS is the operationally simpler choiceIf your storage team can carve NFS exports faster than they can carve LUNs (often the case), and you don't need RDM, NFS is usually the easier provisioning path. Less to coordinate, no LUN dance, no multipathing config, easier to grow.
i
Prefer NFSv4.1 for new deploymentsIf your NAS supports it. NFSv4.1 brings Kerberos auth, multipathing, and better locking. NFSv3 is fine for legacy setups but should not be the default choice for new datastores.
⚠
Things NFS doesn't doRDM (raw device mapping for shared-disk clusters like MSCS). Some specific high-IOPS storage arrays only certify with FC/iSCSI. If a workload needs these, you're on VMFS — full stop.

What it is

NFSNetwork File System — file-storage protocol where a host mounts a remote file system from a NAS appliance over IP. The NAS owns the file system; vSphere just stores files there. is the file-storage option. Instead of the host getting a raw block LUN and putting a file system on it, the host mounts a file system that the NAS appliance owns. vSphere's job is much smaller: just talk NFS, store files.

NFS in vSphere supports two versions: NFS v3 (older, no Kerberos) and NFS v4.1 (newer; Kerberos, multipath, better locking; recommended for new deployments).

What you configure on the host
  • Server address (NAS IP/hostname)
  • Share path (e.g., /vol/vsphere-prod)
  • Datastore name (vSphere's local label)
  • Read-only? (rare for VM datastores; useful for ISO repos)
  • NFS version
  • Kerberos credentials if NFS 4.1 with Kerberos

Why pick NFS over VMFS: simpler to provision (NAS owns the FS), file-level visibility on the array, often easier to grow, uses existing IP network. Why pick VMFS over NFS: FC-style latency, RDM/MSCS support, existing FC investment.

Subtopic 4 of 6

vSAN — hyperconverged storage from local disks

Compute and storage in the same boxes.

📖

Words you'll meet in this section

vSAN has its own dialect — disk groups, FTT, OSA/ESA. Read these once and the rest clicks.

vSAN tool
What it is: VMware's software-defined storage. Pools the local disks (SSDs, NVMe) on every host in a cluster into one shared datastore.
Eliminates the separate storage array. Compute and storage scale together by adding hosts.
HCI · Hyperconverged Infrastructure concept
What it is: A pattern: combine compute, storage, and (sometimes) networking in the same boxes, managed as one. vSAN is VMware's HCI storage layer.
One vendor relationship instead of two (server + array). Easier procurement, simpler operating model.
Disk group hardware
What it is: A vSAN OSA concept. One cache disk (fast SSD/NVMe) plus 1-7 capacity disks (cheaper SSD). Each host has 1-5 disk groups.
Cache absorbs writes; capacity stores data. The two-tier design is what makes OSA fast.
OSA · Original Storage Architecture tool
What it is: The classic vSAN architecture (since vSAN 6.0). Two tiers: cache + capacity disk groups.
Most existing vSAN clusters run OSA. Mature, well-understood.
ESA · Express Storage Architecture tool
What it is: The newer vSAN architecture (vSAN 8). Single-tier — all-NVMe, no separate cache disk.
Higher performance, simpler hardware story. The future direction. New deployments on NVMe-ready hardware should pick ESA.
FTT · Failures To Tolerate config
What it is: A storage-policy setting that says how many simultaneous failures vSAN should survive without losing data. FTT=1 = data has 2 copies; FTT=2 = 3 copies.
FTT is THE knob you turn for resilience. FTT=1 is the default and good enough for most workloads.
Mirroring (RAID-1) concept
What it is: A resilience method where vSAN keeps full copies of data on different hosts. FTT=1 mirroring = 2 full copies.
Simple to reason about, fast to read (any copy can serve I/O). Costs 2× storage for FTT=1.
Erasure coding (RAID-5/6) concept
What it is: A resilience method where vSAN stores data + parity blocks across hosts. RAID-5 (FTT=1, 4+1) costs 1.33×; RAID-6 (FTT=2, 4+2) costs 1.5×.
More space-efficient than mirroring, but slower and uses more CPU. Use for less-performance-sensitive workloads where cost matters.
vSAN object concept
What it is: The unit vSAN distributes — a VM's disk, swap, snapshot, etc. each is an object. Objects are split into components and spread across hosts.
Per-VM granularity. vSAN protects objects independently — different VMs can have different policies.
Storage policy (vSAN) config
What it is: A named profile of vSAN settings (FTT, mirroring vs erasure coding, IOPS limits, encryption) that you apply to a VM. Each VM can have a different policy.
"This database VM gets Gold (FTT=1 mirroring + encrypted); this test VM gets Bronze (FTT=0)." vSAN places data accordingly.
vSAN ReadyNode tool
What it is: A pre-validated server config from a hardware vendor (Dell, HPE, Lenovo) certified by VMware to run vSAN.
Removes the "is this disk on the HCL?" anxiety. Buy a ReadyNode and you know vSAN will run on it.
vsanDatastore config
What it is: The single datastore vSAN creates from the cluster's pooled disks. Visible to every host in the cluster.
All vSAN VMs land here. Capacity = sum of capacity disks across all hosts (minus FTT overhead).

At a glance

Each host contributes a disk group. The pool becomes one vsanDatastore. VM data has multiple copies (FTT=1 mirroring), so one host dying doesn't stop the VM.

What it is

vSANVMware's software-defined storage. Pools the local disks (SSDs, NVMe, HDDs) on each cluster host into one distributed datastore visible to all hosts. The foundation of VMware's hyperconverged infrastructure (HCI) story. takes the local disks (SSDs, NVMe, HDDs) on each host in a cluster and pools them into a single, distributed datastore that all hosts in the cluster see. This is HCI in vSphere terms: compute and storage in the same boxes.

How it works at a high level:

  • Each host contributes one or more disk groups (cache + capacity disks).
  • vSAN distributes each VM's data across the cluster as objects, with copies (mirroring) or parity (erasure coding) for resilience.
  • A storage policy on each VM decides protection level, performance tier, etc.
  • All hosts see one logical datastore: vsanDatastore.

Headline benefits: no external array; scale by adding hosts; per-VM policies; tight vSphere integration (built into ESXi).

Versions: OSA (Original Storage Architecture; cache + capacity disk groups, classic) and ESA (Express Storage Architecture; all-NVMe, single tier, future direction).

Requires its own VLAN (vmkernel adapter with vSAN service enabled on each host) and benefits from 25/100 GbE networking.

Subtopic 5 of 6

vVols — VM-centric storage on smart arrays

Each VM is a storage object the array tracks individually.

📖

Words you'll meet in this section

vVols introduces a few storage terms you won't see anywhere else.

vVols · Virtual Volumes tool
What it is: A storage model where each VM's storage is a separate object the array tracks individually — not just a file inside an opaque LUN.
Per-VM array-native operations: snapshots, replication, encryption all per-VM. The array can apply storage policies natively because it knows what's a VM.
VASA · vSphere APIs for Storage Awareness tool
What it is: The protocol vCenter uses to talk to a storage array's management interface. The array publishes capabilities ("I can do FTT=1, encryption, IOPS limits"); vCenter consumes them.
VASA is what makes per-VM array operations possible. Without VASA, vCenter has no way to ask the array for VM-level features.
VASA Provider tool
What it is: The actual implementation of VASA on the array side. A small service the array vendor runs that vCenter connects to.
Setting up vVols requires registering a VASA Provider in vCenter. Vendor-specific URL and credentials.
Storage Container config
What it is: A vVol-side concept replacing the LUN. A pool of capacity on the array that's exposed for vVols use.
When you create a vVol datastore, vSphere maps to a Storage Container. One container can hold many VMs' vVols.
Protocol Endpoint (PE) config
What it is: The data-path connection between host and array for vVols. A small "access proxy" on the array.
Looks like a tiny LUN to the host. Different from a regular LUN — it's how the host gets to the actual vVol objects, not where the data lives.
vVol object types config
What it is: Five types of vVol per VM: Config (the .vmx-equivalent), Data (each disk), Swap, Memory (suspend state), Other (snapshots, deltas).
Each type can have its own array-side characteristics. Snapshot vVols, for example, can have different replication settings.
Array-native snapshot concept
What it is: A snapshot taken at the array level (using array's own technology) instead of by the hypervisor. Much faster than VMFS-era snapshots.
A vVol snapshot is a small array-side metadata operation. Way faster than VMFS snapshot. Fewer performance penalties.
Per-VM replication concept
What it is: Replication policy applied at the VM granularity, by the array. Different VMs in the same datastore can have different replication targets and RPO.
VMFS replicates LUNs (granular = "all VMs in this LUN"). vVols replicate per-VM (granular = "this one VM goes to DR site").
vVol protocol options protocol
What it is: The data path can be FC, iSCSI, or NFS — vendor's choice. Same vVol concept; different transport.
Use whatever fabric the array supports. The vVol semantics are the same regardless.
vVol-capable array tool
What it is: A storage array whose vendor implements VASA / vVols. Pure Storage, NetApp, Dell EMC PowerStore, Hitachi VSP, IBM, and others.
Not every array supports vVols. Check before designing around it.
SPBM integration concept
What it is: vVols are the cleanest pairing with vSphere's Storage Policy-Based Management. Policies declared in vCenter are honored by the array natively.
Without vVols, SPBM has limited effect on traditional SAN datastores. With vVols, declarative policy actually works.
Per-VM visibility concept
What it is: Storage admins can see real VM names and per-disk usage on the array — instead of opaque VMFS files.
Operationally cleaner. "Which VM is consuming 800 GB?" becomes a question the array can answer directly.

At a glance

Instead of one big VMFS volume hiding many VMs, the array tracks each VM's storage as separate objects (config, disk, swap, snapshot) and applies per-VM policies natively.

Subtopic 6 of 6

SPBM and thin/thick provisioning

Declare what; vSphere finds where.

📖

Words you'll meet in this section

Two related but separate decisions: storage policy (declarative) and provisioning mode (per-disk).

SPBM · Storage Policy-Based Management tool
What it is: A vSphere framework where you declare what storage qualities a VM needs (mirrored, fast, encrypted) and vSphere finds a datastore that can honor it.
Separates "what does this VM need" from "which specific datastore." VMs follow capabilities, not specific physical resources.
Storage policy config
What it is: A named set of storage capabilities — "Gold = FTT=1 mirroring + encrypted + IOPS=0" or similar. Created in vCenter, applied to VMs.
Declarative. You don't pick datastores; you pick the policy and let vSphere place the VM on something compliant.
Capability concept
What it is: One specific storage trait — FTT level, IOPS limit, encryption on/off, latency tier. The atomic units that policies are built from.
Storage arrays publish capabilities to vCenter via VASA. Policies are combinations of capabilities.
Compliance status config
What it is: A continuous check that says whether each VM's actual storage matches its declared policy. "Compliant" = match. "Non-compliant" = drift.
Catch drift quickly. A VM whose storage no longer meets its policy gets flagged in vCenter.
Thin provisioning concept
What it is: A virtual disk only consumes physical storage as the guest writes data. A 100 GB thin disk holding 20 GB of data uses 20 GB on the datastore.
Lets you over-provision — total declared size > datastore capacity, fine as long as actual used stays below. The default and right for most workloads.
Thick provisioning · lazy-zeroed concept
What it is: Full disk size reserved up front, but blocks zeroed lazily on first write. Faster create than eager-zeroed; very slight perf hit on first writes.
Used when you want guaranteed space but don't need eager zero. Mid-tier provisioning.
Thick provisioning · eager-zeroed concept
What it is: Full size reserved AND every block zeroed at creation time. Slowest to create. Required for some features (Fault Tolerance, certain MSCS clusters).
Best subsequent write performance. Use only when needed — costs storage and time up front.
Over-commit concept
What it is: When the sum of all declared VM disk sizes exceeds the datastore's actual capacity. Possible only with thin provisioning.
Saves storage when VMs don't fill their disks. Risky if not monitored — actual usage growing toward capacity = VMs paused.
Datastore alarm config
What it is: A vCenter alert that fires when a datastore reaches a usage threshold (75%, 85%, 95%). Critical for thin-provisioned environments.
Without alarms, an over-committed datastore can fill up silently and pause every VM on it. Always have alarms in place.
FT · Fault Tolerance tool
What it is: A vSphere feature that runs a primary VM and a "shadow" secondary VM in lockstep — instant failover if the primary host dies. Covered briefly in lesson 10 (FT logging).
FT requires eager-zeroed disks. The reason eager-zeroed exists in your toolkit even though thin is the default.
Tag-based placement config
What it is: An SPBM feature where vCenter tags datastores (e.g., "tier-1," "PCI") and policies require certain tags. Different from capability-based placement.
Useful when array doesn't expose enough capabilities via VASA. You manually tag datastores by their actual properties.
Reserved IOPS concept
What it is: A storage-policy capability that guarantees a minimum number of IOPS to a VM (or sets a maximum, for noisy-neighbour control).
Critical workloads get guaranteed performance; misbehaving workloads get capped. Available on vSAN and vVol-capable arrays.

At a glance

Define policies (Gold/Silver/Bronze). Tag each VM. SPBM finds an eligible datastore and places the VM there. Thin or thick is a separate per-disk decision.

★

What it is

SPBMStorage Policy-Based Management — vSphere mechanism for defining storage capabilities (mirroring, IOPS, encryption, etc.) as named policies and assigning them to VMs. The hypervisor finds a datastore that can honor the policy and places the VM there. is vSphere's framework for assigning storage policies to VMs. Instead of "put VM-X on datastore-Y," you say "VM-X needs gold tier" and SPBM places it on a datastore that can honor gold tier.

A storage policy is a list of capabilities the storage must provide:

  • Number of failures to tolerate (FTT)

  • IOPS limit (or reservation)
  • Encryption
  • Tiering / pinning to specific media
  • Vendor-specific features the array exposes
