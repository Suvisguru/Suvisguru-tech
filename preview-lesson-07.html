Words you'll meet in this lesson

SDDC and cloud terminology — three-letter acronyms galore. Plain language for each.

SDDC · Software-Defined Data Center concept
What it is: An architecture pattern where compute, storage, and networking are all virtualized and managed as software. The opposite of fixed-purpose physical infrastructure.
SDDC is the destination VMware has been pulling the industry toward for a decade. vSphere + vSAN + NSX = the SDDC stack.
vSphere (compute layer) tool
What it is: The compute / virtualization layer of the SDDC. Provides VMs that run on pooled CPU and memory.
vSphere virtualized the first leg of the SDDC stool: compute. The other two (storage, network) followed.
vSAN (storage layer) tool
What it is: VMware's software-defined storage. Pools each host's local disks into one shared datastore. Detailed in lesson 11.
Replaces the external SAN/NAS array with a software layer running on the same servers as vSphere. Storage joins compute as software-defined.
NSX (network layer) tool
What it is: VMware's software-defined networking. Virtual networks, virtual firewalls, micro-segmentation — all defined in software, decoupled from physical switches.
Rounds out the SDDC. Lets the network be deployed and reconfigured at software speed instead of cabling speed.
vCenter / vRealize / Aria (management) tool
What it is: The management plane on top — vCenter for vSphere, plus vRealize/Aria suite for monitoring, automation, lifecycle.
SDDC is more than the three infrastructure layers — it's also the unified management on top that lets you treat them as one system.
Public cloud concept
What it is: Compute / storage / networking rented from a provider over the internet — AWS, Microsoft Azure, Google Cloud Platform. Pay per use.
Public cloud is virtualization at hyperscale, with self-service APIs. Every public cloud is built on SDDC-like patterns underneath.
Private cloud concept
What it is: An SDDC you run yourself in your own data center. All the cloud-like behaviors (self-service VMs, pooled resources) without using a public provider.
vSphere + vSAN + NSX is the most common way to build a private cloud. Combine with vCloud Director / Aria for cloud-portal experience.
Hybrid cloud concept
What it is: A mix — workloads run in both your private data center and a public cloud, with some integration between them.
Most large enterprises are hybrid. Pure private was the old world; pure public is rare; hybrid is the practical reality.
VMware Cloud on AWS tool
What it is: A service where AWS runs VMware's full SDDC stack on bare-metal AWS hardware. You get vSphere/vSAN/NSX as a managed service inside AWS.
Lets vSphere shops extend into AWS without rewriting workloads. Hybrid cloud done well — same vSphere on both sides.
Azure VMware Solution · Google Cloud VMware Engine tool
What it is: Same model as VMware Cloud on AWS but on Azure / GCP. Bare-metal hosts running the VMware SDDC stack as a managed service.
All three major clouds offer "VMware as a managed service." Pick by which cloud you're already in.
IaaS · PaaS · SaaS concept
What it is: Three cloud service tiers. IaaS = rent VMs (most VMware-shaped). PaaS = rent app platforms. SaaS = rent finished apps.
vSphere fundamentally provides IaaS. The other tiers build on top.
Multi-cloud concept
What it is: Using more than one public cloud (e.g., AWS + Azure) — sometimes with a private cloud too. Different from "hybrid" in that it's multiple publics.
Avoids single-vendor lock-in. Costs operational complexity. Most large enterprises end up multi-cloud whether they planned it or not.

Section 1 · Concept

Four layers, one architectural pattern

A Software-Defined DatacenterAn SDDC is a datacenter where every infrastructure layer (compute, storage, network, management) is delivered as software running on commodity hardware. The hardware is generic; the intelligence is in software. (SDDC) is a datacenter where every layer of infrastructure is delivered as software running on commodity hardware. Lesson 1 made compute virtualization concrete; SDDC extends the same idea to storage and networking.

Compute virtualization. Many VMs on each host. VMware's product is vSphere.

Storage virtualization. A storage pool that's software, not a specific physical SAN. VMware's product is vSANvSAN is VMware's software-defined storage product. It pools the local disks across many ESXi hosts into a single logical datastore. No separate SAN appliance needed.. AWS equivalent: EBS.

Network virtualization. Switches, routers, firewalls — all in software, riding on top of the physical network. VMware's product is NSXNSX is VMware's network virtualization product. Runs virtual switches, routers, firewalls, and load balancers in software on top of the physical network. Each application can have the network topology it needs without reconfiguring physical hardware.. AWS equivalent: VPC.

Management and automation. A control plane that operates compute, storage, and network as one. vCenter + vRealize. AWS equivalent: AWS Console + IAM.

Every public cloud is an SDDC at hyperscaleHyperscale refers to operating at the scale of major public cloud providers (AWS, Azure, GCP) — millions of servers, multiple regions worldwide, automated everything. The architectural pattern is the same SDDC; the size and rental model are different..

When AWS launched EC2 in 2006, what they had built was an SDDC the size of a city block, rented by the hour. Today every cloud provider operates SDDCs the size of regions. The cloud isn't magic. It's an SDDC at scale, with managed services on top.

Section 2 · Before and after

Three teams or one operational system

Before SDDC

Three vendor stacks, three teams

Compute, storage, network as separate stacks, each procured from a different vendor and managed by a different team. Adding capacity means coordinating three RFPs, three rack-and-stack projects.

New applications wait six months for network configuration approvals. Storage policies live in a Storage team's tooling that's invisible to Compute.

After SDDC (and public cloud)

One operational system

Compute, storage, network, and management are one system. New applications get their full infrastructure stack provisioned by software in minutes. Policy lives in code.

Public cloud takes this even further: you don't even own the hardware.

Section 3 · The analogy, extended

A city, software-defined

The apartment building has been our anchor through the course. Now zoom out.

A traditional datacenter is a city built before zoning laws — roads, plumbing, electrical, residential, commercial all built by different contractors in different decades. Adding a new neighborhood requires coordinating across all of them.

An SDDC is a city designed end-to-end as software. The roads (network) reroute themselves. The water and electricity (storage) flow wherever capacity allows. New neighborhoods (workloads) appear in minutes.

A public cloud is a city this big, run by AWS or Microsoft or Google, where you can rent any neighborhood by the hour.

Quick check

A startup runs entirely on AWS. They never see physical hardware. Are they using an SDDC?

Section 4 · At your level

The same idea, two depths

Imagine a whole city where every street, every pipe, every wire is run by a computer instead of a person. When someone needs a new house, the city's computer just makes one — connects the water, the power, the road. The cloud is a really big city like this, where you can rent a house for an hour. AWS, Microsoft, and Google each run a giant city like this.

An SDDC has four layers: compute (vSphere), storage (vSAN), network (NSX), and management/automation (vCenter, vRealize). The pattern: replace specialized hardware appliances with software running on standard servers, and put a management plane on top.

Public cloud is an SDDC at hyperscale. EC2 is compute. EBS is storage. VPC is network. The AWS console is the management plane. Same pattern; different scale and ownership.

Hybrid cloud runs both at once: workloads on-prem in your own SDDC, plus burst capacity in a public cloud SDDC. The architectural similarity is what makes hybrid mobility technically possible.
