Subtopic 1 of 8

What vCenter Server is and why you need it

From a collection of isolated hosts to a managed fleet.

📖

Words you'll meet in this section

vCenter unlocks a lot of features with their own names. Here's each one in plain language.

vCenter Server tool
What it is: The central management server that knows about every ESXi host, every VM, every datastore in your environment. You log in to vCenter and see everything in one place.
Without vCenter, each host is an island managed via its own Host Client. With vCenter, hosts become one fleet — and you unlock everything below.
VCSA · vCenter Server Appliance tool
What it is: The pre-built form vCenter ships in — a Linux VM (Photon OS) with all vCenter services packaged inside. You install vCenter by deploying this VM on top of an existing ESXi host.
VMware retired Windows-based vCenter in vSphere 7. The appliance is the only modern path. Less moving parts.
Photon OS tool
What it is: VMware's own minimal Linux distribution, used as the base of the VCSA appliance.
You'll see "Photon" referenced in vCenter logs and updates. Same Linux underneath — just curated by VMware.
vMotion concept
What it is: Live migration of a running VM from one ESXi host to another with no downtime. vCenter coordinates the move; the VM's network connections continue.
The signature vSphere feature. Lets you do hardware maintenance, balance load, and move workloads without anyone noticing.
DRS · Distributed Resource Scheduler concept
What it is: A vCenter feature that watches the load on every host in a cluster and automatically vMotions VMs around to balance things. Set-and-forget.
Keeps performance even across the cluster. A new busy VM doesn't pile onto an already-loaded host while another sits idle.
vSphere HA · High Availability concept
What it is: A vCenter cluster feature that automatically restarts VMs on a different host when the host they were on dies. Different from vCenter HA (which protects vCenter itself).
A host failure becomes a 2-3 minute blip for the affected VMs instead of a manual rescue.
Cluster concept
What it is: A group of ESXi hosts vCenter manages as a single resource pool. The unit DRS, HA, and shared storage are scoped to.
"Cluster" is where the fleet behavior lives. You don't think about individual hosts much; you think about the cluster's total capacity.
Template concept
What it is: A frozen, gold-standard VM you've decided is your reference image. You can't power it on; you clone from it. Covered in detail in lesson 12.
Build once, stamp out copies. Eliminates drift across the fleet — every Ubuntu VM starts from the same template.
Content library tool
What it is: A vCenter-managed catalog of templates, ISOs, and OVF appliances. Can be shared across vCenters via subscription.
Multi-site shops keep templates in sync this way. Publish once at a central vCenter; every other vCenter pulls.
Distributed switch (vDS) tool
What it is: A virtual switch defined in vCenter and applied identically to every host. Eliminates per-host network drift. Covered in lesson 10.
vDS is one of the big "fleet management" features vCenter unlocks. Standalone hosts are stuck with the per-host vSS.
vSphere Client tool
What it is: The web UI vCenter serves at https://<vc-fqdn>/ui/. Where you live day-to-day. Manages every host, VM, datastore, and network vCenter knows about.
When people say "log in to vSphere," this is what they mean. One tab, the whole environment.
Audit trail · centralized concept
What it is: A single log that records every admin action — who, what, when, on which object. vCenter has one; standalone hosts each have their own.
Post-mortems and compliance reviews need a single, attribution-rich log. vCenter consolidates it.

At a glance

Same 12 hosts. Without vCenter: 12 islands, no fleet features. With vCenter: one screen, one audit, one set of cluster-wide capabilities.

What vCenter unlocks beyond a standalone host
  • vMotion — move running VMs between hosts with no downtime.
  • DRS — load-balance VMs across a cluster automatically.
  • vSphere HA — restart VMs on a different host if one fails.

Subtopic 2 of 8

vCenter prerequisites: DNS, NTP, network, sizing

Four small things that, if wrong, leave you with a half-broken appliance.

📖

Words you'll meet in this section

Pre-flight checklist vocabulary. Each term shows up later when something goes wrong.

FQDN protocol
What it is: Fully-Qualified Domain Name. The complete name including domain — e.g., vc-01.lab.local, not just vc-01.
vCenter uses FQDNs in certificates and AD binding. Short names create ambiguity and broken integrations down the line.
A record protocol
What it is: A DNS entry mapping a name to an IP. vc-01.lab.local → 10.20.30.50.
When the vCenter installer or any client looks up the name, this is what it follows. Forward DNS.
PTR record protocol
What it is: A DNS entry mapping an IP back to a name. 10.20.30.50 → vc-01.lab.local. Reverse DNS.
vCenter's installer checks BOTH forward and reverse DNS. Missing PTR is the #1 reason vCenter installs fail with confusing cert errors.
NTP protocol
What it is: Network Time Protocol. Keeps clocks in sync across your environment.
vCenter and the host it runs on must share time. Drift breaks Kerberos, certs, log correlation.
IPAM tool
What it is: IP Address Management. The system or spreadsheet that tracks which IPs are in use and which are free in your network.
Reserve vCenter's IP in IPAM before install — otherwise the address gets reissued to something else and you get conflicts.
Port group config
What it is: A named network policy on a vSwitch — VLAN ID, security, teaming. VMs and vmkernels attach to a port group, not directly to a vSwitch. Covered in lesson 10.
vCenter needs a port group on the management network. The wizard asks which one.
Deployment size config
What it is: A pre-set sizing tier you choose during install: Tiny, Small, Medium, Large, X-Large. Each maps to a CPU/RAM allocation and supported scale.
Pick by future fleet, not today's. Scaling up later is a maintenance window; oversized once is forever fine.
Tiny / Small / Medium / Large / X-Large config
What it is: The five concrete sizes. Tiny: 10 hosts/100 VMs · Small: 100/1k · Medium: 400/4k · Large: 1k/10k · X-Large: 2k/35k.
Most production environments pick Small or Medium. Tiny is for labs.
Storage size · Default / Large / X-Large config
What it is: A separate sizing axis for the appliance's own disk. Determines how much room vCenter has for stats, events, alarms, archives.
Heavy event logging or long retention → bigger storage. Default is fine for most.
VAMI · vCenter Server Appliance Mgmt Interface tool
What it is: A separate web UI on port 5480 (e.g., https://vc-01:5480/) for appliance-level operations: backup, patching, network config, health.
Different from vSphere Client (which is on port 443 / /ui/). VAMI is the "admin the OS underneath" interface.
Pre-flight checklist concept
What it is: A list of items to verify before starting the installer — not while it's failing.
Five minutes of pre-flight saves hours of mid-install troubleshooting. The pattern: never start an installer with unknowns.
Future fleet concept
What it is: The size of your environment at the end of vCenter's life — typically 3-5 years out. Use this number to pick deployment size, not today's count.
Resizing vCenter is a maintenance window. Oversizing once costs you nothing measurable but saves a future window.

At a glance

Four-bucket pre-flight. Each one wrong = a vCenter that misbehaves later. Check, then launch.

What it is

You can't just download the VCSA OVA and click Install. The deployer asks for things that, if wrong, leave you with a half-configured appliance you'll end up redeploying. Get four buckets right before you launch.

DNS

The vCenter FQDN must resolve forward and reverse. Missing PTR records produce a vCenter that works but trips AD/SSO/integration in weird ways later. Add A and PTR records before you start.

NTP

The target ESXi host (where vCenter will run) and the appliance itself must be in time-sync. Clock drift between vCenter and its hosts is the source of about 80% of "weird vCenter" problems.

Network

Pick the management port group, IP, gateway, subnet. Pre-allocate the IP in IPAM. If you use a non-default VLAN, make sure it's already on the host's vSwitch.

Quiz · prereqs (2)

⏸ pause animation in 'prereqs' mode
The installer's first screen rejects your FQDN with "name does not resolve." You open a terminal and ping vc-01.lab.local works. What's the most likely cause and the fix?
Answer

Missing reverse DNS (PTR record) for the vCenter IP. The installer checks both forward AND reverse and won't let you continue without both. Add the PTR record (10.20.30.50 → vc-01.lab.local) and retry. ping only uses forward DNS — that's why ping worked but the installer rejected.

The PTR-record-missing failure is the most common first-time-installer experience.

You're sizing vCenter for an environment with 80 hosts today, forecast 250 hosts in 18 months. Which deployment size?
Answer

Medium. Small caps at 100 hosts / 1,000 VMs — fine today, you'll outgrow it. Medium handles 400 hosts / 4,000 VMs and gives headroom. You can scale up later but it's a maintenance window — size for the future.

Subtopic 3 of 8

Stage 1: Deploy the appliance OVA

Land the appliance on a host. Services not started yet.

📖

Words you'll meet in this section

Stage 1 install vocabulary. Each shows up in the wizard.

OVF / OVA config
What it is: Open Virtualization Format. A standard way of packaging a complete VM (config + disks) for distribution. OVF = a folder; OVA = the same content as a single tar archive.
VMware ships VCSA as an OVA. The installer wizard imports this OVA onto your target ESXi host.
VCSA installer tool
What it is: A graphical installer that runs on YOUR laptop (Windows / macOS / Linux), not on the host. It talks to the target ESXi host's API and pushes the appliance OVA onto it.
Most learners are surprised by this — the installer doesn't run on the host. Your laptop is the conductor.
Stage 1 · Stage 2 tool
What it is: Two phases of vCenter install. Stage 1 deploys the OVA onto the target host (creates a powered-on VM). Stage 2 configures and starts vCenter services inside that VM.
Splitting them lets you recover if the second phase fails — the OVA is already deployed and you can resume.
Target host config
What it is: The ESXi host where the new vCenter will live. The installer connects to this host directly (not through any other vCenter) and creates a VM on it.
Pick a host with enough free CPU / RAM / disk for the deployment size you chose. Don't pick a host that's already saturated.
Datastore config
What it is: The storage where the new vCenter VM's files will live. A datastore is a logical chunk of storage attached to the host (covered in lesson 11).
Pick a datastore with enough free space (Small ~14 GB; Large 60+ GB). Avoid a datastore that's already nearly full.
Appliance root password security
What it is: The root password for the Photon Linux OS inside the appliance — different from the host's root password. Used to log in to VAMI and to the OS shell.
You almost never use this directly, but it must be strong and stored. Forget it and you can't update or troubleshoot the OS.
Stage 1 output: powered-on but quiet VM concept
What it is: When Stage 1 finishes, the host has a new VM running Photon Linux but vCenter services aren't started yet. https://vc-01/ returns 503.
The 503 is normal between Stage 1 and Stage 2. Don't panic; click Continue.
Resume mode tool
What it is: An installer option to continue from where you left off if Stage 2 failed or the installer window died. Stage 1 output is durable, so you don't have to start over.
Saves you from redeploying the OVA when Stage 2 hangs.
Time-zone config
What it is: The local time-zone setting for the appliance OS. Default UTC.
Logs and audit timestamps are easier to reason about when set to where the operations team works. Most shops keep UTC.
vSphere API protocol
What it is: The HTTPS REST/SOAP interface every ESXi host exposes (for vCenter and tools to talk to it). The installer uses it to upload the OVA.
No special protocol — just HTTPS. Works through firewalls that allow port 443.

Subtopic 4 of 8

Stage 2: First-time setup

vCenter goes live.

📖

Words you'll meet in this section

Stage 2 starts services and configures identity. New terms below.

SSO domain security
What it is: An internal-to-vCenter domain (default vsphere.local) that holds SSO accounts and groups. Lives entirely inside vCenter — separate from your AD or DNS domain.
Don't make this match your AD domain — that creates name collisions. Keep vsphere.local or pick a clearly-separate name.
administrator@vsphere.local security
What it is: The SSO super-admin account created during Stage 2. Can do anything inside vCenter, including changing SSO itself.
Treat as glass-break: long random password, in a vault, never the daily account. AD-mapped accounts do day-to-day work.
CEIP · Customer Experience Improvement Program config
What it is: An opt-in telemetry program — VMware collects anonymised usage data to improve products.
Your call to opt in. Many security-conscious shops opt out by default.
Postgres tool
What it is: The relational database vCenter stores its inventory and config in. Embedded inside the appliance.
You don't manage Postgres directly — vCenter does. But know that "vCenter database" = embedded Postgres.
Service start order config
What it is: Stage 2 starts vCenter services in a specific order: Postgres first, then SSO, then management services, then the UI.
Why Stage 2 takes 10-15 min: each service has to come up and verify before the next one starts. Don't interrupt.
VAMI · port 5480 tool
What it is: Appliance-management web UI on https://vc-01:5480/ for OS-level operations: backup, patches, network, status. Different from vSphere Client (port 443).
When Stage 2 hangs, VAMI is where you check progress without the installer GUI.
vSphere Client URL tool
What it is: The web URL Stage 2 hands you at completion: https://vc-01.lab.local/ui/. Where you live day-to-day.
Bookmark this. Email it to the team. The "vCenter is up" URL.

vCenter SSO and identity sources

Wire AD as an identity source so admins log in with their corporate identity.

📖

Words you'll meet in this section

Identity and authentication acronyms can pile up fast. Here's each one in plain language before they show up in the prose.

SSO · Single Sign-On security
What it is: A service that authenticates you once with a username and password, then issues a "trust ticket" (a token) that other systems accept without asking for your password again.
vCenter delegates "is this user real?" to SSO. SSO talks to wherever your real users live (AD, LDAP). vCenter never sees passwords.
Identity source security
What it is: A directory of users that SSO checks against — Active Directory, LDAP, OpenLDAP, AD FS federation, or the built-in vsphere.local domain.
Without an identity source, SSO can only authenticate the one built-in admin account. Add your corporate AD as an identity source so real employees log in with their real credentials.
IdP · Identity Provider tool
What it is: The system that holds the master copy of who exists and what their password is. AD is the most common IdP in enterprise. SAML federation providers (Okta, Azure AD, AD FS) are also IdPs.
vCenter SSO is itself an IdP for the local vsphere.local users. Adding AD as an identity source means SSO trusts AD as a *second* IdP for everyone else.
Active Directory (AD) tool
What it is: Microsoft's directory service. Stores every user account, group, computer, and policy in a Windows-centric organization. Most enterprises run on it.
When you "add AD as an identity source," vCenter SSO can now authenticate any AD user. Admins log in to vSphere Client with their normal corporate credentials.
LDAP / LDAPS protocol
What it is: Lightweight Directory Access Protocol. The standard way for software to query a directory ("does user 'alice' exist? what groups is she in?"). LDAPS is LDAP over TLS (encrypted).
vCenter SSO talks to AD over LDAPS. If LDAPS isn't configured (only plain LDAP), passwords would travel unencrypted on the wire — never do this.
Kerberos protocol
What it is: A network authentication protocol that uses tickets. The user's password never leaves their machine; instead the IdP issues a short-lived ticket the application can verify.
"Integrated Windows Authentication" between vCenter and AD uses Kerberos. Clock drift >5 minutes makes Kerberos refuse tickets — a major reason NTP must be right.
Security token security
What it is: A short, signed piece of data SSO issues after a successful login. It says "this user is X, valid until Y." Other services accept the token without re-checking the password.
vCenter does not store passwords. It receives a token from SSO, verifies the signature, and trusts what's inside. That's how delegation works.
vsphere.local config
What it is: The default SSO domain name created during vCenter Stage 2. Lives inside vCenter, separate from your AD or DNS domain. Has one user out of the box: administrator@vsphere.local.
Don't make this match your AD domain — that creates collisions. Keep vsphere.local (or pick a clearly separate name). It's an internal SSO concept.
administrator@vsphere.local security
What it is: The SSO super-admin account created at install time. Has the power to change SSO itself, including removing other admins.
Treat like ESXi root: long random password in a vault, used only when nothing else works. Never the daily account.
Federation · SAML / AD FS tool
What it is: A pattern where two systems trust a third (the federation provider) to authenticate users. Login goes through the provider's UI, not vSphere Client.
If your org already has Okta / Azure AD / AD FS as the SSO front door, federation lets vCenter integrate without making it the IdP itself.
Glass-break account config
What it is: A super-admin account you keep locked away (in a password vault) and only use during emergencies. The "break the glass in case of fire" pattern.
Both ESXi root and SSO administrator should be glass-break. Daily admin happens through named identities, not these.
Service account concept
What it is: An account used by a piece of software (not a human) — backups, monitoring, automation. Has a strong password and only the permissions that one job needs.
Backup tools and integrations should authenticate as service accounts in AD, never as a human user. If a person leaves, their integrations don't break.

Subtopic 6 of 8

Roles, privileges, and permissions (RBAC)

Three building blocks. Mix them up and you'll get permissions wrong.

📖

Words you'll meet in this section

Three concepts that sound similar but are not the same. Get them right and the rest of RBAC clicks.

Privilege concept
What it is: The smallest unit of "can do this." One atomic action, e.g. Virtual Machine.Power.Power On. vCenter has hundreds.
You don't manage privileges directly day-to-day. Roles bundle them so you don't have to.
Role concept
What it is: A named bundle of privileges. "Read-Only" bundles a few hundred view-only privileges; "Administrator" bundles every privilege.
Roles are reusable. You assign Alice "Read-Only," not "these 437 specific privileges."
Permission concept
What it is: A triple: (user-or-group, role, inventory-object). E.g., "Alice + Read-Only + Production folder." This is what you actually create in vCenter.
Privileges + roles are abstract. Permissions are real assignments at real places. Permissions are what you click.
Propagating config
What it is: A flag on a permission that says "apply to this object AND all children below it in the inventory tree." Default: yes.
"Give Alice admin on the entire datacenter" works because propagation is on by default — the permission flows down to every cluster, host, VM beneath it.
Non-propagating config
What it is: A permission that applies ONLY to the specific object it's set on, not to children. The opposite of propagating.
Used for overrides — "Alice has Read-Only on Production (propagating), but specifically on vm-payments she has Power User (non-propagating)."
Override concept
What it is: When two permissions affect the same object, the one set directly on that object wins over an inherited one.
"Alice gets Read-Only on Production folder (propagated) BUT Power User on vm-payments specifically (override)" → Alice can power vm-payments.
Inventory hierarchy concept
What it is: The tree vCenter organises objects in: Datacenter → Folder → Cluster → Host → VM. Permissions attach to nodes in this tree.
Where you set a permission decides what it affects. Setting "Read-Only" at the datacenter level is very different from setting it on one VM.
Predefined roles tool
What it is: Built-in roles vCenter ships with: Administrator, Read-Only, No Access, Virtual Machine User, Virtual Machine Power User, No Cryptography Administrator, etc.
Most environments live with the predefined roles plus a couple of custom ones. Don't reinvent.
Custom role tool
What it is: A role you create yourself by hand-picking which privileges it includes. Available when predefined roles don't quite fit.
Use sparingly. Custom roles drift over time as VMware adds privileges in new versions. Predefined roles update automatically.
Global permission security
What it is: A permission that applies across the ENTIRE vCenter inventory AND to objects that don't appear in the inventory tree (tag categories, content libraries).
Powerful — usually only granted to a small senior admin team. Most permissions should be object-level (folder / cluster / VM).
Group-based permissions concept
What it is: Granting a role to an AD group instead of an individual user. When AD adds someone to the group, they automatically get the role; when AD removes them, they lose it.
Maps cleanly to org structure. Joiner/leaver flows in AD propagate to vCenter automatically.
Least privilege concept
What it is: The principle of giving each user exactly the permissions they need, no more. Junior engineers → Read-Only; ops → Power User; only senior staff → Administrator.
Reduces blast radius. A compromised Read-Only account can only see; a compromised Administrator can destroy.

What it is

vCenter's RBACRole-Based Access Control. vCenter's authorization model: privileges (atomic actions) → bundled into roles → assigned as permissions on inventory objects to specific users/groups. has three building blocks:

  • Privilege — an atomic action. e.g., Virtual Machine.Power.Power On. Hundreds of them.
  • RoleA named bundle of privileges in vCenter. Predefined roles (Administrator, Read-Only, etc.) plus custom roles you create. Roles are reusable; you assign them to people via permissions. — a named bundle of privileges. "Administrator," "Read-Only," "VM Power User," etc.
  • PermissionA (user-or-group, role, inventory-object) triple. e.g., "Alice has Power User role on the Production folder." Can propagate to children or stop at the object. — a (user-or-group, role, inventory-object) triple. "Alice has the VM Power User role on the Production folder."

Inheritance. Permissions can be set as propagatingPermission setting that determines whether the permission applies to child objects in the inventory hierarchy. Default is propagating (yes). Object-level permissions on a child override inherited permissions for that specific object. (apply to the object and all children — the default) or non-propagating (apply only to the object itself).

📖

Words you'll meet in this section

Two related topics — organising the fleet (inventory) and protecting it (backup) — share these terms.

Datacenter (object) concept
What it is: The top-level container in vCenter's inventory tree. Usually one per physical datacenter or per business region (e.g., DC-East, DC-West).
Datacenters are inventory boundaries — clusters and folders live inside them. Permissions and policies often start here.
Folder concept
What it is: A grouping container — VM folders, host folders. Used for organising and for permissions/tags. Nest as deep as makes sense (Production / Web / Frontend).
Folders are how you scope permissions and tag policies. "Give Ops Read-Only on the Production folder" applies to every VM in there.
Cluster (object) concept
What it is: A group of ESXi hosts that share resources and behave as one for HA, DRS, and vMotion.
Cluster is where most fleet behavior lives — DRS rules, HA admission control, EVC. Usually one cluster per workload tier.
File-based backup tool
What it is: vCenter's built-in backup feature. Exports the appliance's database + config as files to a remote target. Configured from VAMI.
Lets you rebuild vCenter from scratch by deploying a fresh OVA + restore. Much faster than rebuilding inventory by hand.
SFTP / FTPS / NFS / SMB / HTTPS protocol
What it is: Five protocols vCenter file-based backup can ship to. Pick whichever your storage / backup team supports.
SFTP and NFS are the most common in production. HTTPS is rare. SMB works but is mostly Windows-shop territory.
Backup retention config
What it is: How many backup snapshots vCenter keeps before deleting old ones. Default 5; configurable.
Set to 30 days minimum for production. Useful when you discover a problem 2 weeks after it started — you can roll back to a known-good state.
Restore config
What it is: The inverse of backup — deploying a fresh VCSA and using a backup file to repopulate everything (inventory, permissions, certs, AD config).
Faster than rebuilding by hand. ~30-45 min from "vCenter dead" to "vCenter restored." Hosts auto-rejoin.
Inventory state concept
What it is: Everything vCenter "knows" — which hosts exist, which VMs, which permissions, which AD bindings. Lives inside the appliance.
When vCenter dies, this knowledge dies with it (until you restore). The VMs themselves keep running on their hosts; vCenter just doesn't know about them.
Pre-operation backup concept
What it is: The discipline of taking a fresh backup before a major operation (upgrade, certificate rotation, large config change).
If the operation breaks something, your rollback is "redeploy + restore" — fast and clean. Without a fresh backup, rollback is "argue with logs for two days."
Encrypted backup tool
What it is: Optional setting that encrypts the backup files with a passphrase before shipping. Without the passphrase, the backup is unrestorable.
Default to encrypted. Store the passphrase in your vault. Backup files leaving the org's network must be encrypted.

What it is — inventory hierarchy

vCenter organizes everything into an inventory tree:

  • Datacenter — top-level container, usually one per physical datacenter or per region.
  • Folder — for grouping (VM folders, host folders) for permissions and tags.
  • Cluster — group of hosts that share resources (HA, DRS, vMotion).
  • Host — an ESXi host added to vCenter.
  • VM — virtual machines on hosts, inside folders.

Networks (port groups, distributed switches) and datastores (VMFS, NFS, vSAN) appear under their datacenter as separate views.

What it is — vCenter backup

vCenter has built-in file-based backup that ships its database and configuration to an external target. Configure it from the appliance VAMI (https://vc-01.lab.local:5480/) under Backup. Schedule it nightly. Keep at least 7 backups, ideally 30.

Supported targets: FTPS, SFTP, HTTPS, NFS, SMB.

A backed-up vCenter restores to a fresh appliance in ~30 minutes — much faster than rebuilding inventory, permissions, certs, and AD binding from scratch.

Take a fresh backup before any major operation: upgrade, certificate rotation, SSO change. If something goes wrong, the rollback is "redeploy + restore," not "argue with logs for two days."

Subtopic 8 of 8

vCenter High Availability (intro)

Three nodes for the management plane: Active · Passive · Witness.

📖

Words you'll meet in this section

High-availability terms. Two of them sound similar but mean very different things.

vCenter HA tool
What it is: A vCenter-specific feature that protects the vCenter appliance itself from a host failure or vCenter VM crash. Three nodes work together: Active, Passive, Witness.
Without vCenter HA, a vCenter outage means manual restore from backup (~30 min). With it, failover happens in 1-3 minutes automatically.
vSphere HA tool
What it is: A cluster-level feature that protects guest VMs by restarting them on surviving hosts if a host dies. Different from vCenter HA.
vSphere HA cares about your application VMs. vCenter HA cares about vCenter itself. Both can be on at the same time.
Active node concept
What it is: The live vCenter — the one clients actually talk to. Has the FQDN, the IP, the running services. Only one Active at a time.
Day-to-day, "vCenter" means the Active node. The other two are quiet.
Passive node concept
What it is: A clone of Active that's kept in sync via continuous replication. Doesn't serve clients — just receives changes and stays ready.
When Active dies, Passive promotes to Active in minutes — taking over the same FQDN/IP. Clients see only a brief blip.
Witness node concept
What it is: A small third node that holds no vCenter data. Its only job is to break ties — confirm whether Active is actually down, or whether the network just split.
Without Witness, a network partition could make Active and Passive both think they're in charge ("split-brain") — disastrous. Witness prevents this.
Replication concept
What it is: The continuous synchronisation of state from Active to Passive. Database changes get streamed in real time so Passive is always close to current.
When Active dies, Passive can take over with minimal data loss — usually seconds, not minutes.
Failover concept
What it is: The act of Passive taking over Active's role when Active dies. Witness confirms; Passive promotes; clients reconnect.
vCenter HA failover is automatic and takes 1-3 minutes. Compared to backup-restore (~30 min), this is dramatically faster.
Split-brain concept
What it is: The catastrophic state where two cluster members both think they're in charge — usually because they can't see each other due to network failure.
Witness exists specifically to prevent this. With Witness, Passive only promotes if Witness agrees Active is dead.
HA network config
What it is: A separate IP network used for replication and heartbeat traffic between the three nodes. Distinct from the management network.
Isolates HA traffic so it can't be choked by other workload traffic. Required for vCenter HA setup.
Three-host placement concept
What it is: The recommendation to place each of the three nodes on a different ESXi host, ideally with different power and network paths.
If two nodes share a host and that host dies, you've lost two-thirds of your HA cluster. Spread them out.
Resource cost config
What it is: vCenter HA costs three full appliance VMs instead of one — roughly 3× the CPU/RAM/disk.
Real cost. Smaller environments often skip vCenter HA and rely on file-based backup-and-restore. Worth it at scale; overkill at 6 hosts.

Quiz · vCenter HA (2)

⏸ pause animation in 'ha' mode
Someone says "we have vSphere HA on every cluster, so we're protected against vCenter going down — vSphere HA will restart it." Are they right?
Answer

Partially. vSphere HA WILL restart the vCenter VM on another host if its host dies — that's worth something (5–10 minutes of unreachable management plane). For full protection, vCenter HA (active/passive/witness) fails over in 1–3 minutes and doesn't depend on vSphere HA's restart cycle. They're complementary, not substitutes.

Confusing vSphere HA and vCenter HA is the most common rookie mistake.

Your environment has 6 hosts and a single vCenter appliance, backup configured nightly. The CTO asks "do we need vCenter HA?" What's a reasonable answer?
Answer

Probably not for that size. vCenter HA costs three appliance VMs and adds operational complexity. For 6 hosts with working backup and a 30-minute restore window the business can tolerate, file-based backup-and-restore is a reasonable choice. Revisit when the environment grows or when management-plane downtime becomes business-critical.
