Words you'll meet in this lesson

Type 1 vs Type 2 and the products that fall under each. Plain-language definitions.

Type 1 hypervisor concept
What it is: A hypervisor that runs directly on hardware — no general-purpose OS underneath. Also called "bare-metal." ESXi, Hyper-V (server), KVM (when used standalone), Xen.
Lower overhead, better performance, more secure. The right choice for production data centers.
Type 2 hypervisor concept
What it is: A hypervisor that runs on top of a regular OS (Windows, macOS, Linux). VMware Workstation, VMware Fusion, VirtualBox, Parallels.
Convenient for desktops — run Windows on your Mac, run a Linux VM on your Windows laptop. Not used for production workloads.
Bare-metal concept
What it is: Synonym for Type 1. Hypervisor installs straight on hardware; nothing else underneath.
"Bare-metal hypervisor" and "Type 1 hypervisor" mean the same thing. ESXi is the canonical example.
Hosted hypervisor concept
What it is: Synonym for Type 2. The "host" is the regular OS the hypervisor sits on top of.
Gets confusing because vSphere also uses "host" to mean the physical server. Context matters: in Type-2 talk, "host OS" = Windows/macOS/Linux on your laptop.
VMware ESXi tool
What it is: VMware's Type 1 hypervisor. Installs bare-metal on a server. The hypervisor in vSphere.
The product the rest of this course is about. Detailed in lesson 8.
Microsoft Hyper-V tool
What it is: Microsoft's hypervisor. On Windows Server, runs as Type 1. On Windows desktop, can be used Type 2-ish for development.
Main competitor to ESXi. Tightly integrated with Windows. Common in Microsoft-centric shops.
KVM tool
What it is: Kernel-based Virtual Machine. The Linux kernel's built-in hypervisor. Used by Red Hat OpenShift Virtualization, Proxmox, and many cloud providers.
Open source and free. The hypervisor underneath much of the open-source / cloud world.
Xen tool
What it is: An older open-source Type 1 hypervisor. Was AWS EC2's hypervisor for years (replaced by Nitro/KVM-based around 2017).
Less common today but still important historically — it powered the early public cloud.
VMware Workstation · Fusion tool
What it is: VMware's Type 2 hypervisors. Workstation (Windows / Linux), Fusion (macOS). Used to run other OSes on a desktop.
Useful for developers and labs. Not for production. Often used to "nest" ESXi inside another ESXi for learning.
VirtualBox tool
What it is: Oracle's free Type 2 hypervisor. Cross-platform, popular for personal labs and education.
Free and easy. Often a learner's first hypervisor experience.
Nested virtualization concept
What it is: Running a hypervisor inside a VM. e.g., ESXi as a VM inside Workstation. Possible but slower than running ESXi natively.
The standard way of building a vSphere lab on a laptop. Real production workloads don't nest.
Hardware overhead concept
What it is: The performance cost of having a hypervisor between your apps and the hardware. Type 1 has lower overhead than Type 2 because there's no host OS in the middle.
For production servers, the few percent matters. For your laptop, it doesn't. That's why production = Type 1, desktops = Type 2.

Section 1 · Concept

Two kinds, one technical difference

There are two kinds of hypervisors. The difference is where the hypervisor sits relative to the host operating system. The choice between them is a choice between performance and convenience.

Type 1 (bare-metalBare-metal means running directly on the physical hardware with no host operating system underneath. A bare-metal hypervisor is itself a minimal, specialized OS.). The hypervisor runs directly on the physical hardware. There is no host operating system. The hypervisor is itself an extremely focused operating system whose only job is running VMs. ESXi is Type 1. So are KVM (built into the Linux kernelThe kernel is the core of an operating system — the part that talks directly to the hardware and manages everything else. KVM (Kernel-based Virtual Machine) lives inside the Linux kernel itself, which is what makes it a Type 1 hypervisor.), Microsoft Hyper-V, and Xen.

Type 1 gives near-native performance. It also has a smaller attack surface. Type 1 is what production datacenters and public clouds run on.

Type 2Type 2 (also "hosted") hypervisors run as an application on top of a regular operating system. The host OS handles the hardware; the hypervisor sits above it as another program. Examples: VMware Workstation, VirtualBox, Parallels. (hosted). The hypervisor runs as an application on top of a regular host operating systemThe host OS is the regular operating system (Windows, macOS, or Linux) that the Type 2 hypervisor runs on top of. The host OS owns the hardware; the hypervisor is just one of many applications running on it.. VMware Workstation, VMware Fusion, Oracle VirtualBox, and Parallels Desktop are all Type 2.

Type 2 is slower because every hardware request goes through one extra layer. But it's dramatically more convenient. A developer can install VirtualBox on their laptop and have a Linux VM running in five minutes.

Production servers and clouds: Type 1. Developer laptops, testing, learning: Type 2.

★
Quick rule of thumbIf the goal is "run server workloads at scale" — Type 1 (ESXi, Hyper-V, KVM). If the goal is "run a different OS on my laptop for an afternoon" — Type 2 (Workstation, Fusion, VirtualBox). Cloud providers all use Type 1 underneath, even though they look like Type 2 from a developer's view.
i
Nested virtualization for learningBuilding a vSphere lab on your laptop? You'll run ESXi (Type 1) as a guest inside Workstation or Fusion (Type 2). Nested. Slower than real, but lets you learn the whole stack on a single machine.

Section 2 · Before and after

Why Type 1 mattered for x86

Before ESX (1999–2001)

Type 2 only

The only hypervisors available for x86 were Type 2 — they ran on top of an OS. Performance was a real problem; running a VM through a hosted hypervisor meant losing 20–30% of the host's performance to OS overhead.

This was tolerable for testing but unacceptable for production workloads.

After ESX (2001 onward)

Type 1 ships

VMware ESX brought bare-metal hypervisors to commodity x86. Production-grade virtualization with near-native performance became available. Type 2 didn't disappear — it remained the right choice for desktops — but production decisively moved to Type 1.

It's where production has stayed for the last 25 years.

Section 3 · The analogy, extended

Building vs. basement

The apartment-building analogy from Lesson 1 splits naturally into the two hypervisor types.

Type 1 is a purpose-built apartment building. The whole structure was designed from day one to be apartments. The superintendent owns the whole building. Tenants live in apartments efficiently, with no compromises.

Type 2 is a single-family house where the owner has converted the basement into a couple of rental units. The owner still lives upstairs, in the main living-space (the host OS). The basement apartments are real, but they share a front door and the renters have to coordinate with the owner about everything.

A purpose-built building (Type 1) is more efficient. The basement-apartment setup (Type 2) is far more accessible.

Section 4 · At your level

The same idea, two depths

There are two kinds of "computer pretend houses." The first kind (Type 1) is built on empty land — the apartments are the whole building. That's faster. The second kind (Type 2) is when you have a regular house and you let some friends live in the basement. That's easier to set up but slower because everyone shares the same front door. Big companies use the first kind. People learning at home use the second kind.

Type 1 hypervisors run directly on the hardware with no host OS. ESXi, KVM, Hyper-V, Xen. Near-native performance, smaller attack surface, used in datacenters and public clouds.

Type 2 hypervisors run as applications on top of a normal OS. VMware Workstation, Fusion, VirtualBox, Parallels. Lose 15–30% to OS overhead, but trivial to install. Used by developers and learners.

The decision between them is almost always made by counting hosts: at scale, Type 1 wins; at desktop scale, Type 2 wins.

Lesson 6 complete

Two kinds of hypervisors. Choose by counting hosts.

  • Type 1 runs directly on hardware. No host OS. ESXi, KVM, Hyper-V, Xen. Near-native performance.
  • Type 2 runs as an app on a host OS. Workstation, Fusion, VirtualBox, Parallels. ~15–30% overhead.
  • Production = Type 1. Desktop / dev / learning = Type 2.
  • You can run Type 1 inside Type 2 for demos. It's slow but valid.
