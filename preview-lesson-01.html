Words you'll meet in this lesson

If any term feels unfamiliar, here's the plain-language version. We're learning the foundation — these will follow you through the whole course.

Virtualization concept
What it is: The technology that lets a single physical computer act as many smaller, independent computers at the same time. Each "computer" is a virtual machine.
Without it, every workload needs its own physical box — wasteful, slow to deploy, hard to move. With it, hardware becomes a flexible pool.
Hypervisor tool
What it is: The software layer that creates and runs virtual machines. It slices a server's CPU, memory, network, and storage and gives each VM its own share.
ESXi (VMware), Hyper-V (Microsoft), KVM (Linux) are hypervisors. Without one, you have no VMs.
VM · Virtual Machine concept
What it is: A software container that looks, to the operating system inside it, exactly like a physical computer — its own CPU, memory, disks, network cards.
VMs are the unit of work in virtualization. You deploy VMs, move VMs, snapshot VMs. The "fake computer" you spend your day with.
Host · physical host concept
What it is: The physical server running the hypervisor. Each host can run many VMs sharing its hardware.
When you hear "we have 12 hosts in the cluster" — those are 12 physical servers running ESXi, each carrying many VMs.
Guest OS concept
What it is: The operating system installed inside a VM — Linux, Windows, etc. From the guest's point of view, it's running on a real computer.
The guest doesn't know it's virtual (in normal cases). That's the whole point — applications work without modification.
Bare-metal hypervisor concept
What it is: A hypervisor that installs directly on hardware, with no general-purpose OS underneath. ESXi is bare-metal. Also called "Type 1."
Lower overhead, better performance than running a hypervisor on top of Windows/Linux. The right choice for production data centers.
Resource pool concept
What it is: The combined CPU/memory/storage capacity of all the physical hosts in a cluster, treated as one big pool VMs draw from.
Lets workloads scale across hardware without caring which specific host they run on.
Capacity planning concept
What it is: The discipline of forecasting how much hardware you need to support workloads — both today and as they grow.
In the physical world, you bought a server per app. In the virtual world, you size the pool — much more flexible but still requires planning.
Isolation concept
What it is: The hypervisor's guarantee that what happens inside one VM stays inside that VM. A crash, virus, or rogue process in VM-A doesn't affect VM-B.
Foundational. Without it, multi-tenant virtualization wouldn't be safe.
Snapshot concept
What it is: A point-in-time saved state of a VM. Lets you roll back if a change breaks things. Detailed in lesson 12.
Powerful safety net for short-term changes. Not a backup — covered properly later.
vSphere tool
What it is: VMware's virtualization platform — ESXi (the hypervisor) + vCenter (the central manager) + the surrounding features (vMotion, HA, DRS).
When people say "vSphere," they mean the whole VMware stack. The full picture comes together in lessons 3 and 5.
ESXi tool
What it is: VMware's bare-metal hypervisor. Installed on each physical host. Detailed in lesson 8.
The actual software doing the slicing of hardware into VMs. Every host in a vSphere environment runs ESXi.

Section 1 · Concept

A computer that acts like many

Virtualization is a technology that lets a single physical computer act as many smaller, fully independent computers at once.

Each of those independent computers is called a virtual machineA virtual machine is a software computer that runs an operating system and applications, but lives inside another computer rather than running directly on hardware. Each VM thinks it owns a complete machine. — a VM. Each one runs its own operating systemAn operating system (Linux, Windows, macOS, etc.) is the software that manages a computer's hardware and runs the actual applications on top. Each VM runs its own operating system, fully separate from the others. and applications, completely unaware that it shares the underlying hardware with others.

A VM thinks it owns a whole computer. A hypervisor maintains the illusion.

The piece of software that makes this possible is called a hypervisorThe hypervisor is the software layer that sits between hardware and virtual machines. It divides the host's CPU, memory, disk, and network into isolated slices and gives each VM its own slice.. It sits between the physical hardware and the virtual machines, taking the real CPU, memory, disk, and network of the physical machine and dividing them into isolated slices. Each VM sees its slice as if it were a complete computer.

Why virtualization exists

Before virtualization, every application that needed an operating system needed its own dedicated physical machine — even though that application typically used only a small fraction of the machine's capacity.

Hardware sat idle. Datacenters grew. Electricity bills climbed.

Virtualization lets many operating systems coexist on a single physical hostThe host is the physical computer that runs the hypervisor and many virtual machines on top. It provides the real CPU, RAM, disk, and network that all the VMs share., fully isolatedIsolation means one VM cannot reach into another VM's memory, files, or processes. The hypervisor enforces this — like the locks on each apartment door keep tenants out of each other's homes. from each other, with the hypervisor managing the sharing behind the scenes. The result is a foundation for everything that came after — cloud computing, modern storage, software-defined networking.

i
Where you'll see this in real lifeAlmost every cloud you've used (AWS EC2, Azure VMs, GCP Compute) runs on virtualization underneath. Every modern data center uses it. Even your laptop probably runs Docker on top of a tiny VM. Once you see virtualization, you start seeing it everywhere.
★
The shift in mindsetOld way: one app per server. New way: many VMs per host, each running its app. The hardware becomes a pool you draw from, not a fixed asset per workload. This single shift unlocks every other modern infrastructure pattern.

Section 3 · The analogy

The apartment building

Picture an old single-family house. The kind a small family of three or four lived in throughout the twentieth century. Bedrooms, a kitchen, a living room, a bathroom — but most of the time, most of those rooms sit empty.

Virtualization is what turns that house into an apartment building.

Click the button to virtualize the house.

Inside the same building footprint, a hypervisor — playing the role of the superintendent — carves the space into separate apartments. Each is fully private: tenants have their own front door, their own kitchen, their own privacy.

They share the building's water, electricity, and internet. But the superintendent handles all of that behind the scenes. Tenants barely notice.

Section 4 · At your level

The same idea, two depths

Pick whichever feels right. They tell the same story; one keeps it simple, the other adds the vocabulary you'll see throughout the course.

Imagine your computer is like a big house, but only one family lives in it. Most of the rooms sit empty most of the time.

Virtualization is like turning that house into an apartment building. Now lots of families live there at the same time, each with their own front door and their own private rooms, sharing the building's water and electricity.

There's a person in charge of the building who makes sure no family bothers any other family. So virtualization just lets one big computer pretend to be lots of small ones.

Virtualization works through a piece of software called a hypervisor that makes one physical computer behave like many independent ones. The hypervisor is the apartment building's superintendent. The physical machine is the building. Each apartment is a virtual machine, or VM, running its own operating system.

The tenants — the operating systems — believe they each have their own building. The superintendent quietly slices the building's water, power, and internet (the host's CPU, RAM, network) and delivers each apartment its share without anyone realizing they're sharing.

This setup exists because the alternative wasted a lot of hardware. Before virtualization, every operating system needed its own physical machine, and most machines ran one application at single-digit utilization — like a five-bedroom house with one person living in it. After virtualization, twenty operating systems live in the same physical box.

In practice, the interesting question is what the superintendent does when twenty tenants all want to run the dishwasher at the same instant. That's resource scheduling, and it's the next lesson.
